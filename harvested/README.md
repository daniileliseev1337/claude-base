# harvested/

Каталог внешних инструментов, оценённых через WebFetch / `gh api` / тестирование в sandbox.

**Notes-only** — здесь не лежит **код** из чужих репо, только описание и оценка.

## Формат заметки

Каждая заметка — отдельный `.md` файл. Имя:

```
harvested/<YYYY-MM-DD>_<repo-or-library-name>.md
```

Шаблон содержания:

```markdown
# <repo-name> или <library-name>

**URL:** https://github.com/owner/repo
**License:** MIT / Apache 2.0 / GPL / no LICENSE / ...
**Активность:** last commit YYYY-MM-DD, X stars, Y forks
**Статус:** candidate / adopted / rejected / archived

## Что делает

Краткое описание (по README).

## Почему может быть полезно

Конкретное применение в наших задачах (1-3 пункта).

## Ограничения / риски

- Лицензия (можно ли раздавать через installer).
- Требования (Python ≥X, библиотеки, OS).
- Качество кода (на основе беглого просмотра).
- Заброшенность / активность / отзывчивость maintainer'а.

## Тест в sandbox (если был)

- Дата теста: YYYY-MM-DD
- Команды: ...
- Результат: что работало, что нет.

## Решение

- Adopt в `agents/` или `skills/` нашей базы — да/нет, обоснование.
- Если нет — почему (overhead, нерелевантно, лицензия, качество).
```

## Дисциплина

- Адаптация в `agents/` / `skills/` — **только вручную**, после явного решения пользователя. Auto не делаем.
- Repos без LICENSE — пишем заметки об **идеях**, код не копируем (переформулируем своими словами, как сделали с `karpathy-guidelines`).
- GPL — не адаптируем (вирусность лицензии портит installer).
- Используем **только во внутреннем круге** (личные ПК + 1-2 коллеги). При расширении — пересматриваем дисциплину.
