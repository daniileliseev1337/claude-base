# Методичка: как работать с Claude Code (наша база `claude-base`)

> Для сотрудников отдела. Описывает, **как у нас всё устроено** (что откуда тянется и куда уходит),
> **как работают** агенты, скиллы и команды, и даёт **справочник ручных команд** с примерами.
> Работает одинаково в **Claude Code CLI** (терминал) и **Desktop**-приложении — у них общая папка `~/.claude`.
>
> Актуализируется хабом; ты получаешь свежую версию автоматически (см. раздел 2). Источник — `docs/methodichka.md` в базе.

---

## 1. Что такое «база» в двух словах

Claude Code «из коробки» — это просто умный ассистент в терминале. Наша **`claude-base`** надстраивает над ним рабочий слой под наши задачи (ОВ/ВК/ЭО/СС-проектирование, ПТО, сметы, снабжение, исполнительная документация, Revit/AutoCAD):

- **Агенты** — узкие специалисты (проектировщик, сметчик, снабженец, ревьюеры и т.д.), которых главный Claude зовёт сам.
- **Скиллы** — методички-инструменты, подгружаются по смыслу запроса (работа с Word/Excel/PDF, спецификации, ПНР/ВОР, OCR сканов…).
- **Слэш-команды** — то, что ты вызываешь руками: `/sync-base`, `/harvest`, `/format`, `/graphify`.
- **Память и правила** — `CLAUDE.md` (свод правил), `memory/` (накопленные уроки), граф знаний.
- **Инфраструктура** — автоматическая синхронизация: ты получаешь обновления и отправляешь обратную связь, ничего не делая руками.

Главное, что нужно понять: **ты работаешь на своей копии базы, а все улучшения собираются централизованно и раздаются всем.**

## 2. Как всё устроено: что откуда тянется и куда уходит

### Модель «хаб ↔ сотрудники» (hub-and-spoke)

```
                ┌───────────────────────────┐
                │   ХАБ (разработчик базы)   │
                │   правит agents/ skills/   │
                │   memory/ CLAUDE.md …      │
                └─────────────┬─────────────┘
                              │ git push (в общий репозиторий на GitHub)
                              ▼
            ╔═════════════════════════════════════╗
            ║      Общий репозиторий claude-base   ║
            ╚═══┬─────────────┬─────────────┬══════╝
   git pull ▲   │ git pull ▲  │  git pull ▲ │   ← раздача обновлений всем
            │   ▼            ▼              ▼
      ┌─────┴────┐    ┌──────────┐    ┌──────────┐
      │ Сотрудник│    │ Сотрудник│    │ Сотрудник│   ← у каждого своя копия ~/.claude
      │   ПК-1   │    │   ПК-2   │    │   ПК-N   │
      └─────┬────┘    └────┬─────┘    └────┬─────┘
            │ feedback     │ feedback      │ feedback
            ▼              ▼               ▼
            ╔══════════════════════════════════════╗
            ║   Feedback-канал (отдельный репо)     ║  → хаб собирает и внедряет
            ╚══════════════════════════════════════╝
```

- **Хаб** — один человек (главный разработчик базы). Только он правит общую базу и пушит изменения.
- **Ты — «consumer»** (потребитель). Твой ПК определяется автоматически: если в `~/.claude` нет файла `.developer-marker` — ты consumer. Тебе **не нужно** ничего коммитить в общую базу руками.

### Что приходит к тебе АВТОМАТИЧЕСКИ

- **При старте сессии** (хук `SessionStart`) запускается `auto-pull` — тихо подтягивает свежую базу от хаба (`git pull`). Новые/исправленные агенты, скиллы, память, команды появляются сами.
- **Личные настройки не затираются**: общие командные настройки (`settings.shared.json`) аккуратно вливаются в твой личный `settings.json` (через `merge-shared-settings`), не трогая твою тему оформления и прочие UI-поля.
- Если хаб добавил **новый инструмент** (MCP-сервер, пакет) — ты увидишь короткое напоминание «запусти `/sync-base`».

### Что уходит ОТ тебя

- **В конце сессии** (хук `SessionEnd`) запускается `auto-push` — отправляет твои **session-reports** (отчёты о проделанной работе) и накопленный **feedback** в общий/feedback-канал.
- **Feedback по базе** (нашёл косяк, придумал улучшение, урок): пишется в `~/.claude/feedback-pending/`, авто-сборщик (`feedback-collector`) отправляет его в **feedback-канал** (отдельный GitHub-репо, токен зашифрован через Windows DPAPI). Хаб периодически собирает (`pull-feedback`) и внедряет полезное в общую базу. **Так твой опыт становится общим достоянием.**

### Граф знаний (навигатор по базе)

В базе есть «карта» её устройства — граф знаний (`graphify`). Когда нужно понять «как у нас устроено X / где это описано», Claude спрашивает граф, а не перечитывает все файлы — это экономит токены и время. Тебе трогать его не нужно; он строится и поддерживается хабом.

### Где что лежит (тебе для ориентира)

| Папка/файл | Что внутри |
|---|---|
| `~/.claude/CLAUDE.md` | Свод правил базы (читается в начале каждой сессии) |
| `~/.claude/agents/` | Доменные агенты-специалисты |
| `~/.claude/skills/` | Скиллы-инструменты и методики |
| `~/.claude/commands/` | Ручные слэш-команды |
| `~/.claude/memory/` | Накопленные уроки и факты |
| `~/.claude/feedback-pending/` | Твой исходящий feedback (уходит сам) |
| `~/.claude/session-reports/` | Отчёты о сессиях (уходят сам) |
| `~/.claude/mcp-manifest.json` | Список инструментов-MCP, которые должны стоять |

## 3. Что происходит в начале каждой сессии

Claude по правилам базы выполняет «процедуру старта» — тебе полезно понимать, что значат строки, которые ты видишь:

1. Читает `CLAUDE.md` (свод правил).
2. **Проверяет MCP-инструменты** (офисные, веб, AutoCAD/Revit…). Если что-то «в холодном кэше» — увидишь `⚠ MCP не подключены: …`. Обычно прогревается само при первом использовании; если мешает — `/sync-base`.
3. Проверяет, что все агенты на месте.
4. Если хаб добавил инструменты — `⚠ запусти /sync-base`.
5. Подтверждает строкой вида `✓ прочитан CLAUDE.md (MCP X/11, agents Y/16)`.

Если видишь предупреждение про MCP или про `/sync-base` — это нормально, следуй подсказке.

## 4. Как Claude выбирает инструмент: 3 способа активации

| Способ | Кто запускает | Как выглядит |
|---|---|---|
| **`auto`** | Главный Claude сам, по смыслу твоего запроса | Ты пишешь задачу обычными словами — нужный агент/скилл подключается сам |
| **`invoke`** | **Ты**, вручную | Пишешь слэш-команду: `/sync-base`, `/format`, `/harvest`, `/graphify` |
| **`both`** | И так, и так | Срабатывает по фразе или по явному вызову |

### Архитектура «двойной проверки» (почему это важно)

Серьёзные артефакты (смета, спецификация, акт, раздел РД) делаются по цепочке:

```
Главный Claude  →  доменный агент (сделал)  →  ревьюер (проверил)  →  выдача тебе
```

То есть результат сначала проверяет независимый ревьюер (`auditor`, `word-checker`, `excel-validator`, `pdf-reviewer`, нормоконтроль) и только потом отдаётся тебе. Это снижает риск ошибок и «выдумок».

### Что ты можешь увидеть по ходу (это не баги, а контроль качества)

- **Роутинг-гейт** — строка `Роутинг: <домен> → агент <имя>` перед доменной работой.
- **Грилинг-выбор** — перед строй-задачей Claude может предложить «догрилить вводные» (задать уточняющие вопросы), чтобы ничего не выдумывать. Можно согласиться или идти сразу к задаче.
- **Вопрос о модели субагента** — для тяжёлых задач Claude спросит, на какой модели запускать помощников (дешевле/дороже). Это про экономию лимита.
- **Гейт сессии** — на длинной сессии предложит начать новый чат (так дешевле по токенам).

## 7. Ручные слэш-команды (вызываешь сам через `/`)

Это то, что **запускается только тобой**. Главный справочник раздела.

### `/sync-base` — обновить базу и проверить инструменты

**Зачем:** одной командой подтянуть свежую базу от хаба + проверить, что всё на месте и доустановить недостающие инструменты. Заменяет все прежние ручные шаги обновления.
**Когда:** увидел подсказку «запусти /sync-base»; начал рабочий день; что-то «отвалилось».
**Что делает:** `git pull` (с обходом корп-прокси) → самопроверка базы → инвентарь и установка инструментов по манифесту (нужное — молча, опциональное — спросит) → отчёт. **Сам ничего не пушит.**
**Пример:**
```
/sync-base
```
Ответ покажет: сколько обновлений пришло, что доустановлено, что требует ключа/прав админа.

### `/harvest` — найти готовый внешний инструмент

**Зачем:** прежде чем писать своё, проверить, нет ли готового open-source решения (GitHub, skills.sh, MCP-реестр).
**Когда:** «нужен инструмент для X», «поищи готовое под эту задачу».
**Что делает:** ищет 2–5 кандидатов, фильтрует по активности/звёздам/лицензии/бесплатности, выдаёт сравнительную таблицу + рекомендацию. Ничего не устанавливает без твоего согласия.
**Примеры:**
```
/harvest extract tables from PDF
/harvest DOCX templates ГОСТ
/harvest Excel formula validator
```

### `/format` — оформить документ по ГОСТ-стилю

**Зачем:** применить аккуратное оформление (ГОСТ 7.32 / упрощённый / с рамкой ВКР / простой) к готовому тексту и получить DOCX (опционально PDF).
**Когда:** надо «причесать» пояснительную записку, отчёт, ответ экспертизе под выдачу.
**Что делает:** спросит стиль → возьмёт шаблон → наполнит контентом → сохранит DOCX (по желанию — PDF).
**Примеры:**
```
/format                     (спросит стиль и что оформлять)
/format full ПЗ-ОВ-блок-А   (стиль «ГОСТ полный», имя файла)
/format light               (упрощённый ГОСТ)
```

### `/graphify` — карта/навигатор по коду или базе знаний

**Зачем:** разобраться в большом незнакомом проекте или своде документов, не перечитывая всё подряд (экономит токены). Строит граф знаний, по которому потом отвечает на вопросы.
**Когда:** «разберись, как тут устроено X», «как связаны A и B», «обнови граф — я поменял файлы».
**Что делает:** строит граф (один раз) → отвечает на вопросы по нему. На маленьком проекте (<100 файлов) предупредит, что выгода мала.
**Примеры:**
```
/graphify <путь-к-проекту>
/graphify query "как тут устроена авторизация"
/graphify <путь> --update     (обновить после правок)
```

> Также у Claude Code есть **встроенные** команды (например, проверка/ревью кода). Наши, командные — это четыре выше; их и держи под рукой.

## 8. MCP-инструменты (коротко — зачем их понимать)

MCP — это «плагины» Claude для работы с внешним: офисные файлы, веб, AutoCAD, Revit, время. Тебе важно знать два момента:

- **Файлы открываются спец-инструментом, а не «как текст»:** `.docx` → Word-MCP, `.xlsx` → Excel-MCP, `.pdf` → PDF-MCP, и т.д. Поэтому не удивляйся, что Claude «по-разному» читает разные форматы — так и надо (читать бинарник «как текст» = мусор).
- **«Холодный кэш»:** при первом обращении инструмент может быть не подключён (`✗ Failed to connect`) — это нормально, он прогревается. Если мешает — `/sync-base`.

Эталон — 11 основных MCP + опциональные (AutoCAD, Revit). Что должно стоять, описано в `mcp-manifest.json`; `/sync-base` это доустанавливает.

## 9. Практические правила для сотрудника

1. **Одна задача — одна сессия.** Длинные сессии дороже по токенам и путаются в контексте. Сменил тему — начни новый чат (Claude сам предложит).
2. **Не выдумываем данные.** Имена, шифры, номера, даты, количества — если их нет, Claude спросит. Нормы (ГОСТ/СП) — только через `norm-lookup`, не «по памяти».
3. **Давай обратную связь.** Заметил косяк базы или придумал улучшение — скажи об этом; это уйдёт хабу и улучшит базу для всех (отдельно делать ничего не надо — система соберёт сама).
4. **Обезличивание.** В материалы, которые уходят в общую базу/feedback, не попадают ФИО, шифры, объекты, заказчики — Claude подставляет плейсхолдеры. Конкретику проекта держим в `FACTS.md` проекта.
5. **Доверяй ревьюеру.** Если Claude говорит, что прогнал результат через проверку (`auditor`/`word-checker`/…) — это часть процесса, не лишний шаг.
6. **Серьёзные документы — нейтральное оформление** (без декоративных цветов), по образцу документа или ГОСТ.

## 10. Что делать, если…

| Ситуация | Что сделать |
|---|---|
| `⚠ MCP не подключены: …` | Обычно прогреется само; если мешает — `/sync-base` |
| `⚠ запусти /sync-base` | Запусти `/sync-base` (хаб добавил инструменты) |
| «База ведёт себя странно / устарела» | `/sync-base` (подтянет свежее + самопроверка) |
| Нужен инструмент, которого нет | `/harvest <что нужно>` |
| Надо красиво оформить документ | `/format` |
| Большой незнакомый проект/свод доков | `/graphify <путь>` |
| Контекст переполнен на длинной задаче | Скажи «давай в новый чат» — соберёт пакет передачи |
| Что-то сломалось и непонятно | Опиши проблему обычными словами — Claude разберётся или подскажет |
