# Knowledge Library — спецификация

## Контекст и проблема

Сейчас агенты (`designer`, `pto-engineer`, `сметчик`, `audit-rd-section`,
`expertiza-responder` и т.д.) и сам Claude **выдумывают содержание
нормативов** когда нет первоисточника на руках. Empirically — session
2026-05-25 показала что даже профильный агент ошибается:

- `сметчик` — путал ПДВ (предельно-допустимые выбросы) и НДС
  (налог на добавленную стоимость), хотя это разные домены.
- `audit-rd-section` — ссылался на СП 76 вместо актуального СП 256
  (электрические сети).
- `pto-engineer` — придумывал маркировку П/В без проверки ГОСТ 21.602.

Эти ошибки **критичны** для строительной отрасли — выдача документа
с придуманной нормативной ссылкой в экспертизу = провал.

Запрос пользователя: **«нужна некая папка где лежит база знаний —
библиотека (не инструменты, не путать), и чтобы у всех сотрудников
Claude имели туда такой же доступ, и чтобы это не жрало токены
нещадно»**.

## Решение в одном предложении

**Тонкий локальный индекс в git claude-base + PDF файлы норм на
Я.Диске + расширенный `norm-lookup` агент = lazy retrieval для
дословных цитат без раздутия контекста.**

## Архитектура

### Файловая структура

**В git claude-base (раздаётся всем 9 ПК через auto-pull):**

```
~/.claude/library/
├── README.md                  ← что это и как пополнять (для людей и Claude)
├── INDEX.md                   ← главный каталог (плоская таблица всех документов)
├── categories/                ← один md-файл на категорию
│   ├── spds.md                ← ГОСТы 21.xxx (СПДС)
│   ├── ov.md                  ← СП по ОВ
│   ├── vk.md                  ← СП по водоснабжению/канализации
│   ├── eo.md                  ← электрика, ПУЭ
│   ├── ss.md                  ← слаботочные системы
│   ├── ppr.md                 ← постановления Правительства РФ
│   ├── prikazy.md             ← приказы Минстроя/Минрегиона
│   └── shablony.md            ← шаблоны / примеры оформления
└── extracts/                  ← (пустая на старте, заполняется при необходимости)
```

**Per-PC, gitignored:**

```
~/.claude/.library-config.json
```

Формат:
```json
{
  "library_path": "C:\\Users\\<USER>\\YandexDisk\\Claude_Library"
}
```

Это **полный путь** до папки Claude_Library на этом конкретном ПК.
Причина — у сотрудников shared папка Я.Диска может иметь префикс
«От Даниила» / другое имя / другой родительский каталог. Полный путь
снимает зависимость от формата.

**На Я.Диске у Daniil (single owner) + расшарено на чтение 8 сотрудникам:**

Daniil создаёт `Claude_Library/` в своём 1ТБ Я.Диске и расшаривает её
через веб Я.Диск.Доступ → инвайт каждому из 8 сотрудников **на чтение**
(не write). У сотрудников после принятия инвайта папка появляется в их
собственном Я.Диск-аккаунте, Я.Диск-клиент на их ПК синхронизирует её
локально (с возможным переименованием на их стороне — но содержимое
идентично).

**Структура внутри (одинаковая у всех 9 ПК благодаря Я.Диск sync):**

```
<YandexDisk>/Claude_Library/
├── spds/
│   ├── GOST_21.101-2020.pdf
│   └── ...
├── ov/
├── vk/
├── eo/
├── ss/
├── ppr/
├── prikazy/
└── shablony/
    ├── primer_PZ_OV.docx
    └── primer_specifikacii_AR.xlsx
```

### Формат `INDEX.md`

Плоская таблица — Claude всегда читает её целиком (~3-12K токенов для
50-200 норм, грузится только когда `norm-lookup` агент вызван).

```markdown
# Library INDEX

| ID | Категория | Документ | Полное название | Файл (rel) | Год | Статус |
|----|-----------|----------|-----------------|------------|-----|--------|
| spds-21.101 | СПДС | ГОСТ 21.101-2020 | Основные требования к проектной и рабочей документации | spds/GOST_21.101-2020.pdf | 2020 | actual |
| ov-60 | ОВ | СП 60.13330.2020 | Отопление, вентиляция и кондиционирование воздуха | ov/SP_60.13330.2020.pdf | 2020 | actual |
| ppr-87 | Постановления | Постановление 87-ПП | О составе разделов проектной документации | ppr/PP_87.pdf | 2008 | actual |
| shablon-pz-ov | Шаблоны | Пример ПЗ ОВ | Пояснительная записка раздел ОВ — типовой образец | shablony/primer_PZ_OV.docx | — | template |
```

**Статусы:** `actual` / `superseded:<id-нового>` / `cancelled` / `template`.

### Формат `categories/<имя>.md`

Расширенные аннотации (3-10 строк на норму) — для случаев когда таблицы
INDEX.md мало; помогает `norm-lookup` агенту понять **куда смотреть**
в PDF:

```markdown
# Категория: СПДС

## ГОСТ 21.101-2020

**Полное название:** Система проектной документации для строительства. Основные требования к проектной и рабочей документации.

**О чём:** Базовые требования к составу, оформлению, шифрам, штампам, форматам листов.

**Когда применять:** Всегда при оформлении ПД/РД.

**Ключевые пункты:**
- §4 — Общие положения
- §5 — Состав комплекта РД
- §6 — Правила выполнения текстовых документов
- §7 — Спецификации
- §A — Формы основных надписей

**Файл:** `spds/GOST_21.101-2020.pdf`
```

## Расширение `norm-lookup` агента

### Tools whitelist (расширение)

```yaml
tools: Read, Bash, Grep, Glob, WebFetch,
       mcp__pdf-mcp__pdf_info,
       mcp__pdf-mcp__pdf_search,
       mcp__pdf-mcp__pdf_read_pages,
       mcp__pdf-mcp__pdf_get_toc,
       mcp__pdf-mcp__pdf_render_pages,
       mcp__word__get_document_text,
       mcp__word__find_text_in_document,
       mcp__word__get_document_outline
```

- `mcp__pdf-mcp__*` — read-only PDF (для дословных цитат норм).
- `mcp__word__*` read-only — для работы с шаблонами `.docx`.
- `mcp__pdf-mcp__pdf_render_pages` — для случаев когда PDF без text
  layer (скан); потом OCR через скилл `image-text-replace`.

### Алгоритм поиска

```
Step 1: Прочитать ~/.claude/.library-config.json
        → library_path (полный путь до Claude_Library на этом ПК)
        Если файла нет → fallback на WebFetch cntd.ru сразу.

Step 2: Прочитать ~/.claude/library/INDEX.md.
        Найти подходящие записи по: ключевые слова в названии,
        категории, ID, году. Если совпадений несколько — все вернуть
        пользователю для уточнения.

Step 3: (Опционально) Прочитать categories/<имя>.md для аннотации
        ключевых пунктов.

Step 4: Собрать абсолютный путь:
        <library_path>\<file_rel>

Step 5: Проверить существование файла.
        Если нет → NOT_DOWNLOADED warning + WebFetch cntd.ru fallback.

Step 6: mcp__pdf-mcp__pdf_search "ключевое слово" → 1-3 фрагмента.
        Если нужен контекст шире → pdf_read_pages для N-N+2 страниц.

Step 7: Вернуть структурированный ответ:
        ─ Документ: <полное название + редакция>
        ─ Пункт: <номер раздела/пункта>
        ─ Цитата: «<дословный текст>»
        ─ Источник: <путь к файлу>
        ─ Достоверность: locally_read_PDF | cntd_ru_WebFetch | OCR_low_confidence | NOT_FOUND
```

### Failure modes (структурированные)

| Код | Условие | Поведение |
|-----|---------|-----------|
| `NO_CONFIG` | `.library-config.json` отсутствует | WARN в лог. Fallback на WebFetch. Подсказка пользователю: «запусти Set-LibraryRoot.ps1». |
| `NO_INVITE` | `library_path` указан, но папка не существует (сотрудник не принял invite) | Сообщить «прими invite от Daniil в веб-интерфейсе Я.Диска, потом запусти Set-LibraryRoot.ps1 заново». Fallback на cntd.ru. |
| `SYNC_IN_PROGRESS` | Папка существует, но в ней нет ни одной подкатегории (Я.Диск ещё качает) | Сообщить «Я.Диск синхронизирует библиотеку, подожди несколько минут». Fallback на cntd.ru. |
| `NOT_IN_INDEX` | INDEX.md не содержит запрошенной нормы | WebFetch cntd.ru попытка; если найдено — ответ с пометкой `cntd_ru_WebFetch`. |
| `NOT_DOWNLOADED` | Норма в INDEX, но конкретный файл на Я.Диске отсутствует (sync ещё не докачал именно этот файл) | Сообщить «файл нормы X не доехал на Я.Диск, проверь sync». Fallback на cntd.ru. |
| `NO_TEXT_LAYER` | PDF — скан (pdf_info: text_layer=false) | Auto-switch на pdf_render_pages + OCR через image-text-replace skill. Пометка `OCR_low_confidence`. |
| `WEBFETCH_BLOCKED` | cntd.ru заблокирован прокси / 403 / timeout | Сообщить честно «не могу подтвердить». Спросить пользователя добавить норму в библиотеку. **Запрет на придумывание.** |
| `CANCELLED` | Норма со статусом `cancelled` | Цитата + явное предупреждение «отменена, для новых проектов не использовать». |
| `SUPERSEDED` | Норма со статусом `superseded:<новый-id>` | Цитата + предупреждение + ссылка на актуальную редакцию. |

**Запрещено:** возвращать «не нашёл» без указания failure mode, или
выдумывать цитату при любом failure. Это противоречит главному
требованию пользователя.

## Правило в CLAUDE.md

Минимальная вставка в секцию «Универсальные правила работы» (~50 токенов
на каждый turn — приемлемо):

```markdown
7. **Нормы — только через `norm-lookup`.** Локальная библиотека норм
   лежит в `~/.claude/library/` (INDEX.md + categories) + PDF на Я.Диске.
   **Не цитировать** ГОСТ/СП/СНиП/ПУЭ/постановления по памяти. Любая
   нормативная ссылка → делегация в `norm-lookup` агента для точной
   цитаты + пункта + редакции.
```

## Per-PC setup helper

`scripts/Set-LibraryRoot.ps1` — интерактивный скрипт по аналогии с
`Set-FeedbackToken.ps1`:

1. Спрашивает **полный путь** до Claude_Library на этом ПК
   (например `C:\Users\Иван\YandexDisk\Общая Claude_Library от Даниила`).
   Default попытка: `$env:USERPROFILE\YandexDisk\Claude_Library`.
2. Проверяет существование (Test-Path). Если не существует — WARN и
   подсказка «прими invite от Даниила в веб-интерфейсе Я.Диска, потом
   запусти этот скрипт заново».
3. **Smoke check:** проверяет что внутри есть хотя бы одна из ожидаемых
   подпапок (spds/ov/vk/eo/ss/ppr/prikazy/shablony). Если ни одной —
   WARN «папка пустая, возможно Я.Диск-клиент ещё не закончил sync».
4. Пишет `~/.claude/.library-config.json`.
5. Финальный smoke: вывести путь и количество найденных PDF.

**На DANIILPC** (owner): скрипт дополнительно создаёт 8 категорийных
подпапок если их нет, потому что Daniel пишет в эту папку.

**На consumer-ПК**: read-only режим, подпапки уже подтянутся через
Я.Диск sync, скрипт не пытается их создавать.

## Раскатка

| Этап | Где | Что |
|------|-----|-----|
| 0 | Daniil в веб-интерфейсе Я.Диска (one-time) | Создаёт `Claude_Library/` в своём 1ТБ Я.Диске. Расшаривает через «Поделиться → Право: только просмотр» → инвайты на яндекс-почту/логины 8 сотрудников. Сотрудники получают invite-нотификацию |
| 1 | git claude-base (этим commit) | Скелет `~/.claude/library/` + helper + расширение `norm-lookup` + правило в CLAUDE.md + CHANGELOG запись |
| 2 | На каждом ПК (next SessionStart) | auto-pull подтягивает изменения. CHANGELOG notification читается Claude'ом в первой реплике |
| 3 | Сотрудник (one-time, ~3 минуты) | (a) В веб-интерфейсе Я.Диска принять invite от Daniil → папка появляется в его Я.Диске. (b) Дождаться окончания первичного sync (Я.Диск-клиент покажет прогресс). (c) Запустить `Set-LibraryRoot.ps1` → конфиг создан |
| 4 | Daniil (постепенно, неделями) | Добавляет PDF в свой Claude_Library/ + обновляет INDEX.md в git + commit |
| 5 | Сотрудник (автоматически) | После auto-pull у него индекс есть; PDF подтянутся Я.Диском в фоне (Daniil добавил → sync → у сотрудника появилось) |

**Read-only safety:** Сотрудники имеют **read-only** доступ к Я.Диск
shared folder. Они не могут случайно удалить или изменить норму.
INDEX.md в git — write также только у Daniil. Это устраняет класс ошибок
типа «сотрудник случайно положил тестовый файл с тем же именем что норма».

## Версионность (новые редакции норм)

- Новая редакция = новая строка в INDEX.md (`status: actual`).
- Старая = меняется `status: superseded:<id-нового>`.
- **Файл старой нормы НЕ удаляется** — нужен для работы со старыми проектами.
- `norm-lookup` при работе со `superseded` обязан предупредить о наличии новой редакции.

## Тестирование

### Smoke test (на DANIILPC после раскатки)

1. Скелет `~/.claude/library/` создан ✓
2. `Set-LibraryRoot.ps1` запущен, `.library-config.json` создан ✓
3. Положить один тестовый PDF в `<YandexDisk>/Claude_Library/spds/`
4. Добавить строку в INDEX.md
5. Заполнить `categories/spds.md` секцию для этой нормы
6. **Restart Claude Code** (агенты грузятся при старте)
7. `Agent(subagent_type="norm-lookup", prompt="что говорит ГОСТ 21.101 про форму основной надписи?")`
8. **Ожидаемое:** структурированный ответ с цитатой + § + путём + `Достоверность: locally_read_PDF`

### Integration test (когда библиотека наполнена ~10 нормами)

1. `Agent(subagent_type="audit-rd-section", prompt="проверь раздел ОВ на соответствие ГОСТ 21.602")`
2. **Ожидаемое:** `audit-rd-section` сам делегирует в `norm-lookup`, получает точные цитаты, использует их в отчёте.

### Edge case tests

- Запрос нормы **не в INDEX** → WebFetch попытка → если не нашёл, явное `NOT_IN_LIBRARY`.
- В INDEX путь к **несуществующему файлу** → `NOT_DOWNLOADED`.
- Норма со `status: cancelled` → явное предупреждение.

### Что НЕ тестируем сейчас (YAGNI)

- Производительность с 200+ нормами — replicate когда наполнится.
- Concurrent access из нескольких ПК — Я.Диск разрулит.

## Метрика успеха

После раскатки и накопления минимум 10 норм:

1. На session где упомянуты ГОСТ/СП/СНиП — `norm-lookup` агент вызван хотя бы раз.
2. В ответах агентов (`designer`, `audit-rd-section`, `сметчик`) — нормативные ссылки **с дословной цитатой и пунктом**, не пересказ по памяти.
3. На запросы по нормам которых нет в библиотеке — агент **честно говорит** «не нашёл», а не выдумывает.
4. На session-report'ах сотрудников — упоминания «использована библиотека норм».

## Out of scope (отдельные задачи)

- **Я.Wiki интеграция** для сотрудников-людей (отдельный канал).
- **Автоматический парсинг новых норм с cntd.ru** в INDEX.md.
- **OCR-индексация полнотекстовых search'ей** для сканов.
- **Кросс-ссылки между нормами** (ГОСТ X ссылается на ГОСТ Y).
- **Чтение библиотеки через Claude Code Slash command** (`/norm <название>`).
- **Распространение через `claude-lite-instaler` Stage 10** — будет добавлено отдельным commit когда base изменения зацементируются.
