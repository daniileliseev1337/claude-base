# project-memory v2 — усиление ядра проекта (дизайн)

**Дата:** 2026-07-09 · **Скилл:** `project-memory` · **Статус:** дизайн согласован, ждёт ревью владельца

## Проблема

Владелец (2026-07-09): «моментами прям ужасно, ничего не происходит, не реагирует на
жалобы». Разложено на 4 боли (владелец отметил ВСЕ, главная — первая):

1. **Не держит контекст проекта** (главная) — стартует будто впервые, не подхватывает
   STATUS/журнал, переспрашивает решённое.
2. **Игнорит доменные правила** — лепит из головы вместо норм/форматов проекта, не в «роли».
3. **Повторяет тупики** — забывает, что уже пробовали и не сработало.
4. **Не реагирует на жалобы** — на «это плохо / не то» продолжает то же, не меняет подход.

**Корень боли #1 (установлено вопросом):** механизм подхвата ядра завязан на `cwd`, а
владелец стартует сессии «по-разному» (то из папки проекта, то из общего окна VS Code с
путём в сообщении). SessionStart-хук ищет `Claude/` в cwd → при старте не из папки проекта
молчит, ядро не попадает в контекст. Усиливать содержание бессмысленно, пока доставка
ненадёжна.

## Цели (проверяемые критерии успеха)

- Ядро проекта попадает в контекст НЕЗАВИСИМО от того, как стартовали (cwd ИЛИ путь в
  сообщении).
- Перед мутирующей работой по проекту роль+критерии+грабли ГАРАНТИРОВАННО прочитаны
  (жёсткий гейт, детерминированный факт чтения).
- Роль проекта явно связывает его с ведущим доменным агентом из 16 (мостик к архитектуре
  main→агент→аудитор).
- Жалоба пользователя ловится как стоп-сигнал.
- Цена «на каждую сессию» измерена и удержана низкой (см. §Производительность).
- Вне папок-проектов памяти — молчаливый no-op (обычные сессии базы не задеты).

## Архитектура — 5 компонентов

Усиление существующего скилла (не переписывание). Текущее ядро (`CLAUDE.md`, `STATUS.md`,
`README.md`, `ЖУРНАЛ СЕССИЙ.md`, root-указатель, хуки session_start/end) остаётся; добавляем
доставку, гейт, файл содержания, реакцию на жалобы.

### ① Надёжная доставка (фундамент, лечит #1)
- **UserPromptSubmit**-логика: извлечь пути из сообщения пользователя (Windows `C:\…`,
  кавычки, относительные) → walk-up от каждого до ближайшего `Claude/ЖУРНАЛ СЕССИЙ.md` =
  проект. cwd-канал (текущий session_start) остаётся вторым.
- Walk-up переиспользует логику из `tools/hooks/session_start.ps1` (один дом).
- **Ловушки** (из первого разворота, память `project_memory_skill_done`): несколько путей в
  сообщении → первый с ядром; session-маркер не перетирать; `session_id` у хука есть.

### ② Блокирующий гейт чтения (лечит #1/#2, жёсткость = блокировка — выбор владельца)
Хук не видит текст ответов Claude, поэтому «подтверди словами» — только мягкий слой.
Настоящая блокировка — на ДЕТЕРМИНИРОВАННОМ факте чтения:
- **Мягкий слой:** нашли проект → инжектим ядро + директиву `СТОП. По проекту <X>
  подтверди: ✓ прочитал КОНТЕКСТ (роль, критерии, грабли)` + маркер `<сессия+проект>=показано`.
- **Регистрация чтения:** PostToolUse ловит `Read(КОНТЕКСТ.md)` этого проекта → маркер=`прочитано`.
- **Жёсткий слой:** PreToolUse на мутацию (Write/Edit/Task) в папке проекта → маркер ≠
  `прочитано` → блок `exit 2` «сначала прочитай КОНТЕКСТ проекта». Блокируется РАБОТА
  РУКАМИ, не чтение и не диалог.

### ③ `КОНТЕКСТ.md` — обязательное ядро содержания (лечит #2/#3)
Bootstrap кладёт с плейсхолдерами; заполняет владелец/агент. Шаблон:

```markdown
# <ПРОЕКТ> — КОНТЕКСТ (обязательно прочитать ДО работы)

## ТВОЯ РОЛЬ
В этой задаче ты выступаешь в роли: <напр. инженер-проектировщик ОВ>.
Домен: <ОВ / ВК / ЭО / СС / ИД / ПТО / сметы / снабжение>.
Ведущий агент: `<имя из 16>` — содержательную работу вести ЧЕРЕЗ него (Task), НЕ инлайн.
Соблюдать: <ключевые нормы / форматы / правила именно этого проекта>.

## КРИТЕРИИ ГОТОВНОСТИ (когда задача «сделана»)
- <проверяемый критерий 1>

## ГРАБЛИ (топ-3 — чего НЕ делать)
- <что уже пробовали и не сработало / не трогать>

## ФАКТЫ
→ FACTS.md (шифр, нагрузки, отметки — там, не дублировать)
```

**Роль-мостик к агентам (уточнение владельца):** секция ТВОЯ РОЛЬ несёт поле
`Ведущий агент` из 16. `bootstrap` при развороте спрашивает домен → мапит на агента:
ОВ/ВК/ЭО/СС→`designer`; объёмы/ВОР/спеки→`pto-engineer`; ИД/акты→`id-engineer`;
сметы→`сметчик`; снабжение/УПД→`снабженец`; письма→`letter-writer`; КП→`kp-writer`;
замечания экспертизы→`expertiza-responder`; Revit→`pyrevit-engineer`. Поле ПРЕДЗАПОЛНЯЕТ
роутинг-гейт глобального CLAUDE.md — main сразу знает домен и агента.

### ④ Опциональные файлы по мере роста (лечит #3, не задавить заполнением)
- `РЕШЕНИЯ.md` — журнал решений владельца (ADR-лайт) + активные развилки.
- `ГРАБЛИ.md` — полный список (топ-3 в КОНТЕКСТ, остальное здесь).
- НЕ обязательны на старте; bootstrap их не создаёт, создаются по потребности.

### ⑤ Реакция на жалобы (лечит #4)
UserPromptSubmit (в том же объединённом хуке, что ①) ловит маркеры недовольства
(«ужасно», «не то», «ничего не происходит», «сколько можно», «не реагируешь») → инжектит
стоп-сигнал: `Пользователь недоволен — НЕ продолжай то же. Разберись что не так, сверься с
КОНТЕКСТ/критериями, предложи иной подход или спроси`. Паттерн `grilling-detector`.

## Производительность и цена (замер 2026-07-09, эта машина)

| Что | Латентность |
|---|---|
| Старт PowerShell (пустой) | ~180 мс |
| Хук с логикой | ~270-350 мс |
| session_start (walk-up) | ~300-320 мс |
| UserPromptSubmit сейчас (3 хука) | ~900 мс/промпт |

**Вывод:** доминанта — СТАРТ PowerShell (~180 мс), не логика (~+100 мс). Цена измеряется в
ЧИСЛЕ ПРОЦЕССОВ. Поэтому механизм **облегчённый — вливаем в существующие хуки, 0 новых
процессов:**
- path-detector + детектор жалоб → в существующий UserPromptSubmit (объединить, не плодить).
- регистрация `Read(КОНТЕКСТ.md)` → в существующий `log-tool-usage` (PostToolUse).
- гейт мутаций → в существующий PreToolUse (`screenshot-source-zoom`) либо один новый (+180мс
  только на мутации, их меньше промптов).
- **инжекция ядра — ОДИН раз на сессию** (маркер), не на каждый промпт → ~1-2K токенов/сессия.
- **мгновенный no-op вне проектов** — первой строкой проверка «есть путь и `Claude/` выше?»,
  нет → выход ~5мс.

Чистая цена облегчённого: +~50-100 мс логики на промпт + ~1-2K токенов однократно на сессию
+ один обязательный Read. Полный механизм (отдельные хуки) отвергнут: +600-1200 мс/промпт.

**Настоящая жертва — надёжность:** больше логики в критических хуках = риск бага (за день
поймали 3 кодировочных бага в хуках). Обязательно в реализации: UTF-8-обвязка stdin первой
строкой каждого трогаемого хука; тест на каждый; no-op-выход проверен.

## Границы (scope)

- Вне папок-проектов памяти (нет `Claude/ЖУРНАЛ` в cwd и не найден путём) — все хуки
  no-op. Обычные сессии базы, чужие проекты — не задеты.
- Гейт блокирует ТОЛЬКО мутации в подтверждённой папке проекта (узкий scope против ложных
  блоков).
- Не трогаем: curate_rot, механику бэкапов, мультидевайс-правила (относительные пути, mtime).

## Тестирование

- Юнит walk-up: путь в сообщении → правильный проект; несколько путей → первый с ядром;
  нет проекта → no-op.
- Гейт: мутация без чтения КОНТЕКСТ → exit 2; после Read → проходит; вне проекта → no-op.
- Реакция на жалобы: маркер во фразе → стоп-сигнал инжектится; обычная фраза → тихо.
- Кодировка: кириллица в путях/выводе под cp1251-консолью → без UnicodeEncodeError (регресс
  уроков дня).
- Латентность: замер объединённого хука ≤ существующего +100мс.
- Bootstrap: КОНТЕКСТ.md кладётся с ролью-плейсхолдером; выбор домена→агент маппинг.
