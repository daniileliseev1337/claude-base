# 1С Research Sub-project — спецификация

## Контекст

Большой проект «выгружать данные из 1С для отчётности и дашбордов
начальству» декомпозирован на 5 подпроектов (A→E). Этот spec покрывает
**только подпроект A** (Research / собрать факты + получить доступ).

**Out of scope этого spec'a:**
- Sub-project B (Access mechanism) — детали извлечения через OData
- Sub-project C (Extract pipeline) — пайплайн регулярного забора
- Sub-project D (Dashboards) — визуализация
- Sub-project E (Maintenance) — мониторинг, ротация credentials

## Цель Research

Получить **read-only доступ** к 1С:ERP фирмы через OData REST API:

1. Передать IT-исту формальное ТЗ (что настроить в 1С).
2. Получить от IT-иста: URL, credentials, версию платформы.
3. Зафиксировать **карту 1С** в обновлённом spec'е (этот файл).
4. Дать сигнал старта подпроекту B.

## Подтверждённые факты (от пользователя)

- В фирме используется **одна** конфигурация 1С — **1С:ERP**
  (Управление предприятием). Содержит **всё** в одной базе:
  бухучёт, снабжение, продажи, производство, сотрудники, проекты.
- Доступ нужен **только на чтение** (read-only).
- IT-исты / 1С-исты **доступны** и готовы помочь. Запросили
  **структурированный список требований**.
- Целевые отчёты для дашбордов начальства:
  - **Финансы**: выручка / прибыль / ДЗ к КЗ / платежи
  - **Проекты**: статусы договоров, сроки, план vs факт
  - **Сотрудники**: зарплаты, загрузка, KPI
- Снабжение / закупки — **НЕ в скоупе** дашбордов (пока).
- На ПК всех 9 сотрудников **нет админ-прав** (ограничение для COM-варианта,
  но OData REST это не требует — поэтому не проблема).

## Подход — OData REST API + выделенный read-only пользователь

Стандартный путь для 1С 8.3:

1. IT-ист создаёт техн. пользователя `claude_readonly` с ролью «Чтение
   всех данных».
2. IT-ист публикует веб-сервер 1С (Apache 2.4 или IIS) если не
   опубликован — стандартная процедура из Конфигуратора.
3. В диалоге публикации ставится галочка «Опубликовать стандартный
   OData-интерфейс».
4. Доступ через HTTPS (Basic Auth):
   `https://<server>/<publish-name>/odata/standard.odata/<EntitySet>`.
5. Мы тянем данные через `requests` + DPAPI-зашифрованный пароль.

Альтернативы (B HTTP-services, C Excel-выгрузки) — на случай если
OData в этой 1С:ERP заблокирован политикой или платформа не поддерживает.

## Риски

- **OData не разрешат политикой безопасности** фирмы — fallback на
  HTTP-services (B) или Excel-выгрузки (C).
- **Старая платформа 1С** (< 8.3.5) — OData может быть глючным.
  Узнать версию первым делом.
- **Self-signed сертификат** — нам придётся выключить verify или
  добавить cert в trusted store на каждом из 9 ПК. Решаемо.
- **Bus factor** — credentials хранятся через DPAPI per-user/per-machine.
  Если Daniil не может войти на свой ПК — никто не дёрнет данные.
  Решение в подпроекте E (Maintenance) — назначить **резервного хранителя**
  credentials или использовать общий vault фирмы (Bitwarden / 1Password).
