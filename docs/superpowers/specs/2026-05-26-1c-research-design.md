# 1С Research Sub-project — спецификация

## Контекст

Большой проект «выгружать данные из 1С для отчётности и дашбордов
начальству» декомпозирован на 5 подпроектов (A→E). Этот spec покрывает
**только подпроект A** (Research / собрать факты + получить доступ).

**Out of scope этого spec'a:**
- Sub-project B (Access mechanism) — детали извлечения через OData
- Sub-project C (Extract pipeline) — пайплайн регулярного забора
- Sub-project D (Dashboards) — визуализация
- Sub-project E (Maintenance) — мониторинг, ротация credentials

## Цель Research

Получить **read-only доступ** к 1С:ERP фирмы через OData REST API:

1. Передать IT-исту формальное ТЗ (что настроить в 1С).
2. Получить от IT-иста: URL, credentials, версию платформы.
3. Зафиксировать **карту 1С** в обновлённом spec'е (этот файл).
4. Дать сигнал старта подпроекту B.

## Подтверждённые факты (от пользователя)

- В фирме используется **одна** конфигурация 1С — **1С:ERP**
  (Управление предприятием). Содержит **всё** в одной базе:
  бухучёт, снабжение, продажи, производство, сотрудники, проекты.
- Доступ нужен **только на чтение** (read-only).
- IT-исты / 1С-исты **доступны** и готовы помочь. Запросили
  **структурированный список требований**.
- Целевые отчёты для дашбордов начальства:
  - **Финансы**: выручка / прибыль / ДЗ к КЗ / платежи
  - **Проекты**: статусы договоров, сроки, план vs факт
  - **Сотрудники**: зарплаты, загрузка, KPI
- Снабжение / закупки — **НЕ в скоупе** дашбордов (пока).
- На ПК всех 9 сотрудников **нет админ-прав** (ограничение для COM-варианта,
  но OData REST это не требует — поэтому не проблема).

## Подход — OData REST API + MCP-экосистема (бесплатный путь)

После harvest 2026-05-26 (см. `harvested/2026-05-26_1c-erp-integration-tools.md`)
выявлен **критический insight**:

> **Все коммерческие коннекторы** (Денвик, ATK BiView, 1С-Рарус) идут
> через **SQL view над базой 1С**, НЕ через OData. Чистый OData
> «зависает на регистрах накопления» — а именно регистры это финансы
> (взаиморасчёты, остатки, продажи). OData годится для **простых
> запросов** (документы, справочники), но **не для агрегатов из
> регистров** в production масштабе.

**Решение пользователя 2026-05-26:** идём бесплатным путём (OData + MCP),
платные коммерческие коннекторы — отложить и взять только если бесплатный
путь упрётся в потолок.

### Двойной трек

**Track 1 (краткосрочный):** Стандартный OData REST API:
1. IT-ист создаёт пользователя `claude_readonly` с ролью «Чтение всех данных».
2. IT-ист публикует веб-сервер 1С (Apache 2.4 / IIS).
3. Включает «Опубликовать стандартный OData-интерфейс».
4. Доступ через HTTPS (Basic Auth):
   `https://<server>/<publish-name>/odata/standard.odata/<EntitySet>`.
5. Мы тянем данные через `requests` + DPAPI-зашифрованный пароль.

**Применимость Track 1:** простые отчёты (списки контрагентов, договоров,
документов реализации/АВР). Прототипы и smoke-тесты. Финансовые регистры
накопления — **под вопросом**, проверим эмпирически.

**Track 2 (параллельно изучаем):** **MCP-экосистема для 1С**:

Свежие репо (апрель-май 2026) реализуют MCP-серверы которые Claude Code
дёргает напрямую — без BI-инструмента в середине. Топ кандидаты:

- **`Untru/1c-mcp`** (★94, CC0) — каталог 40+ MCP-серверов для 1С,
  явная поддержка 1С:ERP. Hub-точка входа.
- **`ROCTUP/1c-mcp-toolkit`** (★140) — toolkit для построения MCP под 1С.
- **`Nikolay-Shirokov/cc-1c-skills`** (★333, MIT) — большая база Claude
  Code skills специально под 1С:Предприятие. **Прямой аналог нашей
  экосистемы `~/.claude/skills/`** — изучить методологию.

**Применимость Track 2:** AI-driven дашборды через Claude. Может
решить проблему регистров если MCP-сервер сам внутри обращается через
HTTP-services / SQL view (а не OData).

### Что отложили (платное)

- **Денвик 1С-Connector** (25k₽ + 2k₽/мес) — fallback если бесплатно не вытянет.
- **1С:Аналитика** (39.7k₽) — fallback enterprise.
- **ATK BiView** (412k₽) — не наш bracket.

Берём только если эмпирически упрёмся в потолок бесплатного пути.

## Метрика успеха Research

- [ ] ТЗ передано IT-исту (`docs/1c-it-requirements.md`).
- [ ] От IT получены: URL, версия платформы, credentials.
- [ ] Credentials зашифрованы через DPAPI в `~/.claude/.1c-config.json`.
- [ ] Smoke test проходит — `requests.get($metadata)` возвращает XML
      с описанием объектов.
- [ ] **Эмпирический тест OData на регистре накопления**:
      `AccumulationRegister_Взаиморасчёты` за месяц → засекаем время.
      Если > 30 сек или 500 ошибка → ставим флаг `odata_aggregates_slow=true`
      и активируем поиск через MCP / HTTP-services.
- [ ] Изучен `Untru/1c-mcp` (Track 2) — что включает 40+ MCP, какие
      из них применимы.
- [ ] Изучен `cc-1c-skills` методологически — есть ли паттерны которые
      перенесём в нашу базу.
- [ ] Раздел «Карта 1С:ERP фирмы» в этом spec'е заполнен фактами.
- [ ] Подпроект B (Access pipeline) может стартовать.

## Риски

- **OData не разрешат политикой безопасности** фирмы — fallback на
  HTTP-services (B) или Excel-выгрузки (C).
- **Старая платформа 1С** (< 8.3.5) — OData может быть глючным.
  Узнать версию первым делом.
- **Self-signed сертификат** — нам придётся выключить verify или
  добавить cert в trusted store на каждом из 9 ПК. Решаемо.
- **Bus factor** — credentials хранятся через DPAPI per-user/per-machine.
  Если Daniil не может войти на свой ПК — никто не дёрнет данные.
  Решение в подпроекте E (Maintenance) — назначить **резервного хранителя**
  credentials или использовать общий vault фирмы (Bitwarden / 1Password).
