# Дизайн: facts-layer — единый источник правды по проекту

- **Дата:** 2026-06-02
- **Статус:** approved (brainstorming), → writing-plans
- **Источник идеи:** принцип #2 агентной архитектуры (пост пользователя 2026-06-02):
  «единый источник правды; факты ≠ CLAUDE.md; свалишь одно в другое — помойка».
- **Закрывает:** дыру в claude-base — у доменных агентов нет единого файла фактов
  на проект, факты размазаны по спецификации/КП/смете/промптам → противоречия.

## Проблема

На реальном объекте «факты» (шифр, цена, нагрузки, отметки, даты, заказчик)
размазаны по нескольким артефактам и промптам агентов. Поменял в одном месте, в
трёх забыл → система себе противоречит. Нет `FACTS.md`, который `designer` /
`сметчик` / `снабженец` / `pto-engineer` / `kp-writer` читают первым.

## Решённые развилки (brainstorming)

1. **Охват:** лёгкий **standalone** механизм — работает даже на одной спецификации
   (как в посте: «файл фактов для простых проектов»). На крупных задачах FACTS.md
   живёт в `_meta/` рядом с 5 артефактами structured-artifacts.
2. **Запись факта:** `ключ → значение → источник` (провенанс). НЕ минимум (риск
   «второй помойки»), НЕ полная с потребителями (impact #7 отложен, YAGNI).
3. **Подход:** лёгкий скилл `facts-layer` + правило-триггер в CLAUDE.md + шаблон.
   (Отклонено: B — целиком в structured-artifacts, противоречит охвату «любой проект»;
   C — программный facts.yaml + reader, переусложнение сейчас.)

## Архитектура и границы

`FACTS.md` = единый источник правды по проекту, **только данные**. Жёсткое
разделение (главное правило — не смешивать):

| Файл | Что | Пример |
|---|---|---|
| **FACTS.md** | факты (данные) | шифр, цена, нагрузка ОВ, отметка, заказчик |
| CLAUDE.md | инструкции «как работать» | роутинг, правила |
| `_meta/DECISIONS.md` | решения «почему» | «выбрали схему Б, потому что…» |
| auto-memory | межпроектные уроки | «WebFetch фейлит» |

**Где живёт:** `<project-cwd>/FACTS.md`. Если активны structured-artifacts —
`<project-cwd>/_meta/FACTS.md` (6-й артефакт). НЕ в `~/.claude/` (это данные
конкретного проекта, не методбаза).

## Формат

Markdown. Frontmatter: `project`, `last_updated`. Тело — таблицы по категориям
(Объект / Заказчик / Технические / Цены / Даты; категории — по факту, не наперёд).
Строка таблицы:

```markdown
---
project: <объект>
last_updated: 2026-06-02
---

## Объект
| Факт | Значение | Источник |
|---|---|---|
| Шифр | ПСИ-158-06-01 | ТЗ п.1.2 |

## Технические
| Факт | Значение | Источник |
|---|---|---|
| Нагрузка ОВ | 48 кВт | расчёт designer 2026-06-02 |

## Цены
| Факт | Значение | Источник |
|---|---|---|
| Цена ВУ-1 | 320 ₽/шт | КП поставщика №45 |
```

**Источник** — провенанс факта: ячейка/пункт ТЗ, пункт ГОСТ/СП, «КП №N»,
«решение пользователя <дата>», «вычислено: <агент> по <входам>». Цель — факт
**трассируется** и перепроверяем (принцип #5 «только оригинал»), а не копирует
источник вслепую.

## Data flow

1. **Старт проекта:** оркестратор грилит пользователя (`domain-grilling`) → пишет
   FACTS.md с источниками. Чего нет — спрашивает, **не выдумывает** (правило #1).
2. **Любая доменная задача/агент:** читает FACTS.md **первым** (целиком — он мал).
3. **Агент вывел новый факт:** append со `Источник = вычислено: <агент> по <входам>`.
4. **Пользователь поменял факт:** правка **в одном месте** (значение + источник +
   `last_updated`) → все агенты подтянут новое при следующем чтении.

## Механизм «читать первым»

- **(а)** Правило в CLAUDE.md: «если в папке проекта есть `FACTS.md` — прочитать
  его первым перед доменной работой; факты править только там».
- **(б)** Оркестратор при спавне агента дублирует в промпте: «сначала прочитай FACTS.md».
- **(в)** Строка-напоминание в шаблоне доменных агентов (`agents/_TEMPLATE.md`).

## Компоненты (юниты)

1. `skills/facts-layer/references/FACTS.template.md` — шаблон.
2. `skills/facts-layer/SKILL.md` — методология: когда создавать, как грилить факты,
   read-first, change-handling, границы (не путать с CLAUDE.md/DECISIONS/memory).
3. CLAUDE.md — правило-триггер (1 строка/мини-секция).
4. `agents/_TEMPLATE.md` — строка «читать FACTS.md первым» (для будущих агентов).

## Обработка ошибок

- Факт нужен, но его нет в FACTS → **грилить пользователя, не выдумывать**.
- Устаревание: `last_updated` во frontmatter; если документ-источник изменился —
  флаг на перепроверку зависимых фактов.
- Конфликт значения в FACTS и в готовом артефакте → FACTS приоритетен (он SoT);
  артефакт правится.

## Отложено (честно, не в этом scope)

- **#7 impact-map** (колонка «потребители» → что перегнать при смене факта). Без неё
  решение «что перезапустить» остаётся у человека. Расширение на будущее — добавить
  колонку `Consumers`, когда реально понадобится выборочный перезапуск.
- Программный reader (`facts.yaml` + скрипт) — если факты станут строго структурными.

## Критерий успеха (верификация)

Факт встречается в **одном** месте; смена в FACTS → все агенты используют новое
значение, без противоречий. Эвал: проект, где цена нужна 3 агентам → меняем цену в
FACTS → проверяем, что все три взяли новую (а не старую из своего промпта).

## Композиция с существующим

- **structured-artifacts** — FACTS.md встаёт 6-м файлом в `_meta/` на крупных задачах.
- **domain-grilling** — механизм первичного наполнения FACTS.md.
- **chains-pattern** — chain может начинать с «заполни FACTS.md».
- **karpathy #1 (не выдумывать)** — FACTS пустой → грилить, не подставлять.
