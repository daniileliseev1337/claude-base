# Сводка моделей Ollama — июнь 2026

> Полный обзор каталога ollama.com/library (~210 моделей): кто что умеет, какие мощности
> нужны, что можем поставить мы. Источники: страница library + карточки моделей
> (выкачаны 2026-06-12), знание семейств до 01.2026. Эталонное железо в сводке —
> **граф-машина базы: RTX 3060 12GB VRAM + 64GB RAM + Ryzen 5 3600** (домашний ПК-хаб).
> Для других ПК отдела (без GPU, 16GB RAM) — только классы «✅ лёгкие» до ~4B.

## Как читать

| Символ | Значит |
|---|---|
| ✅ | влезает в VRAM 12GB целиком — максимальная скорость (20–60+ tok/s) |
| 🟡 | частично в RAM. **MoE с малой активацией — быстро** (15–25 tok/s), плотные — медленно (3–7 tok/s, только batch) |
| ❌ | локально нереально (>45GB или cloud-only) |

Ключевая физика: скорость упирается в то, сколько весов читается на каждый токен.
У плотной (dense) модели — все веса, у MoE — только активные эксперты. Поэтому
**MoE 30B с 3B активных (класс «A3B») — золотой класс для нашей машины**: качество ~30B,
скорость ~3B даже с выгрузкой в RAM.

Размеры указаны для квантования q4 (дефолтный тег), если не сказано иное.
Pull через корп-прокси рвётся TLS-таймаутами — большие модели тянуть с перерывами,
Ollama докачивает с места обрыва.

## Эксплуатационные грабли этой машины (проверено 2026-06-11/12)

1. **OLLAMA_MODELS обязан быть ASCII** — кириллический профиль (`C:\Users\Даниил ПК`) ломает
   llama-server (битый путь). Итоговое расположение: `F:\ollama-models` (52.5 GB; холодная
   загрузка с SATA ~12 с против ~4 c с NVMe — плата разовая, на скорость генерации не влияет).
   На других ПК с кириллическими профилями — та же мина.
2. **`ollama run` из CLI виснет** в окружении корп-прокси: перед генерацией CLI стучится
   на ollama.com. Лечение: `NO_PROXY=localhost,127.0.0.1` либо работать через REST
   (`http://127.0.0.1:11434/api/generate`) — graphify так и делает.
3. **Tray-приложение Ollama не подхватывает свежий env** до перелогина — после смены
   OLLAMA_MODELS перезапускать сервер с явной переменной или перелогиниться.
4. RTX 3060 (cc 8.6) не входит в cuda_v13-сборку Ollama 0.30.6 — работает fallback cuda_v12,
   это штатно, GPU используется.
5. qwen2.5:7b слаб для строгого JSON-вывода на больших чанках — graphify бисектит чанки
   и теряет часть. Для structured-output задач брать модели новее/крупнее (см. шорт-лист).
6. Pull больших моделей через прокси рвётся TLS-таймаутами — докачка работает, просто
   перезапускать pull.
7. **Thinking-модели НЕ ставить на batch-экстракцию.** qwen3.5:9b на graphify: ~5 мин/чанк —
   модель наматывает тысячи скрытых reasoning-токенов перед каждым JSON (capability
   `thinking` в `ollama show`). Для конвейерных structured-output задач брать модели БЕЗ
   thinking: qwen2.5-coder:14b (9.0GB, рекомендация самого graphify), granite4.1:8b.
   Thinking-класс — для интерактива и сложных одиночных задач, не для потока.
8. **Видимость делегированных запросов:** Open WebUI показывает только свои чаты; запросы
   инструментов (graphify и т.п.) идут в Ollama напрямую. Хотим видеть всё — нужна
   LiteLLM-прослойка (см. harvest 2026-06-12).
9. **Замеренные скорости на graphify-чанке (~6k токенов промпта, RTX 3060 12GB, ctx 16384),
   прогон базы 440 doc-файлов / 172 чанка, 2026-06-12:**

   | Модель | Размещение | Темп/чанк | Вывод |
   |---|---|---|---|
   | qwen2.5:7b @ ctx 4096 | VRAM | ∞ (hollow) | контекст 4096 «обрезает» чанк — мусор |
   | qwen3.5:9b | VRAM | ~300 с | thinking сжигает время на скрытые рассуждения |
   | qwen2.5-coder:14b | 84% VRAM / 16% RAM | ~300 с | вылез из VRAM на 16% — скорость рухнула на порядок |
   | **qwen2.5-coder:7b** | **100% VRAM** | **~18 с** | правильная посадка бьёт «ум» в 16 раз |

   **Формула посадки для 12GB VRAM: вес модели (q4) + 2–3 GB на KV-кэш 16k ≤ 11 GB**,
   т.е. плотные ≤ 8B. Итог прогона: 1263 узла / 1277 рёбер, 840k in / 209k out токенов,
   **$0.00**. Кросс-файловых связей 52% (у Claude-эталона 43%), но INFERRED-связей
   втрое меньше (63 против 225) и метки шумнее — локальная экстракция буквальнее Claude.
10. **Кэш graphify ключуется парой файл+модель** — смена `--model` обнуляет кэш,
   все файлы пересчитываются. Модель экстрактора выбирать ДО старта большого корпуса.
