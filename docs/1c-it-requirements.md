# Техническое задание: read-only доступ к 1С:ERP через OData REST API

**От:** Даниил Елисеев (ПТО)
**Кому:** IT-департамент / специалист по 1С
**Дата:** 2026-05-26
**Версия:** 1.0
**Цель:** Получить read-only доступ к данным 1С:ERP для построения
аналитических отчётов и дашбордов руководству.

---

## 1. Контекст

Для подготовки регулярной отчётности руководству (финансы, проекты,
сотрудники) нужно автоматизированно получать данные из 1С:ERP в
аналитические инструменты (Python pandas, Power BI и т.п.).

Сейчас аналогичные отчёты собираются **вручную** — выгрузка из 1С в
Excel, ручное сведение, отправка руководителю. Это:
- Долго (несколько часов на отчёт раз в неделю).
- Ошибки в копировании.
- Невозможно построить дашборды без живых данных.

Решение — **OData REST API** который встроен в платформу 1С 8.3 и
стандартно используется для интеграций.

---

## 2. Что просим настроить (6 пунктов)

### 2.1 Создать в 1С:ERP технического пользователя

- **Логин:** `claude_readonly` (или любой другой по вашим политикам)
- **Роль:** «Чтение всех данных» (типовая роль платформы) либо
  кастомная роль на чтение без права изменения данных.
- **Пароль:** генерируется вами, передаётся нам защищённо (см. п. 3.1).
- **Запретить интерактивный вход** (если возможно) — пользователь
  только для веб-сервиса.

### 2.2 Опубликовать веб-сервер 1С (если ещё не опубликован)

Стандартная процедура из Конфигуратора 1С:

1. Конфигуратор → **Администрирование → Публикация на веб-сервере...**
2. Веб-сервер: Apache 2.4 или IIS (по вашему выбору, у IT обычно уже
   стоит один из них для основной публикации 1С-клиента).
3. Каталог публикации — любой стандартный (например `D:\1c_publish\erp\`).
4. Имя публикации — например `erp` → URL `https://<server>/erp/`.

Если веб-сервер уже опубликован для веб-клиента 1С — отдельная
публикация не нужна, можно использовать ту же.

### 2.3 Включить стандартный OData REST API

В том же диалоге публикации (или в `default.vrd` файле публикации):

- **Поставить галочку «Опубликовать стандартный OData-интерфейс»** /
  `<standardOdata enable="true" .../>` в default.vrd.
- Перезапустить веб-сервер (Apache restart / IIS recycle app pool).
- Проверить что работает: открыть в браузере
  `https://<server>/erp/odata/standard.odata/$metadata` — должен
  вернуть большой XML со списком объектов конфигурации.

### 2.4 Настроить HTTPS

**Категорически не HTTP** — credentials Basic Auth идут в открытую.

Если у вас уже HTTPS на веб-сервере 1С — этот пункт пропускается.
Иначе:
- Или сертификат от внутреннего CA вашей фирмы.
- Или self-signed (тогда мы получим от вас корневой cert для импорта
  в trusted store на ПК пользователей; либо договариваемся что
  `verify=False` приемлемо для внутреннего трафика).

### 2.5 Сетевой доступ

- Адрес 1С-сервера должен быть **достижим с ПК пользователей**
  (через корп-сеть / VPN / прямой).
- Если у вас есть фаервол с whitelist по IP — нам нужно whitelist'нуть
  IP пользователей (9 ПК команды).
- Если 1С во внутренней сети и работает через корп-прокси — нужен
  **bypass прокси** для адреса 1С-сервера (мы такое уже делаем для
  GitHub — настройка тривиальная).

### 2.6 Передать нам параметры подключения

После настройки выслать защищённо (например через корпоративный
Bitwarden / 1Password / запароленный архив):

```yaml
# Параметры подключения к 1С:ERP
base_url:        https://<server>/<publish-name>/
odata_url:       https://<server>/<publish-name>/odata/standard.odata/
user_login:      claude_readonly
user_password:   <пароль>
platform_version: 8.3.X.X        # версия платформы 1С
configuration:   1С:ERP 2.5.X.X  # версия и редакция конфигурации
cert_type:       trusted | self_signed
notes:           # любые особенности (например VPN required, IP whitelist done, и т.п.)
```

## 3. Что мы гарантируем со своей стороны

### 3.1 Безопасность credentials

- Пароль `claude_readonly` хранится через **Windows DPAPI** (CurrentUser scope).
  Расшифровать может **только тот же Windows-пользователь на той же
  машине**. Защищает от случайной утечки конфига.
- Не пересылается через Telegram / WhatsApp / открытую почту. Только
  через корпоративный vault или запароленный архив.
- Не попадает в наш git-репозиторий (исключение `.1c-config.json` в
  `.gitignore`).

### 3.2 Только GET-запросы

Гарантируем что **никаких** POST / PATCH / DELETE запросов к 1С от
нас не будет. Только GET. Это **двойной слой защиты**:
- На стороне 1С — роль «Чтение всех данных» физически не позволит
  изменять.
- С нашей стороны — в коде нет POST/PATCH/DELETE.

### 3.3 Нагрузка на 1С

- Запросы выполняются **по расписанию** (например раз в час или раз в
  день), не в реальном времени.
- Запросы делаются с **`$select`** (только нужные поля) и **`$filter`**
  (только новые / изменённые записи) — не «выгрузить всё подряд».
- Готовы добавить **rate-limiting** на нашей стороне (например не более
  N запросов в минуту) если попросите.
- Не выполняем тяжёлых join'ов / cross-objects запросов которые могут
  заблокировать 1С — только простые точечные.

### 3.4 Логирование

Каждый запрос к 1С логируется на нашей стороне (`~/.claude/auto-sync.log`):
дата, время, endpoint, status-код, размер ответа. Можем поделиться
логом если потребуется аудит.

### 3.5 Минимальные права

Запрашиваем доступ только к тем объектам конфигурации которые нужны
для конкретного отчёта. Если есть **чувствительные** объекты к
которым лучше не давать доступ — можно ограничить кастомной ролью
(не «Чтение всех данных», а узкая «Чтение финансовых документов»).

## 4. Что мы будем делать с данными

Конкретные дашборды для руководства:

| Группа | Объекты конфигурации (предположительно) | Частота refresh |
|--------|----------------------------------------|-----------------|
| **Финансы** — выручка / прибыль / ДЗ к КЗ / платежи | Document_РеализацияТоваровУслуг, Document_ПоступлениеНаРасчётныйСчёт, AccumulationRegister_Взаиморасчёты* | Раз в день |
| **Проекты** — статусы договоров, сроки, план vs факт | Catalog_Договоры, Document_АктВыполненныхРабот, регистры по проектам | Раз в день |
| **Сотрудники** — зарплаты, загрузка, KPI | Catalog_Сотрудники, регистры расчёта зарплаты, регистры рабочего времени | Раз в неделю |

Точные имена объектов мы определим **после получения URL** через запрос
к `$metadata` (это стандартный OData endpoint который возвращает
полный список доступных EntitySet'ов с типами полей).

## 5. FAQ — типовые возражения

### Q: Это безопасно? Не повредит ли 1С?

**A:** OData REST API — это **штатный встроенный интерфейс платформы
1С 8.3**, разработанный самой фирмой 1С специально для интеграций.
Используется тысячами компаний. Read-only пользователь физически не
может изменить данные. GET-запросы не блокируют 1С (это легче
обычного отчёта который запускает любой бухгалтер из клиента).

### Q: Это не нагрузит сервер 1С?

**A:** Запросы делаются с фильтром по дате (только изменения с
последнего раза), не «выгрузка всей базы». Один запрос — десятки или
сотни записей, не миллионы. Можем добавить rate-limiting на нашей
стороне.

### Q: Что если завтра уволится Даниил?

**A:** Credentials хранятся через DPAPI **per-user**. Если хотите
снизить bus factor — можем дополнительно положить credentials в
корпоративный vault (Bitwarden) с доступом для 2-3 человек.

### Q: А если 1С обновится — всё сломается?

**A:** OData REST API — **стабильный** интерфейс платформы (часть
ядра 8.3). Обновления конфигурации 1С:ERP внутри версии 2.5.X →
2.5.Y или 2.5 → 2.6 могут поменять структуру **конкретных
документов** (добавились/удалились реквизиты), но **сам OData
интерфейс** работает идентично. Мы пишем парсер защитно (если поле
отсутствует — пропускаем, не падаем).

### Q: Можно проверить что мы хотим, на тестовой базе?

**A:** Да, если есть тестовая копия 1С:ERP — настройте OData на ней,
дайте нам ту же связку URL+credentials. Мы проверим что наши запросы
работают, потом перенесём на боевую.

### Q: Что если OData окажется недоступен / не поддерживается?

**A:** Есть два fallback варианта (обсудим если потребуется):
- **HTTP-сервисы** — 1С-программист пишет кастомные endpoint'ы под
  наши задачи. Дороже по работе 1С-иста, но более производительно.
- **Excel-выгрузки** — Даниил (или ответственный) раз в день/неделю
  выгружает стандартные отчёты 1С в Excel в общую папку, Claude
  парсит файлы. Не требует ничего от IT, но ручной.

### Q: Нужно ли подписывать NDA?

**A:** Если фирма требует — да, обсудим. Данные не покидают нашу
команду и не передаются третьим лицам.

## 8. Приложение — пример OData запроса (для справки)

Чтобы понять что мы будем делать:

```bash
# Список всех документов реализации за 2026 год, проведённые,
# только нужные поля, JSON:
curl -u "claude_readonly:<pwd>" \
  "https://<server>/<publish>/odata/standard.odata/Document_РеализацияТоваровУслуг?\
\$filter=Date ge datetime'2026-01-01T00:00:00' and Posted eq true&\
\$select=Number,Date,Контрагент,СуммаДокумента&\
\$format=json"
```

Ответ:
```json
{
  "odata.metadata": "...",
  "value": [
    {"Number": "000123", "Date": "2026-01-15T...", "Контрагент": "...", "СуммаДокумента": 150000.00},
    ...
  ]
}
```

Это и есть стандартный механизм. Никакой магии — простой HTTP + JSON.
