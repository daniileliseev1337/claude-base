# Вводная для ветки «токен-экономия» (2026-06-10, dev-сессия хаба)

Факт: сессия хаба за день — **100.2M токенов** (in 395k / out 367k / **cache r 90.2M** /
cache w 9.3M). 5-часовой лимит MAX x5 выбит дважды. Так работать нельзя — нужна
системная починка. Этот файл — вводная для облачной/новой сессии, чтобы не копать заново.

## Диагноз (по составу расхода)

99% расхода — **cache reads**: каждый tool call агентного цикла прогоняет весь
накопленный префикс (системный промпт + история сессии) через кэш. Cache-токены
дешевле обычных, но в лимит подписки считаются. Три множителя дня:

1. **Мега-сессия**: 6 разных задач в одном чате (мерж ветки → построение графа базы →
   верификация PowerShell → плагины/маркетплейс → телеграм-бот → harvest 24 репо).
   Сотни tool calls; к вечеру каждый шаг тащит хвост ВСЕХ прошлых задач.
   Скилл `handoff-to-new-chat` существует и его триггеры (30+ turns, 3+ агентов,
   5+ больших выводов) сработали давно — но Claude его не предложил. Гейта нет.
2. **Системный промпт базы разжирел**: description 16 агентов (по 300-700 токенов
   живых фраз) + ~50 скиллов с длинными триггер-секциями + MCP-инструкции грузятся
   в каждый запрос КАЖДОЙ сессии КАЖДОГО сотрудника. Оценить точно и сжать.
   Дилемма: аудит 2026-06-09 показал, что роутинг живёт именно на description —
   резать нельзя вслепую, нужен бюджет «токены description ↔ качество роутинга».
3. **Разовое**: 11 экстракционных субагентов унаследовали Fable 5 (~11M за 40 мин).
   Уже закрыто правилом (memory: fan-out → явный model haiku/sonnet, батчи ≤6,
   озвучка стоимости). Дострой графа теми же объёмами на haiku стоил ~1.4M.

## Кандидаты решения (для ветки, по убыванию эффекта)

1. **Дисциплина сессий = жёсткий гейт, не привычка.** Одна задача — одна сессия.
   Проактивный handoff: вшить в CLAUDE.md видимую строку-гейт (как роутинг-гейт):
   при смене темы / N turns — Claude ОБЯЗАН предложить новый чат + handoff-пакет.
   Возможно hook-счётчик (PostToolUse уже пишет tool-usage.jsonl — можно считать
   вызовы за сессию и напоминать).
2. **Диета системного промпта**: замерить реальный префикс (description агентов,
   скиллы, MCP-инструкции); сжать description до триггеров (детали — в тело агента,
   оно грузится только при спавне); неиспользуемые скиллы → блоки/плагины
   (механика blocks/ уже есть). Цель: −30-50% префикса. Телеметрия покажет,
   какие агенты/скиллы молчат (aggregate-tool-usage.ps1 уже собирает).
3. **Выключить лишние плагины у всех**: каждый плагин добавляет свои скиллы в
   промпт каждой сессии (у нас сегодня висел twilio с ~70 скиллами). Ревизия
   enabledPlugins на всех ПК; кандидат в /sync-base — инвентарь плагинов.
4. **Субагенты дешёвых моделей по умолчанию** — уже правило; добавить в
   CLAUDE.md/Workflow-секцию явно (сейчас только в личной памяти хаба).
5. **Замерить и наблюдать**: плагин `session-report` (официальный маркетплейс) —
   HTML-разбор сессии: токены, cache efficiency, самые дорогие промпты, субагенты.
   Поставить на хаб, гонять после рабочих сессий.
6. **context-ledger** (github.com/wiztek-llc/context-ledger, из harvest 2026-06-10) —
   обещает 28× до auto-compact: селективная загрузка контекста. Изучить в ветке.
7. Хвост: большие tool-выводы (каталоги, diff'ы) — резать head_limit'ами,
   persisted-output уже помогает; не читать большие файлы целиком без нужды.

## Что НЕ является проблемой
- Граф базы: query дёшев, построение разовое (~1.4M haiku) — не источник расхода.
- weekly-лимит всего 39% — горит именно 5-часовое окно, т.е. пиковая интенсивность,
  не суммарный объём. Дробление по сессиям/паузам решает большую часть.
