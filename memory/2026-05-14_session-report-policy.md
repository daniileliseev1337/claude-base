# Кейс: каждая сессия → session-report + auto-sync статус в начале/конце

**Дата:** 2026-05-14
**Контекст:** Пользователь решил поменять политику session-reports.
Раньше они писались **по запросу** («зафиксируй что мы научились»),
теперь — **автоматически каждую сессию**. Параллельно — снижена
планка обезличивания, потому что репо `claude-base` переключён в
**private**.

## Главное

1. Репо `claude-base` → **private** на 2026-05-14.
2. Каждая сессия Claude **обязана** написать session-report перед
   закрытием.
3. В начале сессии — строка про статус auto-pull (что подтянулось).
4. В конце сессии — прогноз auto-push (что пушнётся).
5. Сырые отчёты — без обезличивания (private-доступ). Обработка в
   обезличенную методику (memory/, skills/) — потом ручная.

## Что изменилось в инфраструктуре

### Было

- **Public репо.** Жёсткое обезличивание для всего что пушится.
- Session-reports писались **по запросу** пользователя (триггерные
  фразы: «запиши в inbox», «зафиксируй»).
- Большинство сессий **не оставляли следа** в `claude-base`.

### Стало

- **Private репо.** Доступ только внутреннего круга.
- Session-reports пишутся **каждую сессию** обязательно.
- Сырые отчёты с конкретикой (имена, шифры, бренды) — разрешены.
- НЕ записываются: пароли, API-ключи, ПДн (паспорта, СНИЛС),
  банковские реквизиты — это `[СЕКРЕТ — не записан]`.
- Пользователь периодически обрабатывает накопленные session-reports
  и выносит ценное в `memory/`, `skills/`, `agents/` (методика).

## Решения по дизайну

### 1. «Каждая сессия» vs «по запросу»

**Цитата пользователя:**
> «у нас была задача чтобы отчёт из каждого чата загружался на GitHub,
> не выборочно нужно снизить планку того что нужно и можно загужать»

**Принятое решение:** Claude **обязан** написать session-report в
конце каждой сессии. Не «по запросу».

**Почему это работает теперь:**
- Private репо → конкретика не утекает наружу.
- «Сырой» материал ценнее «отшлифованного» — потом обработать проще
  чем восстановить контекст.
- Пользователь видит **что Claude реально делал**, не только итог.

### 2. Снижение планки обезличивания

**Цитата пользователя:**
> «снизить планку того что нужно и можно загужать»

**Принятое решение (только потому что private):**
- Имена hosts, бренды, шифры, имена сотрудников нашей команды —
  можно писать.
- Категорически нельзя: пароли, API-ключи (включая GitHub PAT),
  ПДн (паспорт/СНИЛС/банк-реквизиты).
- Имена клиентов/заказчиков (third parties) — стараться обезличивать
  или ставить плейсхолдер `<заказчик>`, потому что **их** данные
  чувствительны независимо от приватности нашего репо.

### 3. Auto-sync статус — в начале и в конце сессии

**Цитата пользователя:**
> «нужно писать в начале и конце каждой сессии что auto sync выолнен
> и мини отчёт что подгрузилось, что выгрузилось»

**Принятое решение:**
- **Начало:** одна строка `Auto-pull: ...` после `✓ прочитан CLAUDE.md`
  — что подтянулось из других ПК (читаю auto-sync.log tail или
  git log).
- **Конец:** прогноз `Auto-push на SessionEnd: ожидается N коммитов
  (managed paths: ...)` или «push не будет (no managed changes)».
  Реальный результат — в **следующей** сессии при старте.

**Что Claude НЕ делает:**
- Не запускает `auto-push.ps1` руками в финале — это работа hook'а.
- Не дублирует commit/push — это нарушение разделения
  ответственности.

### 4. Идемпотентность session-reports

Если за один день+тема уже есть `session-reports/<дата>_<тема>/` —
обновлять `report.md`, не плодить новые папки. Тема может
эволюционировать в течение дня.

### 5. Шаблон отчёта

`~/.claude/session-reports/_TEMPLATE.md` — обязательная структура:
- Запрос пользователя
- Хронология (3-10 пунктов)
- Источники (MCP / скиллы / slash / нормативы / harvest)
- Артефакты для пользователя
- Итерации и ошибки (самый ценный раздел — без приукрашивания)
- Что выдумывал / placeholder
- Цитаты пользователя
- Открытые вопросы
- Auto-sync (начало + конец)

## Уроки (для будущих волн)

1. **Public → private** меняет правила игры. Решение про
   обезличивание зависит от приватности репо. Снимать ограничения
   когда public — нельзя; снимать когда private — нормально.

2. **Сырые материалы важнее отшлифованных** — для обучения и debrief.
   Потерянный контекст не восстановить. Тщательная обработка может
   быть позже (а может и не нужна).

3. **Auto-sync — это инфраструктура, не действие Claude.** Я не
   запускаю push в конце сессии — это делает hook. Я только
   фиксирую **прогноз** и в следующей сессии вижу **результат**.

4. **«Обязательное» правило с триггером** работает надёжнее
   «опционального по запросу». Раньше session-report писался редко,
   потому что триггер был субъективный (Claude должен был «уловить»
   что пользователь хочет debrief). Теперь триггер объективный —
   каждая сессия.

## Связанные

- `~/.claude/CLAUDE.md` USER EXTENSIONS, раздел «Sessions —
  обязательный отчёт каждой сессии».
- `~/.claude/session-reports/_TEMPLATE.md` — шаблон.
- `~/.claude/scripts/auto-push.ps1` — hook, который автоматически
  пушит session-reports.
- `~/.claude/auto-sync.log` — источник для статусов начала/конца.

## Открытое для будущих волн

- **Когда переключать обезличивание обратно** (если репо снова станет
  public когда-то)? Сейчас private — норма.
- **Когда передавать сотрудникам право писать session-report** vs
  они только pull'ят и читают? Сейчас сотрудники-end-users пишут
  свои отчёты, и эти отчёты прилетают мне — хорошо.
- **Автоматизация обработки сырых отчётов в методику** —
  потенциальная следующая волна: скрипт сканирует session-reports/,
  предлагает что вынести в memory/ или обновить в skills/.
