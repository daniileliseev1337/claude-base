## Экономия токенов — практические правила

Эта секция собрана 2026-05-26 после сессии с реальным замером (~3-5M
tokens за день на Opus 4.7 1M context). Проверенные факты, **без**
гипотез про prompt-caching в markdown (это миф — caching работает на
уровне API/SDK, не контролируется пользователем Claude Code Desktop).

### 1. Выбор модели — главный рычаг

В Claude Code Desktop / CLI у пользователя **отдельные квоты** на Opus
и Sonnet (видно в UI: `5-hour limit` для Opus, `Sonnet only` для Sonnet).

**Переключение модели в текущей сессии:** команда `/model sonnet` или
`/model opus` (доступна и в Desktop, и в CLI).

**Когда какая модель:**

| Задача | Модель | Причина |
|---|---|---|
| Чтение/правка одного-двух файлов, find-replace, переименования | **Sonnet** | Достаточно — экономим Opus квоту |
| Word/Excel шаблонные документы по skills | **Sonnet** | Skills делают тяжёлую работу, агент только оркестрирует |
| OCR scan, парсинг УПД (через upd-parser) | **Sonnet** | Логика в skills, модель управляет шагами |
| Запрос к норм-базе через `norm-lookup` | **Sonnet** | Read-only поиск, не reasoning |
| Раскрутка спорного нормоконтроля, ответ экспертизе, проектирование | **Opus** | Reasoning важен |
| Длинная исследовательская сессия с 5+ Agent calls | **Opus** | Subagent качество критично |
| Любая правка кода с нетривиальной логикой | **Opus** | Точность важнее скорости |

**Правило по умолчанию:** **Sonnet** если задача не требует
доказанно Opus-уровня reasoning. Не доказано — берём Sonnet.

### 2. Команды управления контекстом

| Команда | Что делает | Когда применять |
|---|---|---|
| `/compact` | Сжимает накопленный контекст (lossy). | При **50%+** context window UI индикатор. Перед длинной задачей. |
| `/clear` | Полный reset сессии — теряем всё. | Только при pivot на совершенно другую тему. |
| `/model <sonnet\|opus>` | Переключение модели. | При смене типа задачи (см. таблицу выше). |
| `/cost` (если доступна) | Показывает расход текущей сессии. | Sanity-check время от времени. |

### 3. Дисциплина расхода (поведенческие правила)

**Не лить дорогие операции:**
- **Agent (subagent) calls стоят в 3-5х дороже** обычной реплики —
  каждый запускает отдельную сессию с полным контекстом. Использовать
  только когда:
  - Нужна **независимая** проверка (auditor для финального артефакта).
  - Нужно **многошаговое исследование** которое раздует основной контекст.
  - **НЕ** для тривиальной задачи которую решает основной Claude.
- **Один Agent call вместо трёх** — собирать пачку файлов/задач в
  одну инструкцию субагенту, не вызывать его 3-5 раз с похожими TZ.
- **Не делать retry на 529 Overloaded** в loop — это **дублирует**
  стоимость. Лучше задержать сессию или сделать handoff.

**Cascade loading вместо whole-file Read:**
- Сначала `Grep` на ключевые секции файла.
- Потом `Read offset=N limit=50` только нужной части.
- **НЕ** `Read` 400-строчного файла целиком чтобы найти 10 строк.

**Tool outputs через фильтры:**
```powershell
# Плохо: вернёт 30K строк
pip install paddleocr

# Хорошо: только результат
pip install paddleocr 2>&1 | Select-Object -Last 5
```

**Background для длинных команд:**
- `run_in_background=true` для Bash/Agent если результат нужен потом.
- Не блокирует контекст stdout'ом в реальном времени.

### 4. Мониторинг — что смотреть в UI

Перед длинной задачей **проверить индикатор в правом нижнем углу**
(панель Plan usage):

| Показатель | Что значит | Действие |
|---|---|---|
| `Context window X / 1.0M (Y%)` | Текущий контекст в сессии | При Y > 50% в†’ `/compact` или handoff |
| `5-hour limit X%` | Opus квота за 5 часов | При X > 80% в†’ переключиться на Sonnet или handoff в новой сессии через 5h |
| `Weekly В· all models X%` | Total за неделю (Opus+Sonnet) | При X > 70% в†’ беречь до конца недели |
| `Sonnet only X%` | Отдельная квота Sonnet | Если 0% в†’ есть запас для рутины |

**Правило:** **сначала смотрим UI, потом действуем.** Не оценивать «на
глаз» — у меня (Claude) нет видимости в эти показатели, я их сам не
вижу. Только пользователь видит UI.

### 5. Что НЕ работает (антипаттерны)

- вќЊ **Prompt Caching через markdown-комментарии** в CLAUDE.md
  (`<!-- cache_control -->`) — **не существует**. cache_control это
  API-level параметр в request body, недоступен пользователю Claude
  Code Desktop.
- вќЊ **Условный agent dispatcher** в Claude Code (типа «если задача
  X в†’ Sonnet, иначе Opus») — Claude Code не имеет user-level routing.
  Модель управляется командой `/model` для всей сессии.
- вќЊ **Retry в loop при 529 Overloaded** — удваивает стоимость без
  results. Лучше ждать или сделать handoff.
- вќЊ **Несколько Agent calls подряд с похожим TZ** — слить в один.

### 6. Конкретные источники расхода (из retro 2026-05-26)

Что обошлось дорого в реальной сессии:
- 7 вызовов `auditor` с retries и revisions — можно было 2-3 вызова
  с пачкой.
- 11 целых `Read` agent.md по 300-400 строк — можно было cascade.
- 2 `Agent(Explore)` по GitHub — один развёрнутый `WebFetch` дешевле.
- 30+ turns без handoff — надо было разбить на 2-3 сессии.
- **`Extra high` reasoning mode** включён на Opus — это самый дорогой
  режим. Использовать только когда reasoning критичен.

### 7. Модели субагентов и сессионный гейт (обновлено 2026-07-01 — выход Sonnet 5)

**Субагенты: дефолт `claude-sonnet-5`, оркестрировать активно.** До 2026-07-01 правило было
«выбирает пользователь, планка вверх»: дешёвая модель галлюцинировала → каскад правок дороже
(инцидент 2026-06-22), а fan-out на дорогой модели жёг лимит (десяток дорогих субагентов = 5ч-лимит
за 40 мин). **Sonnet 5 (2026-06-30: near-Opus, <½ цены, ниже галлюцинаций/подхалимажа) сбил обе
посылки** → дефолт сместился в сторону параллелизма.

- **Дефолт субагентов = `claude-sonnet-5`** (всё содержательное + ВСЕ ревьюеры: auditor /
  word-checker / excel-validator / pdf-reviewer / audit-rd-section / rd-coordinator / norm-lookup /
  синтез / аудит). Спавнить БЕЗ поштучного `AskUserQuestion`, видимой строкой `Субагенты: claude-sonnet-5 ×N (<тип>)`.
- `haiku` — ТОЛЬКО тупая экстракция / OCR / парсинг / инвентаризация. `opus` (модель сессии) —
  крупные ставки (деньги / экспертиза / сдача / критичные нормы). `fable` — доступен снова (Fable 5, редеплой 2026-06-30): главная модель сессии по выбору владельца, субагентам НЕ назначать (лимиты/цена).
- **Спрашивать `AskUserQuestion` ТОЛЬКО** когда нужен `opus` (крупная ставка) ИЛИ массовый fan-out
  (≈8+ агентов разом — заметный расход 5ч-лимита, предупредить про токены). В вопрос: рекомендация +
  чем грозит модель здесь + цена ошибки + оценка токенов.
- **Оркестрируй активно (главная мощь Opus):** дроби задачу на независимые ветки → параллельный
  fan-out (несколько Task в turn / `parallel` / Workflow), не звать одного и ждать. Зависимости → конвейер;
  барьерная сборка — только когда нужен весь результат разом. Ревьюер-гейт остаётся.
- Автономный режим (cron/loop/headless) → `claude-sonnet-5` без вопроса. Учитывать: токенизатор
  Sonnet 5 даёт ≈ +30% токенов на тот же текст (закладывать в оценку лимитов).

**Сессионный гейт.** Одна задача — одна сессия. Признаки: смена темы ИЛИ длинный
хвост (30+ ходов / 3+ субагентов / 5+ больших выводов). Длину ловит телеметрия-хук
(`log-tool-usage.ps1`, порог 75 вызовов → additionalContext с напоминанием). Смену
темы хук не видит (семантика) — предложить handoff самому одной строкой:
`Гейт сессии: <смена темы | длинная> — рекомендую новый чат (handoff-to-new-chat)`.
Один раз на задачу, решение за пользователем.
