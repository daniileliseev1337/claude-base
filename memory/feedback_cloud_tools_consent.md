# Cloud-инструменты: consent-prompt вместо жёсткого запрета

**Правило:** для инструментов которые отправляют данные на внешние серверы
(Codex/OpenAI, Exa, Firecrawl cloud, Higgsfield) — предлагать **consent-prompt
в моменте** (агент спрашивает разрешение перед отправкой), а НЕ blanket-запрет
обезличивания который блокирует работу.

**Why:** Пользователь (руководитель <организация>, 2026-06-01) предпочитает гибкость.
Готов осознанно принять privacy-риск. Жёсткие правила «запрещено давать
документы с ФИО/шифрами» тормозят работу. Лучше — информированное согласие
пользователя в конкретный момент.

**How to apply:**
1. При интеграции cloud-инструмента — НЕ предлагать жёсткий запрет первым.
   Предлагать consent-механизм: агент перед отправкой спрашивает.
2. **Consent должен быть ИНФОРМИРОВАННЫМ** (это условие пользователя — он
   одобрил честную формулировку, не «не страшно»):
   - честно сказать что данные уходят на внешние серверы (не локально);
   - предупредить если free tier может использовать данные для обучения;
   - флагнуть если в тексте ПДн третьих лиц (ФИО/реквизиты заказчиков, 152-ФЗ);
   - дать опции [Да / Нет / Обезличить сначала].
3. Решение принять/отклонить риск — за пользователем. Моя роль —
   дать точную картину, не блокировать и не подхалимничать («не страшно»).

**Граница:** это не отменяет правило обезличивания для того что **пушится
в claude-base** (agents/skills/memory/session-reports) — там обезличивание
обязательно (CLAUDE.md правило 6). Consent-подход касается **отправки в
сторонние AI-сервисы по запросу пользователя**, не публикации в репозиторий.
