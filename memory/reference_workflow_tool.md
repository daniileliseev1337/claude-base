# Workflow (Opus 4.8) — мультиагентная оркестрация

Загружать по триггерам: «workflow», «оркестрация агентов», «фан-аут», «ultracode»,
«запусти агентов параллельно», «ревью комплекта», «batch обработка N файлов».
Правило проактивного предложения — в CLAUDE.md (USER EXTENSIONS).

## Что это

Детерминированный JS-скрипт, который запускает **много субагентов по программе**
(не модель решает порядок на лету, а скрипт: циклы, условия, фан-аут). Выполняется
**в фоне**, уведомляет по завершении. Для задач на масштаб / исчерпаемость / уверенность.

Отличие от обычного `Agent`: `Agent` — один субагент ad-hoc; `Workflow` — десятки
по конвейеру/параллельно с детерминированным control-flow.

## Когда оправдан (vs обычный заход)

ДА: 3+ независимых подзадач; ревью/аудит комплекта; broad research/harvest;
batch-преобразование N однотипных единиц.
НЕТ: простая/одношаговая/диалоговая задача; один файл; быстрый факт-лукап.

⚠ **Дорого по токенам** (десятки агентов) → только с явного согласия пользователя
(«ultracode» / «запусти workflow»). Самому не запускать.

## Примитивы скрипта (кратко)

- `agent(prompt, opts)` — субагент; с `schema` возвращает валидированный объект.
- `parallel(thunks)` — пачкой одновременно (барьер: ждёт всех).
- `pipeline(items, stage1, stage2, ...)` — конвейер: каждый элемент проходит все стадии
  независимо, без барьера между стадиями (default для multi-stage).
- `phase(title)` — фазы для прогресс-дисплея. `log(msg)` — нарратор.
- Старт скрипта: `export const meta = {name, description, phases:[...]}` (чистый литерал).

Шаблоны: **understand** (параллельные читатели по подсистемам), **review** (измерения →
adversarial verify), **research** (мульти-поиск → синтез), **migrate** (найти места →
преобразовать каждое), **judge-panel** (N попыток → оценка → синтез).

## Разовый vs named

- **Разовый** — инлайн, живёт в рамках запуска. Готовности «у нас» не требует.
- **Named (переиспользуемый)** — файл `~/.claude/workflows/<имя>.js`, зовётся по имени.
  **У нас пока НЕТ ни одного** (папка не заведена). Заводить — по инкременту: когда
  оркестрационный сценарий повторился 2-3 раза и отлажен (как со скиллами).

## Кандидаты в named-workflow (наша практика)

- `base-audit` — аудит claude-base фан-аутом (агенты/скиллы/memory/mojibake/индексы). Уже
  доказал пользу 2026-06-04 (делалось руками одним агентом).
- `review-rd-komplekt` — ревью комплекта РД по разделам: parallel(excel-validator +
  word-checker + norm-lookup) на каждый раздел. Кейс — серия ПНР+ВОР Сит-Центр (15 разделов).
- `batch-akty-reformat` — pipeline переформата N актов (см. [[reference_docx_table_reformat]]).
- `feedback-harvest` — параллельное чтение пачки feedback-отчётов → сводка кандидатов
  (делалось руками в этой сессии несколько раз).

## Связь

- [[chains-pattern]] — named chains (ручные последовательности); Workflow — программная
  оркестрация субагентов (мощнее, но дороже).
- [[structured-artifacts]] — выносить контекст; Workflow часто пишет результаты в артефакты.
- depth-лимит делегации 2 действует и тут: оркестратор → агент-этап → ревьюер.

## Первый боевой запуск (2026-06-04)

`autocad-mcp-harvest` — fan-out по 11 AutoCAD-инструментам (scout свежего + 6 deep-read +
синтез). **8 агентов, ~766K субагент-токенов, ~4.5 мин.** Дал то, что один агент не вытянул бы:
глубокий разбор 11 репо + находка `prumputira` (не было в harvest 15.05). Дорого, но оправдано
на broad-research. Кандидат в named-workflow `harvest-tools` (scout + deep-read + synth — паттерн
переиспользуемый). См. `session-reports/2026-06-04_autocad-mcp-harvest/report.md`.
