# Лицензионные договорённости <организация>

> Reference info на будущее. Источник: пользователь, 2026-06-01.
> Категория: правовые рамки использования open-source инструментов.

## Правило
**AGPL-3.0 и GPL не являются блокером** для harvest-workflow на этой базе.
У <организация> есть договорённости с правообладателями ряда AGPL-инструментов,
позволяющие использование внутри корпоративного контура.

## Условие
**Не выпускать за рамки компании <организация> / <смежная-организация> / смежных компаний.**

На практике это означает:
- ✅ Можно ставить AGPL-инструмент на ПК сотрудников <организация> и смежных
- ✅ Можно интегрировать в `claude-base` (репо остаётся приватным)
- ✅ Можно использовать в проектной работе для клиентов <организация>
- ❌ **Нельзя** публиковать `claude-base` в открытом доступе на GitHub
- ❌ **Нельзя** делиться кодом/плагинами с внешними организациями
- ❌ **Нельзя** публиковать на Claude Code marketplace

## Как применять при harvest
1. Если репо имеет лицензию AGPL/GPL — **не блокировать сразу**.
2. Проверить: входит ли в список зонтика <организация> (этот файл).
3. Если входит — использовать без ограничений в рамках условия.
4. Если не входит — задать вопрос пользователю: «У <организация> есть договорённость
   с правообладателем X? Если нет — пропускаем по лицензии.»

## Override CLAUDE.md
Этот файл переопределяет правило из CORE-секции harvest-workflow:
> «GPL — флаг для согласования; виральность лицензии портит installer»

Уточнение: виральность важна **только при публикации наружу**. Внутри
закрытого корпоративного контура — не действует, если есть договорённость
или прямое разрешение правообладателя.

## Anti-pattern
- ❌ Не использовать как обоснование для копирования чужого кода в `agents/`
  или `skills/` без явного разрешения пользователя. Условие зонтика —
  это про использование инструмента, не про порчу авторства.
- ❌ Не расширять список «смежных компаний» сам. Уточнять у пользователя
  если возникает сомнение.
