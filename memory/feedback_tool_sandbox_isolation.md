# Инструменты Claude работают в изолированном окружении (per-machine)

**Факт (подтверждён 2026-06-01):** мои `Bash` / `PowerShell` tool-команды для
**per-machine установок** (`npm install -g`, `pip install`, изменения PATH,
глобальные пакеты) выполняются в **изолированном окружении инструмента**, а
НЕ на реальном ПК пользователя.

## Кейс
Я запустил `npm install -g @openai/codex` через Bash → `codex --version`
вернул `0.135.0` → я заявил «Codex CLI установлен». Но в РЕАЛЬНОМ PowerShell
пользователя `codex` не найден, и даже папка `%APPDATA%\npm` «не существует».
Мой `Test-Path` видел её (True), пользователь — нет. Разные AppData.
Пользователь установил `npm install -g @openai/codex` сам → заработало.

## Что SHARED (доезжает до пользователя)
- `~/.claude/**` — всё через git (skills, memory, commands, chains, agents)
- `.claude.json` — MCP config (поэтому `claude mcp add exa` сработал у юзера)
- Рабочая папка проекта (файлы, git push)

## Что ИЗОЛИРОВАНО (sandbox-локальное, НЕ доезжает)
- `%APPDATA%\npm` — глобальные npm-пакеты (`npm install -g`)
- pip user packages, системный PATH
- Любая per-machine установка вне `~/.claude`

## Как применять (правило для будущих сессий)
1. **Per-machine инструмент** (CLI через npm/pip, системный пакет) → НЕ
   заявлять «установил» после tool-команды. Дать пользователю команды для
   запуска в ЕГО терминале.
2. **Проверять реальность установки** через признаки доступности у
   пользователя, не через свой tool (`codex --version` в моём Bash ≠ доказательство).
3. **Конфиг через `~/.claude` / `.claude.json` / git** — это реально shared,
   тут tool-действия валидны (Exa MCP add сработал).
4. В чеклистах/инструкциях помечать per-machine шаги как «выполнить вручную
   на каждом ПК», не «уже сделано».

## Карпати-привязка
#5 (помощник, не подхалим) — честно признать «установил» было неверным,
когда пользователь показал что не работает. #4 (верификация) — не заявлять
успех установки без подтверждения на реальной целевой системе.
