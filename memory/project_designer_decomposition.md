# Backlog: stage-decomposition агента `designer`

## Контекст

Источник идеи — аудит чужой базы Claude Code «К-7 (агенты)» от 2026-05-20
(см. отчёт аудит К-7 (от 2026-05-20, отчёт архивирован)). У них монолитный
проектировочный агент разделён на стейджи по pattern Никиты:

- **Stage 1** (`s1-*-fetcher` / `s1-*-collect`) — сбор сырья (read I/O);
- **Stage 2** (`s2-*-parser`) — структурирование в JSON;
- **Stage 3** (`s3-*-analyst`) — расчёт без I/O (pure function JSON→JSON);
- **Stage 4** (`s4-*-writer`) — генерация артефактов (write I/O).

Преимущества разделения для них:
- Stage 3 без I/O → детерминированные тесты promptfoo;
- Возможность параллельного запуска нескольких Stage 3 / Stage 4
  через `teammateMode: tmux` + `AGENT_TEAMS`;
- Чётко локализованные ошибки (где упало — там и чинить).

У нас `designer` сейчас монолит — читает источники, считает, рендерит.
По принципу Karpathy §2 (минимум, без спекуляций) — **не разделять «на
будущее»**, пока не появится реальная необходимость.

## Когда триггер срабатывает

Декомпозиция целесообразна, если выполнено **хотя бы одно**:

1. В 3+ сессиях подряд `designer` упирается в одно и то же ограничение
   (например, не успевает обработать большое ТЗ за один call —
   контекст разрывается).
2. Явный запрос пользователя «параллельный расчёт ОВ + ВК + ЭО + СС»
   с сохранением общих исходных данных.
3. Появилась цепочка `chain:project-doc-pack` в active-статусе и
   потребовался Stage 3 без I/O для интеграции с promptfoo goldens.
4. Был внедрён `teammateMode: tmux` (см. R&D backlog) и нужны
   параллелизуемые узкие агенты.

## Предполагаемое разделение

| Stage | Агент | Что делает | I/O |
|---|---|---|---|
| 1 | `s1-stroy-collect` | Собрать ИРД, чертежи, нормативы | read |
| 2 | `s2-stroy-parser` | ТЗ → JSON (нагрузки, площади, требования) | no I/O (input — текст ТЗ) |
| 3 | `s3-stroy-calc` | Расчёт по разделам ОВ/ВК/ЭО/СС | **no I/O** |
| 4 | `s4-stroy-docx-writer` | ПЗ + спецификация (JSON → DOCX) | write |

При активации через `Agent(subagent_type=…)` — каждый с явным
verify-критерием в промпте (см. [[karpathy-guidelines]] §4).

## Связи

- `chain:project-doc-pack` (см. `~/.claude/chains/project-doc-pack.md`) —
  пока stub. При активации backlog'а — станет настоящим
  Stage 1→2→3→4 pipeline.
- `chains-pattern` skill (методология) — описывает 4-стейджевый паттерн
  декомпозиции.

## Анти-паттерны

- **Не создавать** пустые stub-агенты заранее. Только когда триггер сработал.
- **Не лезть** в существующий `designer` чтобы «причесать под будущее
  разделение» — это нарушит Karpathy §3.
- **Не делать** декомпозицию частично (например только Stage 4 без
  остальных) — это создаст несимметричную архитектуру.
