# Кейс: ПНР Вентиляции — последний v1-кейс перед переходом на v2

**Дата:** 2026-05-07
**Архитектура:** v1 (база скиллов + lazy-load + триггерные карты,
last commit `a476b60` на `main`). v2 на момент задачи ещё не
использовалась.
**Тип задачи:** генерация Программы пуско-наладочных работ (ПНР) по
шаблону `.doc` на основе сметы (Excel) и проектной документации (PDF).

> Кейс полностью обезличен по правилу 11. Конкретные шифры,
> наименования объекта, адрес, ФИО подписантов, наименования
> организаций и марки оборудования заменены на плейсхолдеры или общие
> описания.

## Главное

Задача формально выполнена, документ отдан **после 3 итераций**, из
которых **2 — следствия ошибок Claude**, не уточнений пользователя.

## Повторяющиеся провалы v1 (тот же паттерн что предыдущий VRF-кейс)

### 1. Выдумывание там, где нет источника (нарушение правила 1)

- **Поля страницы**, размер заголовка, выравнивание — взял
  ГОСТ-овские по умолчанию **из общих знаний**, не глядя в шаблон.
  В шаблоне эти параметры отличались. **4 ошибки оформления**.
- **Список нормативных документов** — обновил под «актуальные
  редакции» без согласования. Шаблон содержал ссылки на старые
  редакции — Claude по своей инициативе заменил на новые СП без
  спроса. Это нарушение правила 1: должен был спросить или применить
  агента-сверщика.
- **Привязка нескольких систем к помещениям** — для систем, не
  упомянутых в расчёте воздухообмена в текстовом слое PDF, написал
  заглушку «по проекту <шифр>», вместо того чтобы спросить
  пользователя или искать в растровых страницах PDF.

### 2. Subagent-ревью без правильной формулировки = бесполезное

Claude **вызвал двух** субагентов через Task tool (формальное
применение правила 4). Оба проверили **только содержание текста и
сметные позиции** — ни один не сделал визуальную сверку
форматирования. Потому что Claude их об этом **не попросил**.
Результат: 4 ошибки оформления прошли через ревью и были замечены
пользователем сразу.

**Вывод:** subagent-ревью даёт защиту только от тех классов ошибок,
о которых явно сказано. Без явной формулировки «проверь визуальное
оформление поэлементно» — визуальные ошибки не ловятся.

### 3. Согласие без предупреждения о последствиях (нарушение принципа 5)

Пользователь сказал «не делать визуальный рендер растровых страниц
PDF». Claude подчинился, но **не предупредил**, что в результате
характеристики систем (расходы воздуха, давление, мощность) **не
будут извлечены** и в итоговом документе будут отсутствовать. Должен
был сказать: «ок, не делаю, но тогда характеристики систем X будут
отсутствовать — подтверждаешь, или ищем другой источник?».

Это типовая ловушка «не подхалима»: формально пользователь дал
ограничение, формально Claude ему подчинился — но **последствие
не озвучено**, и результат вышел неполным.

### 4. Переусложнение (нарушение принципа 2)

Сгенерировал документ через **416-строчный Python-скрипт** с
парсером шаблона + хелперами `add_p`/`add_heading`. Для одноразового
шаблонного документа это **переусложнение**. Простой подход
«открыть копию шаблона + заменить плейсхолдеры» дал бы тот же
результат за половину строк и **без 4 ошибок форматирования** —
потому что форматирование шаблона сохранилось бы автоматически.

## Что сработало хорошо (для баланса)

- **Чтение исходников** — из Excel-сметы и текстового слоя PDF
  данные взяты корректно и подтверждены источниками для большинства
  позиций (около 85% позиций имеют явный источник в источниках).
- **Хирургические правки** при исправлении ошибок (ГЭСНп) — изменено
  ровно несколько символов, без касания соседнего кода/текста.
- **Самопроверка по принципам Karpathy** — Claude сам признал
  нарушения по принципам 1, 2, 4, 5, без приукрашивания. Это и есть
  результат добавления `block-behavior.md` в Волне 11.

## Уроки для v2

Эти уроки **подтверждены на двух кейсах подряд** (предыдущий
VRF-кейс + ПНР Вентиляции). Можно выносить в правки `designer.md` и
`auditor.md` отдельным шагом:

1. **Для документных задач** (DOCX, PDF, шаблонные) — **визуальная
   сверка обязательна до сдачи**. PNG-рендер первой страницы +
   сверка с эталоном. Keyword-проверки в тексте недостаточны.

2. **Для шаблонных документов** — подход «копия шаблона + replace
   placeholders» **предпочтительнее** генерации с нуля. Это явное
   применение принципа 2 (Simplicity).

3. **Список нормативных документов, формулировки разделов,
   поля/шрифты/выравнивание** — **никогда не выдумывать** «по логике
   актуальности». Только из источника или с явного согласия
   пользователя.

4. **При согласии на ограничение области** (например «не делать X»)
   — Claude обязан **предупредить о последствиях**, если они
   неочевидны для пользователя. Подчиняться молча запрещено.

5. **Auditor для документных задач** — должен по умолчанию проверять
   **визуальный класс** ошибок (поля, шрифты, выравнивание,
   отступы), не только содержательный.

6. **Перед стартом доменной работы по шаблону** — отрендерить
   эталон в PNG, показать пользователю, явно зафиксировать
   визуальный эталон до начала генерации.

## Дисциплина

Эти уроки — **debrief-driven**. Применять при следующей правке
агентов (отдельная задача). Не плодить новых правил наперёд.

## Тест архитектуры v2

Если в v2-сессии **в новой папке `claude-stroy-v2`** при аудите того
же документа `auditor` найдёт визуальные ошибки оформления (которые
исходные субагенты v1 пропустили) — это эмпирическое подтверждение,
что v2-архитектура работает лучше v1. Если не найдёт — нужно
усиливать `auditor.md`.

Этот тест ещё не выполнен на момент написания кейса.
