## Sessions — обязательный отчёт каждой сессии

**Решение пользователя 2026-05-14:** репо `claude-base` теперь **private**,
обезличивание смягчено — Claude может писать в `session-reports/`
**сырые** отчёты (без жёсткой санитизации), потому что доступ только
у нашего внутреннего круга. Пользователь обрабатывает и оставляет в
методике (`agents/`, `skills/`, `memory/`) только нужное.

### Правило: каждая сессия пишет session-report

**Обязательно**, не «по запросу пользователя». Триггер — закрытие
сессии (последний ответ Claude перед концом контекста или явным
«закрываем»). Под закрытием понимается:

- Пользователь сказал «всё», «достаточно», «закрываем», «спасибо» как
  финал, «hand-off» и т.п.
- Тема явно завершена (пользователь подтвердил результат, новых
  вопросов нет)
- Контекст близко к compaction — лучше зафиксировать сейчас, чем
  потерять детали после compaction

**Где сохранять:**
```
~/.claude/session-reports/<YYYY-MM-DD>_<краткая_тема_kebab-case>/
  в”њв”Ђв”Ђ report.md        в†ђ основной отчёт (см. _TEMPLATE.md)
  в””в”Ђв”Ђ artifacts/       в†ђ опционально, обезличенные образцы артефактов
```

**Идемпотентность:** если за этот же день+тему уже есть папка —
**обновлять** существующий `report.md`, не плодить новые. Тема может
эволюционировать в течение дня.

**Структура `report.md`:** см. `~/.claude/session-reports/_TEMPLATE.md`.

**Что писать честно (без приукрашивания):**

- Что **делал** — хронология (3-10 пунктов).
- **Источники**, использованные реально: MCP-серверы по именам,
  скиллы по триггерам, slash-команды, нормативы/каталоги из библиотеки,
  harvest-находки если были.
- **Артефакты** — пути к финальным DOCX/PDF/XLSX в cwd сессии пользователя.
- **Итерации и сломалось** — где промахивался, что переделывал. Это
  ценнее любого «всё прошло гладко».
- **Что выдумывал** — если в моменте подставил placeholder вместо
  спрашивания у пользователя — указать.
- **Цитаты пользователя** — дословные, если были важные («не молча»,
  «не переусложняй», корректировки подхода).
- **Открытые вопросы** — что осталось на следующую сессию.

**Обезличивание (смягчённое для private):**
- **Не нужно** обезличивать имена hosts, шифры объектов, бренды
  оборудования, имена сотрудников в команде — это для нашего разбора.
- **Не писать** только: пароли, API-ключи (включая токены GitHub),
  ПДн (паспорта, СНИЛС), банковские реквизиты. Если такое промелькнуло
  в сессии — пометить `[СЕКРЕТ — не записан]`.
- Цели обработки потом пользователем — превратить сырой отчёт в
  обезличенный кейс в `memory/`, или в обновление скилла в `skills/`,
  или просто оставить как есть для будущего разбора.

### Правило: статус auto-sync в начале сессии

В первой реплике сессии (сразу после строки `✓ прочитан CLAUDE.md
(MCP: X/9)` из STOP-процедуры) — добавить **одну строку** про auto-pull.

**ВАЖНО (2026-05-21):** перед формированием отчёта про auto-pull,
прочитать `~/.claude/auto-sync.log` через `Get-Content -Tail 10` и
найти **последнюю** запись `auto-pull:` или `auto-push:`. Если она:

- Заканчивается на `DONE` — hook отработал корректно, описываем
  результат (ok/FAILED/pulled N commits).
- Заканчивается на `WARN: previous hook was interrupted` или
  `start` без последующего `DONE` — **предыдущая** операция была
  **прервана** (таймаут/kill). В первой реплике явно сказать:
  ```
  ⚠ Предыдущий auto-sync был аварийно прерван — запусти на этом ПК:
    cd $HOME\.claude
    git -c http.proxy="" -c https.proxy="" pull origin main
  ```
- Содержит `WARN: pull reported success but HEAD still behind` —
  pull вернул "ok" но коммиты не подтянулись (network glitch?).
  Тоже предложить manual pull.

Это критично: молчаливо отставшая база приводит к тому что пользователь
работает на устаревшей методике (см. инцидент 2026-05-21 на DELISEEV-PC).

```
Auto-pull: подтянуто N коммитов (last: <hash7> from <host> — <тема>)
```
или
```
Auto-pull: без изменений (HEAD == origin/main)
```
или
```
Auto-pull: FAILED (см. ~/.claude/auto-sync.log; продолжаю в read-only к claude-base)
```

**Источник информации:**
- Прочитать tail `~/.claude/auto-sync.log` (последние 10 строк) — там
  что записал auto-pull hook при старте этой сессии.
- ИЛИ `git log HEAD~5..HEAD --pretty=format:'%h %ad %s' --date=short`
  в `~/.claude/` — последние 5 коммитов, видно что свежее в базе.

### Правило: статус auto-sync в конце сессии

В **последнем** ответе перед закрытием темы — указать **прогноз**
auto-push:

```
Auto-push на SessionEnd: ожидается push N коммитов
(managed paths: <список изменённых>) в†’ origin/main.
В следующей сессии увижу результат в логе.
```
или
```
Auto-push на SessionEnd: managed paths не менялись в†’ push не будет
(только session-report этой сессии, если он попадает в managed).
```

Реальный результат push'а увижу при следующем старте сессии
(см. правило «статус auto-sync в начале сессии»).

### Когда правило не применяется

- Сессия длится <2 реплик и явно не содержит работы (например, просто
  поздоровались) — session-report не нужен.
- Пользователь явно сказал «не пиши отчёт по этой сессии» — пропускаем.

## Audit-trail вызванных агентов (требование, 2026-05-26)

В каждом `report.md` обязательно секция **«Audit-trail вызванных
агентов»** с таблицей: `#` / агент / триггер фразы / цель / verdict /
tokens (грубо).

**Цель:** проверять что 15 наших агентов реально срабатывают на типовых
задачах + ловить случаи когда основной Claude **должен был** вызвать
агента, но сделал сам.

Правила:
- Если агентов в сессии не было — явно `Agent calls: 0 (основной Claude справился сам)`.
- Если ожидался агент но не вызвал — отдельная пометка `Ожидался <name>, не вызвал. Причина: <гипотеза>`.
- Verdict: `PASSED` / `NOT PASSED` / `NEEDS REVISION`.

Шаблон таблицы — в `~/.claude/session-reports/_TEMPLATE.md` секция
«Audit-trail вызванных агентов».
