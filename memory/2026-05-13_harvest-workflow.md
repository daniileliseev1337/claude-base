# Кейс: настройка harvest-workflow с проактивными триггерами

**Дата:** 2026-05-13
**Контекст:** Пользователь заметил что Claude **не идёт активно искать**
внешние инструменты под новые задачи, несмотря на CORE-секцию
`Harvest-workflow` в CLAUDE.md. Причина — CORE-формулировка была
**реактивная** («когда возникла потребность в инструменте»), Claude
по дефолту прыгал к «напишу сам».

## Главное

В USER EXTENSIONS добавили **проактивные триггеры** + создали
slash-команду `/harvest <query>`. Теперь:
- Первое появление нового типа файла в сессии → триггер
- Старт новой задачи → триггер
- Claude **спрашивает** пользователя (не молча ищет) — поискать
  готовое или сразу работать своими
- Источники в обратном порядке (GitHub первым, см. ниже)

## Ключевые решения по дизайну

### 1. «Не молча, а спрашивать» через AskUserQuestion

**Альтернатива:** автоматический фоновый поиск перед каждой новой
задачей.

**Push back:** каждая рутинная задача → 10-30 секунд лишнего поиска,
часто без пользы. Тяжело для коротких задач (правка одной строки,
быстрое чтение файла).

**Принятое решение:** при триггере спросить пользователя одним
вопросом `AskUserQuestion`. Y/N. Если «найди» — `/harvest`. Если
«сразу» — пропуск + запомнить в контексте.

**Тест помощника-не-подхалима:** «согласился бы я если бы это
предложил коллега, а не пользователь?» — да, экономит время.

### 2. GitHub первым (временно)

**Изначальное предложение:** Anthropic available skills →
MCP registry → GitHub. Логика — built-in инструменты безопаснее,
проверены Anthropic.

**Корректировка пользователя (цитата):**
> «GitHub так как он полюбому найдёт на маркетплэйсе или у нас
> необходимый инструмент и тогда мы сможем отыскивать полезные
> инструменты. Когда база будет готова и в ней будут хорошие
> надежные инструменты то мы поправим очередность. Потому что в
> работе я вижу что наши инструменты не работают как хотелось бы.»

**Принятое решение:** на 2026-05-13 — **GitHub первым**, Anthropic
skills последним.

**Условие переключения обратно:** когда наша база скиллов разрастётся
до >5 проверенных, реально применённых в задачах. Тогда Anthropic
skills (как самый безопасный быстрый источник) — первыми, GitHub —
последним.

**Сигнал из контекста:** «в работе наши инструменты не работают
как хотелось бы» — обратная связь, что текущие скиллы (`stroy-*`,
`karpathy-guidelines`, и т.п.) недостаточно полезны в живой работе.
Это **отдельная** проблема, не закрытая сейчас. Стоит вернуться к
этому через debrief после следующей реальной задачи: какие скиллы
сработали, какие нет, что улучшить.

### 3. 2-5 кандидатов, не «5-100»

**Изначальная формулировка пользователя:**
> «5-100 не важно инструментов, берёт их сохраняет, и начинает работу
> над задачей и пробует каждый инструмент по ходу работы исключая
> плохо-работающие, не точные.»

**Push back:**
- 100 кандидатов = часы поиска и evaluation
- Запуск десятков чужих скриптов = security risk (особенно для
  MCP-серверов, которые получают доступ к контексту)
- Большинство GitHub-репо под популярные темы — мусор (заброшенные,
  не работают, кривая лицензия)

**Принятое решение:** 2-5 кандидатов с фильтром (≥50 stars, активный
<12 месяцев, лицензия в whitelist). Меньше 2 — не застревать, выдать
что есть или сказать прямо «не нашёл подходящего».

Пользователь согласился: «100 это я преувеличил, у нас достаточно
хорошая инструкция по выбору, 2-5 кандидатов то что нужно, сильно
не застревать если не можешь найти больше 3».

### 4. Не плодить агента, оставить как инструкцию + slash-команда

**Альтернатива:** создать `harvest-scout` агента в `~/.claude/agents/`.

**Push back:** пока ни одного реального успешного harvest нет, агент
наперёд = переусложнение по правилу из CORE «доменные агенты
добавляются по факту реальной повторяющейся задачи, не наперёд;
критерий: ≥3 успешных применения main без агента».

**Принятое решение:** harvest это **инструкция + slash-команда**.
Если в реальной работе через harvest пройдут 3+ задач и станет
очевидно что нужен отдельный агент (например, для параллельного
сбора данных от нескольких источников) — создадим тогда.

## Алгоритм триггера

1. Проверить `~/.claude/skills/` и `~/.claude/agents/` — есть ли свой
   инструмент.
2. Если есть — работать им, harvest не нужен.
3. Если нет — `AskUserQuestion`: «Поискать готовые инструменты или
   работать сразу?»
4. Если «найди» — `/harvest <query>`.
5. Если «сразу» — пропустить, запомнить решение в контексте сессии
   (чтобы не переспрашивать для того же типа задачи).

## Источники в порядке (на 2026-05-13)

1. **GitHub** — `WebSearch site:github.com` или `gh api search/repositories`
2. **MCP registry** — `mcp__mcp-registry__search_mcp_registry`
3. **Anthropic available skills** — список `anthropic-skills:*` в system context

## Куда сохранять

- **Per-session:** `session-reports/<YYYY-MM-DD>_<тема>/harvested/<repo>.md`
- **Накопленная база:** `~/.claude/harvested/<repo>.md` (auto-sync на все ПК)

## Уроки

1. **Различие реактивных и проактивных триггеров.** Объективный триггер
   (новый тип файла / новая задача) надёжнее субъективного («когда
   возникла потребность») — последний слишком зависит от того, что
   Claude в моменте считает «возникшим».

2. **«Спрашивать пользователя» дешевле, чем «молча работать».** Один
   `AskUserQuestion` экономит десятки секунд автоматического поиска
   когда задача рутинная и harvest не нужен.

3. **Порядок источников зависит от зрелости базы.** Пока своих
   проверенных инструментов мало — GitHub первым. По мере накопления
   проверенной методики — Anthropic skills и наши `~/.claude/skills/`
   первыми.

4. **Обратная связь о работающих/неработающих скиллах — отдельная
   тема.** Сигнал «наши инструменты не работают как хотелось бы»
   стоит превратить в задачу: после следующей реальной работы сделать
   debrief по каждому применявшемуся скиллу — что зашло, что нет, что
   менять.

## Связанные

- CORE-секция `Harvest-workflow` в `~/.claude/CLAUDE.md` — что искать,
  фильтры, что разрешено auto.
- USER EXTENSIONS секция `Harvest-workflow — проактивные триггеры` —
  когда искать.
- Slash-команда `~/.claude/commands/harvest.md`.
- Заметки находок: `~/.claude/harvested/` (общая) +
  `session-reports/<...>/harvested/` (per-session).
- Изолированный sandbox: `~/.claude/_sandbox/`.
