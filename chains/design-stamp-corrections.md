# chain:design-stamp-corrections

## Назначение

Полный цикл применения замечаний проверяющего (<организация> / экспертиза /
заказчик) к набору проектных Excel-спецификаций одного шифра.
Закрывает типовой кейс: PDF замечаний → разбор → правки в Excel →
синхронизация штампа → донор-фикс сломанных формул → пакетирование
П/Р версий.

**Триггер на стороне пользователя:** «применил замечания <организация> к
АРМ-019, нужно обновить спецификации». **Триггер на стороне
артефактов:** один шифр (например `АРМ-019-000-ТФ`) + 1-N PDF
с замечаниями + 4-10 Excel-файлов спецификаций по субшифрам.

## Шаги

### Stage 1 — locate (collect)

Найти все Excel-файлы шифра (например `АРМ-019-000-ТФ.С*`,
`АРМ-019-000-ТФ.Р*`). Обычно — на Яндекс.Диске или сетевом ресурсе.

**Tool calls:**
- `Glob` или `Bash find` по pattern шифра
- спросить пользователя через `AskUserQuestion` если файлы в нескольких
  местах одновременно

**Verify:**
- Список Excel-файлов непуст.
- Сопоставление с ожидаемой структурой `<шифр>.<субшифр>.<С|Р>.xlsx`
  (С = строительная часть, Р = расчётная). Если ожидание не выполняется
  — поднять открытый вопрос пользователю.

### Stage 2 — extract замечания из PDF (collect/parse)

Прочитать PDF с замечаниями. **Ловушка:** многие PDF выпущенные через
PScript5/Distiller имеют CID-encoding и `pdf_read_all` возвращает
текст-мусор.

**Tool calls:**
- `mcp__pdf-mcp__pdf_info` → проверить producer
- если producer PScript5/Distiller → **сразу** `pdf_render_pages` 180 DPI,
  потом OCR через [[image-text-replace]] EasyOCR (см. [[pdf-scan-extract]]
  chain — он может быть использован как nested).
- иначе — `pdf_read_all`

**Verify:**
- Список замечаний с номером, страницей, цитатой, рекомендацией.
- Если PDF возвращает CID-мусор и OCR тоже не справляется (низкое
  качество) — эскалация пользователю.

### Stage 3 — изучение архитектуры Excel (analyse)

Многие проектные xlsx имеют **многолистовую архитектуру**:

- лист `Спецификация` — основной редактируемый
- лист `На печать` — формулы `INDIRECT`/`VLOOKUP` от `Спецификация`
- лист `Данные` — параметры (заказчик, шифр, ГИП, версия, дата) +
  ссылки в штамп
- штамп — формулы от `Данные`, **не** от `Спецификация`

**Tool calls:**
- `mcp__excel__get_workbook_metadata` для каждого файла
- `mcp__excel__read_data_from_excel` лист «Данные» — понять параметры

**Verify:**
- Все 4 листа (Спецификация / На печать / Данные / штамп) обнаружены
  или явно зафиксировано отсутствие конкретного листа.

> **Ловушка `mcp__excel`:** на больших файлах `read_data_from_excel`
> возвращает max-tokens overflow (>64K символов). Тогда переключиться
> на `openpyxl` напрямую через `Bash python -c "..."`. См. [[excel-helper]].

### Stage 4 — правки спецификации (act)

Применить замечания построчно к листу `Спецификация` через **openpyxl**
(не через mcp__excel — он перезаписывает формулы как строки на больших
файлах).

**Tool calls:**
- `python -c "from openpyxl import load_workbook; wb = ...; ws = wb['Спецификация']; ws.cell(row=N, column=M, value=...); wb.save(...)"`
- если правки массовые (несколько десятков) — отдельный python-скрипт
  в cwd, не однострочник

**Verify:**
- Каждое замечание из Stage 2 имеет explicit запись в audit-trail
  (что меняли, в какой ячейке).
- `wb.save` без ошибок.
- Прочитать обратно через `mcp__excel__read_data_from_excel` (с offset
  если файл большой) — убедиться что правки на месте.

> **Ловушка `MergedCell`:** в openpyxl MergedCell **read-only**, формулы
> через corner-ячейку диапазона.

> **Ловушка PowerShell + python:** при stdout кириллицы из python в
> PowerShell — `sys.stdout.reconfigure(encoding="utf-8")` в python-скрипте.

### Stage 5 — синхронизация штампа из листа «Данные» (act)

Замечания часто меняют параметры проекта (версия, дата ревизии, ГИП,
шифр). Эти параметры → лист `Данные` → формулы штампа подтянутся
автоматически. **Но** если в предыдущих сессиях кто-то hard-code'нул
значения в штамп (поверх формул) — синхронизация **не сработает**.

**Tool calls:**
- `python openpyxl` — записать новые значения в листе `Данные`
- проверить штамп на каждом листе — формулы или константы?

**Verify:**
- Лист `Данные` обновлён.
- Если штамп = формулы → синхронизация прошла автоматически (OK).
- Если штамп = константы → переход на Stage 6 донор-фикс.

### Stage 6 — donor-fix hard-coded формул штампа (repair)

**Узкая ловушка проекта:** кто-то ранее правил штамп вручную, openpyxl
сохранил введённые значения как строки/числа, не формулы. Теперь штамп
не подтягивается из `Данные`.

**Решение через donor-файл:**

- Открыть **эталонный** файл шифра (часто `<шифр>.Р.С.П.xlsx` — паспорт
  расчётный, который точно никто не правил).
- Прочитать формулы штампа через `openpyxl` `cell.value` (вернёт
  строку `=Данные!B5` если формула).
- Применить **те же формулы** ко всем целевым файлам через
  `ws.cell(...).value = '=Данные!B5'`.
- Сохранить.

**Tool calls:**
- python-скрипт `apply_donor_stamp_formulas.py` (типовой шаблон,
  ~30 строк)

**Verify:**
- В каждом целевом файле штамп = формулы (открыть через openpyxl,
  убедиться что начинается с `=`).
- Открыть в LibreOffice/Excel вручную для **визуальной** проверки
  что штамп заполнился правильными значениями из `Данные`.

> **Антипаттерн:** не **копировать вычисленные значения** из донора в
> целевой файл — это просто перенос той же hard-coded ошибки. Копируем
> **формулы**, не values.

### Stage 7 — малый штамп (СПДС-правило)

Малый штамп на повторяющихся листах должен показывать **только 2
последних** записи об изменении (а не 3 или 4). Это правило ГОСТ
21.101 / СПДС.

**Tool calls:**
- python-скрипт обнаружить малый штамп (обычно ячейки `B15-B17`)
- удалить старые записи, оставить 2 актуальные

**Verify:**
- Малый штамп содержит ровно 2 записи (например `Изм.1` и `Изм.2`,
  не `Изм.0` + `Изм.1` + `Изм.2`).

### Stage 8 — пакетирование П/Р (save)

Финальная структура папок:

```
<шифр>/
├── С/                   # строительная часть
│   ├── <шифр>.001.С.xlsx
│   └── <шифр>.002.С.xlsx
└── Р/                   # расчётная часть
    ├── <шифр>.001.Р.xlsx
    └── <шифр>.002.Р.xlsx
```

**Tool calls:**
- `Bash mv` или `Move-Item` для переноса в правильные подпапки

**Verify:**
- Все целевые файлы лежат в правильной С/Р подпапке.
- Прогнать `excel-validator` агента на 1-2 случайных файла из
  каждой папки — final sanity check.

## Когда НЕ использовать

- Один Excel-файл без замечаний — это просто правка, не chain.
- Замечания **по чертежам** (DWG/PDF чертежей) — отдельный flow,
  здесь только Excel-спецификации.
- Шифр без `Данные` листа — нет смысла Stage 5/6 (донор не применим).
  Использовать прямую правку через [[excel-helper]].
- Новый проект с нуля — это `chain:project-doc-pack` (когда активируется
  backlog) или `pto-engineer` + [[spec-writer]] скилл.
