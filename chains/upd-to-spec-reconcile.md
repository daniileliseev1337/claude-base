# chain:upd-to-spec-reconcile

## Назначение

Применить **новый УПД от поставщика** к Excel-реестру объекта:

- распарсить УПД в JSON позиций;
- сопоставить по артикулу (D) с реестром;
- обновить S (отгружено от поставщика) и зеркально J (отгружено в ЛС);
- применить правила: накопление S/J, переименование артикулов, возвраты, услуги;
- запись + валидация без перезаписи истории и без потери инвариантов.

**Когда НЕ использовать:** УПД для нового объекта без реестра → сначала
`spec-writer` создаёт реестр / лист «Спец. N», только потом этот chain.

## ⚠ Главные инварианты (нарушение → провал stage)

1. **Накопление, не перезапись.** S/J/I/R накапливаются по партиям.
2. **S ↔ J зеркало.** При обновлении S — синхронно обновить J на то же кол-во штук.
3. **Услуги не идут в продажу.** Для строк «услуги» G/H пусто; P/R как закупка.
4. **Закупочная цена O — только из подписанной спеки.** Не выдумывать из
   расчётных xlsx с надбавками.
5. **Возврат ≠ полное списание.** Возврат → E=0, каскадно H/P/S/J/R=0,
   strikethrough всей строки, **не удалять**.

## Шаги

### Stage 1 — parse (collect)

Распарсить УПД через `upd-parser` → JSON
`{upd_number, upd_date, seller, buyer, items: [...], totals}`.

**Verify Stage 1:**

- `buyer.inn == "<ИНН организации>"` — иначе эскалация: «УПД не на нашу
  организацию, это нормально?»
- Сумма `items[].amount` ≈ `totals.sum_without_vat` (до копейки).
- `len(items) > 0` — иначе stop.
- Если `is_scan == True` — предупредить «УПД сканом, OCR confidence X,
  сверь визуально перед записью».

### Stage 2 — load reference (collect)

1. **Текущий реестр** (xlsx) — через `excel-helper` zip-разведку. Есть
   drawings → Pre-flight как в `vendor-logo-inserter` Stage 0 (бэкап обязателен).
2. **Подписанная спецификация поставщика** — эталон закупочной цены **O**.
3. **Найти целевой лист «Спец. N»**. Несколько Спец → спросить пользователя.

**Verify Stage 2:** файл открывается, бэкап сделан, лист определён явно,
подписанная спека недоступна → warning + O из реестра как fallback с пометкой.

### Stage 3 — match (extract)

Для каждой `item`: поиск по `D` (артикул) — exact → normalized → fuzzy
(Levenshtein > 0.85, **подтверждение пользователя обязательно**).

Не нашёлся → **возможно переименование артикула.** НЕ заявлять «swap» без
визуальной сверки PDF. `pdf_render_pages` страницы таблицы УПД, спросить
пользователя: переименование (заменить D + красным «(Было <старый>)») или
новая позиция.

Собрать **план обновления** как JSON (`matches`, `unmatched`, `renames`).

**Verify Stage 3:** план собран, все unmatched явно классифицированы
(никаких silent fallback на «new»).

### Stage 4 — compute updates (apply rules)

1. **Прочитать текущее S**, извлечь `N_старое` (из формулы `=O523*11` → 11).
2. **Записать S** = `=O_row × (N_старое + N_новое_по_УПД)`.
3. **Сразу записать J** = `=G_row × (N_старое + N_новое_по_УПД)` (зеркало).
4. **Услуги** (unit = работа/монтаж/ввод в эксплуатацию): P,R как закупка;
   G,H **пусто**. Применяется на этапе compute, не «исправляется потом».
5. **Q (дата отгрузки)** — обновить на `upd_date`.
6. **N (№ счетов)** — текст с переносами, **добавить строку**, не заменять.
7. **Переименование** — заменить D, дописать красным курсивом «(Было <старый>)».
8. **Возвраты** (отрицательная qty / пометка) — E→0 каскадно H/P/S/J/R→0,
   strikethrough. **НЕ удалять строку.**

**Verify Stage 4:**

- Для каждой обновлённой S есть зеркальная J с тем же N (инвариант 2).
- Для услуг G/H пусты (инвариант 3).
- `Σ(новый_S − старый_S)` ≈ `totals.sum_without_vat × E_ratio` до копейки;
  расхождение > 1 руб → эскалация.
- План покрывает 100% позиций УПД.
- **Контрольная сумма документа:** `Σ внесённых позиций == «Всего к оплате» УПД`
  + число позиций совпадает + все листы многостраничного скана учтены (см.
  `skills/upd-parser/scripts/verify_document_entry.py`). Cross-check ценой:
  `цена_УПД(без НДС) × 1.22 = O` (НДС 22% с 2026; для закупка=продажа `×1.22 = G`).

### Stage 5 — write + post-verify

1. **Запись** через `excel-helper`: без drawings → `mcp__excel__write_data_to_excel`;
   с drawings → openpyxl с явным `Image()` API либо ZIP-хирургия ячеек. Учесть
   ловушку delete_rows + merge (excel-helper §15) если строки удаляются.
2. **Сохранить план** в `session-reports/<...>/artifacts/upd-plan-<upd_number>.json`.
3. **Спавнить `excel-validator`** на обновлённый файл.
4. **NOT PASSED** → стоп, эскалация, откат на бэкап опционально.

**Verify Stage 5:**

- excel-validator: PASSED с явным списком проверенных пунктов (общее «всё ок»
  не принимается — failure-mode строгий).
- Spot-check 2-3 строк глазами: S/J согласованы, услуги пусты в G/H,
  переименования с пометкой.
- Файл открывается без warning'ов.

## Output

- **Обновлённый реестр** в исходном месте + бэкап рядом.
- **План обновления** JSON в artifacts.
- **Отчёт excel-validator** с явным списком пунктов.

## Когда НЕ использовать

- УПД на новый объект без реестра — сначала `spec-writer`.
- УПД от поставщика не привязанного к Спец (закупка на офис) — другой реестр.
- УПД с непрозрачной структурой (OCR confidence < 0.5) — до нормального исходника.

## Цитаты пользователя, формирующие инварианты

- «Если вносим отгрузку в сторону нашей организации, заполняем аналогичную
  отгрузку в адрес ЛС, не дожидаясь УПД» → инвариант S ↔ J.
- «Ранее эти позиции были отгружены частично и это было в таблице, почему ты
  игнорировал что в строках уже были записи?» → инвариант accumulate, не overwrite.
