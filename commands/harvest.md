# /harvest — поиск внешних инструментов

Использование: `/harvest <query>` или `/harvest` (спрашивает что искать).

Примеры:
- `/harvest extract tables from PDF`
- `/harvest DOCX templates ГОСТ`
- `/harvest Excel formula validator`
- `/harvest CSV manipulation Revit`

## Когда использовать

- Пользователь сам зовёт явно («поищи инструмент для X»).
- Триггер из CLAUDE.md USER EXTENSIONS «Harvest-workflow — проактивные
  триггеры» сработал и пользователь сказал «найди».
- Перед сложной задачей — пользователь хочет проверить что в open-source
  ничего готового нет, а потом уже писать своё.

## Алгоритм

1. **Если query не передан** — спросить через `AskUserQuestion` что
   ищем (краткая формулировка задачи или ключевые слова).

2. **Поиск в трёх источниках** в порядке приоритета (на 2026-05-13):

   ### A. GitHub (основной)

   - `WebSearch` с запросом `site:github.com <query>` — даёт список
     URLов и кратких описаний.
   - Параллельно (если есть `gh` CLI и аутентификация):
     `gh api "search/repositories?q=<query>&sort=stars&order=desc&per_page=10"`.
     Это даёт структурированные данные: name, owner, stars, pushed_at,
     license, description.
   - Собрать до 10 кандидатов из топа.

   ### B. MCP registry

   - `mcp__mcp-registry__search_mcp_registry` с keyword из query.
   - Полезно для протокол-специфичных задач: «нужен MCP-сервер для
     работы с <формат>».
   - Если MCP registry недоступен — пропустить.

   ### C. Anthropic available skills (fallback)

   - Просмотреть `anthropic-skills:*` в моём system context.
   - Найти что-то близкое по описанию.
   - Это **встроенные** и бесплатные — если совпадение есть, это идеал.

3. **Фильтр кандидатов** (для GitHub в первую очередь):

   - **last commit <12 месяцев** — активный.
   - **≥50 stars** или явная репутация автора (Anthropic / Microsoft /
     известные мейнтейнеры).
   - **Лицензия:**
     - MIT/Apache-2.0/BSD/ISC/CC0/Unlicense → зелёный свет
     - GPL → флаг для согласования с пользователем (вирусная лицензия)
     - нет LICENSE → заметка пишется, **код не копируется**

4. **Если после фильтра <2 кандидатов** — НЕ застревать. Выдать что
   есть (1 кандидат) или сказать прямо: «По <query> не нашёл подходящих
   инструментов с приличными метриками. Лучше написать самим / уточнить
   формулировку запроса.»

5. **Записать заметки** в `session-reports/<YYYY-MM-DD>_<тема>/harvested/`.
   Один файл на кандидата (`<repo-name>.md`) со структурой:

   ```markdown
   # <repo name>

   - **URL:** https://github.com/owner/repo
   - **Stars:** 1234
   - **Last commit:** 2026-03-15
   - **License:** MIT
   - **Описание:** <одно предложение что делает>

   ## Зачем смотрели

   <короткий контекст из текущей задачи>

   ## Оценка

   - Подходит? Да / Под условием / Нет
   - Сильные стороны: ...
   - Слабые стороны / риски: ...
   - Решение: используем / держим в уме / отбросили
   ```

6. **Выдать пользователю сравнительную таблицу**:

   ```
   | # | Источник | Имя | Stars | Last commit | License | Краткое описание |
   |---|----------|-----|-------|-------------|---------|------------------|
   | 1 | GitHub   | ... | 2.3k  | 2026-02     | MIT     | ...              |
   | 2 | GitHub   | ... |  340  | 2026-04     | Apache-2| ...              |
   | 3 | MCP      | ... |  -    | -           | MIT     | ...              |
   ```

   + одна строка рекомендации:
   - «**Рекомендую #1** — активный, MIT, точно под наш сценарий.»
   - Или: «**Все слабые** — лучше написать самим, см. план в моих
     следующих шагах.»

7. Пользователь выбирает — либо «возьми #N», либо «попробуй #2 и #4»,
   либо «никакой, делаем своё».

## Что НЕ делать

- **Не запускать код найденных репо** в основной системе. Только в
  Windows Sandbox или временном venv (см. CORE-секция `Harvest-workflow`
  → «Изолированные действия»).
- **Не копировать файлы из чужих репо** в `~/.claude/agents/` или
  `~/.claude/skills/` без явного решения пользователя.
- **Не импортировать через pip/npm** найденные пакеты без согласия.
- **Не «100 кандидатов по очереди»** — это переусложнение, риск
  безопасности и часы потраченного времени. 2-5 хватит.
- **Не застревать при <2 кандидатах** — выдать что есть и двигаться
  дальше.

## Адаптация в нашу базу (после успешного применения)

Если в реальной задаче кандидат показал себя отлично и захотелось его
«закрепить» в нашей базе как переиспользуемый инструмент:

- Только из внутреннего круга (личные ПК + 1-2 коллеги). При расширении
  на третьих лиц — пересматриваем дисциплину.
- Только лицензии MIT/Apache/BSD/ISC/CC0/Unlicense.
- «Нет LICENSE» — переформулировать идеи своими словами, не копировать
  код.
- GPL — не брать (вирусность портит installer).
- Адаптация — отдельный шаг с согласием пользователя, не автоматически
  из `/harvest`.

## Связанные

- CLAUDE.md CORE `Harvest-workflow` — что/где/безопасность
- CLAUDE.md USER EXTENSIONS `Harvest-workflow — проактивные триггеры` —
  когда запускать
- `~/.claude/harvested/` — общая накопленная база
- `session-reports/<...>/harvested/` — per-session находки
- `~/.claude/_sandbox/` — изолированная зона для тестов внешнего кода
