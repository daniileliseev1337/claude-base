# ОСИ ПТО — стратегия работы над томом ИД (зоны · порядок · протокол возврата)

> **Объект-широкий регламент.** КАК блок ПТО доводит том ИД до отправки PDF заказчику.
> Каждая **ось = своя зона ответственности** (в перспективе — свой агент/ряд агентов).
> Сессия читает это, чтобы знать **СВОЮ роль** и **кому передаёт**.
> Источник: договорённость 2026-06-22 (первый пилотный том ИД). Граф осей (graphify) построён — рядом в `graphify-out/` (узлы=оси, рёбра=передачи/гейты/возвраты). Источник истины правил — этот markdown; граф — навигатор.
> ⚠ Дополняет, не отменяет `Порядок работы по тому ИД (блоки).md` (линейный runbook блоков 1-12) — здесь governance-слой (зоны + возвраты).

---

## 0. Сквозной закон (зачем оси)
**Каждая ось работает ТОЛЬКО в своей оси.** Нашёл расхождение в ЧУЖОЙ зоне →
**пишет блокер** в `02_СОСТАВ` со ссылкой на профильную ось, **СВОЁ доделывает, чужое НЕ чинит.**
Иначе двойные правки и сломанный инвариант **ПВ ≡ ВСО ≡ ВОР ≡ ИС**.

Проблема, которую закрываем: сейчас ни одна ось не знает, **в каком виде том был изначально** — все знают только замечания ТН. Ось Б даёт общий «базовый слой знания о томе».

## 1. Состав осей (зоны ответственности)

### Рамочная ось (вокруг всех, не последовательная)
- **Ось Р — ТН-замечания.** Реестр замечаний + карандашных пометок (**вход**) + сводная верификация «**все устранены**» (**выход** — контрольные точки внутри осей 5 и 6). Владелец учёта: `01_СТАТУС`.

### Базовый аудит (грунт под все оси)
- **Ось Б — Независимый аудит СУЩЕСТВУЮЩЕГО тома.** ПОСЛЕ сбора+утверждения замечаний — жёсткая **постраничная** независимая проверка ОРИГИНАЛА тома (skill `id-tom-priemka` + **Workflow**, opus, PNG-режим). Выдаёт:
  1. **Карту структуры тома** — инвентарь страниц/документов (чего сейчас не знает ни одна ось);
  2. **Дефекты СВЕРХ замечаний ТН** — что сделано неправильно, а ТН упустил;
  3. **Где ТН ошибся** — спорные/неактуальные замечания.
  Побочно — **карта страниц** (лист↔страница) для оси 5 (листаж/пересборка). Независимость: смотрит том «с нуля», не зная решений других осей.

### Содержательные оси
- **Ось 1 — Состав** (ВОР ≡ ВСО ≡ ПВ): наименования / кол-ва / инвариант. Источник истины: **СВОД (факт закупки/монтажа)** + РД.
- **Ось 2 — Документы качества:** действующий документ на каждую позицию + реквизиты в реестрах (№ / дата / наименование). Действительность — по **дате акта**, не по сегодня.
- **Ось 3 — Графика ГЧ (DWG):** исп. схемы — ВОР-таблицы (OLE→нативная ACAD_TABLE), длины / Ø ►◄ / арматура / УГО, зачеканка гильз + расчёт пены, выноски под факт-ПВ, спец-узлы (напр. трассировка циркуляционного трубопровода в ИТП/ЦТП).
- **Ось 4 — Акты (нормоконтроль оформления):** даты акта; **даты/номера приказов**; подписанты; **подчёркивания**; **приложения = тому, что прописано в акте**; шифры; ссылки на **СП**; расхождения с **ИД и РД**.

### Интеграция и контроль
- **Ось 5 — Электронная пересборка PDF:** структурный реестр + **листаж** + страницы; собирает электронный том по решениям осей 1-4 (+ карта страниц оси Б); сверяет «**все приложения на местах + все замечания отражены**». Это ЭЛЕКТРОННАЯ сборка Claude'ом — **не** печать человеком.
- **Ось 6 — Финальная приёмка:** `id-tom-priemka` + **Workflow** (постранично, opus, PNG) собранного тома + сверка с осью Р «**все ТН-замечания закрыты**».

### Передача человеку (вне осей-агентов)
- **Выдача / печать:** СВОДКА правок старый→новый для подшивки + физическая замена листов человеком. Том — в корень папки.

## 2. Порядок осей (минимум перехлёста)

```
Р(вход) → Б(базовый аудит) → 1 Состав → 2 Документы → 3 Графика → 4 Акты → 5 Пересборка → 6 Приёмка → Р(выход) → Выдача/печать
```

Логика порядка:
- **Состав — фундамент:** при полной сверке с СВОД+РД ниже почти нет возвратов.
- **Графика (3) — главный источник возврата в Состав(1)/Документы(2)** (визуально вскрывает «позиция на схеме есть, в ВОР нет», нестыковки длин/Ø). Её гейт — самый жёсткий.
- **Акты (4)** — после состава+документов (приложения и шифры ссылаются на них).
- **Пересборка (5)** — только при **нуле открытых кросс-блокеров** по осям 1-4 (иначе листаж разъедется).
- **Б (базовый аудит)** идёт рано (после замечаний), но его находки сверх-ТН распределяются по содержательным осям как обычные блокеры.

## 3. Протокол возврата (гейт между осями) — главный механизм

1. Ось нашла чужое расхождение → **сразу блокер** в `02_СОСТАВ` (поле: `источник-ось → целевая-ось → суть`).
2. Ось **доделывает своё**, чужое не трогает.
3. По завершении оси — **рапорт пользователю**: «в блокерах N изменений по оси X».
4. **Пользователь блокирует** переход к следующей оси → возврат в ось X (та же или **новая сессия**).
5. Ось X доводит; если всплыло изменение ещё одной оси → **снова блокер** → круг повторяется **до нуля кросс-блокеров** (фикс-точка / полная унификация).
6. Ноль кросс-блокеров → переход к следующей оси.

> Цель порядка (раздел 2) — чтобы таких кругов было **минимум**.

## 4. Workflow (мультиагент)
- **Ось Б** и **Ось 6** — Workflow по умолчанию (постраничный fan-out, opus, PNG).
- **Ось 4 (акты)** — Workflow-кандидат: каждый акт проверяет свой агент параллельно (даты/приказы/приложения/шифры) → синтез ведомости.
- **Ось 3 (графика)** — точечные autocad-задачи, обычно не fan-out.
- При подписке **x20** — Workflow предлагать явно (огромное преимущество по полноте).

---

## 5. Где живёт учёт
- Замечания / блокеры / статусы осей → **`01_СТАТУС`** тома.
- Состав / документы / листаж / каскад / **кросс-блокеры между осями** → **`02_СОСТАВ`** тома.
- Справка / нормы → **`03_СПРАВКА`** тома.
- Эта стратегия — **объект-широкая** (одна на все тома). Граф осей (graphify) — позже.

## 6. Будущее
Оси → **агенты** (или ряд агентов), действующие строго по этому регламенту и перекидывающие задачи друг другу через **блокеры** до полной унификации и устранения замечаний. Карта осей будет собрана как **graphify-граф** (узлы=оси, рёбра=передачи/гейты, атрибут=зона+владелец-файл) — после доработки базы под graphify; тогда каждая сессия сможет query'ить «кто я, от кого принимаю, кому отдаю».
