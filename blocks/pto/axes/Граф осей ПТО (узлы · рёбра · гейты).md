# Граф осей ПТО — узлы · рёбра · гейты

> **Что это.** Курированный (ручной, без автогенерации) граф процесса «блок ПТО доводит том ИД до отправки PDF».
> Узлы = оси (зоны ответственности), рёбра = передачи и возвраты, атрибуты = зона + владелец-файл + инструмент + гейт.
> **Источник:** `ОСИ ПТО — стратегия (зоны, порядок, протокол возврата).md` (истина — там; этот файл — навигационный слой над ней).
> **Как пользоваться (query глазами):** найди свой узел в §1 → колонки «принимает от / отдаёт» = твои соседи; «гейт» = условие перехода; «владелец-файл» = куда писать. Возвраты — §3.
> Диаграмма: `Граф осей ПТО (диаграмма).svg` (рядом).

## 2. Рёбра-передачи (прямой порядок)

```
Р(вход) → Б → 1 Состав → 2 Документы → 3 Графика → 4 Акты → 5 Пересборка → 6 Приёмка → Р(выход) → Выдача
```

| Ребро | Что передаётся |
|-------|----------------|
| Р → Б | реестр замечаний + пометки (база знания «что не так») |
| Б → 1/2/3/4 | дефекты сверх-ТН как блокеры (распределение по содержательным осям) |
| Б → 5 | карта страниц (лист↔страница) для листажа |
| 1 → 2 | согласованный состав (что есть в томе по факту) |
| 2 → 3 | позиции с документами (что обеспечено) |
| 3 → 4 | выверенная графика (схемы под факт) |
| 4 → 5 | выверенные акты (приложения/шифры) |
| 5 → 6 | собранный электронный том |
| 6 → Р | подтверждение «все замечания закрыты» |
| Р → Выд | разрешение на печать + СВОДКА правок |

**Логика порядка:** Состав — фундамент (полная сверка с СВОД+РД → мало возвратов ниже). Графика (3) — главный вскрыватель нестыковок → её гейт жёстче всех. Акты (4) — после состава+документов (ссылаются на них). Пересборка (5) — только при НУЛЕ открытых кросс-блокеров 1-4.

## 3. Рёбра-возвраты (протокол кросс-блокеров) — главный механизм

> Любая ось, найдя расхождение в ЧУЖОЙ зоне → **пишет блокер в `02_СОСТАВ`** (поле `источник-ось → целевая-ось → суть`), **своё доделывает, чужое не чинит**. Переход к следующей оси разрешает **пользователь** только при НУЛЕ открытых кросс-блокеров.

| Возврат (ребро) | Типовая причина | Документированный пример |
|-----------------|-----------------|--------------------------|
| **3 → 1** | «позиция на схеме есть, в ВОР нет»; нестыковка длин/Ø | напр.: таблица ВОР в DWG не перевставлена после правок состава |
| **3 → 2** | на схеме изделие, на которое нет/просрочен документ | — |
| **Б → 1/2/3/4** | дефект сверх-ТН попадает в профильную ось | (ось Б ещё не прогонялась — заполнится) |
| **любая → 2** | вскрылась замена изделия → меняется документ | напр.: замена модели изделия тянет замену её сертификата/паспорта |
| **гейт перед 5** | есть открытый кросс-блокер по 1-4 | блокирует пересборку (листаж разъедется) |
| **6 → Р** | приёмка нашла незакрытое замечание | возврат в профильную ось |

**Цикл сходимости:** ось пишет блокер → доделывает своё → рапорт пользователю «N блокеров по оси X» → пользователь блокирует переход → возврат в X (та же/новая сессия) → круг до **нуля кросс-блокеров** (фикс-точка). Порядок §2 нужен, чтобы кругов было минимум.

## 4. Гейты (узлы-условия перехода)

| Гейт | Где | Условие открытия |
|------|-----|------------------|
| G-вход | Р → Б | замечания собраны и размечены |
| G-Б | Б → содержательные | находки сверх-ТН распределены по блокерам |
| **G-5 (жёсткий)** | 4 → 5 | **0 открытых кросс-блокеров** по осям 1-4 |
| G-6 | 6 → Р | 0 дефектов приёмки |
| **G-выход** | Р → Выд | **все 13 п. ТН + все пометки закрыты** (сводная верификация) |

## 5. Атрибут «Workflow» (мультиагент)

| Ось | Режим | Модель (план) |
|-----|-------|---------------|
| **Б** | Workflow по умолчанию (постраничный fan-out, PNG) | opus обе фазы (том со сканами) |
| **6** | Workflow по умолчанию (постранично, PNG) | opus обе фазы |
| **4** | Workflow-кандидат (агент/акт параллельно → синтез) | sonnet/opus по согласованию |
| **3** | точечные autocad-задачи, обычно НЕ fan-out | — |
| 1, 2, 5 | обычный заход (каскад/реестры/сборка) | — |

> Решение о модели субагентов и о запуске Workflow — **за пользователем** (токен-дисциплина). При x20 Workflow предлагать явно.
