# ЗНАНИЯ ПТО — практика подсчёта объёмов и источники факта

> **Постоянный файл-знаний блока ПТО.** Копит практику пользователя (customer zero) по тому,
> ОТКУДА в ИД берётся факт-объём и КАК считается, — чтобы знание не терялось между сессиями.
> **Дополнять при каждом новом объяснении. Спорное — сверять с пользователем, не выдумывать.**
> **МЕТА-ПРАВИЛО ВЕДЕНИЯ:** каждое новое открытие — правило, утверждение, стратегия, порядок действий —
> **сначала согласовать с пользователем, затем записать сюда.** Цель — полный свод правил, чтобы пользователю
> не объяснять одно и то же по многу раз. Файл должен расти, а не терять.
> Общий для всех томов ИД. Работает в паре с `DESIGN — процесс ИД по срезам (2026-07-05).md` (текущий
> процесс) и методичкой `Обучение составления ИД.md`. ⚠ Осевой подход (`axes/`, «ОСИ ПТО») — ПРОШЛЫЙ,
> источником правил не считать.
>
> Статус пунктов: ✅ подтверждено пользователем · ⏳ на сверке (моя формулировка) · ❓ пробел, спросить.

## 1. Иерархия объёмов: ПВ — корень ✅

- **ПВ (предъявительская ведомость) — первоисточник фактических объёмов ИД.** В ней ВСЕ позиции
  (оборудование + материалы + изделия), разбитые **ПО АКТАМ**.
- **ВСО** = только **оборудование** (проекция ПВ, приложение к АОМР). **ВОР** = объёмы работ/материалов
  по акту (проекция ПВ).
- Историю формирования: рабочие ВОР и ВСО были **нарезаны из ПВ** (ПВ первична, они производны).
- Инвариант **ПВ ≡ ВСО ≡ ВОР ≡ ИС** — это сходимость проекций; истина живёт в **ПВ**.
- ✅ Подтверждено пользователем (2026-07-05): «внутри ПВ сидят все ВСО и ВОРы; все ВОР и ВСО,
  что сейчас есть, я когда-то формировал из ПВ». ПВ разбита **по актам**.
- ⚠ В файлах СТАРОГО осевого подхода записано «источник истины: СВОД + РД» — формулировка **огрублена
  и вводит в заблуждение** (см. §2): корень — ПВ, не СВОД и не РД. Старые осевые файлы правилами не считать.

## 2. Факт vs проект — откуда истина ✅

- **ИД несёт ФАКТ** (что и сколько реально смонтировано), не проект. Проектная спека (СО/РД) — только
  **отправная точка**; по факту бывает другое оборудование/количество → «сверять по спеке» нельзя.
- **Единого «правильного» файла-эталона объёмов НЕ существует.** Правдивее всего — рабочие ПВ/ВОР/ВСО тома.
- **СВОД оконечки = 100 % факт, но ограничен:** несколько позиций + количества на ВСЕ строения → берём
  из него ТОЛЬКО наименования/марки реально смонтированного, **не количества** на конкретный корпус.
- ⚠ Файлы вида `СО для вадима\…Водоснабжение 27.11.xlsx` — это **старые неактуальные ПВ прошлых версий
  ИД**, а не проектная СО. Не принимать за проектный источник.

## 3. Подсчёт длин трубопровода ✅

Два правила пользователя (критичны — фиксировались после двух ошибок; см. `[[feedback_is_lengths_from_dwg]]`):
1. **Длины меряются по DWG** (подложка 1:1 = факт по размерам), **НЕ** из внешних замеров монтажника. DWG и есть факт.
2. **Размеры/длины — ТОЛЬКО на ПЛАНЕ, НЕ на аксонометрии.** Горизонтальные длины на аксонометрии
   схематичны → **неверный источник**.
   - **По аксонометрии считаются только ВЫСОТНЫЕ длины** вертикальных участков (верхняя отметка − нижняя отметка).
   - Горизонтальные длины — только с плана.
- Σ метража по Ду из ВОР — **самопроверка, не эталон**; расхождение Σфакт vs ВОР — норма, **не подгонять**
  (тонкий момент — проговаривать, не чинить молча).
- ⚠ Другой том (напр. Р-4) — НЕ эталон методики; опираться на это правило, не копировать чужой том.

## 4. Фасонка (фитинги) ✅

- **Фасонка считается по АКСОНОМЕТРИИ** — она даёт наглядность опусков/подъёмов трубы.
  Пример: опуск «Г» = **2 отвода** соответствующего Ду + вертикальный опуск N м.
- **Переходы Ø (►◄)** — по местам смены диаметра (переход = точка ►◄ на схеме).
- Именно для этого аксонометрия и нужна (наряду с высотными длинами, §3): **не** для линейных размеров,
  а для узлов / фасонки / отметок.

## 5. Арматура

- Запорная/спец. арматура — по ВОР (наименование + Ду + количество), сверяется со схемой
  (символы на слое арматуры). Расхождение «на схеме меньше, чем в ВОР» = замечание, добор на схему.

## 6. Железные правила ✅

- **Факт первичен** — сперва проверить, актуально ли замечание/расхождение (часть могла быть сделана).
- **Не подгонять под сходимость** — расхождение факт ↔ проект это РЕЗУЛЬТАТ сверки, а не ошибка.
- **Любой вопрос по количеству — проговаривать с пользователем** (откуда цифра, надёжность источника,
  где несостыковки). Не выдумывать, не брать из головы.

## 7. Общее vs специфика корпуса

- Часть решений — специфика конкретного корпуса (маркер «⭐ специфика <корпуса>», напр. Т4 в ИТП у Молота).
  Не переносить специфику на другие тома без подтверждения пользователя.

## 8. Каскад правки состава + инженерная рамка ✅

- **Инженер, не робот.** Прежде чем править объём — понять узел (сифон — на каждый приёмник стоков;
  гибкая подводка — на каждый подвод воды к смесителю, обычно 2: ХВС+ГВС). Правка должна быть физически
  осмысленной и проверяемой по факту, а не «добавить, потому что в ВСО есть».
- **Порядок каскада при правке состава:** Акт → ВСО → ВОР → ПВ → сравнительный («Было-Стало») →
  Реестр-опись → таблицы на DWG-схемах.
- **Контрольные суммы** из ВСО (напр. сифоны N, подводки M) обязаны совпасть в ВОР и ПВ (инвариант).
- **Data Link в DWG НЕТ** — после правки ВОР таблицу/ведомость на исп. схеме перевставлять вручную
  (OLE → нативная ACAD_TABLE).
- **Затронул позицию ВОР/ПВ из чужого блока — не чинить на месте:** регистрировать блокер, устранять
  в профильном блоке (правило `[[feedback_user_rules_docs_cascade]]`).

## 9. Процесс ИД по срезам (рамка работы) ✅

> Полный процесс — в `blocks/pto/DESIGN — процесс ИД по срезам (2026-07-05).md`. Здесь — свод, чтобы не терять рамку.

- **Суть:** не «горизонтальными слоями» (все схемы тома → все акты → вся ПВ, с прыжками и потерями),
  а **вертикальными срезами** — берём единицу, доводим **от и до**, фиксируем, следующая. Локальный
  контекст, монотонный прогресс, повторяемость.
- **Единица среза** = акт: ВиВ-1 оборудование · ВиВ-3 трубопровод · ВиВ-4 теплоизоляция · ВиВ-5 заделка.
  Объёмы среза — из ВОР-на-схеме + ПВ по актам. АОУСИТО — общий акт (объёмов нет), ведётся сквозняком.
- **Конвейер среза — 11 шагов** (детали DESIGN р.5): 1) инвентарь акта (работы/объёмы/приложения) →
  2) добор недостающих документов → 3) акт (даты/СП/материалы/подписанты/оформление) → 4) сверки дат →
  5) реестр под акт → 6) наименования документов с предметом → 7) графика DWG (ВОР/выноски/штампы/зум,
  схема с объёмами) → 8) оформление приложений → 9) дозапись в накопители (инвариант ПВ≡ВСО≡ВОР) →
  10) фиксация PDF (замена страниц; изъятие = решение пользователя) → 11) гейт среза (полнота + порядок + виджет).
- **3 сквозных накопителя:** ПВ · Реестр-опись · АОУСИТО (+ реестр замечаний). Дозапись после каждого
  среза; финал и перевод в PDF — в самом конце.
- **Подводные камни (ямы)** ведутся в DESIGN р.12: верифицируемое-первым (правила дат/пагинации уже в
  методичке), полигон = копия чистого оригинала, исходники не дублировать, виджет живой (не снимок),
  файловая гигиена, грабли `Move-Item` со спецсимволами. **Новую яму — дописывать в DESIGN.**

## 10. Документация качества на позицию ✅

- **На КАЖДУЮ позицию, которую закрывает акт (= каждую позицию ВОР), должен быть документ качества —
  хотя бы какой-то.** Нет документа = разрыв = замечание ТН.
- **Стандартный комплект на позицию:**
  - **Паспорт** — широкое понятие: под него подходит масса документов с разными/похожими названиями
    (паспорт изделия, технический паспорт, руководство/инструкция по эксплуатации, техническое описание,
    информационное письмо и т.п.). Хотя бы такой документ — обязателен.
  - **+ Сертификат соответствия или декларация ЕАЭС — ЕСЛИ ТРЕБУЕТСЯ** (по ПП РФ №2425 / ТР ТС).
    - Обязательное подтверждение НЕ требуется → **отказное письмо**.
    - Контакт с питьевой водой → **СГР** (РКТС 299).
- Требуется ли СС/декларация и по какому основанию — **уточнять через `norm-lookup`**, не по памяти.
- **Складирование:** папка «Паспорта и сертификаты» → подпапка «под что прикладывается» → имя файла =
  материал/изделие + тип документа (правило `[[feedback_user_rules_docs_cascade]]`).

## 11. Правила дат (документ ↔ акт, приказ ↔ акт) ✅

- **Документ качества действует на дату акта:** дата **окончания** действия — **не раньше** начальной
  даты акта; дата **начала** действия — **не позже** последней (главной) даты акта. Даты «с-по» акта —
  на его 2-й странице.
- **Не привязываться к формулировке:** бывает «с/по», «с/до» и др. — смотреть смысл, не букву.
- Документ с **одной датой** (паспорт / сертификат качества / инф. письмо) и **отказное** — без срока;
  важна лишь **дата выдачи ≤ дата акта**.
- **Приказы** о назначении ответственных — действующая редакция на дату работ (приказ **не позже** даты
  акта); № и дата приказа сверяются со **сканом-первоисточником**, не по трекеру/памяти.
- Действительность считать **по дате акта, не по сегодня.** Детали — `[[feedback_user_rules_docs_cascade]]`
  + методичка `Обучение составления ИД.md` р.26.

## 12. Дашборд / виджет — постоянная актуализация ✅

- После **каждого изменения** статуса **или каскада** изменений — **актуализировать ВСЕ вкладки виджета.**
- Механика: правки в `journal.json` (данные-правда) → `python build_status.py` (регенерация HTML).
  **HTML руками НЕ трогать** (перезатрётся).
- Вкладки: **Статус тома · Карта акта · Замечания ТН** (детали DESIGN р.8).

## 13. Несколько устройств — ЖУРНАЛ СЕССИЙ обязателен ✅

- Над рабочими папками объекта работают **несколько устройств** (ПК, ноутбук) в разное время;
  проектная память Claude — per-PC, база синкается с задержкой → координация в общем файле
  **`<корень объекта>\Claude\ЖУРНАЛ СЕССИЙ.md`**. Сессии стартуют из папки `Claude\` объекта —
  журнал прямо в cwd; локальные правила — **`Claude\CLAUDE.md`** (в cwd, читается гарантированно);
  в корне объекта — CLAUDE.md-указатель (страховка для запусков из других подпапок).
- ⚠ **Корень объекта на каждом устройстве СВОЙ** (Я.Диск-синк: на одном `F:\…`, на другом
  `C:\Users\…\Yandex.Disk\…`) → пути в журнале/заметках/промптах писать **относительно корня
  объекта**, не абсолютными.
- **В начале сессии** — прочитать верхние 2–3 записи журнала; **в конце сессии / перед handoff** —
  дописать свою запись СВЕРХУ (дата · устройство · тема · сделано · файлы · что следующей сессии).
  **Сессия без записи = незакрытая.**
- Локальные правила папки — в корневом `CLAUDE.md` объекта (читается автоматически из любой подпапки).
- Я.Диск: перед работой после другого устройства дать синку доехать; облачные плейсхолдеры
  гидратировать перед чтением/правкой.
