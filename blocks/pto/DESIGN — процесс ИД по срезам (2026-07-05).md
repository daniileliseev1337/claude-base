# DESIGN — процесс закрытия тома ИД по срезам (2026-07-05)

> Статус: **черновик на согласование** (HARD-GATE brainstorming). Агентов не создаём до утверждения.
> Замена «горизонтального» осевого прохода на **вертикальные срезы**. Полигон обкатки — **копия** тома Молот/Треугольник (оригинал не трогаем; сравниваем со старым грязным результатом).
> Идеи осевого governance переупакованы во **вход → конвейер → выход**; сама папка `axes/` — в архиве блока (`_archive/axes/`, 2026-07-06), источником правил не считать. Свод правил практики — `ЗНАНИЯ ПТО — практика подсчёта и источники факта.md`.

## 1. Суть разворота

Было: горизонтальные слои — «все схемы по тому» → «все акты» → «вся ПВ». Отсюда прыжки ось4→ось1→ось3, «потерялось, кто где», грязный результат.

Стало: **вертикальный срез** — берём единицу (акт), доводим *от и до*, фиксируем, следующая. Локальный контекст, монотонный прогресс, повторяемость → **агента можно довести до идеала**.

## 2. Полная рамка (оси переупакованы)

```
ВХОД:   Замечания ТН (ось Р)  +  Анализ существующей версии тома (ось Б)
   ▼
СРЕЗЫ ×N (оси 1–5):  конвейер из 11 действий   ← анализ графики/ВОР/DWG питает шаг 1/7
   │   сквозное: 3 накопителя · реестр замечаний · журнал/виджет
   ▼
ВЫХОД:  финал накопителей → PDF → Приёмка готового тома как ТН (ось 6, skill id-tom-priemka)
```

## 3. ВХОД

### 3.1 Замечания ТН (ось Р) — драйвер всех изменений

**Три канала от ТН** + отдельный тип:
| Канал | Как снимаем | Риск |
|---|---|---|
| 1. Общий список с подписью (обобщённо: «по схемам/актам») | текст-парсинг | — |
| 2. Стикеры-наклейки на листах | аннотации PDF / vision если скан | текст-слой не видит |
| 3. **Рукописные пометки поверх листа** (черкает у места, зачёркивает на акте, вопросы) | **рендер листа → vision** | **текст-слой слеп → «терялись»** |
| + тип `наша находка` | из анализа тома/срезов | смешать с ТН |

Правила слоя: каждое замечание фиксируется **дословно** (для рукописных — вырезка изображения пометки, не только текст) + **расшифровка рядом** (как поняли — отдельно от оригинала, против «фантома пересказа»). Всё в **реестр замечаний** → распределяется по срезам/актам → трассируется к изменениям (двусторонняя увязка замечание↔изменение). Обобщённое замечание из списка увязывается со своими стикерами/пометками.

### 3.2 Анализ существующей версии тома (ось Б) — **вариант А, гибрид**

- **Лёгкий структурный скан всего тома** на входе: карта (где какой срез, границы), межсрезовые дубли/пропуски, общая логика.
- **Глубокий разбор — перед каждым срезом** (в конвейере).
- Любая находка сразу → реестр замечаний (тип `наша находка`), чтобы не терялась независимо от уровня.

## 4. Единица среза

- **ВиВ‑N акты** (ВиВ‑1 оборудование, ВиВ‑3 трубопровод, ВиВ‑4 теплоизоляция, ВиВ‑5 заделка) — **закрывают материалы/объёмы**. Приложения: паспорта, сертификаты, схема с объёмами. Объёмы только из двух мест: **ВОР на схеме + ПВ по актам**.
- **АОУСИТО** — общий акт, **объёмов нет**. Приложения: общие схемы (без объёмов) + акты испытаний. Ведётся **сквозняком**: рамка → дозапись после каждого среза → финал в конце.

## 5. Конвейер среза (последовательный, не fan-out)

Каждое действие — своя сущность: **агент** (суждение/веб), **скрипт** (детерминизм), **гейт** (проверка). Цепочкой; параллелить можно только внутри шага (напр. добор N документов). Срезы между собой — тоже по одному (накопители сквозные).

| # | Действие | Сущность | Закрывает изъян |
|---|---|---|---|
| 1 | Инвентарь акта — работы, объёмы (ВОР‑на‑схеме + ПВ), перечень приложений | агент + скрипт‑экстрактор | — |
| 2 | Добор недостающих документов качества | агент (веб/каталог) | — |
| 3 | Акт: составить / править / проверить — даты, СП, материалы, подписанты, оформление | агент | — |
| 4 | Сверки дат — акт↔документы, приказы↔акт (по записанным правилам) | скрипт | — |
| 5 | Реестр под акт — вести; ветка: не заводится → сверка+правка акта по факту | агент + скрипт | — |
| 6 | Наименование документов с предметом | агент | реестр без предмета |
| 7 | Графика — анализ DWG/PDF: ВОР, выноски, штампы, **правильное зуммирование**; схема ВиВ (с объёмами) / общая → АОУСИТО | агент + скрипт | scramble, текст‑слой А3 врёт |
| 8 | Оформление приложений — порядок, единый формат | скрипт + гейт | жирность вразброс |
| 9 | Дозапись в накопители (ПВ / АОУСИТО / реестр‑опись), инвариант ПВ≡ВСО≡ВОР, diff по ячейкам | скрипт | рассинхрон ПВ↔ВСО |
| 10 | Фиксация — PDF замена страниц, учёт в журнал (изъятие = решение пользователя) | скрипт + решение юзера | самовольное удаление; карта=страницы≠позиции |
| 11 | Гейт среза — полнота + порядок папок + обновить виджет | гейт | ложный PASS на сканах; «потерялось» |

## 6. Сквозные накопители + пагинация

`ПВ` · `реестр‑опись` · `АОУСИТО` + `реестр замечаний`. Дозапись после каждого среза, финал и перевод в PDF — в конце. **Реестр‑опись = агрегация** реестров‑под‑акт + схемы (скрипт).

**Пагинация — скрипт‑нумератор.** Нумерация считается по **живому листу** (двусторонняя печать: лист = лицо + оборот, пустые обороты), **не по странице PDF**. Модель «лист ↔ страницы PDF ↔ двусторонка» формализуется в tools-слой (точную логику печати даёт пользователь). Пересчёт на каждой фиксации, чтобы ссылки реестра/АОУСИТО/виджета не наврали.

## 7. ВЫХОД

Финализация трёх накопителей → PDF (последними) → **приёмка готового тома как ТН** (ось 6, skill `id-tom-priemka`, постранично, opus, PNG). Финальный доменный ревьюер комплекта — `id-engineer` + `auditor` (архитектурный ревьюер-гейт обязателен).

## 8. Учёт изменений — вне PDF (виджет, 3 вкладки)

- **Журнал** (`journal.json`) — источник правды. PDF остаётся чистым.
- **Виджет** (`status.html`, автономный, открывается без Claude — в т.ч. печатником):
  1. **Статус тома** — прогресс срезов, три накопителя, лог, статус папок.
  2. **Карта акта** — фильтр по срезу → состав по порядку подшивки: файл+путь, страницы в томе, формат листа (А4/А3), статус, блок «добрать/откуда/куда вставить».
  3. **Замечания ТН** — № замечания, где написано, расшифровка, наше решение; тип (ТН / наша находка). На вкладках изменений — **перекрёстная метка** «относится к замечанию №…». Замыкает контур: замечание не теряется, висит с открытым статусом до закрытия.

## 9. Структура папок (по актам, зеркалит срезы)

```
<Объект> — <Том>/
├── 00_Исходники/            ПВ, реестр-опись, DWG, схемы, приказы (стартовое сырьё)
├── 01_ВиВ-1_оборудование/
│   ├── Акт/  ├── Схемы/  ├── Документы качества/  └── Реестр/
├── 02_ВиВ-3_трубопровод/   (та же структура)
├── 03_ВиВ-4_теплоизоляция/
├── 04_ВиВ-5_заделка/
├── 05_АОУСИТО/              общие схемы, акты испытаний
├── 06_Журнал/              journal.json + status.html
├── 07_Замечания/           исходные замечания ТН (список + скрины стикеров/пометок)
└── _К ПЕЧАТИ <дата> (<Объект> <Том>)/     финал
```

Файловая дисциплина — **жёсткое правило**: структура заводится в начале, ведётся параллельно; гейт (шаг 11) не закрывает срез при бардаке.

## 10. Переиспользование существующего

- `id-engineer` (глава семьи) — финальный доменный ревьюер + `auditor`.
- `id-tom-priemka` (skill, оси Б/6) — вход (скан) и выход (приёмка).
- `id-volume-cascade`, `id-scheme-cascade` + `id-scheme-toolkit`, `id-volume-graph` — по профильным шагам.
- Правила сверки дат (шаг 4) — из существующих записанных, не выдумывать (источник укажет пользователь).
- OCR/vision рукописных пометок — `image-text-replace`; извлечение по типу листа — `doc-extract`; сверка ВОР↔СО — `co-verify`.

## 11. Полигон и открытые вопросы

- **Полигон:** копия Молот (Р‑1) / Треугольник (Р‑4). Оригинал не трогаем.
- **Состав/границы агентов** финализируются из ПЕРВОГО прогона (не наперёд — «агенты из практики», BLOCK.md).
- Открыто до старта: где ведём рабочие ПВ/реестр‑опись; точная логика нумерации/печати (для скрипта‑нумератора); спека карты акта (цветность/итог форматов/двусторонняя/готовность).

## 12. Ямы из обкатки (2026-07-05, срез ВиВ‑1 на копии Молота — дополняется)

- **Правила входа уже записаны — не спрашивать вслепую (verifiable‑first).** Сверка дат (шаг 4) =
  методичка `Обучение составления ИД.md` р.26 (действительность по ДАТЕ АКТА, не today; период «с‑по» /
  одна дата / отказное). Пагинация (шаг 10/пагинация) = р.22 + р.6.4 (двусторонка 1 лист=2 стр.; листаж
  по типам; `G[n]=F[n-1]+G[n-1]`; опись — обложка‑вкладыш вне листажа тома). DESIGN ссылается на эти
  разделы, а не «источник укажет пользователь».
- **Голосовой канал ТН — ПРОВЕРЯТЬ наличие речи, не предполагать.** Видео обхода могут быть без
  закадрового пояснения (чистая визуальная фиксация пометок/помещений). Метод: VAD; при 0 сегментов
  контроль `vad_filter=False` + фильтр галлюцинаций Whisper («Продолжение следует…», «Спасибо за
  просмотр», «Субтитры…») = маркер тишины. Если речи нет — видео = **визуальный** канал (кадры), не
  транскрипт. Инструмент: `local-video-digest` + **PySceneDetect** (harvest 2026-07-05; кадры по смене
  сцены, fallback интервал при <3 сцен).
- **Грабли ASR на пилот‑машине:** GPU‑путь faster‑whisper мёртв (`cublas64_12.dll` не загружается) →
  `device=cpu, compute_type=int8, model=large-v3`. Держать в tools‑слое скилла (`--device cpu`, опция VAD).
- **Полигон = копия ЧИСТОГО оригинала (как сдано ТН), не осевой результат.** Осевую пересборку держать
  только для финального сравнения качества «грязный vs по срезам».
- **Исходники не дублировать физически.** Многопозиционный ВСО (десятки позиций), десятки документов
  качества, крупный акт‑docx — работать на месте в копии тома; структура среза (`01_ВиВ‑N/{…}`,
  накопители, `07_Замечания`) хранит рабочие артефакты и реестр, а `00_Исходники` = навигатор на исходники.
- **Разбор ВСО как якорь среза оборудования:** позиции ВСО (наименование/марка/завод/кол‑во/помещение)
  сразу дают инвентарь (шаг 1) и увязку с замечаниями качества — читать ВСО первым в срезе ВиВ‑1.
- **Виджет ДОЛЖЕН быть «живой» (авто), не снимок.** Реализация: `journal.json` (данные‑правда) +
  `build_status.py` (генератор: json + скан папок → перерисовывает status.html, 0 ручной правки HTML) +
  **SessionStart‑хук** (при старте сессии регенерит виджеты всех томов). Правишь journal.json → перегенерил.
  HTML руками не трогать. Три вкладки — см. р.8.
- **Вкладка «Замечания» — по актам с дублированием сквозных.** Сквозные (даты, должность тех.заказчика,
  «правки карандашом», предъявить, ПВ) повторяются в КАЖДОМ применимом акте (селектор акта → его замечания).
  Нагляднее: видно, что править в конкретном акте. Замечания собираются из ВСЕХ каналов (список/стикеры/
  рукописные 01‑11 фото/видео‑визуал/наши находки/данные пользователя) — не только оборудование.
- **Файловая гигиена копии‑полигона (боль пользователя).** Копия тома = дубль всей осевой папки со всем
  хламом; НЕ вести артефакты вперемешку в корне. Сразу раскладка по структуре: 00_Исходники (всё сырьё,
  вкл. Молот вода/канализация), 01‑05 срезы, 06_Журнал (виджет+journal), 07_Замечания, `_АРХИВ осевого`
  (старое — переместить, НЕ удалять). Классификацию старое/исходник/новое показать пользователю до раскладки.
- ⚠ **Грабли перемещения файлов (инцидент 2026-07-05):** `Move-Item` со спецсимволом в имени (`—`, `→`)
  может молча не сработать; НИКОГДА не делать `Remove-Item -Recurse` после Move без проверки, что источник
  реально пуст (потеряли реестр+справку, восстановили из контекста). Правило: только Move с per‑файл
  Test‑Path; удаление — точечное, не рекурсивное; спецсимволы (`→`) в именах не использовать.
