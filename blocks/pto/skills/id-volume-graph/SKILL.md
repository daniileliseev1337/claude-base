# id-volume-graph — граф устройства тома ИД

Курированный граф знаний (формат graphify) о том, **КАК сшит многоактный том
исполнительной документации** инженерной системы. Методология, **обезличено** (без
объектов/шифров/ФИО/артикулов — только типы и зависимости). Источник — методичка
ИД (разд. 1–26) + карта эталонного тома, сверено пользователем и проверено `auditor`.

## Зачем
ИД по природе — паутина зависимостей. Вместо длинной «простыни» в промпте id-engineer
**спрашивает граф**: что тянется к акту, какой каскад правок, где разрыв замечания ТН.
Граф **редактируемый** (правишь `id_volume_extract.json` → пересборка) и **queryable**
оффлайн (BFS/path/explain, без облака и без локальной LLM).

## Как спрашивать (оффлайн, из любого cwd)
```bash
python ~/.claude/blocks/pto/skills/id-volume-graph/tools/id_graph.py query   "<слова из меток узлов>"
python ~/.claude/blocks/pto/skills/id-volume-graph/tools/id_graph.py path    "Позиция (оборуд./материал)" "Документ качества"
python ~/.claude/blocks/pto/skills/id-volume-graph/tools/id_graph.py explain "Каскад изменений"
```
- `query` матчит по **меткам узлов** (подстрока+IDF) → BFS-обход соседства. Спрашивать
  словами модели (АОСР, ВСО, ПВ, реестр, документ качества, замечание, каскад, норма),
  не инстанс-терминами (конкретные гильзы/артикулы в графе НЕТ — это методология).
- `path A B` — кратчайшая цепочка зависимостей между двумя узлами.
- `explain X` — все связи узла + источник + сообщество.

Полезные запросы id-engineer:
- `query "АОСР документ качества реестр схема норма"` — что нужно к акту скрытых работ.
- `query "замечание разрыв требования заказчика"` — типология замечаний ТН (разрывы).
- `explain "Каскад изменений"` — что потянет правка состава.
- `path "Спецификация РД/СО" "ВСО (вед. смонтир. оборудования)"` — происхождение ведомостей.
- `query "инвариант согласованы ПВ ВСО ВОР ИС"` — что должно совпадать (узел инварианта).

> `path`/`explain` берут **точную метку узла** (копируй из вывода `query`, со спецсимволами
> матч ненадёжен) — для них надёжнее короткие чистые метки («Каскад изменений», «Документ
> качества»). `query` устойчив к формулировке (подстрока+IDF) — это основной интерфейс.

## Модель (кратко)
- **Том** = N комплектов (вокруг актов) + 3 прошивки (ПВ, Реестр-опись, Журнал работ) + орг.часть + титул.
- **Цепочка комплекта:** Журнал →освидетельствует→ Акт →(п.3)→ Реестр №X →опись→ Документы качества; Акт →(п.4)→ ИС.
- **Типы актов:** АОСР (скрытые) · АОМР (оборудование) · АОУСИТО (главный, агрегирует акты + протоколы; свои общие схемы).
- **Ведомости:** ПВ — master; ВСО/ВОР — её проекции; **инвариант ПВ ≡ ВСО ≡ ВОР ≡ ИС** (нарушение = разрыв = замечание ТН).
- **Замечание ТН** = разрыв требование↔факт (6 типов разрывов).
- **Каскад:** правка состава → Акт→Реестр→ВОР→ВСО→ПВ→Реестр-опись→таблицы DWG.
- Принципы: факт первичен · не выдумывать/артикул по каталогу · унификация · пересборка PDF в конце · двусторонняя печать (1 лист = 2 стр PDF) · Data Link ушли.

God-узлы (хабы): Документ качества · Акт · ИС · Позиция · Замечание ТН · Требования заказчика.

## Редактирование (граф живой — дополнять по мере новых томов/нюансов)
1. Правишь `~/.claude/blocks/pto/skills/id-volume-graph/id_volume_extract.json` (узлы/рёбра/гиперрёбра,
   формат graphify extraction; `confidence`=EXTRACTED для явной методологии, INFERRED для синтеза).
2. Пересборка: `uv tool run --from graphifyy python ~/.claude/blocks/pto/skills/id-volume-graph/tools/rebuild.py`
3. Визуал (из папки скилла): `graphify export html` → `graphify-out/graph.html`.

## Артефакты
- `graphify-out/graph.json` — граф (queryable).
- `graphify-out/graph.html` — интерактивная визуализация (браузер, без сервера).
- `graphify-out/GRAPH_REPORT.md` — god-узлы, мосты, сообщества.
- `id_volume_extract.json` — редактируемый исходник.

## Границы (важно)
- Это **методология** (как устроен том вообще), НЕ инстанс конкретного объекта.
  Факты проекта (шифры, артикулы, количества, реквизиты) — в `FACTS.md` проекта и
  исходниках, НЕ сюда (граф обезличен и таким остаётся).
- Граф отражает **исправленную** модель: ИС 1:1 со своим актом (не шарятся), у АОУСИТО
  свои общие схемы (методичка §26 ошибочно утверждала шаринг — здесь исправлено).
- Нормы (СП/ГОСТ/РД) граф называет как методологические ссылки; точные цитаты/редакции —
  всегда через `norm-lookup`, не по графу.
- Пока охват = **ВиВ** как эталон; расширять на ОВ/ЭО/СС по мере работы.
