## Зона блока

Каскады задач ПТО: объёмы → ВОР → спецификации → исполнительная документация → том ИД.
Поэлементная проработка каскадов: где скрипт, где агент, где субагент — на боевых
задачах хаба, затем шлифовка через feedback.

Процесс доведения тома ИД — **«по срезам»**: `DESIGN — процесс ИД по срезам (2026-07-05).md`
(вход → конвейер 11 шагов → выход). Свод правил практики — `ЗНАНИЯ ПТО — практика подсчёта
и источники факта.md` (постоянный README-знаний; новые правила — через согласование с пользователем).
Прежний governance-слой осей — **АРХИВ** (`_archive/axes/`), источником правил не считать.

Осевой governance — **АРХИВ** (2026-07-06, `_archive/axes/`; не читать как правила). Текущий процесс —
**вертикальные срезы**: `DESIGN — процесс ИД по срезам (2026-07-05).md` (вход: замечания ТН + анализ

## Семья доменных агентов ИД (план + решение 2026-06-21)

Задумка (черновик 2026-06-08 «семья ИД-агентов вокруг графа») — siblings вокруг графа `id-volume-graph`, оркестрирует main по named chain (НЕ вложение друг в друга — лимит 2 уровня):

- **id-engineer** (ЕСТЬ) — акты/журналы/реестры/опись. **Глава семьи: пишет своё + финальный доменный ревьюер комплекта** (согласованность всего тома + работа других агентов), запускать на **сильной модели** (opus/fable). Это доменный ревьюер 2-го уровня (main → генератор → id-engineer-ревью), лимит глубины не нарушает.
- **id-remarks** (новый) — разбор/ведение замечаний ТН (распознать фото/видео, классифицировать по графу, реестр + ответы ТН).
- **id-cascade** (новый) — каскадная правка состава (ВСО/ВОР/ПВ, инвариант ПВ≡ВСО≡ВОР≡ИС, журнал каскада, грабли openpyxl merge/9pt в tools-слой).
- **id-docs** (новый) — документы качества (инвентарь от собранного PDF, добор из каталогов, артикул↔каталог, унификация между строениями).
- **id-count** (позже) — подсчёт позиций по ИС (метод по PDF/цвету не финализирован).

**РЕШЕНИЕ 2026-06-21 (пользователь): создание новых агентов ОТЛОЖЕНО.** Сначала отладить полное закрытие тома на практике (подводные камни ещё вылезают на пилотных томах), и формализовать процессы в агентов, когда они устоятся. Согласуется с принципом ИДЕИ «не плодить агентов спекулятивно — выделять, когда задача доказала, что тяжёлая и повторяемая». **Триггер к созданию:** процесс закрытия тома пройден до сдачи ≥1 раза, грабли каждого слоя задокументированы (идут в tools-слой соответствующего агента). Источник задумки (обезличенный) — архив объекта-пилота (ИДЕЯ семьи), методология — skill `id-volume-graph`.

## Разграничение триггеров (защита от роутинг-коллизий)

- «посчитай объёмы / ВОР / спецификация» → `pto-engineer` (как сейчас, вне блока).

- «измени состав / приведи в соответствие / инвариант ПВ≡ВСО≡ВОР» → `id-volume-cascade`.
- «таблицу ВОР в чертёж / перевставить ведомость в DWG / OLE не правится / не вмещается в лист»
  → `id-scheme-cascade` (W-слой, skill `id-scheme-toolkit`).

- «прими том / приёмка тома / проверь постранично / независимый аудит оригинала» → skill `id-tom-priemka` (оси Б и 6; Workflow, opus, PNG).
