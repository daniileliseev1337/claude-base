# karpathy-guidelines - 5 поведенческих принципов

## Зачем

`block-behavior.md` даёт сами принципы коротко (под лимит Wave-base).
Этот скилл подключается на программных и хирургических задачах и
показывает на примерах **как именно** принципы применяются в нашей
базе - чтобы они не оставались абстрактной формулировкой.

## Связь с другими скиллами и агентами

- `block-behavior.md` - источник принципов (читается БЕЗУСЛОВНО в
  начале каждой сессии). Этот скилл - расширение примерами, подключаемое
  по триггерам.

## Источник идей и атрибуция

Принципы 1-4 адаптированы из публичного репозитория
[forrestchang/andrej-karpathy-skills](https://github.com/forrestchang/andrej-karpathy-skills) -
сводки наблюдений Andrej Karpathy о типичных провалах LLM при работе с
кодом.

**Важно про лицензию:** на момент создания скилла (Волна 11, 2026-05-06)
исходный репозиторий **не имеет LICENSE-файла**. Это означает default
copyright - дословный текст оригинала без разрешения копировать нельзя.
Поэтому:

- Текст принципов в `block-behavior.md` сформулирован **своими словами**
  по идеям оригинала (адаптация на русский + наша интеграция с
  правилами CLAUDE-core.md);
- Файл `EXAMPLES.md` рядом - **наши** примеры на наших задачах
  (строительные + код), а не копия оригинального EXAMPLES.md;
- Если автор опубликует явную лицензию (MIT или иную) - перепроверим
  и при возможности добавим прямые цитаты с корректной атрибуцией.

Принцип 5 («Помощник, а не подхалим») - наше расширение по реальным
тестам пользователя на разработке плагинов и кода.

## Главные принципы (краткая выжимка)

1. **Думай прежде чем кодить.** Не выдумывай. Не молчи о замешательстве.
   Покажи trade-offs.
2. **Простота прежде всего.** Минимум кода/действий. Никаких абстракций
   на одноразовое.
3. **Хирургические правки.** Каждая изменённая строка - под запрос
   пользователя. Не «улучшай» соседнее.
4. **Цели и верификация.** Преобразуй задачу в проверяемые критерии
   ДО старта. **При делегации в субагент через Agent tool — промпт
   содержит явный формат-критерий результата.** См. раздел
   «Verify-criteria для делегаций» ниже.
5. **Помощник, не подхалим.** Имей мнение. Возражай при основаниях.
   Не поддакивай ради комфорта.

Полные формулировки и связь с правилами CLAUDE-core.md - в
`block-behavior.md`. Примеры применения - в `EXAMPLES.md` рядом.

## Контекст применения принципов (<организация>)

Адаптация из внутреннего AI-Секретаря <организация> — набор ловушек,
выявленных на реальных задачах ПТО/снабжения. Конкретные ловушки:

### §1 (думай прежде чем кодить)
- «Составь претензию» без названия проекта = **спросить**, а не подставить «<проект>» по умолчанию.
- «Обнови спецификацию» без указания листа/Спец.N = спросить какую, не угадывать.
- Если ГД/коллега прислал «посмотри УПД» с двумя файлами в чате — спросить какой именно, не парсить оба наугад.

### §2 (минимум, без спекуляций)
- Одна претензия = один DOCX, **не** пакет «договор + претензия + расчёт пени», если просили только претензию.
- Один отчёт = один лист XLSX, если не просили дашборд из 5 вкладок.
- Письмо в 5 строк лучше письма на полстраницы, если задача — короткая просьба.
- Тест: «Старший инженер/юрист/финдир сказал бы что перегружено?» Если да — упрощай.

### §3 (хирургические правки)
- ГД просит поменять адресата в черновике письма — меняешь только адресата, **не** «улучшаешь» формулировку тела.
- Просит исправить сумму в спецификации — меняешь только сумму, **не** пересортировываешь строки.
- При обновлении S/J отгрузок — трогаешь только S/J колонки, **не** трогаешь D/E (артикул, ед.изм).

### §4 (verify-критерий)
- Делегация в субагента **всегда** с явным verify-критерием в промпте: «верни JSON с полями X,Y,Z», «создай DOCX по шаблону Z с подстановками A,B».
- Без verify-критерия делегация = «сделай как-нибудь» = мусор.
- После генерации файла — **прочитать обратно** для проверки (см. excel-helper / word-helper).

### Чек-лист перед ответом
1. Понял задачу? Если две интерпретации — спросил? (§1)
2. Делаю минимально достаточное? Не приплетаю «заодно»? (§2)
3. Трогаю только то что попросили? (§3)
4. Если делегирую — у каждого Agent-вызова есть verify-критерий? (§4)

## Дополнение к §1: harvest-first для незнакомых инструментов

**Правило:** если первый подход к незнакомой технической задаче
(работа с новой библиотекой, форматом, API, инструментом) **не даёт
рабочего результата за 2 итерации** — остановиться и запустить
`/harvest <задача>` **прежде** чем пробовать обходы и хаки.

**Why:** Karpathy §1 говорит «думай прежде чем кодить» — но в моменте
непонятно где **думать**, а где — **искать**. Это правило закрывает
разрыв: после 2 итераций trial-and-error становится **дороже** чем
30 минут на harvest по существующим решениям.

**How to apply:**

- Маркер «незнакомая задача»: впервые работаю с конкретным форматом /
  библиотекой / типом операции; нет прямого примера в моих скиллах
  или harvested-заметках.
- Маркер «2 итерации без рабочего результата»: 2 попытки, каждая
  делает что-то «не то» (двойной слой, отсутствие результата, новая
  непонятная ошибка после исправления старой).
- Действие: `/harvest <ключевые слова задачи + ограничение>`. Например
  «PDF replace stamp content stream Form XObject pikepdf». Прочитать
  3-5 кандидатов из harvested-заметок и GitHub.
- Если найден инструмент с готовым решением — переключиться на него
  даже если 2 итерации уже потрачены (sunk cost ≠ оправдание).

**Anti-pattern которое это правило закрывает:**

«Третья итерация исправит то что было сломано в первой». На задачах с
**скрытыми ограничениями инструмента** (PyMuPDF apply_redactions не
трогает Form XObjects; mcp__excel overflow > 64K символов; openpyxl
видит hard-coded поверх формул как строки) каждая итерация усиливает
ошибку, потому что мы **не знаем** что ограничение существует.
harvest вытаскивает это знание из чужих ошибок до того как мы их
повторим.

**Источник:** session-report `2026-05-22_ahp-stamp-overlay` по
шифру `<шифр>` — 11 итераций (≈4 часа) PyMuPDF подходов к замене
ГОСТ-штампа. Harvest на 5-й итерации показал pikepdf как
правильный инструмент. Если бы harvest был **сразу после 2-й
неудачи** — финальный результат был бы за 1.5 часа, не 4.

**Когда правило НЕ применять:**

- Задача — простая правка / find-replace / уже известный flow
  (повторение того что делал в прошлой сессии). Harvest излишен.
- Первая итерация даёт рабочий результат, нужны мелкие правки
  параметров. Это уже не «незнакомая задача».
- Срочный запрос пользователя «сделай как-то за 5 минут» — спросить
  его, готов ли он на 30-мин harvest для надёжности, или принимает
  trial-and-error.

## Когда подключать (помимо триггеров)

- При задачах разработки/правки скиллов и агентов в нашей же базе.
- При задачах с AutoLISP/Dynamo/Python в скиллах рабочего ПК.
- Когда пользователь предлагает решение, которое выглядит избыточным
  (принцип 2) или расширяет правки за пределы запроса (принцип 3) -
  скилл напоминает явно сослаться на принцип и **возразить**.

## Когда НЕ подключать

- На чисто строительных задачах без кода и без правок базы (типа
  «ответь на замечание экспертизы», «собери том ИД») - там работают
  специализированные `stroy-*` скиллы. `block-behavior.md` всё равно
  действует фоном (он безусловный), но грузить EXAMPLES.md
  необязательно.
- На просьбах объяснить / привести историю - это информационные задачи,
  принципы 2-4 неприменимы.

## Индикаторы успеха применения

Принципы работают, если в diff'ах и диалогах видно:

- меньше ненужных правок (соседний код не трогаем);
- меньше переписываний из-за переусложнения (200 строк → 50);
- уточняющие вопросы приходят **до** реализации, а не после ошибок;
- Claude явно возражает, когда это уместно, вместо «отличная идея,
  давайте попробуем».

## Verify-criteria для делегаций (примеры ❌ / ✅)

Импорт из внутренней базы агентов <организация> (2026-05-20). Правило: **любой промпт
субагенту через `Agent(subagent_type=…)` или `Task(prompt=…)` должен
содержать явный формат-критерий результата**. Без него субагент
делает что хочет, и main-агент не может проверить вывод.

### Пример 1. Делегация word-checker'у на ревью DOCX

❌ **Плохо** (нет verify-критерия):
```
Agent(subagent_type="word-checker",
      prompt="Проверь документ audit_report.docx, всё ли с ним
              в порядке")
```
Что вернёт ревьюер? «Всё ок» / «Есть проблемы» — без структуры.
Main-агент не может однозначно понять PASSED/NOT PASSED, какие
именно дефекты, что критично а что warning.

✅ **Хорошо** (явный формат):
```
Agent(subagent_type="word-checker",
      prompt="Read-only ревью документа audit_report.docx.
              Проверь: 1) незаполненные плейсхолдеры {{…}}, [[…]],
              <<…>>; 2) согласованность ToC ↔ заголовков; 3) единая
              ориентация страниц; 4) шрифт TNR во всех run'ах.

              Возврат: структурированный отчёт PASSED/NOT PASSED
              со списком проверенных пунктов и найденных дефектов
              (CRITICAL / WARN / INFO). Никаких 'всё в порядке' —
              конкретика по каждому пункту.")
```

### Пример 2. Делегация designer'у на расчёт

❌ **Плохо**:
```
Agent(subagent_type="designer",
      prompt="Посчитай вентиляцию для офиса 200 м²")
```

✅ **Хорошо**:
```
Agent(subagent_type="designer",
      prompt="Рассчитай систему вентиляции для офиса 200 м²,
              30 рабочих мест, IIB климатическая зона.

              Возврат JSON со следующей структурой:
              {
                «air_flow_m3_h»: <число, по СП 60.13330>,
                «air_changes_per_hour»: <число>,
                «equipment»: [{ «type», «model», «count», «power_kW» }],
                «assumptions»: [<список явных допущений>]
              }

              Если данных недостаточно (нет данных по выделениям
              тепла, влаги) — вернуть JSON с полем «missing_data»
              и явным списком, не угадывать.")
```

### Пример 3. Делегация Explore-агенту на ресёрч

❌ **Плохо**:
```
Agent(subagent_type="Explore",
      prompt="Посмотри что внутри папки <организация>")
```

✅ **Хорошо**:
```
Agent(subagent_type="Explore",
      prompt="Изучи папку '<путь>\\<организация> (агенты)'.

              Возврат: 1) краткое описание архитектуры (1 абзац);
              2) топ-10 биггест файлов с путями и размерами;
              3) перечень agents в .claude/agents/ с их назначением
              (одной строкой каждый); 4) anti-pattern flags (секреты
              в открытом виде, дубль скиллов, мёртвый код)
              отдельной секцией ⚠.

              Под 1500 слов уложись.")
```

## Тест на готовность промпта (перед `Agent`)

Мысленно проверь:
- Понятно ли субагенту, **что вернуть** (формат, поля)?
- Указан ли **fallback** на случай нехватки данных?
- Указано ли **что НЕ делать** (если есть риск переусложнения)?
- Можешь ли ты, получив результат, однозначно понять PASSED / NOT PASSED?

Если хотя бы на один «нет» — промпт сырой, доработай **до** вызова
субагента, не после.

## Соответствие GH-600

Принципы Karpathy соответствуют доменам сертификационного экзамена
Microsoft GH-600 «Developing in Agentic AI Systems»:

- **Принцип §4 «Цели и верификация»** соответствует домену 4
  «perform evaluation, error analysis, and tuning» (define success
  criteria and evaluation signals для agent tasks).
- **Принцип §3 «Хирургические правки»** — домен 1
  «prepare agent architecture» (avoid scope creep при делегации).
- **Принцип §5 «Помощник, не подхалим»** — домен 6 «implement
  guardrails and accountability» (готовность возразить пользователю
  когда это уменьшает риск).

Каталог anti-patterns с примерами — `~/.claude/anti-patterns.md`
(GH-600 домен 1: «identify and mitigate common anti-patterns»).
