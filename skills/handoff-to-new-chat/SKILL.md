# handoff-to-new-chat — передача контекста в новый чат

## Зачем

Длинные продуктивные сессии часто упираются в context window. Когда
контекст близок к лимиту — Claude начинает «забывать» ранние решения,
повторять рассуждения, теряет точность. Лучше **до** этого передать
работу в новый чат с компактным briefing.

Этот скилл — **методология формирования handoff-пакета** и **правила**
когда его активировать.

## Когда подключать

### Пользовательские триггеры (явное намерение)

- «давай в новый чат», «handoff», «передай в новую сессию»
- «контекст полный», «контекст забит», «много нагрузил»
- «закрываем чат, продолжим завтра/потом»
- «дай промпт для нового чата»

### Proactive triggers (Claude сам замечает)

> **⚠ Калибровка под размер context-window.** Пороги ниже (5+ Read, 3+ Agent
> и т.п.) калиброваны под **~200K context** (Sonnet, Haiku, старый Opus).
> Для **Opus 4.7 1M context** шкала растягивается ~в 5 раз — пороги
> срабатывают намного раньше реального CRITICAL.
>
> **Правило для 1M-моделей:** ориентир — индикатор `Context window X / 1.0M (Y%)`
> в Claude Code UI (видит пользователь, не Claude). Эвристики ниже
> использовать как **сигнал дисциплинироваться** (cascade loading, Agent
> для исследований), **не** как автоматический trigger handoff'а. Handoff
> предлагать только при реальном Y > 60% по UI или явном запросе
> пользователя.

Сигналы организованы по **двум уровням утилизации контекста** — warning
и critical. Claude Code напрямую не показывает % использования
context-window для Claude (только пользователю), поэтому без UI-данных
ориентир — эвристические proxy-метрики **с поправкой на размер модели**.

#### WARNING level (~60% утилизация) — **отметить про себя**

Накопление 1-2 из списка → продолжать, но **дисциплинировать** работу
(см. ниже «Дисциплина контекста»). Handoff пока не нужен.

| Признак | Порог WARN |
|---|---|
| Read больших файлов (> 200 строк) | 3-4 за сессию |
| Agent (subagent) вызовов | 2 |
| User turns в сессии | 20-25 |
| Большие Bash/PS outputs (> 200 строк stdout) | 3 |
| Edit/Write разных файлов | 3-4 |

#### CRITICAL level (~70% утилизация) — **proactive handoff**

При **двух и более одновременно** из списка — Claude **сам** через
`AskUserQuestion` предлагает handoff:

| Признак | Порог CRIT |
|---|---|
| Read больших файлов (> 200 строк) | 5+ за сессию |
| Agent (subagent) вызовов с большими ответами | 3+ |
| User turns в сессии | 30+ |
| Большие Bash outputs (> 200 строк stdout/stderr) | 5+ |
| Edit/Write разных файлов | 5+ |
| Параллельные branches задач | 3+ |

Эвристики **неточные** — но несколько признаков сразу — надёжный
сигнал. **Не ждать пока compaction съест ранние решения** — handoff
до critical всегда полезнее compaction после.

> **Заимствовано из GSD Redux** (`/gsd-health --context`) —
> двухуровневая система warning/critical. У них программный счётчик
> токенов, у нас эвристика, но логика та же.

### Когда НЕ подключать

- Сессия короткая (< 10 turns), мало действий — handoff overhead не оправдан.
- Пользователь явно сказал «доделай в этом чате» — респектим.
- Текущая задача в **самой середине** technical step (например в pytest run, mid-Bash) — дождаться завершения step, потом handoff.

## Алгоритм handoff (5 шагов)

### Шаг 1 — Спросить подтверждение (если proactive)

```yaml
question: "Контекст почти полный (X признаков перегруза). Готовим handoff в новый чат?"
header: "Handoff"
options:
  - label: "Да, готовь отчёт + промпт (Recommended)"
    description: "Финализирую текущий step, потом делаю summary и промпт."
  - label: "Не сейчас, доделай задачу"
    description: "Продолжаем — но имей в виду риск падения качества."
  - label: "Закрываем сейчас (короткий handoff)"
    description: "Минимальный summary без детализации, для экстренного перехода."
```

### Шаг 1b — Определить размер контекста (подписка) ⚠ КРИТИЧНО

Размер нового чата зависит от подписки. На **PRO окно ~200K** — подробный
handoff-пакет переполняет новый чат сразу же, он становится неработоспособен.
Поэтому **спросить подписку** (если не известна из памяти/контекста):

```yaml
question: "Какая у тебя подписка? От этого зависит размер пакета для нового чата."
header: "Подписка"
options:
  - label: "MAX x20"
    description: "Большое окно (до 1M на Opus) → подробный FULL-пакет."
  - label: "MAX x5"
    description: "Большое окно → подробный FULL-пакет."
  - label: "PRO"
    description: "Малое окно ~200K → УРЕЗАННЫЙ LITE-пакет, иначе новый чат переполнится."
```

- **MAX x5 / x20** → режим **FULL** (Шаг 3, полный шаблон).
- **PRO** → режим **LITE** (Шаг 3, урезанный шаблон). Опора на disk + cascade.
- Ответ запомнить на сессию (не переспрашивать). Если пользователь уже
  называл подписку — не спрашивать повторно.

### Шаг 2 — Финализировать текущий step

- Закрыть начатые TaskCreate/TaskUpdate.
- Если в working tree есть uncommitted important changes — закоммитить и запушить.
- Если запущен background agent — дождаться или сохранить task-id для resume.

### Шаг 3 — Собрать handoff-пакет

**Выбрать режим по Шагу 1b:**
- **FULL** (MAX x5/x20) — полный шаблон сразу ниже.
- **LITE** (PRO) — урезанный шаблон в конце Шага 3.

#### Режим FULL (MAX x5/x20) — подробный пакет

Структура:

```markdown
# Handoff: [тема] (закрытие сессии <YYYY-MM-DD HH:MM> с <host>)

## TL;DR (3-4 строки)
- Что делали (1-2 строки)
- Что закрыли (1 строка)
- Что осталось (1 строка)

## Состояние артефактов
- Файлы на Desktop / в репо: <пути + 1 строка описания>
- Коммиты в origin/main: <hash + краткое описание>
- Pending tasks (TaskList): <список>

## Открытые вопросы
- Вопрос 1 → нужно решение пользователя или action.
- Вопрос 2 → ...

## Промпт для нового чата

> Привет. Это продолжение сессии "<тема>" с <host>.
> 
> Что было сделано:
> - <bullet 1>
> - <bullet 2>
> 
> Текущее состояние:
> - <файлы / коммиты>
> 
> Что нужно сделать:
> - <task 1>
> - <task 2>
> 
> Контекст лежит в:
> - ~/.claude/session-reports/<YYYY-MM-DD>_<тема>/report.md (полный)
> - <другие relevant файлы>
> 
> Перед началом — STOP-процедура CLAUDE.md (выполнить ПОЛНОСТЬЮ: MCP/agents-сверка,
> роутинг-гейт — обязательны). Если <next step> — доменная/строй-задача и вводные
> неполны, начни с domain-grilling (вводные по одному), потом работай. Затем продолжаем с <next step>.
```

#### Режим LITE (PRO ~200K) — минимум inline, опора на диск

Цель: новый чат стартует с **МАЛОГО** промпта, детали подтягивает с диска
cascade loading'ом по необходимости. Полный `report.md` всё равно
сохраняется (Шаг 4) — но в промпт нового чата тащим только скелет.

```markdown
# Handoff LITE: [тема] (<YYYY-MM-DD HH:MM>)

## Промпт для нового чата (компактный)

> Продолжение сессии "<тема>". Подписка PRO — экономь контекст с первого хода.
>
> Цель: <1 строка>.
> Сделано: <2-3 ключевых пункта, НЕ история>.
> Следующий шаг: <1-2 конкретных действия>.
> Блокеры/решения: <только критичные, 1-2 строки>.
>
> ВСЕ детали — в `~/.claude/session-reports/<...>/report.md`.
> НЕ читать его целиком. Cascade: `Grep "^##" report.md` → `Read offset/limit`
> только нужной секции по мере надобности.
>
> STOP-процедуру CLAUDE.md выполни ПОЛНОСТЬЮ (сверки MCP/agents и роутинг-гейт обязательны;
> «кратко» = объём отчёта ОДНОЙ строкой, НЕ тщательность). Если <next step> доменный —
> начни с domain-grilling. Затем к <next step>.
```

**Что НЕ включать в LITE-промпт** (всё это — только в `report.md` на диске):
- Историю итераций, обоснования решений, отвергнутые варианты.
- Полные списки файлов/коммитов (только 1-2 самых нужных + «остальное в report.md»).
- Подробные открытые вопросы (только текущий блокер).

**Целевой размер LITE-промпта: < 1500 токенов** (~1 экран). Больше — ужимать
дальше и выносить на диск. На PRO лучше «слишком кратко + cascade», чем
«подробно + переполнение».

### Шаг 4 — Сохранить session-report

В `~/.claude/session-reports/<YYYY-MM-DD>_<тема>/report.md` — **полный**
отчёт (не trimmed) **в обоих режимах**. Это страховка: новый чат (особенно
LITE на PRO) подтягивает детали отсюда cascade loading'ом. LITE урезает
только ПРОМПТ, не сам report.md.

Если папка уже существует от предыдущего handoff в той же теме —
**обновить** существующий `report.md`, не плодить новые.

### Шаг 5 — Выдать пользователю

В одном сообщении:
1. **TL;DR** (4 строки)
2. **Промпт для нового чата** (готовый к копи-пасте, в code block)
3. **Указатель** на полный `session-reports/.../report.md`

Не повторять полный текст — пользователь увидит его в новом чате через
session-report. Здесь — только briefing.

## Дисциплина контекста (профилактика перегруза)

Чтобы handoff не понадобился слишком часто — соблюдать:

### 1. Cascade loading — читать только нужную секцию

Не `Read("big_file.md")` целиком на 1000 строк. Сначала **узнать
структуру** (Grep на заголовки, Read первых 30 строк с frontmatter),
потом **точечно** читать релевантную секцию через `Read offset=N limit=50`.

**Пример:**
```
# Плохо — раздувает контекст:
Read("session-report.md")  # 500 строк целиком

# Хорошо — cascade:
Grep "^##" session-report.md  # узнал секции
Read("session-report.md", offset=120, limit=60)  # только нужная секция
```

Этот принцип особенно важен для больших артефактов: session-reports,
PLAN.md, REVIEW.md, отчёты доменных агентов.

### 2. Structured artifacts вместо «простыни» в чате

Для проектов в 3+ фазы — выносить контекст в **структурированные
артефакты на диске**, не держать всё в чате:

| Артефакт | Размер | Зачем |
|---|---|---|
| `ROADMAP.md` | 5-15 KB | Общий план проекта, 1 фраза на фазу |
| `STATE.md` | 1-3 KB | Текущее состояние, что сделано, что осталось |
| `PLAN.md` | 10-40 KB | Детальный план активной фазы |
| `REVIEW.md` | 5-20 KB | Findings с line refs (после audit) |
| `DECISIONS.md` | 5-15 KB | Design decisions + rationale (зачем выбран X) |

Хранить в `~/.claude/session-reports/<...>/` или в cwd рабочего проекта.

**Cascade loading работает на этих артефактах**: агент A читает только
PLAN.md, агент B читает только STATE.md + REVIEW.md, и т.д. — каждый
видит **минимум нужного**.

> **Заимствовано из GSD Redux** — structured artifacts как контейнеры
> контекста. У GSD это формализованные artifacts (ROADMAP/STATE/PLAN/
> REVIEW/DECISIONS/AI-SPEC), у нас — те же артефакты под наш профиль
> (документы вместо кода).

### 3. Read больших файлов через offset/limit

Не `Read("big_file.py")` целиком на 1000 строк. Использовать `Grep`
для поиска нужного места, потом `Read offset=N limit=50`. Это
техническая реализация cascade loading (§1).

### 4. Tool outputs через фильтры

```bash
# Плохо — может вернуть 30000 строк
pip install paddlex[ocr]

# Хорошо — только последние строки
pip install paddlex[ocr] 2>&1 | tail -5

# Ещё лучше — что нас интересует
pip install paddlex[ocr] 2>&1 | grep -E "Successfully|ERROR|conflict"
```

### 5. Длинные исследования через Agent

Если нужно прочитать 5+ файлов или сходить в 3+ места — **делегировать
в `Agent(subagent_type="Explore" или "general-purpose")`**. Subagent
вернёт только summary, основной контекст не раздувается.

**Пример:** «изучить структуру 7 файлов в external repo» → `Agent(Explore)`,
не последовательные WebFetch'и в main thread.

### 6. Не дублировать рассуждения

Если уже описал план — не пересказывать его в каждом ответе.
Ссылаться: «по плану из выше, шаг 3...».

### 7. Background для долгих команд

`run_in_background=true` для процессов > 30 секунд. Они не блокируют
и не наполняют контекст stdout'ом в реальном времени.
