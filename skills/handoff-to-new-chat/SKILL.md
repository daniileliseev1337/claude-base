# handoff-to-new-chat — передача контекста в новый чат

## Зачем

Длинные продуктивные сессии часто упираются в context window. Когда
контекст близок к лимиту — Claude начинает «забывать» ранние решения,
повторять рассуждения, теряет точность. Лучше **до** этого передать
работу в новый чат с компактным briefing.

Этот скилл — **методология формирования handoff-пакета** и **правила**
когда его активировать.

## Когда подключать

### Пользовательские триггеры (явное намерение)

- «давай в новый чат», «handoff», «передай в новую сессию»
- «контекст полный», «контекст забит», «много нагрузил»
- «закрываем чат, продолжим завтра/потом»
- «дай промпт для нового чата»

### Proactive triggers (Claude сам замечает)

При **двух и более** из следующих признаков в одной сессии — Claude
**предлагает** handoff через `AskUserQuestion`:

| Признак | Порог |
|---|---|
| Read больших файлов (> 200 строк) | 5+ за сессию |
| Agent (subagent) вызовов с большими ответами | 3+ |
| User turns в сессии | 30+ |
| Большие Bash outputs (> 200 строк stdout/stderr) | 5+ |
| Edit/Write разных файлов | 5+ |
| Параллельные branches задач (image-text-replace + GH-600 + proxy fix...) | 3+ |

Эвристики **неточные** — context-window напрямую не виден Claude. Но
несколько признаков сразу — надёжный сигнал.

### Когда НЕ подключать

- Сессия короткая (< 10 turns), мало действий — handoff overhead не оправдан.
- Пользователь явно сказал «доделай в этом чате» — респектим.
- Текущая задача в **самой середине** technical step (например в pytest run, mid-Bash) — дождаться завершения step, потом handoff.

## Алгоритм handoff (5 шагов)

### Шаг 1 — Спросить подтверждение (если proactive)

```yaml
question: "Контекст почти полный (X признаков перегруза). Готовим handoff в новый чат?"
header: "Handoff"
options:
  - label: "Да, готовь отчёт + промпт (Recommended)"
    description: "Финализирую текущий step, потом делаю summary и промпт."
  - label: "Не сейчас, доделай задачу"
    description: "Продолжаем — но имей в виду риск падения качества."
  - label: "Закрываем сейчас (короткий handoff)"
    description: "Минимальный summary без детализации, для экстренного перехода."
```

### Шаг 2 — Финализировать текущий step

- Закрыть начатые TaskCreate/TaskUpdate.
- Если в working tree есть uncommitted important changes — закоммитить и запушить.
- Если запущен background agent — дождаться или сохранить task-id для resume.

### Шаг 3 — Собрать handoff-пакет

Структура:

```markdown
# Handoff: [тема] (закрытие сессии <YYYY-MM-DD HH:MM> с DANIILPC/<host>)

## TL;DR (3-4 строки)
- Что делали (1-2 строки)
- Что закрыли (1 строка)
- Что осталось (1 строка)

## Состояние артефактов
- Файлы на Desktop / в репо: <пути + 1 строка описания>
- Коммиты в origin/main: <hash + краткое описание>
- Pending tasks (TaskList): <список>

## Открытые вопросы
- Вопрос 1 → нужно решение пользователя или action.
- Вопрос 2 → ...

## Промпт для нового чата

> Привет. Это продолжение сессии "<тема>" с <DANIILPC>.
> 
> Что было сделано:
> - <bullet 1>
> - <bullet 2>
> 
> Текущее состояние:
> - <файлы / коммиты>
> 
> Что нужно сделать:
> - <task 1>
> - <task 2>
> 
> Контекст лежит в:
> - ~/.claude/session-reports/<YYYY-MM-DD>_<тема>/report.md (полный)
> - <другие relevant файлы>
> 
> Перед началом — STOP-процедура CLAUDE.md, потом продолжаем с <next step>.
```

### Шаг 4 — Сохранить session-report

В `~/.claude/session-reports/<YYYY-MM-DD>_<тема>/report.md` — **полный**
отчёт (не trimmed). Это страховка: если новый чат потеряет нюанс —
можно открыть полный отчёт.

Если папка уже существует от предыдущего handoff в той же теме —
**обновить** существующий `report.md`, не плодить новые.

### Шаг 5 — Выдать пользователю

В одном сообщении:
1. **TL;DR** (4 строки)
2. **Промпт для нового чата** (готовый к копи-пасте, в code block)
3. **Указатель** на полный `session-reports/.../report.md`

Не повторять полный текст — пользователь увидит его в новом чате через
session-report. Здесь — только briefing.

## Дисциплина контекста (профилактика перегруза)

Чтобы handoff не понадобился слишком часто — соблюдать:

### 1. Read больших файлов через offset/limit

Не `Read("big_file.py")` целиком на 1000 строк. Использовать `Grep`
для поиска нужного места, потом `Read offset=N limit=50`.

### 2. Tool outputs через фильтры

```bash
# Плохо — может вернуть 30000 строк
pip install paddlex[ocr]

# Хорошо — только последние строки
pip install paddlex[ocr] 2>&1 | tail -5

# Ещё лучше — что нас интересует
pip install paddlex[ocr] 2>&1 | grep -E "Successfully|ERROR|conflict"
```

### 3. Длинные исследования через Agent

Если нужно прочитать 5+ файлов или сходить в 3+ места — **делегировать
в `Agent(subagent_type="Explore" или "general-purpose")`**. Subagent
вернёт только summary, основной контекст не раздувается.

### 4. Не дублировать рассуждения

Если уже описал план — не пересказывать его в каждом ответе.
Ссылаться: «по плану из выше, шаг 3...».

### 5. Background для долгих команд

`run_in_background=true` для процессов > 30 секунд. Они не блокируют
и не наполняют контекст stdout'ом в реальном времени.

## Структура handoff-пакета — пример

См. `~/.claude/session-reports/_TEMPLATE.md` (если есть) или
`~/.claude/session-reports/2026-05-19_image-text-replace-v3-production/
report.md` — наш «эталон» подробного session-report для сессии после
16 итераций.
