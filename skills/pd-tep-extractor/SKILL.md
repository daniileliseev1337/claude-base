# pd-tep-extractor — ТЭП из ОПЗ в структурированный JSON

## Когда подключаться

- **Перед** любым расчётом сметы / объёмов / спецификации на новом объекте.
  Это default-первый шаг — без ТЭП любой объёмный расчёт делается «на угад».
- При получении нового пакета ПД от заказчика (вытащить ключевые цифры
  один раз, потом ссылаться на JSON).
- Сверка нашего расчёта со сторонней оценкой — нужны базовые цифры из эталона.

## ⚠ Главное правило

**Не угадывать. Никогда.** Реальный инцидент: работа шла на придуманной
цифре апартаментов, в ОПЗ была другая (существенно меньшая). Завышение в
1,5–2,7× по нескольким листам ИОС. Установка: работать только с проверенной
информацией из проектной документации.

**Это означает:**

- Если поле X не найдено в ОПЗ → НЕ подставлять значение из косвенных данных
  (число щитов, стояков, расчёт из общей площади и т.п.).
- НЕ округлять «на глаз».
- В выходном JSON для каждого поля X — либо `{value, cite}` либо
  `{value: null, status: "not_found", action_required: "ask user"}`.

**Указание «работай без уточняющих вопросов» НЕ отменяет это правило.**
«Не уточнять» = не задавать вопросы по неоднозначностям в данных;
**не равно** «угадывать базовые числа».

## Workflow

### Stage 1 — найти ОПЗ в пакете ПД

1. Если пользователь дал прямой путь к PDF ОПЗ → шаг 2.
2. Если дана **папка с ПД** — найти ОПЗ:
   - Файлы вида `*ОПЗ*.pdf`, `*Пояснительная*.pdf`, `*Раздел_1*.pdf`,
     `*Том_1*.pdf`, `*ПЗ*.pdf`.
   - Если нашлось несколько кандидатов → спросить пользователя
     (`AskUserQuestion`), какой брать.
3. Если ОПЗ нет в наборе ПД → **stop, эскалация пользователю**:
   «Раздел №1 (ОПЗ) не найден в пакете. Прислать или работать без ТЭП
   запрещено правилом не-угадывания».

**Verify Stage 1:** путь к ОПЗ зафиксирован, файл — PDF с text-layer.

### Stage 2 — определить тип PDF (text vs scan)

Стандартная проверка из `doc-extract`:

```python
import pdfplumber
with pdfplumber.open(opz_path) as pdf:
    text_total = sum(len((p.extract_text() or "").strip()) for p in pdf.pages)
    n_pages = len(pdf.pages)
is_scan = text_total < 50 * n_pages
```

- **text PDF** → Stage 3 (через `pdf-mcp`)
- **scan PDF** → routing на `image-text-replace` EasyOCR + smart cap,
  потом Stage 3 на полученном тексте + координатах.

**Verify Stage 2:** определён тип, для скана зафиксирована OCR confidence
(если < 0.7 на ключевых страницах — warning).

### Stage 3 — найти таблицу ТЭП в ОПЗ

ТЭП в ОПЗ обычно идёт **в первых 10-30 страницах** в виде таблицы
«Технико-экономические показатели». Стратегии поиска по надёжности:

1. **`pdf_search`** через `pdf-mcp` по ключевым словам:
   `"технико-экономические показатели"`, `"ТЭП"`, `"Основные показатели"`,
   `"Площадь застройки"`, `"количество квартир"`.
2. **Если найдено** — извлечь окружающие страницы (`pdf_read_pages` ±2).
3. **`pdf_get_toc`** — искать раздел «Технико-экономические показатели».
4. **Fallback** — `pdf_read_all` для коротких ОПЗ (< 20 стр) или scan.

**Verify Stage 3:** найдена страница(ы) с таблицей ТЭП, цитата заголовка
зафиксирована.

### Stage 4 — извлечь поля в JSON с cite

**Базовый набор полей ТЭП (минимум):**

| Поле | Типичное название в ОПЗ | Единица |
|---|---|---|
| `object_name` | «Наименование объекта» | строка |
| `address` | «Адрес», «Местоположение» | строка |
| `building_area` | «Площадь застройки» | м² |
| `total_area` | «Общая площадь здания» | м² |
| `living_area` | «Жилая площадь», «Площадь квартир» | м² |
| `construction_volume` | «Строительный объём» | м³ |
| `apartments_count` | «Количество квартир/апартаментов» | шт |
| `buildings_count` | «Количество корпусов/блоков» | шт |
| `floors_count` | «Этажность» | шт (или `{min, max}`) |
| `stages` | «Очерёдность строительства» | список / null |
| `thermal_power` | «Тепловая мощность» | кВт / Гкал/ч |
| `electric_power` | «Расчётная нагрузка» | кВт |
| `water_consumption` | «Расход воды» | м³/сут |

Для каждого поля **в JSON**:

```json
{
  "field": "apartments_count",
  "value": 118,
  "unit": "шт",
  "cite": { "page": 10, "quote": "Количество квартир — 118 шт." },
  "confidence": "high"
}
```

**Если поле не найдено:**

```json
{ "field": "thermal_power", "value": null, "status": "not_found",
  "action_required": "ask user or stop calculation if needed" }
```

**Verify Stage 4:**

- Каждое заполненное поле имеет `cite.page` и `cite.quote` (не пустые).
- Никакого `confidence: high` без точной цитаты в ОПЗ.
- Подозрительные значения — пометить `confidence: low` и в `warnings`.
- Несколько корпусов с разными показателями — `"per_building": [...]`,
  не суммировать молча.

### Stage 5 — sanity checks и эскалация

1. `building_area > 0`, `total_area >= building_area * floors_count`
   (warning если расхождение > 50%).
2. Средняя площадь квартиры = `living_area / apartments_count` в диапазоне
   25–250 м² (warning иначе).
3. `electric_power / apartments_count` — обычно 5–30 кВт/кв (warning иначе).

Проверки **не блокируют** запись JSON — формируют `warnings`.

**Verify Stage 5:** JSON собран, warnings есть либо явное `"warnings": []`.

## Output

- **JSON файл** в cwd: `<объект>-tep.json`.
- **Краткая таблица в чате** для подтверждения пользователем.
- **Запись в `facts-layer` FACTS.md** проекта с ключевыми цифрами + ссылкой на JSON.

## Использование вместе с другими

- **Перед `spec-writer`** для нового объекта — сначала pd-tep-extractor.
- **При проверке сметы** — JSON ТЭП в шапку листа как `источник: ОПЗ стр.<page>`.
- **`designer` агент** получает JSON ТЭП как input.

## Когда вызывать pdf-reviewer

После Stage 4 — спавнить `pdf-reviewer`: «проверь что для значения X на
стр.Y действительно есть такая цитата в ОПЗ» (защита от галлюцинаций cite).

## Ловушки

1. **Разнолифтные здания** — `floors_count` это структура, не число. НЕ усреднять.
2. **Очерёдность строительства** — ТЭП отдельно на каждую очередь, не сливать.
3. **ОПЗ с правками экспертизы** — брать актуальную версию, спросить если непонятно.
4. **Апартаменты ≠ квартиры** юридически — зафиксировать тип, не суммировать.
5. **Площади с/без балконов** — зафиксировать в `cite.quote` или warning.
6. **TOC vs реальные страницы** расходятся — сверять `cite.page` через `pdf_read_pages`.

## Tools (слой 3) — готовые скрипты

Детерминированные проверки вынесены в `tools/tep_validate.py`
(skill-development правило 2) — не переписывать inline:

```bash
python skills/pd-tep-extractor/tools/tep_validate.py detect ОПЗ.pdf        # Stage 2: text|scan
python skills/pd-tep-extractor/tools/tep_validate.py check <объект>-tep.json # cite + sanity (Stage 4-5)
```

`check` возвращает exit 2 и список, если есть заполненное поле без cite
(нарушение главного правила) либо sanity-warnings. Или `import`:
`detect_pdf_type`, `validate_cites`, `sanity_checks`, `validate`.
`detect` требует `pdfplumber` (системный Python, не uv-tool graphify).
