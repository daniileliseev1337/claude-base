# spec-writer — генерация спецификаций по объектам <организация>

## Когда подключаться

- Генерация нового листа «Спец. N» в Excel-реестре объекта.
- Создание отдельного DOCX к договору ЛС («ворд»-файл, например «№ 39 от 12.05.26.docx»).
- Перегенерация спецификации после изменений в позициях (новый артикул, добавилась позиция, etc.).

## ⚠ Терминология

Не путать:
- **«ворд» / «ворд файл»** = **DOCX к договору ЛС** (отдельный документ)
- **«Спец. N»** = **группировка позиций как лист в Excel-реестре**

Если пользователь говорит «сделай ворд для Сит-центра» — генерация DOCX, не xlsx. Если «добавь Спец. 5» — добавление листа в существующий xlsx-реестр.

## Эталон стиля по объекту

| Объект | Эталон стиля | Notes |
|---|---|---|
| Сит-центр | «№ 39 от 12.05.26.docx» (НЕ «№ 36») | Образец стиля = конкретный docx объекта; уточнить путь у пользователя |
| Балашиха | TBD (ждём вводные) | Эталон ещё не зафиксирован — спросить |
| Прочие | спросить эталон у пользователя | Не угадывать |

**Перед генерацией:** если эталон неизвестен — спросить пользователя через `AskUserQuestion` («какой файл взять как образец стиля?»). Это §1 Karpathy — не выдумывать.

## Workflow

### Сценарий A — DOCX «ворд» к договору ЛС

1. Открыть **эталонный файл** (например «№ 39 от 12.05.26.docx») через `word` MCP или `python-docx`.
2. Скопировать как шаблон новой спецификации в cwd с понятным именем (например «№ 40 от 20.05.26.docx»).
3. Подставить:
   - Шапка: номер, дата, контрагент (ЛС), реквизиты
   - Таблица позиций: D (артикул), E (ед.изм), кол-во, цена, сумма
   - Подвал: итог, НДС, прописью
4. **Read-back verify** через [[word-helper]] — проверить незаменённые `{{плейсхолдеры}}`, наличие всех позиций.
5. Сохранить артефакт в cwd сессии (НЕ в `~/.claude/`).

### Сценарий B — добавление листа «Спец. N» в Excel-реестр

1. Открыть существующий реестр объекта через `excel` MCP или `openpyxl`.
2. Определить N — следующий номер после максимального «Спец. K» (грепом sheetnames).
3. Скопировать структуру с предыдущего листа «Спец. (N-1)» (заголовки, шрифт, ширины колонок).
4. Заполнить позиции из входного `list[dict]`.
5. **Соблюдать правила накопления:**
   - **S/J/I/R колонки — пустые по умолчанию** для нового листа (история отгрузок начинается с 0). НЕ копировать значения из предыдущей Спец.
   - **D (артикул) — правило переименования (R):** если позиция уже была в другой Спец и поставщик переименовал артикул → заменить на новый + дописать красным курсивом «(Было <старый>)». Старое имя не теряем — оно нужно для сверки с прошлыми отгрузками/УПД.
6. **Read-back verify** через [[excel-helper]] — `ws.max_row`, формулы, типы.

### Сценарий C — обновление существующей Спец после прихода УПД

1. Парсинг УПД → [[upd-parser]] → JSON позиций.
2. Сверка с текущей Спец: артикулы (D), кол-во (по поз.).
3. Применить правила S/J/I/R (критичные, нарушение = порча истории отгрузок):
   - **S→J зеркало:** если обновился S (отгружено) — сразу обновить J на ту же величину. S и J двигаются синхронно, не оставлять рассинхрон.
   - **Сохранение истории:** S/J/I/R накопительные — сначала **прочитать существующее** значение и **прибавить** дельту, никогда не перезаписывать «с нуля».
   - **«Включить в стоимость»:** формулировка «включить X в стоимость Y» = арифметика по цене G, а НЕ обнуление: G_новая = G_старая + X/E (где E — кол-во ед.). Это не «оставить с G=0».

## Использование helper

```python
import sys
from pathlib import Path
sys.path.insert(0, str(Path.home() / ".claude/skills/spec-writer/scripts"))
from write_spec_xlsx import write_spec_sheet, verify_spec_sheet

# Добавить новый лист «Спец. 5» в реестр Сит-центра
write_spec_sheet(
    workbook_path="Реестр_Сит-центр.xlsx",
    sheet_name="Спец. 5",
    items=[
        {"D": "ABC-123", "E": "м", "qty": 1000, "price": 35.50},
        {"D": "XYZ-456", "E": "шт", "qty": 200, "price": 70.00},
    ],
    template_sheet="Спец. 4",  # скопировать шапку с предыдущего листа
)

# Verify
issues = verify_spec_sheet("Реестр_Сит-центр.xlsx", "Спец. 5", expected_rows=2)
if issues:
    raise RuntimeError(f"Spec verify failed: {issues}")
```

## Tools (слой 3)

Папка `scripts/` — детерминированный слой инструментов скилла (3-й слой стандарта: **Description** «когда» + **Instructions** «как» + **Tools** «чем»). Повторяемая логика записи листа вынесена в код, а не генерируется моделью каждый раз.

- `scripts/write_spec_xlsx.py` — `write_spec_sheet(...)` (запись листа «Спец. N» по шаблону предыдущего) и `verify_spec_sheet(...)` (read-back проверка строк/формул/типов). Подключение — см. блок «Использование helper» выше.

## Output

- **xlsx-реестр** — обновлённый файл в том же месте где был (с новым листом «Спец. N»).
- **DOCX «ворд»** — новый файл в cwd сессии (или указанной пользователем папке). **НЕ в `~/.claude/`** — это артефакт сессии.

## Чек-лист перед сдачей

1. Эталон стиля подтверждён (правильный файл-образец) — §1 Karpathy
2. Делаю только то что попросили — добавил лист, не пересортировал старые позиции — §3
3. S/J/I/R пустые / накопленные правильно
4. Артикулы (D) — если переименование → красным с «(Было ...)», старое имя сохранено (правило R выше)
5. Read-back verify прошёл (через [[excel-helper]] / [[word-helper]])
6. Артефакт сохранён в cwd, не в ~/.claude/
7. (Опционально) спавнить [[excel-validator]] / [[word-checker]] для финальной проверки

## Связь с другими скиллами/правилами

- **Excel часть:** [[excel-helper]] (методология) + read-back verification
- **DOCX часть:** [[word-helper]] (методология); если нужен фирменный бланк <организации> — взять шаблон из самого эталонного docx объекта (отдельного template-файла в базе нет)
- **Парсинг входящих УПД для обновления Спец:** [[upd-parser]]
- **Стили оформления (ГОСТ etc.) если на сдачу:** [[stroy-formatting]]
- **Архивация в Я.Диск:** [[yandex-disk-uploader]] (FileType=CONTRACT для ворда / SPEC для xlsx если такой будет)

## Ловушки

1. **Не перезаписывать историю отгрузок** — критично: S/J/I/R читать и прибавлять, не обнулять (правило «Сохранение истории» в Сценарии C).
2. **Стиль НЕ из «№ 36 от ...»** для Сит-центра — точно из «№ 39 от 12.05.26.docx». Это разные стили, путались в прошлом.
3. **«Включить в стоимость»** ≠ «оставить с G=0» — это арифметика по цене G (правило «Включить в стоимость» в Сценарии C).
4. **При генерации сразу нескольких Спец подряд** (партии) — каждый раз read-back verify, не батчить save без проверки.
