# chains-pattern — методология именованных цепочек

## Зачем

Импортировано из базы К-7 (агенты) — в их CLAUDE.md `chains/` является
**first-class сущностью оркестратора**. Цепочка распознаётся по
триггерам, читается main-агентом перед действием, выполняется
последовательно с verify на каждом шаге. У нас аналогичная папка
`~/.claude/chains/` создана 2026-05-20 (см. отчёт
`~/Desktop/K-7_audit_report.docx` раздел 4.2).

Этот скилл — методология **создания новых** chains.

## Когда подключать

- Пользователь описывает задачу с явной многошаговостью (3+ stages
  с зависимостями).
- Claude замечает что одна и та же последовательность шагов повторяется
  в 3+ сессиях — сигнал формализовать flow.
- При планировании нового сценария который будет переиспользоваться
  командой.

## Когда НЕ подключать

- Одношаговая задача (просто чтение, простая правка).
- Задача из 2 шагов без явных зависимостей.
- Ad-hoc запрос, который вряд ли повторится.
- Если уже есть подходящий chain в `chains/` — использовать его,
  не плодить новые.

## 4 правила создания chain

### 1. Декомпозиция по 4 стейджам

Любой chain раскладывается на 4 stage по К-7-pattern:

- **Stage 1 (collect)** — чтение источников (I/O).
- **Stage 2 (extract)** — структурирование в JSON.
- **Stage 3 (analyze)** — расчёт без I/O (pure function).
- **Stage 4 (act)** — действия в мире (запись, отправка, рендер).

Не все 4 stage обязательны. Минимум 2: collect → act. Stage 3 особо
ценен — даёт детерминированную JSON-функцию, тестируемую через goldens.

### 2. Verify-критерий на каждом шаге

Каждый шаг chain имеет **явный verify-критерий** в формате, который
main-агент может проверить:

- «JSON вернул поля X, Y, Z»;
- «документ не содержит строк {{…}}»;
- «count > 0»;
- «sum > 0 и ИНН валиден (10 или 12 цифр)».

Без verify-критерия шаг = «сделай как-нибудь». См.
[[karpathy-guidelines]] §4 + раздел «Verify-criteria для делегаций»
с примерами ❌/✅.

### 3. Точечный fallback на main при провале

На любом шаге если verify не прошёл — chain **останавливается**,
main эскалирует пользователю. Не пытаться «скрыть» проблему и
идти дальше.

### 4. Именование

Формат: `chain:<verb>-<noun>`. Примеры:

- `chain:docx-from-template` ✅
- `chain:pdf-scan-extract` ✅
- `chain:debt-followup` ✅ (К-7-пример)
- `chain:fix-stuff` ❌ (слишком общо)
- `chain:helper` ❌ (нет verb-noun)

## Структура файла chain

`chains/<name>.md` — markdown с YAML frontmatter:

```yaml
---
chain: <name>
status: active | stub | deprecated
created: YYYY-MM-DD
triggers:
  - «фраза 1»
  - «фраза 2»
related_skills:
  - [[skill-a]]
  - [[skill-b]]
depends_on:
  - [[memory-entry-if-any]]
---
```

Тело:

- **## Назначение** — 1-2 абзаца, что делает chain.
- **## Шаги** — 4 stage (или меньше) с явным verify на каждом.
- **## Когда НЕ использовать** — анти-паттерны.

Опционально: `## История` для трекинга изменений.

## Алгоритм создания нового chain

1. Подтвердить с пользователем что задача действительно повторяемая
   (Karpathy §2 — не создавать «на будущее»).
2. Декомпозировать задачу на stages — проверить, что Stage 3 (no I/O)
   имеет смысл (если нет — это 2-stage chain).
3. Для каждого stage сформулировать verify-критерий.
4. Создать `chains/<name>.md` с YAML и шагами.
5. Добавить запись в `chains/README.md` (таблица).
6. Если используются новые скиллы — обновить cross-references.
7. Протестировать chain в одной сессии — после успешного применения
   запись остаётся active. Если применение показало что декомпозиция
   неверная — переделать на ходу или пометить status: deprecated.

## Соответствие GH-600

Принципы chains-pattern напрямую соотносятся с доменами Microsoft
сертификационного экзамена GH-600 «Developing in Agentic AI Systems»:

- **Домен 1** «prepare agent architecture and SDLC processes» — наша
  4-стейджевая декомпозиция (collect → extract → analyze → act)
  соответствует «define boundaries between planning, reasoning, and
  action» из GH-600.
- **Домен 1** «configure agent to output a structured plan» — наш
  chain в `chains/<name>.md` и есть тот самый structured plan, который
  main-агент читает целиком перед действием.
- **Домен 1** «validate agent plans» — наш verify-критерий на каждом
  stage.
- **Домен 5** «orchestrate multi-agent coordination» — `chains/` как
  first-class сущность оркестратора.

Конспект экзамена на русском — `~/Desktop/GH-600_study_guide_ru.docx`.
