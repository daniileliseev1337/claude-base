# svor-vor-works-base — база правил работ для СВОР/ВОР

## Суть
Единый источник наименований строк-работ в сопоставительных ведомостях (СВОР) и ведомостях
объёмов работ (ВОР): xlsx-база «типы позиций → ключевые слова → состав работ». Пайплайн
детерминированный: извлечение блоков → матчинг типов → план правок → win32com → верификация.

## База правил
- **Живой экземпляр** — в папке текущего объекта (файл `baza_rabot_vor_svor*.xlsx`); путь
  спросить/найти рядом с ведомостями. Для нового объекта скопировать из
  `templates/baza_rabot_vor_svor.xlsx` (эталон последней версии, 36 типов).
- Лист «База работ», колонки: A №, B тип позиции, C группа позиций, D ключевые слова (через
  `;`), E логика, F **работы для ВОР/СВОР** (из неё собирается наименование), G расширенный
  состав, H что учитывать в объёмах, I исключения.
- **После изменения живой базы — обновить копию в `templates/`** (иначе новые объекты стартуют
  со старой версии).

## Правила формирования наименования (колонка D ведомости)
- Составное: элементы F через запятую, последний через « и », затем объект в родительном
  падеже: «Установка, монтаж, подключение, настройка и проверка работоспособности коммутатора».
- Объект при ПЕРЕИМЕНОВАНИИ — хвост существующей работы после глагольного префикса («Монтаж »,
  «Пусконаладка », «Настройка и запуск », «Монтаж (оконцевание) »…) — уже в род. падеже,
  не склонять заново. При СОЗДАНИИ с нуля — родовое имя позиции в род. падеже (формирует
  модель, не скрипт), вставка строк — метод memory `reference_svor_add_works_win32com`.
- Особые типы: **2 (нематериальные: лицензии/сертификаты/подписки/опции ПО)** и
  **7 (крепёж/монтажные комплекты/расходники)** — строку-работу НЕ формировать / удалить;
  **33 (строительные: бурение, штробление)** — наименование не менять; **34** → фикс.
  «Маркировка кабелей и оборудования»; **36** → «Огнезащитная герметизация проходок»;
  **32 (заземление)** → «Монтаж <объект>».

## Пайплайн (tools/)
1. `extract_blocks.py --files "<glob>" --out blocks.json` — блоки по вертикальному merge
   колонки A (ВОР: работа+позиция; СВОР: до 4 строк — 2 работы «до/после» + 2 позиции).
2. `make_plan.py --base <база.xlsx> --blocks blocks.json --out plan.json` — матчинг типов
   (ключи из базы, фразовый матч) + план: replace/delete/skip. Печатает статистику,
   непокрытые пары и **прочёс подозрительных** («работа не похожа на тип») — их разбирать
   ГЛАЗАМИ до применения; новые ложняки лечить ключами базы или FIXUPS в скрипте.
3. `apply_plan.py --plan plan.json --src "<папка исходных>" --dst "<папка результата>"` —
   win32com (НЕ openpyxl: относительные формулы H/I СВОР), правки снизу вверх, при удалении
   строк перенос №/производителя/модели из якоря merge.
4. `verify_plan.py --plan plan.json --blocks blocks.json --dst "<папка результата>"` —
   блоки 1:1 против плана, тексты, переносы, формулы самоадресные. 0 ошибок = можно на ревью.
5. Ревьюер-гейт: excel-validator + auditor. Аудитору ЯВНО писать: строки смещены — сверять
   по содержимому, не по номерам; какая папка новая. Ревьюеру давать Bash-инструкции
   (Excel MCP чанками сжигает лимит на больших листах).

## Матчинг — накопленные грабли (полный разбор: memory `reference_baza_rabot_vor_svor`)
- Только фразовый матч (слова ключа подряд, стем-префикс русских, оба порядка 2-словных).
  Независимый матч слов по тексту = ложняки («релейный модуль» ← «релейный выход… модуль»).
- Короткие ключи («ПО», «ОК») — только словом целиком. Опасные одиночные ключи в описаниях:
  «панель», «настенный», «силовой», «сигнализации», «кабель» (в скобках описания), «кресло»
  (ловит «крестообразный»), «клавиатура», «лицензия» у серверов «в комплекте с лицензиями».
- Тай-брейк равного скора — более специфичный (больший №) тип. FIXUPS: специфичные подстроки
  раньше общих.
- После ЛЮБОГО изменения матчера — прочёс подозрительных по типам с удалением (2/7) и
  кабельными формулами (25/24/3/8/9), не только спот-чек.

## Развитие базы
Непокрытая позиция → предложить новый тип (9 колонок по образцу) / доп-ключ существующего →
подтверждение пользователя → внести в живую базу → обновить `templates/`. Согласование
больших правок удобно через черновик-xlsx: пользователь красит текст (красный = удалить,
зелёный = добавить), читать `openpyxl load_workbook(rich_text=True)` → `TextBlock.font.color`.
