# revit-testbed — сессия-полигон приёмки Revit-процессов

Роль: жёсткая приёмка стороннего инструмента на живом стенде, чтобы в команду
уезжало только проверенное. Вырос из практики живого проекта (шаблон + библиотеки
+ Revit-MCP); прототип приёмки — FamFab (скилл → JSON → валидатор → executor).

## Стенд (что должно быть)

- Живой Revit (2024+) + pyRevit Routes (MCP `Revit-Connector`);
- Рабочий шаблон и эталонные библиотеки — ТОЛЬКО как источник копий, не полигон;
- Отработанные механики (детали — memory `reference_revit_mcp`): загрузка семейств
  через family-doc, CopyElements с FailuresPreprocessor и печатью TransactionStatus,
  8.3-пути, кириллица-через-файл, батчи 20–25 элементов, подавление диалогов.

## Протокол приёмки (ядро, 5 пунктов — все обязательны)

1. **Только копия/изолированный документ.** Тест НИКОГДА не на рабочем шаблоне —
   на копии файла или чистом тестовом документе (family-doc / new project).
2. **Фиксация до/после.** Счётчики элементов (FilteredElementCollector по
   категориям), хэши файлов — до запуска и после.
3. **Проверка транзакций.** Смотреть TransactionStatus после Commit; return-значениям
   инструмента НЕ верить (молчаливые фейлы — обычное дело, см. anti-pattern A11.2).
4. **Отчёт по форме:** работает / ограничения / ловушки → находки в memory базы
   (пополняем каталог ловушек, не теряем).
5. **Гарантированный откат.** Файл-копия + close без сохранения; после теста
   стенд в исходном состоянии (проверить п.2 повторно).

## Чек-лист прогона end-to-end (пример: генератор контента)

- [ ] Вход подготовлен (изображение/описание/JSON) — зафиксирован в папке прогона;
- [ ] Промежуточный артефакт получен и провалидирован кодом (если инструмента
      без инспектируемого артефакта — это находка против него, anti-pattern A11.1);
- [ ] Исполнение в изолированном документе, статусы транзакций напечатаны;
- [ ] Результат осмотрен глазами (геометрия/параметры/размеры), «невозможные»
      значения (0, пусто) трактуются как ошибка, не как норма;
- [ ] До/после сверено, откат выполнен;
- [ ] Отчёт: вердикт (принять / принять с ограничениями / отклонить), список
      ловушек, рекомендация по форме внедрения (as-is / форк / только паттерн).

## Кто исполняет

Агент **pyrevit-engineer**, режим «приёмка стороннего инструмента» (описан в самом
агенте). Оркестратор даёт: что принимаем, где стенд, какой тестовый документ,
критерии вердикта. Финальная приёмка поведения — всегда на реальной модели.
