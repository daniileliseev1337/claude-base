# supplier-due-diligence — проверка контрагента и подлинности его документов

## Когда подключаться
Перед включением поставщика в реестр / подписанием договора / приёмкой присланных им
документов качества. Особенно если поставщик новый, цены/условия «слишком хорошие», или
документы вызывают сомнение.

## Что детерминированный код, что — через web (честно)
- **Подлинность присланного PDF** → рабочий скрипт `tools/verify_pdf_metadata.py` (offline, pikepdf).
- **Реальность фирмы / санкции / домен** → через web-инструменты Claude (см. секцию «Веб-доступ» ниже).

## Веб-доступ
Канон (лестница exa→firecrawl→fetch→playwright→WebFetch, гео/egress-модель, проверка `%PDF`) —
[[feedback_web_direct_access]] + CLAUDE.md §«Веб-доступ». Ключевое для due-diligence: росс. госсайты-реестры
(ЕГРЮЛ/АРШИН/pub.fsa) при иностранном egress → скилл [[ru-gov-access]] (страницы/PDF); **ДАННЫЕ из API
реестров = только браузер** (playwright с RU-IP), curl=403 на любом IP.

## Workflow

### Шаг 1 — Реальность юрлица (через web)
- **РФ:** ФНС ЕГРЮЛ/ЕГРИП — `egrul.nalog.ru` (по ИНН/ОГРН): существует ли, статус (действующее /
  ликвидация), руководитель, дата регистрации, ОКВЭД (профиль бьётся с поставкой?).
  Углублённо (если есть доступ): Контур.Фокус / СПАРК — арбитраж, банкротства, аффилиаты.
- **Зарубеж:** OpenCorporates (`opencorporates.com`) — 200+ реестров.
- **Verify:** юрлицо найдено, действующее, зарегистрировано не «вчера под сделку», профиль
  деятельности соответствует. Не найдено / ликвидируется → FAIL.

### Шаг 2 — Санкционный скрининг (через web)
- OpenSanctions (`opensanctions.org`) — по названию/ИНН/руководителю (OFAC, EU, UN, 40+ списков).
- **Verify:** в списках нет. Есть → BLOCKER, эскалация пользователю.

### Шаг 3 — Легитимность домена из КП/реквизитов (через web)
- `crt.sh/?q=<домен>` — все SSL-сертификаты домена и поддомены (без регистрации).
- SecurityTrails / DNSDumpster — исторические DNS, возраст, смена владельца.
- **Verify:** домен не «свежесозданный под сделку» (история > нескольких месяцев), почта на
  своём домене. Свежий домен + gmail/mail.ru вместо корпоративного → WARN.

### Шаг 4 — Подлинность присланных документов (СКРИПТ)
Для каждого присланного PDF (сертификат/паспорт/декларация):
```bash
python ~/.claude/skills/supplier-due-diligence/tools/verify_pdf_metadata.py "<file.pdf>"
```
Скрипт читает метаданные и флагует: правка в графическом/PDF-редакторе, изменение ПОСЛЕ
создания, метка сканера+редактора (подмена текста на скане), будущая дата, вычищенные
метаданные. exit 1 = есть флаги.
- **Сверка реквизитов:** № и дату сертификата проверить в официальном реестре —
  АРШИН (`fgis.gost.ru`) для СИ, Росаккредитация (`pub.fsa.gov.ru`) для деклараций/сертификатов.
- **Verify:** метаданные чисты ИЛИ флаги объяснимы; реквизиты есть в реестре. Иначе FAIL
  «документ не подтверждён».

### Шаг 5 — Вердикт
Свести в краткий отчёт: фирма (реальна/нет), санкции (чисто/стоп), домен (ок/свежий),
документы (подтверждены/флаги). Итог: ✅ можно работать / ⚠ с оговорками / ⛔ не подтверждён.

## Tools (слой 3)
- `tools/verify_pdf_metadata.py` — проверка подлинности PDF по метаданным (pikepdf, offline).
  Протестирован: чистый PDF → 0 флагов; Photoshop+правка → ловит. Зависимость pikepdf (в базе).

## Связь с базой
- Усиливает agent `снабженец` (проверка поставщика) и `auditor` (due-diligence в ревью).
- Добыча самих документов — skill [[doc-finder]] + memory [[feedback_web_direct_access]].
- Источник правила «не доверять документу без проверки» — feedback_cert_sourcing_fabrication.

## Ловушки
- Чистые метаданные ≠ гарантия подлинности (могли вычистить). При сомнении — ELA
  (FotoForensics/Forensically, web) + визуальная сверка + реквизиты в реестре.
- ЕГРЮЛ/АРШИН — росс. госсайты: страницы/PDF через скилл `ru-gov-access`, а ДАННЫЕ из их API =
  только браузер (playwright с RU-IP); curl даёт 403 на любом IP (см. секцию «Веб-доступ»).
- Многомодельные/многосерийные документы (один сертификат на серию) — НЕ дефект.
