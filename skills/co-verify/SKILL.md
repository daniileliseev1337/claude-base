# co-verify — координатная верификация СО из проектных PDF

## Когда брать
Нужно ПРОВЕРИТЬ числа/состав ВОР или спецификации против исходной СО в **векторном**
PDF, либо сравнить две редакции ПД. Это альтернатива вижн-аудиту: прошлый опыт —
визуальное чтение сотен строк не масштабируется и пропускает однотипные ошибки
(кейс: 9 завышений кол-ва среди 546 строк, найдены этим методом, см. `examples/`).

**Ключевой принцип:** для векторного PDF **text-layer авторитетнее глаза**.
`find_tables(strategy="lines")` строит сетку по векторным линиям таблицы и читает
реальный текст ячеек. Рендер лишь растеризует тот же слой — им только адъюдицировать
единичные спорные ячейки, не читать всю таблицу.

## Инструменты (слой 3 — основная ценность, в `tools/`)
- `co_engine.py` — движок: `extract_co(pdf)` → строки СО; `pair_by_position_then_name`
  (для сверки), `pair_by_name_sequence` (для diff редакций); нормализация qty/unit/
  гомоглифов. Не привязан к проекту.
- `co_verify.py` — CLI: сверить один СО-PDF с данными.
  `python tools/co_verify.py --pdf СО.pdf --json data.json`
  `python tools/co_verify.py --pdf СО.pdf --xlsx ВОР.xlsx --sheet Рабочий --cols A,B,F,G`
  Код выхода 0 = чисто, 1 = есть расхождения. Печатает QTY / UNIT / EXTRA / MISSING.
- `co_diff.py` — CLI: diff двух редакций.
  `python tools/co_diff.py --old старое.pdf --new новое.pdf`

## Процедура
1. **Извлечь** СО координатно (`extract_co` / `co_verify`). Если `extract_co` вернул 0
   строк — проверь детект страницы (`is_co_page`: нужны «Поз./Позиция» + «Кол.» +
   «Наименование и техническ») и шапку (`map_columns`).
2. **Сверить** детерминированно: где данные и PDF совпали — оба верны; где разошлись —
   это кандидаты. Метод сужает сотни строк до горстки.
3. **Адъюдицировать** каждое расхождение визуально по КОНКРЕТНОЙ странице: отрендерить
   её (`fitz get_pixmap(matrix=Matrix(150/72,150/72))` — поворот применится сам) и
   прочитать нужную строку. Только так — не доверять ни данным, ни своему экстрактору слепо.
4. **Замкнуть цепочку** при правке: исходник(PDF) → данные(json) → артефакт(xlsx).
   После правки данных пересобрать xlsx и заново прогнать `co_verify` (должно стать 0).

## Ловушки (обойдены в движке, помни при отладке)
- Страницы А3 повёрнуты на 90° (rotation=90; координаты в неповёрнутом пространстве —
  find_tables и get_pixmap справляются).
- Переносы в шапке («Коли-\nчество») → дегифенизация перед матчем ключа колонки.
- Название в строках данных уезжает на ±1 колонку → название = склейка ячеек от
  позиции до тип/код (не одна ячейка).
- Номера позиций в PDF НЕнадёжны: опечатки (напечатано «2.11» вместо «3.11»),
  пустые номера у реальных строк, рестарт нумерации per-СО в мульти-СО → матчинг
  гибридный (позиция + название + гомоглифы), для diff — по последовательности названий.
- **Виды работ** (Монтаж/Прокладка/Установка) keyword-правилом НЕ верифицируются
  надёжно (слова «винт/кабель/труба» внутри описаний оборудования → ложные флаги) —
  нужен семантический ревьюер.

## Чего НЕ делает
- Не читает сканы без текстового слоя → OCR в `image-text-replace`.
- Не генерирует ВОР/спецификации → `pnr-vor-helper`, `spec-writer`.
- Не проверяет ШТАМП/мету (объект, ФИО, шифр) — это отдельная сверка текста штампа
  (geo: искать «еверодонецк»/«алашиха» в тексте всех СО-страниц).
