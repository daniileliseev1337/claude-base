# stroy-formatting — оформление итоговых документов

## Когда применять

Этот скилл — **поведенческое правило**, активирующееся когда Claude
собирается выдать пользователю **финальный** DOCX или PDF документ:

- Пояснительная записка (ПЗ) к проекту
- Ответ заказчику или экспертизе на замечание
- Техническое задание (ТЗ) заказчику или подрядчику
- Статья или доклад на конференцию
- Отчёт по командировке / визитной поездке
- Служебная записка

**ОБЯЗАТЕЛЬНО ПЕРЕД сохранением финального файла** Claude вызывает
`AskUserQuestion` с выбором стиля. Без явного ответа — не сохранять.

## Когда НЕ применять

- Markdown-черновики (`.md`) — это рабочий формат, не финал.
- Промежуточные `.py`, `.json`, `.csv` — данные, не документы.
- Excel-спецификации (`.xlsx`) — у них свой ГОСТ (Р 21.110) через
  `stroy-specs-xlsx`, не этот скилл.
- AutoCAD `.dwg` — чертёжная документация, оформление по ГОСТ Р 21.101
  внутри самого DWG.
- Лог-файлы, дампы.
- Если пользователь явно сказал «сделай как у Word по умолчанию» —
  применяй `plain-clean` без вопросов.

## 4 шаблона стиля

Все лежат в **`~/.claude/formatting-templates/`** (синхронизируются
auto-push hook'ом на все ПК):

### 1. `gost-report-full.docx` — ГОСТ 7.32 полный (без рамки)

**Параметры (наследует от ВКР-эталона):**
- A4, поля: левое 25 мм, правое 10 мм, верх/низ 20 мм
- Основной текст: Times New Roman 14 pt, по ширине (justify),
  отступ первой строки 12.5 мм, межстрочный интервал 1.5
- Заголовок 1 уровня: TNR 14 pt **bold**, по центру, без отступа
- Заголовок 2/3 уровня: TNR 14 pt bold, слева, с отступом
- Подписи к рисункам/таблицам: TNR 12 pt, по центру
- TOC: TNR 14 pt
- **БЕЗ рамки** — header'ы заменены пустыми, на полях ничего

**Под что:** ПЗ к проекту для экспертизы, серьёзные ответы заказчику,
ТЗ, отчёты для контролирующих органов, академические статьи.

### 2. `gost-report-light.docx` — ГОСТ упрощённый

Структурно идентичен `full`, отличается **правилами применения**:
- Допустимо использовать TNR 12 pt для табличных подписей и сносок
- Отступ первой строки можно 10 мм вместо 12.5
- Заголовки можно слева без принудительного центрирования
- Подписи к рисункам — короче, без обязательного «Рис. N — ...»

**Под что:** внутренние документы, рабочие ответы менее формальным
адресатам, технические заметки, не для экспертизы.

### 3. `gost-report-with-border.docx` — ВКР-стиль с рамкой

Полная копия `full` плюс:
- **Рамка на каждой странице**: чёрный прямоугольник, граница 1.75 pt,
  размер 18.2 × 28.3 см (с отступом ~7 мм от края страницы с 3 сторон,
  ~14 мм справа)

**Под что:** дипломные работы, ВКР, документы для учебных заведений
которые требуют рамку. **Не используй** для рабочих проектных
документов — заказчики/экспертиза не ожидают рамку в ПЗ.

### 4. `plain-clean.docx` — простой аккуратный

С нуля, минимум стилей:
- A4, поля 25 мм со всех сторон
- TNR 14 pt
- Word-стандарт межстрочный 1.15
- Без кастомных заголовков (стандартные Heading 1/2/3 от Word)
- Без рамок, без специфической нумерации

**Под что:** служебная записка, внутренняя заметка, быстрый ответ
коллеге, любой документ где «просто читаемо и не нужно прям ГОСТ».

## Алгоритм применения

Когда сработал триггер (Claude собирается выдать финальный DOCX/PDF):

1. **Спросить пользователя через AskUserQuestion** (обязательно):

   ```
   question: "В каком стиле оформить документ '<имя>'?"
   header: "Стиль"
   options:
     - label: "ГОСТ полный (без рамки)"
       description: "Times New Roman 14, поля 25/10/20/20, отступ 1.25 см, межстрочный 1.5. Для ПЗ, экспертизы, серьёзных ответов."
     - label: "ГОСТ упрощённый"
       description: "Те же базовые правила ГОСТ, но мягче. Для внутренних документов."
     - label: "ГОСТ с рамкой (ВКР)"
       description: "Полный ГОСТ + рамка 1.75 pt на каждой странице. Для дипломов и учебных работ."
     - label: "Простой аккуратный"
       description: "TNR 14, поля 25 со всех, Word-стандарт без специфики. Для служебных записок."
   ```

2. **Получив ответ** — выбрать шаблон:
   - `gost-report-full.docx`
   - `gost-report-light.docx`
   - `gost-report-with-border.docx`
   - `plain-clean.docx`

3. **Применить шаблон** через гибрид python-docx + word-MCP:

   ```python
   import shutil
   from docx import Document
   from pathlib import Path

   src = Path.home() / ".claude" / "formatting-templates" / f"{chosen_style}.docx"
   dst = Path.cwd() / f"{document_name}.docx"
   shutil.copy(src, dst)
   doc = Document(str(dst))
   # ... наполнить контентом: doc.add_heading, doc.add_paragraph, doc.add_table
   doc.save(str(dst))
   ```

   ИЛИ через word-MCP — `mcp__word__add_heading`, `mcp__word__add_paragraph`,
   `mcp__word__add_table` — если документ уже открыт и удобно поэтапно.

4. **Сохранить в cwd сессии** (или явно указанной пользователем папке).
   **Не в `~/.claude/`** — выдача всегда рядом с проектом пользователя.

5. **Если нужен PDF** — после сохранения DOCX:
   - Использовать `mcp__word__convert_to_pdf` если доступен
   - Или fallback на docx2pdf через python (требует MS Word или
     LibreOffice локально)

6. **Сообщить пользователю** конечный путь и краткое резюме:
   - Какой стиль применён
   - Где сохранён файл
   - (При наличии PDF) — путь к PDF

## Когда стиль уже известен (без вопроса)

Если пользователь **явно** в запросе указал стиль («сделай ПЗ по
ГОСТ полному», «нужен ВКР-формат с рамкой», «просто аккуратно»), —
не спрашивай, используй указанный стиль.

## Расширение списка стилей

Когда понадобится 5-й стиль (например, для отдельной категории
документов вроде «коммерческое предложение»):

1. Создать новый шаблон через python-docx или копию из эталона
2. Положить в `~/.claude/formatting-templates/<имя>.docx`
3. Добавить описание в этот SKILL.md (раздел «4 шаблона стиля»)
4. Закоммитить в `claude-base` — auto-sync разнесёт на все ПК

## Failure-modes (важно)

- **Шаблон не существует** в `~/.claude/formatting-templates/` →
  предупредить пользователя, спросить хочет ли он скопировать из
  эталона или сгенерировать с нуля.
- **python-docx падает** на копировании шаблона → попробовать
  через прямой `Copy-Item` + `mcp__word__add_*`, без python-docx
  trick'ов.
- **Не удалось спросить через AskUserQuestion** (например, в
  автономном режиме) → дефолт `gost-report-full` + явное упоминание
  в финальном ответе «применил дефолтный стиль, потому что не мог
  спросить».
