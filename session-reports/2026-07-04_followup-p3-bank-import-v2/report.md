### 2. Умный импорт банковской выписки v2 — brainstorming → spec → план
- Изначальный запрос — **real-time канал банк→сайт**. Проверка выполнимости (research):
  - У банка нет публичного API для физлица-покупателя (API платёжного сервиса — только для мерчантов, отдаёт входящие платежи клиентов, не личные траты).
  - Открытый банкинг РФ (стандарты ЦБ, методы для физлиц) — вступает в силу 01.10.2026, банк как поставщик пока не подключён.
  - У владельца email/SMS на операции не приходят — только push в приложении. Единственный real-time путь — перехват push (Android Notification Listener → вебхук), хрупкий. **Отложено.**
- Итерация переопределена в **умный импорт выписки**: докрутка существующего PDF-парсера + классификация типов операций (оплата / перевод-себе / перевод-физлицу с обезличиванием ПДн) + обучаемый категоризатор-цепочка (выученное → MCC → словарь с границами слов → [LLM опц.] → ручной review) + хранение выученных правил в БД + дедуп.
- **Ключевой инсайт про LLM:** владелец платит с одного банка, доход приходит на другой, назначение перевода пишет на стороне банка-источника — в выписку принимающего банка оно не попадает. Вывод: LLM не восстановит данные, которых нет в источнике; для переводов — только ручной разбор. LLM оправдан лишь как слой для незнакомых мерчантов (оплаты), и то опционален при обучаемом словаре. → детерминированное ядро + ручной review, LLM вне scope v1.

## Реализация банка v2 (выполнена в этой же сессии, subagent-driven)
По выбору владельца реализация пошла в этой же сессии (не в новом чате). 15 коммитов (`b7c3870..cbc0687`), 147 тестов, build green. 5 групп задач (D1-D5), каждая: fresh implementer → task-review → fix-subagent. Финальный opus whole-branch review поймал регрессию. Merged в main `26e0b86` по «го» (Merge в main, deploy отложен). Ledger: `.superpowers/sdd/progress.md`.

Ценность subagent-driven (находки, пойманные ревью до merge):
- **Утечка ПДн** при переводе физлицу без запятой (`split(",")[0]` возвращал всю строку с ФИО) — D2 I3.
- **minX-регрессия**: при выносе парсинга в чистую функцию потерялась координатная фильтрация правой колонки → текст правой колонки мог попасть в описание. Восстановлено эвристикой «строка без букв = не описание».
- **Потеря income-guard** при рефакторинге: `categorizeByDict` лишился `type`-параметра, расход с ключом «возврат средств» получал категорию «Прочий доход» (income). Поймано финальным opus-review, восстановлено.
Каждая — исправлена + покрыта тестом до merge. Per-task ревью их не видели (узкий контекст одной задачи); whole-branch review критичен.

Урок процесса: для транскрипции готового кода из плана прямая реализация была бы дешевле армии субагентов, НО subagent-driven окупился именно на ревью-гейтах — три реальные регрессии (одна — PII) пойманы независимыми проверяющими. Модель: implementer sonnet, финальный review opus.
