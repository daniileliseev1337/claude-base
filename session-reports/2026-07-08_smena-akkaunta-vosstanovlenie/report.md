## Уроки / грабли (переиспользуемые)

- **Что переносится при смене аккаунта:** вся файловая память машины (~/.claude,
  папки проектов, транскрипты projects) — аккаунт-агностична; в аккаунте живут
  только подписка/модели, облачные сессии, чаты claude.ai и проекты Claude Design.
  Облачное без экспорта теряется.
- **Реестр Desktop-сессий per-account** — см. механику в п.1; лечится копированием
  файлов, бэкап обязателен.
- **`cleanupPeriodDays` по умолчанию 30 дней** — тихо стирает транскрипты
  (на этом ПК съел апрель–май). На новых машинах ставить 365 сразу.
- **Экспорт claude.ai не содержит Code-сессий** — их носитель только локальный
  `~/.claude/projects/` каждого устройства (и session-reports в базе).
- **Дизайн-проекты Claude Design восстановимы реплеем** design_chats из экспорта —
  паттерн «write_file+str_replace_edit реплей» работает; правки на холсте и
  uploads — слепая зона.
- **Кириллица в аргументах shell из тулов бьётся о кодировку** (и Bash, и
  PowerShell): пути с кириллицей в командах → FileNotFound при живом файле.
  Обход: короткие имена (`C:\Users\88F3~1`), поиск пути через os.listdir в
  python-скрипте, файл скрипта вместо инлайн-кода.
- **`--bare` у headless claude отключает OAuth/keychain** — для проверки логина
  запускать без `--bare`.
- **Отключённый аккаунт**: локальный `claude auth status` может показывать
  loggedIn:true (кэш), реальную живость проверять коротким `-p` запросом.

## Где сломался / ограничения
- DesignSync в Desktop-сессии требует design-авторизацию, `/design-login` —
  только интерактивный терминал; OAuth-клики принципиально за владельцем.
  Заливка дизайн-системы в новый аккаунт на момент отчёта — в ожидании
  (перелогин CLI запущен, дальше /design-login + push по «готово»).
