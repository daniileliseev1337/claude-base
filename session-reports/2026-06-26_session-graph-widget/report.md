# Session report — 2026-06-26/27 — Виджет «карта понимания» (мост Claude↔пользователь)

Брейншторм-сессия (superpowers:brainstorming): от идеи «виджет после грилинга» до зафиксированной
архитектуры. Длинная, диалоговая, с реальными прототипами и фактической проверкой. Артефакты — в
`Desktop\widget-spike\` (для просмотра с телефона) и `harvested/` рядом.

## Что выработано (итог)

**Принцип (ядро):** виджет = ЗЕРКАЛО понимания Claude. Показывает то, что Claude реально понял, чтобы
пользователь сверил ДО работы. Изначальная боль — «не понимали друг друга». **Форма следует за
содержанием** задачи, а не фиксирована.

**Архитектура — 3 слоя:**
1. Карта понимания (Claude эмитит: сущности, связи, уверенность, опц. координаты — единый формат).
2. Виды (рендер под содержание): дашборд / схема / план / граф.
3. Канал — MCP-app (рендер в чате + возврат правок встроен).

**Движки — деление СНЯТО (ключевой разворот в конце):** не «лёгкий/тяжёлый движок», а ось «форма содержания».
- **Vanilla HTML/CSS-дашборд — по умолчанию** (как сам виджет карты): невесом, полный контроль; при
  архитектуре shell+data генерится так же дёшево, как mermaid (Claude льёт только данные).
- **React Flow** — только для БОЛЬШОГО интерактивного графа (сотни узлов). На элементарном дереве избыточен.
- **Fabric/canvas** — только для пространственного плана (Revit-данные, как в прошлой Revit-сессии владельца).
- **mermaid** — под вопросом: плюс (дешёвая генерация) исчезает при vanilla+shell, минусы (3.5 МБ, слабый
  визуал) остаются. Возможно не нужен.

**Два режима на общем каркасе** («и то, и то»): лёгкая «сверка» (данные от Claude) и тяжёлый «проект»
(реальные данные Revit/AutoCAD, экспорт координат назад в CAD).

## Факты (проверено, не из головы)
- Harvest 8 библиотек через GitHub API (stars/license/push актуальны на 26.06): mermaid 88.9k MIT,
  Excalidraw 126k MIT, React Flow 37k MIT, Fabric 31k MIT, Cytoscape 11k MIT, vis-network 3.6k Apache,
  Konva 14k, tldraw 48k (не-OSI лицензия). React-троица (RF/Excalidraw/tldraw) требует React.
- Учение-спайк: один граф на Fabric vs React Flow, рендер в playwright. RF выиграл (fitView/zoom/MiniMap
  из коробки; Fabric обрезает, механику пишем сами). RF UMD живёт в одном HTML БЕЗ сборки (опровергнут мой
  прежний страх «нужен bundler»).
- MCP-app («show widget» десктопного Claude Code) = Apps SDK: tool отдаёт данные + ui-resource HTML,
  хост рендерит iframe в чате; `app.sendMessage` = встроенный возврат правок. CSP блокирует CDN →
  всё инлайнить (наш standalone уже годится). Деталь — `harvested/channel-mcp-app.md`.

## Где сломался / уроки
- **Главный провал — наскакал по 5 слоям разом** (форма/движок/канал/масштаб/тип контента) → запутали
  обоих. Урок: на брейншторме держать ОДНУ ось за раз; «и то, и то» от пользователя = сигнал, что я
  склеиваю разные продукты в один выбор.
- **mobile-баг:** ранние прототипы (proto-fabric/reactflow) БЕЗ `<meta name="viewport">` → на реальном
  телефоне пусто (Safari рендерит 980px и масштабирует). Карта/формы работали — у них тег есть. Факт
  подтверждён playwright-замером (root 390×793, узлов 12, но `viewport_meta: НЕТ`). Урок: **виджет
  ОБЯЗАН иметь meta viewport** — владелец смотрит результаты с телефона (remote control).
- **Файлы — сразу в одну папку на рабочем столе** с понятными именами, не в Public с копированием
  потом. Владелец смотрит с телефона.
- Verifiable-first сработал дважды: не согласился «это телефон» — проверил, нашёл реальную причину
  (viewport); не поверил числам README — взял метаданные из GitHub API.
