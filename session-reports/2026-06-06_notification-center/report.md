# Session report — Центр уведомлений (in-app inbox)

**Дата:** 2026-06-06
**Проект:** self-hosted dashboard (React+Vite+Supabase, FRP-туннель на VPS)
**Тема:** проектирование и реализация in-app inbox, дополняющего Web Push.

## Что сделано
Полный цикл: brainstorming → спек → writing-plans → реализация (subagent-driven) →
деплой → E2E → merge. Фича — durable-лента уведомлений (колокольчик с badge +
dropdown + Realtime + отметка прочитанным), чтобы пропущенный системный push
оставлял след в приложении.

**Архитектура (подход A):** edge-функция рассылки push для каждого события считает
базовый список получателей БЕЗ флаг-фильтра → батч-INSERT строки inbox каждому
(durable), затем push фильтрует по per-тип флагам. Inbox ловит всё всегда; флаги
управляют только push (это и есть «страховка от эфемерности»).

**Артефакты:**
- БД: миграция таблицы `notifications` (RLS: read/update только свои; INSERT только
  service_role; в Realtime-публикации) + миграция retention-cron (чистит только
  прочитанные старше 7 дней; непрочитанные бессрочно).
- Edge: рефактор — хелперы базового списка (`baseIds`/`baseApproved`) + `insertInbox`,
  батч-INSERT во всех ветках событий перед отправкой push.
- Фронт: lib-модуль (fetch/count/markRead/markAllRead/subscribe) + компонент-колокольчик
  (dropdown, badge, Realtime INSERT/UPDATE) + интеграция в top-bar.
- Verify-скрипты (durable-активы): RLS, service_role POST (путь записи), event→inbox.
- Cleanup: убран мёртвый код; добавлен `.gitattributes` (`*.sh` → LF).

## Верификация (evidence-based)
- RLS_OK: клиентский INSERT → 403; чужое UPDATE → 0 строк; своё read/update — ок.
- service_role POST в таблицу → строка создана (проверен путь записи edge).
- Регрессия push: реальное событие через edge → `{sent:2, inbox:1}` (push на 2
  устройства + 1 строка inbox из одного вызова).
- E2E на устройстве (владелец): колокольчик, push, позиция панели — подтверждены.
- Сборка зелёная; merge в main + push; main = origin/main; рабочее дерево чистое.

## Где сломалось / инциденты
1. **iOS PWA кэш.** После деплоя долго виден старый бандл (Service Worker отдаёт
   закешированную оболочку). Лечится холодным перезапуском PWA (иногда дважды) или
   проверкой в Safari. Каждый деплой → новый хеш бандла → нужен повторный холодный старт.
2. **Панель уезжала за край на мобильном.** v1 (`right:0`) у левого колокольчика уходила
   за левый край; v2 (`left:0`+кламп) могла задеть правый. Итог — `position:fixed`
   привязка к вьюпорту (`left:8/right:8`, ширина auto), `top` по замеру низа колокольчика
   (getBoundingClientRect). Урок: для dropdown у переменного по позиции триггера на мобиле
   надёжнее fixed-к-вьюпорту, чем absolute-anchor.
3. **Прод «упал» при живых контейнерах.** FRP-туннель завис после сетевого блипа между ПК
   и VPS (`i/o deadline reached`, сам не восстановился даже после возврата связи). Фикс —
   рестарт службы frpc (`systemctl restart frpc`). НЕ связано с кодом/деплоями.

## Уроки (грабли среды self-hosted/WSL)
- Кириллический путь, захваченный в переменную через PowerShell→wsl, теряется → `cd` по
  глобу `/mnt/f/*/<repo>` + относительные ASCII-пути; глоб в АРГУМЕНТЕ команды надёжен.
- Сложный inline-bash с `$()`/SQL-кавычками через PowerShell→wsl бьётся → файл-скрипт;
  для psql избегать одинарных кавычек (`\d+`, запросы без WHERE-литерала).
- В контейнере reverse-proxy (kong) нет `curl` → бить с WSL-хоста.
- `psql -At` к INSERT…RETURNING печатает тег `INSERT 0 1` → извлекать значение grep'ом.
- CRLF в `*.sh` ломает запуск под WSL (`set: pipefail: invalid option`); причина —
  autocrlf на checkout; durable-фикс — `.gitattributes` `*.sh text eol=lf`.
- «Сайт упал» + живые контейнеры = проверить туннель (рестарт frpc), а не код.
- Само-curl публичного адреса с машины-клиента туннеля даёт 000 (hairpin) — проверять с
  внешнего устройства.

## Процесс
Subagent-driven исполнение плана: свежий исполнитель на задачу + контроль спек-комплаенса
(чтение каждого файла контроллером) + код-ревью качества субагентом на содержательном коде
(edge, фронт), а для дословных SQL/bash — контроллер-проверка. Дословные блоки кода в плане
резко сократили итерации. Прод-шаги (миграции к живой БД, деплой edge/фронта, E2E, merge) —
под явным контролем владельца по чеклисту команд; код+коммиты — автономно.
