# Сессия 2026-05-13 — Сводная «было-стало» по DOCX-спецификациям

## Постановка

Даны две Word-таблицы:
- **БЫЛО** (стадия П): 1042 позиции, 43 шкафа `ШК.N`
- **СТАЛО** (стадия РД): 1270 позиций, ~47 типов шкафов с разбивкой по узлам сети

Нужно построить XLSX-сводную «было-стало» бок-о-бок с попозиционным сопоставлением. Эталон-пример частично заполнен человеком.

Контрольные суммы исходников: 787 444 730,91 ₽ vs 2 539 204 832,40 ₽ (≈3× прирост).

## Ключевые архитектурные находки

1. **Структура примера-эталона** — 2-уровневая иерархия:
   - Якорь группы: номер `N` в названии (`ШК.N` ↔ `Тип N*`)
   - Внутри якоря: позиции матчатся по **функциональному описанию**, не по производителю/артикулу

2. **N в БЫЛО ≠ N в СТАЛО семантически**:
   - БЫЛО: `ШК.N` — физический номер шкафа на схеме (1..43)
   - СТАЛО: `Тип N` — номер типовой конфигурации шкафа
   - Совпадение по числу — приблизительный якорь, не гарантия эквивалентности

3. **Повторы типов**: `Тип 12` встречается несколько раз для разных узлов (Л1, Л2, Л3, Л4). Объединение их в одну корзину приводит к перескокам номеров — нужно матчить только с первым вхождением, остальные → отдельный блок «Только в РД».

4. **Diff alignment vs жадный матчинг**: жадный (v3) разрешает пересечения (`БЫЛО[5]↔СТАЛО[2]` и `БЫЛО[2]↔СТАЛО[7]` одновременно), что нарушает порядок. Needleman-Wunsch DP запрещает пересечения и автоматически выбирает максимально полезное непротиворечивое подмножество.

## Что сломалось / на чём итерировал

1. **Самописный OOXML не открывался Excel'ем** (v1 первая версия):
   - Пустые `<row r="N"></row>` — нарушение OOXML schema
   - `<sheetViews>` после `<sheetData>` — неверный порядок элементов
   - `tabSelected="1"` без `<bookViews>` в workbook.xml
   - Решение: добавлен `<dimension>`, переупорядочены элементы, пустые row пропущены

2. **Кодировка PowerShell scripts**:
   - `.ps1` без BOM не читался — Windows-1251 fallback крашился на кириллице
   - Решение: пересохранение с UTF-8 BOM

3. **excel-validator subagent** не имел доступа к excel MCP — не смог провести валидацию. Делал самопроверку через MCP excel напрямую из main.

4. **MCP write_data_to_excel** имеет лимит на размер `data` параметра — для 2447 строк × 28 колонок пришлось разбивать на блоки.

5. **MCP `excel__create_worksheet` + openpyxl save** перезаписывает мой самописный OOXML — стили частично теряются. Решение: финальные правки (cols, freeze pane) делать через прямую правку sheet.xml в zip после openpyxl-операций.

## Выводы на будущее

1. **Для подобных задач** (двусторонний документ с fuzzy alignment) — стартовать сразу с Python+rapidfuzz+openpyxl+NW, не тратить итерации на PowerShell+OOXML
2. **Самописный OOXML** — рискованно для финального артефакта; openpyxl значительно надёжнее. Самописный OOXML оправдан только когда openpyxl недоступен или нужен прямой контроль ширин/панелей.
3. **Anchor matching по числам в названии** — слабая эвристика, но рабочая. При отсутствии явного маппинга от пользователя — приемлемая base line.
4. **5 итераций алгоритма** были нужны: пользователь уточнял требования по результатам каждого прогона (визуальный шум → перемешивание → перескоки номеров). Это нормальный итеративный процесс.
