# bom-diff (ttran-tech)

- **URL:** https://github.com/ttran-tech/bom-diff
- **Звёзды:** 9
- **Последний коммит:** не показан явно в README
- **Лицензия:** MIT (зелёный свет)
- **Язык:** Python (Flask + openpyxl + **fuzzywuzzy**)

## Что делает

Web-приложение (Flask) для сравнения BOM (Bill of Materials) разных
форматов. По README:

- сравнивает Agile PLM BOM с другими форматами BOM;
- сопоставляет **количество, ревизию, reference designators**,
  множественные поставщики;
- подсвечивает несоответствия;
- предлагает скачать отчёт сравнения;
- использует **fuzzywuzzy** для нечёткого сравнения строк.

## Чем подходит под нашу задачу

- **Единственный из найденных** инструментов с **встроенным fuzzy
  matching** — то есть умеет «Hikvision DS-2CD2143G2-I» и
  «HIKVISION DS2CD2143G2I» свести в одну позицию.
- Прямо позиционируется под BOM — самый близкий аналог нашей
  спецификации из всего найденного.
- MIT, Python, openpyxl — стек совместим с тем что уже есть у
  пользователя.

## Чем НЕ подходит / риски

- **9 звёзд** — далеко от фильтра ≥50. Личный проект одного автора.
- Заточен под **Agile PLM** (Oracle product lifecycle) — формат
  ввода отличается от наших docx-таблиц. Парсинг придётся переделывать.
- Web-приложение, а не библиотека/CLI — для встраивания в скрипт
  потребуется выдрать ядро.
- Активность не подтверждена.
- Качество кода и покрытие тестами не верифицированы.

## Оценка: **MAYBE — как источник идеи fuzzy-нормализации**

Не брать целиком (низкие звёзды, заточен под Agile PLM). Но **взять
идею**: добавить в наш пайплайн слой fuzzywuzzy/rapidfuzz для
нормализации производителя+модели **перед** передачей в datacompy.
Этот шаг закрывает главную слабость datacompy на нашей задаче.
