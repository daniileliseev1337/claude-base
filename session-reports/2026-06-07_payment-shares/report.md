# Сессия 2026-06-07/08 — daniil-dashboard: доли оплаты + несколько исполнителей + уборка доступа

## Что сделано (всё на проде)

Личный дашборд (React+Vite+self-hosted Supabase). Полный цикл brainstorm → спек → план →
subagent-driven реализация → финальный ревью → деплой → живая проверка владельцем.

1. **Кластер «доли оплаты» (крупная фича, поглотила отложенный баг рассинхрона стадии/оплаты).**
   - Таблица `project_shares`: полиморфный участник (юзер системы / клиент / внешнее имя через
     CHECK «ровно один адрес»), доля в % или в деньгах, владелец = ОСТАТОК (строкой не хранится).
   - RLS: владелец видит все доли проекта, участник — только свою. Приватный `SECURITY DEFINER`
     RPC `get_my_shares()` отдаёт участнику строго {название, моя доля, получено, остаток} —
     чувствительное (заказчик, полная сумма, чужие доли) не утекает.
   - Чистый модуль расчётов (vitest): доля→деньги, остаток владельца, получено-пропорция,
     KPI «Получено» и дебиторка по моей доле (с фильтром по владельцу — иначе двойной учёт
     расшаренных чужих проектов), агрегат «мои доли в чужих проектах».
   - Баг стадии «Оплачен»: автозаполнение «оплачено = сумма договора» на уровне формы.

2. **Несколько исполнителей** (фидбэк из живой проверки): поле «Исполнитель» (одно) →
   список (несколько), `jsonb`-колонка `[{name,userId}]`, текстовое поле-производная для
   совместимости. Autocomplete возвращает в т.ч. самого себя; push-уведомление каждому новому
   исполнителю-юзеру.

3. **Уборка задвоения доступа**: режим видимости «Избранные» дублировал «Команду проекта»
   (оба давали просмотр) — убран (−132 строки UI + клиентские обёртки RPC). Остался единый
   механизм точечного доступа. Миграция данных безопасна (таблица выбранных была пуста).

## Где сломался / грабли

- **SW-кэш PWA при preview.** Главная боль итеративной проверки: Service Worker кэширует старый
  build, и владелец видел прежнюю версию — даже «в инкогнито», если то же окно держит SW.
  Полчаса ушло на root-cause (код и бандл были свежие — виноват кэш браузера).
- **verify-скрипт давал ложный FAIL**: сравнение строк, а `numeric` из Postgres вернул
  `30000.000000000000` → строковое `"30000.0"` ≠ `"30000"`. Значения были верны, ломалось только
  сравнение (фикс: приведение к int при извлечении).
- **WSL psql через PowerShell**: `-F'|'` и SQL-кавычки бьются — использовать `-At` (дефолтный
  разделитель) без явного `-F`.
- Несколько раундов фидбэка из живой проверки (несколько исполнителей; «долю себе» наглядно;
  задвоение доступа) — фича обросла доработками уже после «готово», как и предсказывал владелец
  («понятно станет, когда начнут пользоваться»).

## Уроки (в скилл / память)

- **Обход SW-кэша для итеративной preview-проверки**: каждую новую сборку отдавать на НОВОМ порту
  (`npm run preview -- --port <новый> --strictPort`) — новый origin = свежий SW, старый кэш не
  действует. Надёжнее, чем просить чистить SW или «инкогнито» (инкогнито помогает только в НОВОМ
  окне). Записано в память проекта.
- **Живая проверка вскрывает то, что не ловят автотесты** (UX, ментальная модель пользователя):
  закладывать раунды доработок после формального «готово», не считать фичу закрытой до реального
  клика пользователем.
- **Для дословно-специфицированного кода и БД-слоя верификация исполнением (тесты/RLS-скрипт) +
  контроль diff контроллером эффективнее, чем механический прогон двух ревью-субагентов** —
  экономит токены без потери качества; ревью-субагентов держать для содержательных кусков
  (интеграция в монолит, финальный полный ревью).
