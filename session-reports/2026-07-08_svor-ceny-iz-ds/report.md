# Сессия 2026-07-08 — внесение цен из ДС (docx) в СВОР (xlsx), раздел «мультимедиа»

## Задача
В комплект из 8 СВОР (сопоставительные ведомости, 12 граф A–L) внести цены из 8 спецификаций
ДС (docx-таблицы «№|Производитель|Модель|Описание|Ед.|Кол-во|Цена с НДС|Стоимость с НДС»):
две новые колонки M/N («Цена за ед.» гр.13 / «Стоимость» гр.14), цены ТОЛЬКО на строки
оборудования/материалов (работы без цен), N = живая формула =ROUND(M*G,2) от объёма «с учётом
изменений», строка «ИТОГО за раздел» перед подписями, выход в отдельную папку. Пилот — пара
«раздел ММ ↔ СВОР МС», ядро переиспользуется на остальные 7 пар.

## Ключевые уроки (переиспользуемые)
1. **Матчинг ДС↔СВОР — по описанию, № не ключ.** Нумерация расходится (в ДС дыра в 34 номера
   без позиций — вычеркнули без перенумерации; сумма при этом сходится с итогом).
   Алгоритм: монотонный указатель + ОКНО (40 строк пула), fallback «производитель+модель»
   в окне, затем строгий fuzzy (ratio≥0.95 + ед.изм + кол-во), глобальный добор только по
   точному описанию;模ель глобально — только при единственном кандидате.
2. **Глобальный прыжок указателя = лавина**: одна нематчащаяся позиция (описание в СВОР
   отредактировано: другая ревизия железа) утаскивает указатель к дублю в другом разделе,
   и весь остаток файла матчится «вне порядка». Окно вместо глобального скана лечит.
3. **Model-fallback ворует строки РАБОТ**: строки-работы наследуют производителя/модель через
   вертикальный merge блока → ключ (производитель, модель) у работы совпадает с оборудованием,
   а работа стоит ВЫШЕ. Фильтровать кандидатов по глагольным префиксам работ.
4. **Гомоглифы**: кириллические а/е/о/р/с/у/х/к в латинских моделях и наоборот — фолдить с
   ОБЕИХ сторон (translate-таблица после lower()); но тогда и ВСЕ сравнения с русскими
   литералами («Раздел», «Составил», «Итоговая сумма») обязаны идти через ту же нормализацию —
   иначе тихо ломается детект секций/подписей/итога.
5. **Дубли описаний с разными ценами в ДС** (8 шт: мониторы, камеры, кабели по залам) —
   именно поэтому порядок следования обязателен в ключе матчинга.
6. **Одна строка СВОР на две позиции ДС** (та же опция в двух залах, в СВОР учтена одна):
   монотонность отдаёт строку «своему» залу, второй экземпляр честно попадает в непокрытые.
7. **Ручные привязки**: manual_overrides.json (ds_num→строка) — механизм нужен, но овверрайд
   ОБЯЗАН проверять, что строка не принадлежит другой позиции по точному совпадению
   (мой первый овверрайд отнял строку у законного владельца — отменён).
8. **win32com, русская локаль**: `Range.NumberFormat = "#,##0.00"` кидает COM-ошибку —
   падать в `NumberFormatLocal = "# ##0,00"`. Вставка строк через `Rows.Insert` — формулы/merge
   сдвигаются корректно; verify обязан сравнивать merge с учётом сдвига (+2 ниже точки вставки).
9. **Запись колонок в конец (M/N) через COM безопасна**: не трогает формулы граф 1–12
   (self-referencing IF/ROUND) — подтверждено сравнением с эталоном (счётчик формул + выборка).
10. **docx-числа**: «Кол-во» в ДС бывает «шт» при фактических метрах — сверять ЧИСЛА
    (кол-во ДС == объём СВОР), а не доверять метке единицы; безобидно, если числа равны.
11. **excel-validator без Bash**: в сессии субагента может не оказаться Bash и кэша значений
    (read_data_from_excel отдаёт формулы) — планировать так, чтобы детерминированная verify-стадия
    оркестратора закрывала кэш/ширины/merge-diff заранее, а ревьюеру оставлять форму и сравнение
    с эталоном.
