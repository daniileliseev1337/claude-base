# Перечень неточностей ПТО-базы (валидация 2026-06-20)

Все места, где практик (customer zero) правил/уточнял ответ доменного агента `id-engineer`
по 5 вопросам ИД. Цель — внести в граф ИД (`blocks/pto/skills/id-volume-graph/id_volume_extract.json`)
при следующей актуализации (с ревью `auditor` + rebuild). Обезличено.

Степень: 🔴 существенная (модель ошибочна) · 🟡 уточнение (сужение/неполнота) · 🔵 не ошибка, а разъяснение ПО vs практика.

## 🔴 1. Мандаторность ВОР (SD-100 Q3) — главная — ✅ ВНЕСЕНО В ГРАФ 2026-06-20
- **База:** ВОР привязана только к АОСР; у АОМР ВОР «нет / не типично».
- **Практика:** ВОР — у **каждого акта, который закрывает работы** (несёт материалы/оборудование),
  включая **АОМР**. ВОР живёт на ИС и должна полностью соответствовать ВСО (там, где ВСО есть).
- **В граф:** ребро «ВОР ← закрывающий работы акт» (АОМР + все АОСР); ВСО — только у АОМР;
  АОУСИТО и акты испытаний/промывки/ПНР — без ВОР и ВСО (фиксируют событие, не состав).

## 🔴 2. Правило дат паспорт↔акт (SD-105 E) — ✅ ВНЕСЕНО В ГРАФ 2026-06-20
- **База:** статус срока документа качества — абсолютный («истёк/истекает» на сегодня),
  серверное разнесение состояния в хранилище.
- **Практика:** срок релевантен **относительно ДАТЫ АКТА**, не абсолютно. У актов разные даты —
  один паспорт для одного акта просрочен, для другого нет. Абсолютный флаг в хранилище справочен;
  ОСНОВНАЯ валидация — документ действителен на дату акта (проверка при привязке паспорт↔акт).
- **В граф:** правило «документ качества действителен на дату акта» как инвариант привязки.

## 🔴 3. Движок «было-стало» РД↔ИД (SD-102 Q2a)
- **База:** «Поз.» спецификации РД трактовалась как навигационный номер позиции.
- **Практика:** отдельный ordinal «Поз.» нужен для **движка «было-стало» РД↔ИД** (diff позиций:
  новые/изменённые/удалённые). Номера позиций РД и ПВ совпадать не будут — это норма.
- Это продуктовая специфика ПО, которой в графе методологии нет (граф про ручную практику).

## 🟡 4. Перенос акта между разделами (SD-116 Q1в)
- **База:** привела сценарий «перенос акта в другой раздел РД со сменой шифра» через пример
  переноса позиций (кейс <позиция>).
- **Практика:** в том примере переносились **материалы между существующими актами** (АОМР→АОСР),
  номера актов не менялись — это **не перенумерация**. Смена шифра реальна, но через
  **копирование акта-аналога** в другой раздел, а не перенос материалов.

## 🟡 5. Стоимость переподписания подписанного акта (SD-116 Q4)
- **База:** переподписание подписанного акта «дни/недели», делают «только при грубых ошибках».
- **Практика:** при **переформировании ИД** по требованию заказчика (разбить акт на два / добавить
  акт), когда работы уже приняты, переподписание проходит **легко, без повторных перепроверок** —
  сканы подтверждают принятые работы, меняется лишь концепция ИД.

## 🟡 6. Стадия П в пилоте (SD-102 Q1)
- **База:** «ТЗ / проектная / заключение экспертизы — вне зоны ИД, в пилот не входит».
- **Практика:** **стадия П** — важная отдельная стадия проектирования со своей спецификацией
  (может отличаться от РД); учитывать в пилоте хотя бы как факт наличия (`kind: project`).
  ИД строится по РД (первично), но П игнорировать нельзя.

## 🟡 7. Кросс-томный поиск документов качества (SD-105 G)
- **База:** фильтры по «разделу РД» и «поставщику» как равнозначные.
- **Практика:** фильтр по **поставщику** — да; по разделу РД — умеренно; важнее
  **кросс-томный поиск** «в каком томе/строении уже есть документ на эту позицию» (даёт реальную
  экономию за счёт унификации документов между строениями).

## 🔵 8. Привязка паспорта (SD-98) — разъяснение, не ошибка
- **База:** документы качества привязаны к акту (из физики реального тома — подшиты за актом).
- **Уточнение:** в ПО привязка **паспорт → позиция РД → позиция ПВ → акт** (через позиции).
  У ручной практики задачи такой увязки нет (документ просто подшит за актом) — у ПО есть, и это
  его ценность (через неё работают проверка комплектности и правило дат). Не баг модели — задача ПО.
