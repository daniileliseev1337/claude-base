# 2026-07-09 — <объект>: таблица «было-стало» СОТ, блок «СТАЛО <дата>» по последним спекам

## Задача
Вести версионную таблицу «было-стало» (протокол ТДЦ, xlsx, один лист): слева БЫЛО, правее
столбцовые блоки «СТАЛО <дата>». По 5 последним спецификациям раздела СОТ (.xlsm по блокам
здания, разные изм.) добавить новый блок «СТАЛО 09.07.2026» — состав/кол-ва/модели, БЕЗ цен
(решение пользователя: цены живут в СВОРах, таблица кормит их количествами — «цепь взаимосвязей»).
Прошлые блоки не трогать. Плюс живой дашборд задачи и усиление ядра (персистентные стейты).

## Что сделано
- Ядро `tools/bylo_stalo.py`: parse-specs (кэш формул «Кол.», BLOCKER при отсутствии кэша),
  parse-bs (детект блоков по «№ п/п» в шапке — расширяемо на N блоков), зонный diff
  (ZONE_MAP: подсекция спеки → corpus-секции листа), журнал `work/BS/runs.jsonl`.

  verify (14 195 проверок: значения нового блока, пустота ценовых, формулы ВСЕГО всех блоков,
  ширины, merge со сдвигом, ПОЯЧЕЕЧНАЯ неизменность старых блоков с учётом вставки).
- `apply_bs.py` (win32com на копии): Rows.Insert, ширины/формат PasteSpecial с блока-образца,

  пользователя. Всё локализовано до строк автоматически зонной сверкой.

## Ключевые уроки (переиспользуемые)
1. **Гомоглиф-норм на границе построения, не использования.** Урок «все литералы через norm()»
   из памяти укусил ДВАЖДЫ в новом коде (ключи ZONE_MAP, множество SUBSECTIONS): словари/
   константы с кириллицей нормализовать в момент ПОСТРОЕНИЯ структуры
   (`{norm(k): [norm(x)…]}`), тогда места использования не смогут ошибиться.
2. **NFKC превращает «№»→«No»** — ключи зон/секций после norm() внезапно латинизируются;
   сравнивать можно только norm↔norm, никогда norm↔сырое.
3. **`sys.stdout = TextIOWrapper(...)` в импортируемом модуле ломает импортёра**
   (ValueError: I/O operation on closed file — GC первого враппера закрывает buffer).
   Правильно: `sys.stdout.reconfigure(encoding="utf-8")` в try/except.
4. **Раскладка замен моделей — два ключа.** Агрегат спеки ищется по НОВОЙ модели, а
   строки-носители в старом блоке — по СТАРОЙ (carrier_key). Одноключевая логика даёт
   «спека N не раскладывается на 0 строк».
5. **Инвариант-гейт плана окупается мгновенно:** «Σ плана == Σ спек по каждой модели, все
   позиции потреблены, иначе план не сохраняется» — поймал ошибку п.4 до записи в файл.
6. **Зонная сверка > агрегатной.** Расхождения ±1 по модели, невидимые на сумме файла,
   локализуются до конкретной зоны и строки, и большинство «спорных» привязок решаются
   числовым доказательством (по комплектам «станция+ИБП+SNMP»), а не вопросом.
7. **Столбцовое версионирование xlsx:** новый блок = зеркало 13 колонок правее; вставка
   строк двигает обе стороны синхронно и расширяет SUM автоматически; verify обязан
   доказывать неизменность прошлых блоков поячеечно СО СДВИГОМ (old_row→new_row).
8. **PasteSpecial(formats) блока-образца переносит merge/границы/числовые форматы разом**,
   значений не несёт — идеально для «один в один по виду»; ширины отдельно
   (xlPasteColumnWidths), разделительную колонку — копией узкого разделителя-образца.
9. **Дашборд как чистая функция от стейтов** (`build()` читает все JSON + журнал) — «живой»
   означает: любой прогон ядра меняет стейты, пересборка одной командой, redeploy тем же
   file_path. Ничего не рисуется руками.
10. **Спека с формулами в «Кол.»** (H7=H6-H8): парсить два экземпляра книги (data_only
    True/False), кэш отсутствует → BLOCKER, не гадать.

## Продолжение сессии — цепь «СТАЛО → СВОР/ВОР» (та же дата)

По инварианту пользователя «СПЕКА = было-стало = СВОР = ВОР = СВОР+цены» построена цепь:
подсветка изменений в было-стало (4 функциональных цвета, verify заливок), зонный план
правок СВОР/ВОР (по 44 на файл: qty/model/zero/insert/price_fix/work_fix), apply win32com
на копии в «Актуальные <pair>», «без цен» = детерминированная производная «с ценами»
(минус графы цен и ИТОГО), chain-verify: агрегаты моделей всех трёх файлов == 5 спекам ==
СТАЛО (PASS), recount ИТОГО == Σ граф стоимости == прежний итог + расчётная дельта плана
копейка в копейку. Вставки блоков — по шаблонам БЛОКОВ ТЕХ ЖЕ МОДЕЛЕЙ из самих файлов
(работа+позиция, fallback по имени позиции) — ноль выдуманных текстов работ.

## Уроки цепи (переиспользуемые, продолжение нумерации)
19. **Гомоглиф-дисциплина — 4 укуса за сессию:** ZONE_MAP-ключи, SUBSECTIONS-литералы,
    regex «^Раздел:», ключи REPLACEMENTS с кириллицей в моделях. Правило усилено: ЛЮБАЯ
    константа/ключ/паттерн, сравнивающийся с norm()/mkey()-строкой, строится ЧЕРЕЗ ту же
    функцию в момент определения (`{B.mkey(k): v}`, `re.compile(re.escape(B.norm(...)))`).
20. **Классификатор строк-работ — первое слово целиком, не префикс:** «Монтажный комплект»
    и «Устройство защиты» — позиции; глагольный префикс ловит ложняки, слово из множества
    {монтаж, установка, …} — нет.
21. **verify не смеет мутировать стейты пайплайна:** parse_blocks(save=False) для копий;
    перезапись *_blocks.json результатами сломала шаблоны вставок при повторном apply
    (симптом: «нет шаблона», дубли вставок).
22. **Вставка строк на ГРАНИЦЕ диапазона SUM не расширяет его** — формулу итога переписывать
    после вставок на актуальный диапазон (=ROUND(SUM(N20:N{itog-1}),2)).
23. **Вставные блоки: якорь получает маркер-число (−1), финальная перенумерация колонки A
    делает №№ сквозными** — без числа в якоре блок невидим для парсера merge-блоков.
24. **Ценовая лестница вставок/замен:** ДС → та же модель в этом же файле → рыночная ×1,30 →
    пусто+эскалация; ревизия того же изделия наследует old_price, смена класса — по лестнице.
    Денежные замены — пакетом на подтверждение с Decimal-дельтой ДО записи.
25. **Зонная сверка вскрывает «долги» прошлых этапов:** «лишние/недостающие» в СВОР оказались
    ровно теми «12 позициями ДС без строк», что висели вопросом — инвариант закрыл их
    вставками с ДС-ценами, а «внутренние» ошибки (работа=2 при позиции=1, цена чужой модели
    на строке) нашлись как побочный эффект.
