# Session report — экспорт спецификаций оборудования (СО) из томов ПД в Excel

**Дата:** 2026-07-15
**Задача:** этап 2 бюджета `<объект>` — извлечь СО (ГОСТ 21.110) 12 разделов (раздел «Сети связи»
ИОС5.x + раздел ТХ) из томов ПД (PDF) в один Excel: лист на раздел + сводный. Опора на СО, не ВОР.
Цены не навешиваются (следующий этап).

## Что сделано
- Один Excel: Сводный + 12 листов-разделов, единая структура колонок (№/наименование/тип/код/
  завод-изготовитель/ед/кол-во/масса/[ценовые плейсхолдеры]/примечание). Итого извлечено ~414 позиций.
- Извлечение — **детерминированным скриптом** (движок coordinate-таблиц по векторным линиям), НЕ fan-out LLM.
- Пройдено двумя ревьюерами (форма xlsx + содержательная сверка кол-в с PDF-СО).

## Метод (переиспользуемо)
1. **Deterministic-first.** Таблицы СО в векторном PDF читаются `page.find_tables(strategy="lines")` +
   динамический маппинг колонок по тексту шапки. Дёшево, 0 галлюцинаций, точнее ручного чтения. Fan-out
   LLM по разделам не нужен — один Python-скрипт по всем томам.
2. **Разведка формы ДО генерации.** Сначала прогон-детект: где лист СО в каждом томе, повёрнут ли (rot),
   сколько строк, единая ли шапка. Реальная форма может расходиться с ТЗ (у нас: листы оказались НЕ повёрнуты).
3. **Верификация выборкой против досье** + два независимых ревьюера (форма + содержание vs источник).

## Грабли / уроки
- **Штатный `extract_co` движка co-verify НЕ мапит колонку «Завод-изготовитель» и «Масса».** Для бюджетных
  задач (где изготовитель заменяется на вендора) нужен свой маппинг колонок поверх движка. Файлы базы не
  править — переиспользовать только нормализаторы, свой экстрактор в папке проекта.
- **Детект СО-страницы:** штатный `is_co_page` требует «Кол.» с точкой, но в шапке перенос «Коли-\nчество» →
  детект падает. Надёжнее: заголовок «Наименование и техническ» + успешный маппинг колонок (pos+qty).
- **Индексы колонок плавают между томами** (завод в col9 или col10) → маппинг только динамический по шапке, не по фиксированному индексу.
- **Спецификация может отсутствовать как таблица.** В одном разделе (вертикальный транспорт) отдельного
  листа СО не было — позиции собраны из СО смежного тома + опросного листа. Разведка это вскрыла, не ТЗ.
- **Пронумерованные подзаголовки-группы** («Оборудование», «Кабель и материалы») в СО имеют номер позиции —
  показывать номер, иначе в нумерации «дырки» и ложные замечания ревьюера.
- **Расхождение ревьюеров разрешать фактом, не мнением.** Один ревьюер дал NOT PASSED из-за суммы кол-ва
  (посчитал вручную по десяткам строк с qty>1 и ошибся на 1). Разрешено детерминированным пересчётом
  (openpyxl суммирует колонку) — объективно, воспроизводимо. Не резолвер-агент, а машинная проверка.
- **Семантическая фильтрация «наша часть».** Где в СО перемешаны наши и чужие позиции по помещениям —
  классификация по ключевым словам наименования + вынос спорных в отдельный справочный блок (решение владельца).

## Дополнение — ценовой пилот (следующий этап, та же сессия)
Задача: навесить рыночные цены на извлечённую спецификацию (2 бюджета: бренды проекта / вендоры пула компании).
- **Дедуп ДО поиска:** N строк спецификации → M уникальных артикулов (у нас 414 → 198). Цену искать на уникальный, размножать на вхождения. Экономит ~2×.
- **Пилот 1 раздела перед полной фазой:** прогнать цены по одному представительному разделу, оценить реализуемость/качество/токены, ТОЛЬКО потом масштабировать. Пилот дал 28/30 цен, откалибровал метод.
- **Fan-out снабженцев (веб-поиск цен):** источник — рыночные каталоги/дистрибьюторы; каждая цена с URL+уверенностью+датой; не нашёл → «не найдено» + что проверено (не выдумывать). Группировать по поставщику (1 каталог = 3-5 позиций).
- **Грабли матчинга цены обратно на строки:** матчить по АРТИКУЛУ, не по наименованию (два «Кабель» с разным типом → частичный name-матч подставил чужую цену). Урок: ключ матча — артикул, name-фолбэк строгий.
- **web_get.py требует Bash** — на Windows у субагента его нет; веб-поиск шёл через semantic-search MCP напрямую, работает.
- **Вывод по вендор-замене:** там где бренды проекта уже импортозамещены/совпадают с пулом — колонка «вендор» = «бренд проекта»; реальные замены только в узлах с импортом.
- **Тяжёлую полную фазу (сотни артикулов, ~1,5-2 млн токенов) — в новый чат + Workflow**, с оценкой стоимости ДО старта.
