# 2026-07-14 — Экспертиза крупного тома РД СКС (320 стр.) через Workflow

## Что сделал
Экспертиза последнего (5-го) тома РД СКС-ЛВС-ТФ <объект> — 320 стр., 349 МБ, Изм.13.
Пайплайн: extract постранично (PyMuPDF) → детерминированные сверки скриптами (суммы кабеля
по ~9800 линиям КЖ, привязка страница→лист, парс ~980 позиций спецификаций) → Workflow
(16 читателей по блокам страниц + auditor + audit-rd-section + артикульщик, все sonnet;
адверсариальная верификация чанками по 4; итого 42 агента, ~8,1 M токенов субагентов) →
свой визуальный ГОСТ-аудит рендеров → генератор findings из JSON-результата Workflow →
скрины → docx → ревью word-checker (PASS) + auditor (PASSED после 1 правки).
Итог: ведомость 62 замечания (5 крит.), 53 скрина. Вердикты артикулов перенесены из
предыдущего тома без повторного веб-поиска (+18 новых через exa).

## Что сработало хорошо
- Deterministic-first: суммы кабеля/парс спецификаций скриптами ДО фан-аута — агенты получили
  готовые числа, а кросс-сверка (медь в СО ×2 от КЖ) стала бесспорной.
- Схема Workflow «читатели → кросс → адверсариальный verify чанками по 4» отработала чисто:
  86/91 подтверждено, все 5 REFUTED — реальные артефакты текст-слоя (верификаторы с рендером).
- Дубли результатов Write'ом на диск (wf_reports/*.json) — страховка от потери результатов Workflow.
- Точечная доверка после правки: SendMessage тому же auditor вместо полного повторного аудита.

## Где сломался / грабли
1. **«Разрывы нумерации» в таблицах = потери текст-парсера на многострочных записях.** И читатель,
   и верификатор подтвердили несуществующий дефект по текстовому слою; опровергла только
   рендер-проверка. Правило: числовые/структурные утверждения субагентов о таблицах чертежей —
   пересчитывать скриптом и/или проверять рендером ДО вставки в findings (пересчёт также поправил
   количество и перечень листов со служебной заглушкой формата).
2. **Лишний лист-дубль в томе сдвигает привязку страница→лист** у всех последующих блоков.
   Сначала детект дублей страниц (hash), потом раздача диапазонов читателям.
3. **PyMuPDF find_tables на векторных планах A0 — минуты на страницу** (320 стр. ≈ 40+ мин).
   Решение: планы выгружать текстом без таблиц, find_tables — только на табличных листах
   (КЖ/спецификации). Фоновый Bash-таймаут 10 мин убивает долгий extract — дробить на диапазоны.
4. find_tables сливает многоколоночные таблицы (3 секции КЖ в одну ячейку) — для таких листов
   парсер по тексту (4-строчные группы) надёжнее.

## Источники
Локальная выгрузка тома (page_NNN.txt + table JSON), эталонные скрипты серии томов,
перенос вердиктов артикулов из ведомости предыдущего тома, exa для новых артикулов.
