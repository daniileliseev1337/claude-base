# Session-report — сквозная верификация трёх бюджетов A/B/C (СС-проект)

> Обезличено для общей базы (`<объект>`, `<шифр>`, суммы — как метод, не как факт объекта).
> Реальные данные, пути и числа — в памяти проекта `<проект>/Claude/` (ЖУРНАЛ/STATUS/ПЛАН-БЮДЖЕТА).
> Дата: 2026-07-16. Этап 5 закрыт (дашборд A/B/C), готовится этап 6 (верификация).

## Контекст задачи

Бюджет проекта по зоне слаботочки: 12 разделов (ИОС5.1–5.10 + ТХ1 + ТХ2), опора на
**спецификации оборудования (СО, ГОСТ 21.110)**, не ВОР. Три оценки на раздел:
- **A** — по смете заказчика (ЛСР ГрандСмета РИМ, формат `.gge` = XML).
- **B** — по спецификации стадии П (бренды проекта), цены рыночным веб-поиском.
- **C** — та же СО с заменой завода-изготовителя на вендоров подрядчика (пул К-7).

Всё сведено в один xlsx (лист «Сводный» + 12 листов-разделов + лист детализации A).
Дашборд — self-contained HTML (Artifact), свод A/B/C + детализация C.

## Метод по каждому бюджету (что проверять в этапе 6)

### A — по смете (.gge)
- Итоги берутся из узла `EstimatePrice/` каждого `.gge`: `Equipment/Total`,
  `Summary/Materials/Total`, `Summary/Total`.
- **об+мат** = Equipment/Total + Summary/Materials/Total; **полная** = Summary/Total; обе ×1.22 (НДС).
- Один ЛСР на раздел (нумерация последовательная). Технологический раздел — сборный:
  большая часть (лаборатория/мебель/пищеблок) НЕ наша → вычленять по составу СО.
- ⚠ ГРАБЛЯ: часть нужных позиций в смете проходит тегом `<Material>`, а не `<Equipment>`
  (напр. настенный экран) — при сборе оборудования теряются. Проверять ОБА тега.
- Сверка: `.gge`-итоги должны совпасть с Excel-дублями ЛСР 1:1.
- ⚠ Скрипты A-пересчёта прошлого этапа были в session-scratchpad (не в папке проекта) —
  для верификации, скорее всего, придётся ВОССОЗДАТЬ парсер по этому методу.

### B — спецификация ПД (цена × количество)
- По каждой позиции СО: сумма = кол-во (из СО) × цена (веб, розница РФ с НДС).
- ⚠ ГРАБЛЯ (укрупнённые единицы): если позиция сметы/СО в «100 шт» или «10 м» —
  натуральное кол-во = множитель × значение; цену и кол-во держать в одной базе.
- ⚠ Матчинг цен на лист — по АРТИКУЛУ (tip), не по наименованию; ключ (name, tip) дословно.
- ⚠ Двойной счёт в своде: строка-ИТОГО раздела не должна попадать в SUM (диапазон без итога).
- Обезличенные позиции (ПК/МФУ без бренда) — по типовой рыночной технике, помечены в примечании.
- 17 артикулов общие для 2+ разделов — цену искать на уникальный, размножать на вхождения.

### C — вендоры К-7
- Замена только где вендор реально есть; иначе **C = B** (пометка «нет аналога / бренд ПД»).
- ⚠ Класс аналога решает цену (варифокал vs фикс-камера — ×2; NGFW vs SOHO-роутер — ×60 нерелевантно).
- Не занижать при неясной конфигурации ПД → оставлять C = B + флаг.
- Публичные цены — веб (Exa напрямую, последовательно, не параллельный fan-out — иначе
  free-limit exa + single-browser playwright коллизия). Непубличные — ждут КП (ориентир C=B).

### Свод (лист «Сводный») и дашборд
- D/E разделов = SUM позиций листа; строка ИТОГО = SUM разделов; C = A(об+мат), H = A(полная).
- ⚠ Формулы правит только **win32com** (реальный Excel) — openpyxl рвёт кэш формул (D/E → None).
- ⚠ MCP-excel отдаёт формулы как ТЕКСТ без кэш-значения → числа из формул проверять
  ПЕРЕСЧЁТОМ из сырых ячеек листов-разделов, не доверять формуле «на глаз».
- ⚠ Сумма построчно-округлённых ≠ округление итога — судить по неокруглённому totals.

## Задача этапа 6 (постановка)

Независимая **сквозная** верификация ВСЕХ вычислений трёх бюджетов на ошибки:
от источников (`.gge`, СО-PDF, веб-цены) до итогов Сводного и дашборда. Не только
«дашборд == Сводный» (это уже подтверждено до рубля в этапе 5), а вся цепочка:
1. A — пересчёт каждого раздела из `.gge` (оба тега, ×1.22, вычленение ТХ по СО), сверка с Excel-дублями.
2. B — пересчёт кол-во×цена по каждой позиции, единицы (укрупнённые!), нет двойного счёта, ключи матчинга.
3. C — корректность замен/флагов C=B, класс аналогов, суммы.
4. Свод — формулы, диапазоны SUM, НДС единообразно, базы не смешаны (об+мат vs полная).
5. Кросс: единицы измерения, размножение общих артикулов на вхождения, обезличенные позиции.

**Deterministic-first:** пересчёт Python-скриптами (точное сравнение), НЕ fan-out LLM.
Ревьюер-гейт: `auditor` (+ `excel-validator` при правках xlsx). Расхождение ревьюеров → резолвер.

## Критерий готовности этапа 6
Каждый из трёх бюджетов пересчитан из первоисточника независимым скриптом; расхождения
с текущими числами Сводного/дашборда перечислены построчно (или подтверждён 0 расхождений);
найденные вычислительные ошибки исправлены через win32com (кэш B/C сохранён) + пересборка
дашборда; финальное ревью PASSED.

## Грабли-сводка (для быстрого старта)
`.gge` итоги в EstimatePrice · экран=Material не Equipment · openpyxl рвёт кэш → win32com ·
MCP-excel формулы без кэша → пересчёт из сырья · укрупнённые единицы (100 шт/10 м) ·
матчинг по артикулу не имени · класс аналога решает цену C · параллельный веб-fanout убивает
exa/playwright · сумма округлений ≠ округление суммы.

## Результаты верификации (этап 6 ВЫПОЛНЕН)

Сквозная верификация проведена независимыми Python-скриптами (openpyxl + xml.etree, deterministic-first,
не fan-out LLM). Снимок текущего свода снят дважды (формулы + кэш-значения). **Вывод: вычислительных
ошибок в трёх бюджетах и своде — НОЛЬ; одна MINOR-несогласованность входных цен (~0,006%).**

### A — по смете (.gge): 0 расхождений
- 10 разделов ИОС: `(Equipment/Total + Summary/Materials/Total)×1.22` = столбец A(об+мат); `Summary/Total×1.22`
  = A(полная) — совпало со сводом ДО КОПЕЙКИ по всем 10 (парсер воссоздан, старые скрипты были в scratchpad).
- Технологический раздел (сборный, 82% чужое): оборуд/материалы = ресурсы `<Equipment>`/`<Material>` внутри
  расценок `<Item>`, не отдельные позиции. Наша часть (ИТ+оргтехника+мультимедиа+связь) верифицирована
  **subset-sum** по ресурсам сметы — единственное точное подмножество (19 позиций) даёт заявленную сумму
  оборудования до рубля; экран найден как Material-ресурс; лифты — как один Equipment-ресурс ×2. Единственность
  набора = сильное подтверждение границы «наше/чужое», а не случайное совпадение сумм.
- Методологическое допущение (не дефект, заявлено в детализации): для технолог. части полная = об+мат
  (монтаж/ПНР по нашей части в общей смете не выделен) → занижает справочную «полную» по этим строкам; на
  сравнение с B/C (база об+мат) не влияет. Стоит явно продублировать в примечании свода.

### B — спецификация ПД: 0 вычислительных, 1 MINOR
- Кол-во на листах ↔ первоисточник СО = **1:1 по порядку позиций** (все разделы). Сверка по ИМЕНИ даёт ложные
  срабатывания на дубликатах наименований (одна номенклатура в разных строках/помещениях с разным кол-вом) —
  сверять ПО НОМЕРУ ПОЗИЦИИ, не по имени.
- **Укрупнённых единиц НЕТ** (только шт/м/кг/комплект) — грабля «100 шт/10 м» не материализовалась, множитель
  не нужен. Проверять всё равно (в этом проекте кол-ва в натуральных ед.).
- Сумма позиции = кол-во×цена на всех строках; свод по разделу = SUM строк данных; двойного счёта нет
  (единственная строка-ИТОГО на пилотном листе корректно исключена ограниченным диапазоном SUM).
- Справочные позиции (мебель/интерьер вне зоны) имеют кол-во, но пустую цену → не входят в SUM.
- **MINOR (к согласованию, не правил):** общие артикулы (в 2+ разделах) — пилотный раздел, заполнявшийся
  отдельно и раньше, хранит для 5 материалов/крепежа цену, отличную от основной фазы (мастика/дюбель/трубы/
  кабель). Нетто-эффект приведения к единой цене ≈ 0,006% бюджета, разнонаправленный. Это несогласованность
  ВХОДНЫХ данных (обе цены рыночные), не арифметическая ошибка; правка требует решения «какая цена эталонная»
  = деньги → вопрос пользователю, не менять молча. Принцип «цена на уникальный, размножать на вхождения»
  соблюдён для 12 из 17 общих артикулов.

### C — вендоры подрядчика: 0 расхождений
- **Полнота идеальна:** каждая позиция с ценой B имеет и цену C (0 позиций «с B без C») — C ничего не теряет.
- Дельта (итог C − итог B) = сумма всех замен `Σ кол-во×(цена_C − цена_B)` **до копейки** → C полностью
  выводится из B + замены, ни одна не потеряна/лишняя.
- Замены осмысленны, класс аналога корректен (премиальный СКС-вендор дороже generic; смена ТВ-бренда дешевле;
  аналог→IP-телефон дороже — смена класса; варифокал-камера). Где вендора нет → C=B (флаг).

### Свод/кросс
- Формулы SUM без захвата строк-ИТОГО и без обрезки данных; базы не смешаны (об+мат отдельно, полная отдельно);
  НДС применён единообразно (смета ×1.22 к цене без НДС; B/C — розница уже с НДС).

### Уроки этапа 6
1. **Сверять кол-во по НОМЕРУ ПОЗИЦИИ, а не по наименованию** — дубли имён (номенклатура по помещениям,
   разные модули с одним названием) дают массовые ложные расхождения при матчинге по имени.
2. **Вычленение «наше/чужое» из сборной сметы проверяется subset-sum** по ресурсам (Equipment/Material внутри
   расценок): единственность подмножества, дающего заявленную сумму, = доказательство границы.
3. **Дельта C−B = Σ замен до копейки** — исчерпывающая проверка вывода C из B (полнота + арифметика замен).
4. **Несогласованность входных цен ≠ вычислительная ошибка** — если каждая строка кол-во×цена верна, а расходятся
   лишь цены одного артикула между источниками, это к согласованию (деньги → пользователю), не «ошибка расчёта».
5. **Ревьюер без исполняемого кода** (JS-калькулятор вместо Python) досчитывает крупные листы с риском ручной
   транскрипции → давать ему извлечённое сырьё крупного листа для дуплекса, граница «не досчитал» закрывается
   не как дефект, а до-фасовкой.

## Методичка/навигатор + деловой стиль xlsx + дашборд-вкладки

- **Ограничение канала уточнено:** запрет — на публикацию через сайт вендора (Artifact-инструмент) и на
  браузер-автоматизацию, НЕ на HTML-формат. Локальные HTML-файлы в папке проекта — можно (пользователь
  открывает сам). Первая интерпретация «запрет HTML вообще» была шире, чем нужно — поправлено.
- **Методичка/навигатор первым листом** в мастер-книгу бюджета: итоги, кликабельный навигатор по листам
  (внутр. гиперссылки) + по файлам папки + по структуре проекта — «где что смотреть».
- **Единый деловой стиль** (переиспользуемый форматтер): Times New Roman, границы, автоперенос, вертикальное
  центрирование, autofit высоты, нейтральная шапка. Зафиксирован в память, чтобы не собирать заново каждый раз.
- **Грабля инструментов:** субагенты-ревьюеры на Windows не имеют Bash → не читают кэш формул (data_only) и
  стили openpyxl. Их «не смог проверить» — граница спавна, не дефект: закрывается до-фасовкой (txt-выгрузка
  вычисленных значений / дамп стилей), ревьюер читает файл через Read и сверяет сам. Файл с формулами (кэш
  соседних бюджетов) форматировать только win32com — openpyxl рвёт кэш.
