# Session-report — сквозная верификация трёх бюджетов A/B/C (СС-проект)

> Обезличено для общей базы (`<объект>`, `<шифр>`, суммы — как метод, не как факт объекта).
> Реальные данные, пути и числа — в памяти проекта `<проект>/Claude/` (ЖУРНАЛ/STATUS/ПЛАН-БЮДЖЕТА).
> Дата: 2026-07-16. Этап 5 закрыт (дашборд A/B/C), готовится этап 6 (верификация).

## Контекст задачи

Бюджет проекта по зоне слаботочки: 12 разделов (ИОС5.1–5.10 + ТХ1 + ТХ2), опора на
**спецификации оборудования (СО, ГОСТ 21.110)**, не ВОР. Три оценки на раздел:
- **A** — по смете заказчика (ЛСР ГрандСмета РИМ, формат `.gge` = XML).
- **B** — по спецификации стадии П (бренды проекта), цены рыночным веб-поиском.
- **C** — та же СО с заменой завода-изготовителя на вендоров подрядчика (пул К-7).

Всё сведено в один xlsx (лист «Сводный» + 12 листов-разделов + лист детализации A).
Дашборд — self-contained HTML (Artifact), свод A/B/C + детализация C.

## Метод по каждому бюджету (что проверять в этапе 6)

### A — по смете (.gge)
- Итоги берутся из узла `EstimatePrice/` каждого `.gge`: `Equipment/Total`,
  `Summary/Materials/Total`, `Summary/Total`.
- **об+мат** = Equipment/Total + Summary/Materials/Total; **полная** = Summary/Total; обе ×1.22 (НДС).
- Один ЛСР на раздел (нумерация последовательная). Технологический раздел — сборный:
  большая часть (лаборатория/мебель/пищеблок) НЕ наша → вычленять по составу СО.
- ⚠ ГРАБЛЯ: часть нужных позиций в смете проходит тегом `<Material>`, а не `<Equipment>`
  (напр. настенный экран) — при сборе оборудования теряются. Проверять ОБА тега.
- Сверка: `.gge`-итоги должны совпасть с Excel-дублями ЛСР 1:1.
- ⚠ Скрипты A-пересчёта прошлого этапа были в session-scratchpad (не в папке проекта) —
  для верификации, скорее всего, придётся ВОССОЗДАТЬ парсер по этому методу.

### B — спецификация ПД (цена × количество)
- По каждой позиции СО: сумма = кол-во (из СО) × цена (веб, розница РФ с НДС).
- ⚠ ГРАБЛЯ (укрупнённые единицы): если позиция сметы/СО в «100 шт» или «10 м» —
  натуральное кол-во = множитель × значение; цену и кол-во держать в одной базе.
- ⚠ Матчинг цен на лист — по АРТИКУЛУ (tip), не по наименованию; ключ (name, tip) дословно.
- ⚠ Двойной счёт в своде: строка-ИТОГО раздела не должна попадать в SUM (диапазон без итога).
- Обезличенные позиции (ПК/МФУ без бренда) — по типовой рыночной технике, помечены в примечании.
- 17 артикулов общие для 2+ разделов — цену искать на уникальный, размножать на вхождения.

### C — вендоры К-7
- Замена только где вендор реально есть; иначе **C = B** (пометка «нет аналога / бренд ПД»).
- ⚠ Класс аналога решает цену (варифокал vs фикс-камера — ×2; NGFW vs SOHO-роутер — ×60 нерелевантно).
- Не занижать при неясной конфигурации ПД → оставлять C = B + флаг.
- Публичные цены — веб (Exa напрямую, последовательно, не параллельный fan-out — иначе
  free-limit exa + single-browser playwright коллизия). Непубличные — ждут КП (ориентир C=B).

### Свод (лист «Сводный») и дашборд
- D/E разделов = SUM позиций листа; строка ИТОГО = SUM разделов; C = A(об+мат), H = A(полная).
- ⚠ Формулы правит только **win32com** (реальный Excel) — openpyxl рвёт кэш формул (D/E → None).
- ⚠ MCP-excel отдаёт формулы как ТЕКСТ без кэш-значения → числа из формул проверять
  ПЕРЕСЧЁТОМ из сырых ячеек листов-разделов, не доверять формуле «на глаз».
- ⚠ Сумма построчно-округлённых ≠ округление итога — судить по неокруглённому totals.

## Задача этапа 6 (постановка)

Независимая **сквозная** верификация ВСЕХ вычислений трёх бюджетов на ошибки:
от источников (`.gge`, СО-PDF, веб-цены) до итогов Сводного и дашборда. Не только
«дашборд == Сводный» (это уже подтверждено до рубля в этапе 5), а вся цепочка:
1. A — пересчёт каждого раздела из `.gge` (оба тега, ×1.22, вычленение ТХ по СО), сверка с Excel-дублями.
2. B — пересчёт кол-во×цена по каждой позиции, единицы (укрупнённые!), нет двойного счёта, ключи матчинга.
3. C — корректность замен/флагов C=B, класс аналогов, суммы.
4. Свод — формулы, диапазоны SUM, НДС единообразно, базы не смешаны (об+мат vs полная).
5. Кросс: единицы измерения, размножение общих артикулов на вхождения, обезличенные позиции.

**Deterministic-first:** пересчёт Python-скриптами (точное сравнение), НЕ fan-out LLM.
Ревьюер-гейт: `auditor` (+ `excel-validator` при правках xlsx). Расхождение ревьюеров → резолвер.

## Критерий готовности этапа 6
Каждый из трёх бюджетов пересчитан из первоисточника независимым скриптом; расхождения
с текущими числами Сводного/дашборда перечислены построчно (или подтверждён 0 расхождений);
найденные вычислительные ошибки исправлены через win32com (кэш B/C сохранён) + пересборка
дашборда; финальное ревью PASSED.
