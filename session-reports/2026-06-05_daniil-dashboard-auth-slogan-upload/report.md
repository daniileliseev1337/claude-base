## 2. Auth A+ (решение владельца: autoconfirm + админ-сброс, без SMTP)
- **Два бага регистрации:** (а) `MAILER_AUTOCONFIRM=false` (письма-заглушка);
  (б) **триггер создания профиля `on_auth_user_created` отсутствовал** — функция
  `handle_new_user()` была, но триггер потерян при миграции БД с облака (pg_dump не
  переносит триггеры auth-схемы). Без него регистрация ломалась бы даже с autoconfirm.
- **Миграция** (новая, в репо): восстановление триггера + `admin_reset_password`
  (SECURITY DEFINER, `is_admin()`-гейт, pgcrypto bcrypt `gen_salt('bf',10)`, лог в
  activity_log, фикс search_path). Применена к живой БД.
- **Серверная правка (вне репо):** `ENABLE_EMAIL_AUTOCONFIRM=true` в `.env` self-hosted
  Supabase + пересоздание контейнера auth.
- **Фронт (9 правок):** хелпер сброса, кнопка-ключ + модалка в админке, текст
  после регистрации «ждите одобрения», подсказка «Забыли пароль? → к администратору».
- **Верификация на живой БД (E2E через GoTrue signup + psql):** профиль создаётся,
  `approved=false` (барьер цел), email подтверждается без письма, сброшенный пароль даёт
  валидный bcrypt (GoTrue примет), не-админ получает отказ, RLS не сломана. Тест-юзер очищен.
- **Техдолг (НЕ трогали):** `admin_*` RPC, `is_admin`, `handle_new_user`, `log_activity`
  живут в живой БД, но отсутствуют в репо-миграциях — отдельная задача.

## 3. Баг загрузки файлов (edge-функция файлового хранилища)
- **Симптом:** прикрепление файла → non-2xx; позже — «долго грузит и сбрасывается».
- **Root cause #1 (память):** upload-ветка буферизовала весь файл в base64 в памяти
  worker'а (`req.json()` → `atob` → `Uint8Array.from` per-char ≈ 5× размера) → лимит
  памяти worker'а (150 МБ) → kill. download работал, т.к. стримит — это и подсказало.
- **Фикс (стриминг):** фронт шлёт сырой File бинарно, метаданные в заголовках; функция
  стримит тело запроса напрямую в WebDAV (`duplex:'half'`). Ноль буферизации.
  Редеплой функции (cp в volumes + restart edge-runtime), E2E-тест обновлён под binary
  + тест 25 МБ. Верификация: 25 МБ upload+download, целостность совпала, права/RLS ок.
- **Root cause #2 (timeout, всплыл при UI-тесте через туннель):** `workerTimeoutMs=60с`
  — лимит времени жизни worker'а. При стриминге функция активна всё время передачи; через
  двойной путь браузер→VPS→туннель→ПК крупный файл по домашнему upload-каналу не успевал
  за 60с → kill. Локальный E2E проходил (localhost быстрый).
- **Исход:** для типовых файлов (≤5–10 МБ) стрим-фикс достаточен — владелец подтвердил,
  что файлы качаются и прикрепляются. Поднятие worker-timeout до 5 мин **заблокировано
  авто-классификатором** как правка shared-инфры без явного согласия; владелец отказался
  трогать инфру (для его файлов и так работает). Оставлено как известная граница: очень
  крупные файлы через туннель могут упереться в 60с — вернуться при необходимости.
- **Грабля:** ПЕРВЫЙ запрос после restart edge-runtime падает (cold-start isolate),
  на тёплом — ок. Учитывать при редеплое функций.

## Среда / грабли (подтверждено в сессии)
- Кириллица «Сайт» в пути: надёжно — redirect-glob `/mnt/f/*/` в bash, файл-скрипты,
  `sed 's/\r$//'` перед запуском скриптов, записанных Write на Windows (CRLF).
- Сложный SQL через psql — гонять файлом (`< /mnt/f/tmp/q.sql`), не inline-кавычками.
- Self-hosted Supabase = софт в Docker на VPS/ПК владельца, НЕ облако (частый вопрос).
- Авто-классификатор блокирует правки shared-инфры (edge-runtime config, restart) без
  явного согласия владельца — правильно; объяснять и спрашивать, не обходить.
- Транзиентные `EUNKNOWN: fsync` при Write на F: — повтор/перечитывание помогает.

## Disaster recovery (2026-06-05/06) — сбой диска F: + потеря/восстановление git

**Что случилось:** при merge feature/6.4b→main на сбойном диске F: повредился кэш —
удалились/побились `.git/HEAD`, `.git/index`, ref `feature/6.4b`, `index.html`, правки
`App.jsx`, объект коммита `d60aa84`, `node_modules` (tailwind `normalizeScreens`).
WSL 9p показывал «призраков» (Invalid argument), Windows FS — `does not exist`. Диагноз:
повреждение кэша/FS, не чистая физика (большинство файлов целы). Владелец: диск так иногда
себя ведёт, обычно проходит. Нужен `chkdsk F: /f`.

**Восстановление (git — ТОЛЬКО с Windows-стороны, WSL 9p ненадёжен):**

3. `git checkout` вернул удалённые 7 миграций (0002-0008) + спек tz из 89c929a.
4. `App.jsx`: рабочая копия откатилась к 89c929a, ext4-бэкап оказался ПУСТ (9p-сбой при cp) →
   правки воспроизведены ВРУЧНУЮ из переписки (12 Edit: импорт KeyRound, слоган вход+хедер,
   adminResetPassword, resetUser-состояние, doResetPassword, кнопка-ключ, модалка сброса,

**Инцидент 502:** `wsl --shutdown` (рефреш диска) убил VBS-держатель WSL-VM
