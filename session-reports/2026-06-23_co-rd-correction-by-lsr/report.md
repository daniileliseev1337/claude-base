# Корректировка РД (спецификаций СО) по ЛСР — раздел слаботочки (ТМС)

Дата: 2026-06-23. Объект: <объект> (мед. учреждение), раздел <слаботочка/ТМС>, шифр `<шифр>-X_CO`, организация `<организация>`.

## Задача
Откорректировать 4 спецификации оборудования и материалов (СО, блоки A/B/C/D) рабочей
документации по локальному сметному расчёту (ЛСР), в котором колонка «Правка РД (согласно ЛСР)»
содержала указания. Выдать 2 PDF (с жёлтыми примечаниями + чистый), копии стадии П для тома ПД.

## Что сделано
- 4 xlsx СО скорректированы (стадия Р), 2 PDF, Word-отчёт (audit trail), 4 xlsx СО стадии П.
- Осталось (вынесено в новый чат): вставка СО стадии П в том ПД (замена страниц) + ревизионные
  облачка на изменённых позициях + запись в Реестр изменений.

## Ключевые уроки (переиспользуемые)

### 1. ЛСР количество — в УКРУПНЁННЫХ единицах измерителя расценки (КРИТИЧНО, деньги)
В ГРАНД-Смете кол-во материала в колонке «Количество» дано в единицах измерителя расценки:
«100 шт.», «10 м», «1000 шт.», «100 компл.». Натуральное (штучное) кол-во = G × множитель единицы.
Прямой перенос G в спецификацию = катастрофа (коннекторов стало бы 0.2 вместо 20). Большинство
«расхождений» ЛСР↔СО мнимые — вопрос пересчёта единиц. ВСЕГДА читать колонку «Единица измерения»
и умножать. Деталь вынесена в memory [[reference_lsr_unit_multiplier]].

### 2. openpyxl + файлы с INDIRECT/расчётной моделью = НЕ применять openpyxl-сохранение
- openpyxl при save СБРАСЫВАЕТ кэш всех формул → лист «На печать» (INDIRECT) даёт None до пересчёта.
- openpyxl-сохранённый файл иногда НЕ ОТКРЫВАЕТСЯ в Excel headless вообще (повреждение, хотя openpyxl
  его читает). Excel COM Open падает «Метод Open завершён неверно».
- Excel COM в одной PowerShell-сессии рвёт соединение между файлами на сетевом диске (RPC_E_DISCONNECTED),
  и переиспользование инстанса → чётные файлы молча не пересчитываются (exit 0, но calcChain не создан).
- РАБОЧЕЕ РЕШЕНИЕ: применять правки ЧЕРЕЗ ЖИВОЙ Excel (Python win32com) на ОРИГИНАЛАХ (кэш цел,
  calcChain цел): открыть, менять значения/заливки через COM, ForceFullCalculation + CalculateFull ×2,
  Save. Локальные копии (не сетевой диск). Excel сам сохраняет кэш INDIRECT и не ломает файл.
- Диагностика повреждения: сравнить calcPr в xl/workbook.xml (рабочий vs битый: `fullCalcOnLoad="1"`
  + отсутствие calcChain.xml = Excel не пересчитал).

### 3. Спецификация СО — параметрическая модель; правка кол-ва требует заморозки каскада
Кол-ва считаются формулами от драйверов (камеры=141+2+6 → патч-корды=G4×2; люк → вставка=8×люк;
витая пара = ROUNDUP(.../305)×305 — округление до БУХТ). При замене формулы драйвера на значение
ЛСР зависимые пересчитаются и уйдут от сметы. «Один в один с ЛСР» = целевые → значения ЛСР,
зависимые ЗАМОРОЗИТЬ их текущими (=ЛСР) значениями. Транзитивное замыкание зависимостей строить заранее.

### 4. Адверсариальная верификация ловит реальные BLOCKER
3 ревьюера (excel-validator + auditor + pdf-reviewer на Opus) поймали инверсию: буквальное
применение «поменять местами наименование» сломало уже-правильную пару длина↔количество
(оригинал xlsx был верным, инструкция относилась к старому тому). Без ревьюера ушло бы в выпуск.

### 5. Гибрид детерминизм + LLM: матчинг скриптом, осмысление агентами
Сопоставление ЛСР↔СО (155 позиций) — детерминированный fuzzy-матч (difflib + бонус за партномер),
спорные (много-к-одному по диаметрам, секции помещений) — на адъюдикацию агентам Opus с полным
контекстом блока. Реальных правок оказалось ~18 из 155 «указаний» (111 уже верны после пересчёта единиц).

## Где сломался / потратил время
