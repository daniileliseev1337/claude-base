# 2026-07-02 — Актуализация спецификаций СОТ под новую редакцию заказчика + разница двух ценовых вариантов

## Задача
Объект <объект>: три xlsx-спецификации (листы Видео / Охранные датчики / ВОЛК).
Источник количеств — спецификация заказчика v2.4.1; два «наших» ценовых файла
(вариант «30» и вариант «+7») актуализировать по колонке F (Кол-во), не трогая
наименования/артикулы/цены. Затем — попозиционная разница между двумя вариантами
(нужна заказчику решения по «подрядным» позициям). Этап 2 (генерация КП) — отложен
пользователем явно.

## Что сделал
1. Дамп трёх книг (openpyxl, data_only=False + кэш) → сопоставление позиций ПО ПОРЯДКУ
   в разделах (артикулы у заказчика и у нас различаются — матчинг по позиции, не по артикулу).
2. Выявил агрегацию: у заказчика 41 позиция Видео, у нас 39 — АРМ КПП (станция+2 монитора)
   у нас слит в общие позиции (2+1=3, 8+2=10) — сверка сошлась, правка не нужна.
3. Правка: 8 ячеек F на файл (5 Видео, 3 Датчики), ВОЛК без изменений. Скрипт с
   assert старого значения + маркер артикула в строке; бэкапы в scratchpad.
4. Сплошная верификация F против источника (детерминированный маппинг, 57 позиций × 2 файла).
5. Файл разницы «вариант30 vs вариант+7»: попозиционно цена/сумма/дельта + итоги по листам.
6. Ревью: excel-validator (PASS) + auditor (NOT PASSED → 1 MINOR: str(None) в двух ячейках
   файла разницы → исправлено на «-», перепроверено).

## Результат
- Разница вариантов — только цены листа «Видео» (27 позиций), Δ итога ≈ 5.62 млн ₽;
  листы Датчики и ВОЛК идентичны до копейки.
- Обе спецификации актуализированы, формулы/merged целы (проверено diff'ом с бэкапом).

## Уроки / грабли
- **Ревьюер тоже ошибается в деталях:** excel-validator в INFO-секции заявил, что бэкап
  «наши+7» уже содержал новые значения (diff пустой) — факт-чек openpyxl показал обратное
  (старые значения на месте, правка реальная). Вердикт PASS не пострадал, но INFO-утверждения
  ревьюеров стоит факт-чекать, если они противоречат собственным assert'ам.
- str(None) при генерации отчётных xlsx: ветка «else str(g)» превращает None в текст 'None' —
  в отчётные таблицы для «нет значения» писать «-» сразу.
- openpyxl-пересохранение уменьшает файл на ~40% (вылетает calcChain и пр.) — данные целы,
  но для файлов «на выдачу заказчику» стоит одно контрольное открытие в Excel.
- Кол-во позиций у заказчика и подрядчика может расходиться из-за агрегации (АРМ 2+1) —
  перед правкой сверять СУММУ по агрегату, а не кричать «расхождение состава».

## Этап 2 (та же сессия, по команде пользователя): 2 КП docx
- Вторая вводная в тот же день: одна позиция (опоры) в источнике 3→11 — точечная правка + SendMessage
  тому же auditor-агенту (дёшево, контекст сохранён) вместо нового полного аудита.
- Генерация 2 КП по docx-образцам: копия + хирургическая правка python-docx. Объединённое КП:
  deepcopy tbl-элемента между документами + ОБЯЗАТЕЛЬНЫЙ пустой <w:p/> между таблицами (иначе Word склеит).
- 4 ценовых развилки (формулы СМР не выводились из образца) → AskUserQuestion ДО генерации, решения
  зафиксированы и переданы аудитору как «утверждённые пользователем», не отсебятина.
- Ревью: word-checker PASS + auditor PASSED (построчная сверка 57 позиций, независимый пересчёт итогов).

## Уроки / грабли (этап 2)
- Дата/фразы в docx разбиты между runs → замена спан-алгоритмом по конкатенации runs;
  python-docx run.text маппит <w:br/> в '\n' в обе стороны — переносы выживают при перезаписи.
- Строки «ИТОГО» в таблицах merged → позиционные cells[i] врут; искать ячейку по ожидаемому
  содержимому (merged-safe, с assert на число совпадений).
- Edit-tool на файлах с NBSP (\xa0) в литералах не матчится с виду идентичной строкой —
  точечный патч питон-скриптом по номерам строк надёжнее.
- Образец КП содержал сумму, не соответствующую своей же формуле (ТЗР «7%» ≠ 7% базы) —
  на «один в один» задачах сверять производные суммы образца и эскалировать пользователю.
- Ревьюер ошибся в INFO («диff пустой»)— факт-чек показал обратное; INFO-утверждения ревьюеров
  проверять, если противоречат собственным assert'ам.

## Этап 3: вариант КП «−N» с подгонкой итога + удаление нулевых строк
- Требование: итог ниже ровно на заданную сумму ТОЛЬКО за счёт СМР (цена одной позиции + ТЗР),
  прочее заморожено. Решение: сохранить утверждённое правило «ТЗР = 7% от суммы работ» → цена
  позиции вычисляется из уравнения однозначно, без произвола. Одно уравнение — одно неизвестное.
- Удаление строк с кол-вом 0 из docx-таблиц + сплошная перенумерация двух таблиц; самопроверка
  в генераторе: пересчёт суммы строк СМР = ИТОГО, непрерывность нумерации assert'ом.
- Вопрос пользователя «почему цена не поменялась» → урок: openpyxl-правка оставляет формулы без
  кэша, предпросмотрщики показывают старое/пустое → после правки xlsx прогонять COM-пересчёт
  (Excel открыть-пересчитать-сохранить), чтобы новые суммы были видны без открытия в Excel.
- Итоговая сводка дня — HTML-виджет (Artifact), ссылка в памяти проекта.
