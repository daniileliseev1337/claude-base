Если `git push origin main` падает, и в окружении есть подозрительная штука (прокси, странный credential helper, истёкший PAT) — нельзя сразу атрибутировать ошибку первой подозрительной штуке. В сессии 2026-05-14 я увидел `Proxy CONNECT aborted` и записал в session-report инцидент про прокси, а на втором проходе без прокси выяснил, что **реальная причина — 403 от GitHub на аккаунт без push-прав**. Прокси был лишь верхним слоем.

**Why:** ложная атрибуция причины уходит в session-report и память — Даниил/будущая сессия будет искать решение не в той области. И собственно потерянное время на «починку прокси», когда нужно было поправить GitHub permissions.

**How to apply:**
1. **Раунд 1 — с текущим окружением.** Записываю exact error message (включая код, hostname, пользователя из remote-ответа). Не делаю выводов.
2. **Раунд 2 — снять подозрительный слой 1** (прокси). `Remove-Item Env:HTTP_PROXY,Env:HTTPS_PROXY,...; git -c http.proxy= push`. Записываю новый error message.
3. **Раунд 3 — снять слой 2** (credentials, если нужно). И т.д.
4. **Только когда видна цепочка причин — фиксирую в session-report.**

Корень практически всегда находится на самом нижнем слое, до которого надо честно дойти. Сетевые ошибки часто **маскируют** auth-проблемы — потому что без сети auth-запрос даже не доходит до сервера.
