# Session report — Сверка реестра документов качества трубопровода с PDF тома ИД

Дата: 2026-06-20. Домен: ИД (исполнительная документация), блок pto. Объект: ОКН (музей), том водоснабжения, <шифр тома>, акт АОСР монтажа трубопровода.

## Задача
Сверить документы качества трубопровода (реестр-опись «Реестр №2», 44 позиции) с фактически подшитыми паспортами/сертификатами в собранном PDF тома; проверить соответствие материалам (ВОР) и закрыть замечания технадзора (грувлок, трубы, СИМТЕК, краны, PEX).

## Метод (паттерн — кандидат в скилл)
**Гибрид: инлайн-скаут структуры → Workflow OCR+верификация → анализ vs источники.**
1. Инлайн-скаут (main): по собранному PDF (392 стр.) нашёл границы блока акта (акт→реестр→схемы→паспорта); вычислил кумулятивную карту «позиция реестра → страницы PDF» по столбцу «кол-во листов» (сумма листажа точно легла на конец блока — подтвердило карту); рендер страниц паспортов в PNG через PyMuPDF (0 токенов).
2. Workflow-проход 1 (haiku ×9, батчи): OCR-распознавание 44 документов по рендерам — тип/номер/дата/изготовитель/изделие/листаж + сверка с реестром.
3. Workflow-проход 2-3 (sonnet ×9): адверсариальная верификация ВСЕХ 44 по сканам с нуля («не доверяй прошлому проходу»).
4. Workflow-проход 4 (sonnet ×2): покрытие по диаметрам + сверка спорных позиций со спецификацией проекта.
5. Финал: привязка позиций к актам по ПВ (предъявительская ведомость) — решило судьбу спорных документов.

## Ключевые уроки
1. **haiku на OCR сканов сертификатов — ненадёжен в деталях:** верно берёт номера/даты/типы/листаж, но галлюцинирует изготовителей и путает тип изделия (задвижку назвал «краном», фланцы — «грувлоком», крепёж — «трубами», выдумывал названия заводов). Для документов под сдачу — обязательна верификация sonnet по сканам. Вывод: haiku для черновой инвентаризации, sonnet для содержательной верификации критичного.
2. **Реестр-опись без наименований изделий** (только тип/номер/дата документа) — типовая ловушка: восстановление наименований по сканам впервые вскрывает чужие документы (из других систем/узлов) и несоответствия.
3. **Сверять по ПВ, не по одному ВОР:** ПВ привязывает каждую позицию к своему акту (колонка № акта). Это надёжный критерий «относится ли документ к данному акту»: нет позиции в ПВ → документ чужой/ошибочный. Не удалять документы вслепую — сверять с ПВ/ВОР; есть позиция, но документ не тот → заменять.
4. **Кумулятивная карта листажа** — дешёвый и точный способ локализовать документы в сшитом PDF без полного OCR (если сумма «кол-во листов» из реестра ровно ложится на границу блока — карта верна).
5. **Сверка строений (унификация) имеет пределы:** документы переносимы между строениями только при совпадении номенклатуры. Здесь головное строение использует иные трубы (полипропилен ПК Контур) и краны (другой бренд), чем проверяемое (PEX Rehau + др. краны) — унификация не сработала, добор только веб/поставщик.

## Находки (обобщённо)
- ~30% документов реестра — на изделия, отсутствующие в ПВ/ВОР данного акта (арматура смежного узла, изделия других систем) → к удалению/переносу.
- Расхождения записей реестра с фактом: неверный тип документа, опечатки дат (в т.ч. срок «по» раньше «с»), дубль одного сертификата под двумя позициями, просроченные документы.
- GAP: на часть фактически смонтированных материалов (полимерные трубы, отдельные компенсаторы) документов в реестре нет — добрать.

## Токены
4 Workflow-прохода, ~1.5М токенов субагентов (haiku OCR + sonnet верификация). Оправдано требованием полноты (документы под сдачу технадзору) и режимом максимальной тщательности. haiku-проход дал каркас дёшево; sonnet-проходы — надёжность.

## Артефакты (в рабочей папке тома)
Отчёт сверки (полная карта 44 документов), трекер изменений тома (секция трубопровода), обновлён реестр замечаний и HANDOFF.
