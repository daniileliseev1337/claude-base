## Разведённые конфликты (deconflict-ревизор)

- «проверь раздел ОВ перед сдачей» был ОДНОВРЕМЕННО лидирующим триггером у auditor, audit-rd-section и rd-coordinator — разведено: audit-rd-section = один раздел vs нормы ГОСТ/СП; rd-coordinator = связи МЕЖДУ ≥2 разделами; auditor = сверка цифр/моделей артефакта с источником-чертежом. Из auditor и rd-coordinator generic-лид удалён, у каждого добавлен router.
- «проверь/глянь спецификацию (ВОР/КП/смету) перед отправкой» пересекался у auditor и excel-validator — разведено: excel-validator проверяет ФОРМУ файла (формулы/ячейки/дубли), auditor сверяет СОДЕРЖАНИЕ (количества/коды/ТЗ) с источником. Фразы-артефакты у excel-validator квалифицированы «как ФАЙЛ — формулы целы?».
- «посчитай ...» (общий глагол) у designer / pto-engineer / сметчик — разведено по объекту: designer = физ-величины и подбор оборудования (воздухообмен/нагрузки/сечение); pto-engineer = объёмы/метраж/материалы по готовому чертежу; сметчик = деньги (стоимость/расценка/смета). У каждого добавлена одна строка-разграничитель.
- НДС и цены у снабженца vs сметчика — разведено по направлению документа: входящая накладная/УПД/КП поставщика (НДС в приходе) → снабженец; НДС/НР/СП в СМЕТЕ → сметчик.
- «КП» двусмысленно — разведено по направлению: НАШЕ исходящее КП заказчику → kp-writer; разбор ВХОДЯЩЕГО КП от поставщика → снабженец. У обоих квалифицированы лиды и добавлен router.
- «акт выполненных работ» / «ведомость выполненных работ» у id-engineer vs сметчик — разведено: КС-2/КС-3 для оплаты по расценкам → сметчик; АОСР/АОМР освидетельствование скрытых работ/материалов для приёмки качества → id-engineer.
- norm-lookup vs audit-rd-section — разведено: norm-lookup отдаёт ТОЧНУЮ цитату ОДНОГО пункта / проверку одной ссылки; audit-rd-section ревьюит ЦЕЛЫЙ раздел на соответствие нормам и сам зовёт norm-lookup за цитатами (композиция, не конкуренция).
- word-checker «нормоконтроль/вычитка docx» мог спутаться с audit-rd-section (нормоконтроль раздела) и auditor (содержание) — квалифицировано до «нормоконтроль ФОРМЫ / как ФАЙЛ» + router: нормы→audit-rd-section, факты→auditor.
- pdf-reviewer «проверь чертёж в pdf» мог притянуть content-проверки — квалифицировано «как ФАЙЛ» + router: количества/коды/нормы в чертеже → auditor / audit-rd-section.
- «проверь смету» у сметчика vs excel-validator vs auditor — разведено в пост-ревью сметчика: формулы/ячейки → excel-validator, корректность кодов ГЭСН-ФЕР и коэффициентов vs источник → auditor.
- letter-writer vs expertiza-responder (оба пишут в экспертизу) — разведено по признаку: обычное письмо/запрос → letter-writer; на руках ПЕРЕЧЕНЬ ЗАМЕЧАНИЙ и нужен табличный ответ → expertiza-responder.
- pto-engineer старый черновик сам содержал «проверь раздел ПД / нормоконтроль» — передано audit-rd-section (pto-engineer — генератор, не ревьюер).
