## Брейншторм (метод superpowers:brainstorming, спек одобрен)
Спек: `docs/superpowers/specs/2026-06-26-visitor-demo-design.md`. Решения владельца:
1. Демо visitor = **пустые вкладки + помощник** (не фейк-данные) — UI реальный, риск §11 «мок ≠ UI» снят.
2. Помощник = **пошаговый тур с подсветкой** (coach-marks), переиспользуемый под будущий навигатор.

4. Авто-approve visitor — **через расширение триггера** `handle_new_user_meta` (не RPC).
5. Подсветка тура — **своя, без новых npm** (корп-прокси на `npm install`).

## Ключевое решение по безопасности (verifiable-first)
`has_role()` (Ф1) **в RLS-политиках не используется** (grep по миграциям чист) → запись роли сама по себе
доступа к данным не даёт; вход всё равно гейтится `approved`. Поэтому подделка `role=employee` в meta
бесполезна (триггер одобряет только visitor). Утечку данных visitor закрывает код-гейт + не-рендер
реальных компонентов (надёжнее, чем «реальные вкладки с пустыми данными» из спека §4 — усилено осознанно).

## Увязка заказчика (коммит 9bd0920) — грилинг 2026-06-26
Найден разрыв: роль `user_roles.client` (вид/портал) и привязка `clients.user_id` (данные «Мои заказы»)
жили отдельно — заказчик с ролью без привязки видел пустой портал. Решение (грилинг, 4 вопроса):
**единый источник истины — заказчик = аккаунт, привязанный к записи в `clients`.**
- **Миграция 20260626_0005:** `set_client_user` при привязке → +роль client (не стирая employee) + approved
  + закрыть заявку; при отвязке → снять client (если это была последняя привязка). `set_user_roles` больше
  НЕ трогает client (`delete ... where role <> 'client'`) — чекбоксы не сотрут роль-от-привязки.
  Синхронизация: убраны битые роли client без привязки (была 1 — Test).
- **Фронт:** убран чекбокс «Заказчик» из админки (роль идёт от привязки), вместо него read-only бейдж «Заказчик ✓».
- **Цепочка данных (была и осталась):** Проект --client_id--> запись clients --user_id--> аккаунт;
  портал «Мои заказы» = `get_my_client_projects`. Назначение заказчика проекту (ProjectForm client_id) не трогали.
- **Сценарий (решение владельца):** самрег «Заказчик» → «ХОЧЕТ: ЗАКАЗЧИК» в админке → сотрудник создал запись
  заказчика + проекты → админ «Привязать аккаунт» в карточке заказчика → заказчик активирован, видит свои заказы.
