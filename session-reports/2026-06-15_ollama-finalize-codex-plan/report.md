# 2026-06-15 — Локальные модели: финализация инфраструктуры + план codex/GLM-5.2

Продолжение сессии «Ollama-делегирование» (DANIILPC, RTX 3060 12GB / 64GB RAM, ПК-хаб).
Цель: смоук-тест MCP local-llm, разбор моделей по кейсам, эксперимент think:false.
По ходу всплыло новое направление — codex-плагин + облачный GLM-5.2.

## Что сделано

1. **Смоук-тест MCP local-llm + найден и починен баг.** `list_local_models`/`ask_local`
   таймаутили. Прямой HTTP к гейтвею :4000 работал мгновенно — отличие только в обходе прокси.
   Корень: httpx в `server.py` читал env-прокси (`trust_env=True`), а `HTTP(S)_PROXY` заданы
   при пустом `NO_PROXY` → запросы к localhost шли через corp-прокси → 503/таймаут. Фикс:
   `trust_env=False` в обоих httpx-вызовах. Верифицировано отдельным процессом: list 0.79с,
   ask → «Москва», запрос виден в дашборде LiteLLM (`/spend/logs`). **Нужен реконнект MCP**
   пользователем (hot-reload нет) — иначе нативный tool в чате держит старый код.

2. **Эксперимент think:false — доказан и оформлен.** gemma4:12b «съедает» ~220 скрытых
   reasoning-токенов на запрос (в `reasoning_content` их нет — чистая потеря). `"think": false`
   в теле запроса пробрасывается LiteLLM в Ollama: качество ответа идентично, скорость ×5,
   out-токены ×12 (239→19). Оформлено как алиас `gemma4-12b-extract` (think:false зашит в
   `litellm_params`) — проверено после рестарта: out_tokens=28 (vs 200+ с thinking). Снимает
   граблю «thinking не для batch» — gemma4 теперь кандидат в graphify-экстракторы (была качество-лидером
   бенча, забракована из-за скорости).

3. **Скачаны и зарегистрированы 3 модели** (выбор пользователя): `glm-ocr` (OCR УПД/ИД),
   `qwen3-vl:8b` (vision чертежей), `qwen3-embedding:0.6b` (RAG норм). + алиас extract.
   Конфиг `F:\llm-stack\litellm-config.yaml`, рестарт контейнера `llm-stack-litellm-1`.
   Гейтвей теперь видит 11 моделей. translategemma отложена.

4. **Память** обновлена: `project_graph_ollama_delegation` (был помечен блокером
   OLLAMA_MODELS и consumer-ПК — оба устарели: блокер закрыт, ПК стал хабом); новая
   `feedback_localhost_proxy_trust_env` (родня git-прокси грабли).

5. **Codex-плагин исследован** (по видео пользователя). Официальный `openai/codex-plugin-cc`
   + экосистема (cc-suite, magic-cc-codex-worker, codex-toolkit). Claude делегирует cross-review
   независимой фронтир-модели (GPT или GLM-5.2 через z.ai OpenAI-endpoint). Внедрение — отдельной сессией.

## Где сломалось / уроки

1. **trust_env прокси-грабля** (главный) — см. feedback-память. Любой локальный Python-сервис
   за corp-прокси: `trust_env=False`, не полагаться на `NO_PROXY` в env.
2. **Первый MCP-вызов** на холодном `uv run --with` окружении может таймаутить и сам по себе —
   но в этот раз реальная причина была прокси (воспроизвёлся на 2-м вызове).
3. **Коммуникация (на себя):** вывалил на пользователя выбор технической реализации (где
   прописать think:false — 4 опции) — он не понял «зачем», раздражился («5-й раз читаю»).
   Вывод: реализационные развилки решать самому + объяснять суть простым языком; спрашивать
   только содержательное (что качать, приватность). Применено в остатке сессии.
4. **GLM-5.2** (вышла 13.06.2026): 744B — локально не встанет, только облако/free-API. «Сильнее
   Fable 5» — не подтверждено (Z.ai бенчей на запуске не дала; предыдущая GLM-5 уступала Opus 4.5
   на SWE-bench). Полезна как облачный cross-review при необходимости.
