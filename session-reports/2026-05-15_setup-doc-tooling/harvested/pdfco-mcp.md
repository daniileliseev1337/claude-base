## Что делает

Тонкий MCP-клиент для облачного API **PDF.co**. Сам ничего не считает локально — все операции уходят на серверы PDF.co. Тулы:
- **pdf_merge** — объединение из PDF / DOC / XLS / images
- **pdf_split** — разделение по индексам или диапазонам страниц
- **Form fields**: чтение, заполнение, **создание новых fillable элементов**
- **Annotations**: текст, изображения, формы, ссылки на внешние сайты, вложение других PDF
- **Watermarking**: в списке тулов **не** упоминается явно

## Почему подходит нам

По набору операций — **очень хороший матч** (merge/split/forms/annotations). Но...

## Как подключить

```powershell
uvx pdfco-mcp
```

**Перед использованием обязательно:**
1. Зарегистрироваться на https://pdf.co
2. Получить API-ключ
3. Прописать ключ в env-переменной MCP-конфига.

## Подводные камни

- **Cloud-only, платный API.** Это deal-breaker для большинства наших задач:
  - Содержимое PDF (чертежи, спецификации, переписка с экспертизой, сметы) отправляется на чужие серверы — для проектной документации это **прямой риск конфиденциальности**.
  - Тарификация по запросам — для рутинной правки нескольких PDF в день стоимость незначительна, но при пакетной обработке (например, рендер 100 чертежей) счёт быстро растёт.
  - **Без интернета не работает** — у пользователя на стройплощадке нет.
- **9 stars** — даже для официального репо компании это мало, говорит о слабом community вокруг MCP-обёртки (хотя сам сервис PDF.co с большой клиентской базой существует с 2014).
- Lock-in: уйти с PDF.co на локальный движок — переписывать вызовы.

## Вердикт

⚠️ **Не брать как основной**, но **держать в уме** для двух случаев:
1. **Aggressive OCR / structured extraction** из сложных сканов — у PDF.co качество чище, чем у Tesseract.
2. **Сертификация / подписание PDF** — облачные сервисы умеют DigiCert/USERTrust, локально это головняк с настройкой OpenSSL.

Для рутинной работы с проектной документацией — **категорически нет** из-за отправки данных в облако.
