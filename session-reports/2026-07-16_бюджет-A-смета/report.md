# Session report — бюджет A (по смете) для объекта СС/ТХ

> Обезличено. Объект → `<объект>`, шифр → `<шифр>`, компания → `<фирма>`.
> Дата: 2026-07-16. Тема: этап 4 бюджета проекта — колонка A (стоимость по сметной документации заказчика).

## Задача
Собрать бюджет A («по смете заказчика») по 12 нашим разделам (слаботочка ИОС5.1–5.10 + ТХ1/ТХ2) и добавить
колонку A к существующим B (спецификация ПД, бренды) и C (спецификация ПД с заменой на вендоров пула).
Три бюджета должны быть сопоставимы; опора — спецификация оборудования (СО).

## Ключевой поворот
Первичный комплект смет содержал по нашей зоне **только ПНР** (пусконаладку) — детальных смет на оборудование+монтаж
сетей связи не было (стоимость свёрнута в укрупнённую объектную смету здания). Провёл грилинг, вынес развилку
пользователю. Пользователь **перезагрузил полный комплект смет** — появились локальные сметы по разделам.

## Что оказалось источником A
- Локальные сметы здания **ЛСР-02-01-08…17** (по одной на систему ИОС5.1→08 … 5.10→17) + **02-01-19** (технологическое оборуд.).
- Формат **`.gge`** — это XML ГрандСмета (метод РИМ, ресурсно-индексный). Итоги лежат в
  `/Construction/Object/Estimate/EstimatePrice/`: `Building/Total`, `Mounting/Total`, `Equipment/Total`,
  `Summary/Materials/Total`, `Summary/Total`. Всё в рублях, без НДС.
- Дубли тех же смет: Excel-версии (`.xlsx`, читаются openpyxl — шапка со «Сметная стоимость / оборудования / …»),
  PDF (полный том), `.gsfx` (бинарный ГрандСмета — не читать).
- Сводная смета вида «ПС-XX» оказалась **сметой на ПИР** (проектные работы), не на оборудование — для A не годится.

## Метод (детерминированный, без LLM-fan-out)
- Парсинг `.gge` (xml.etree) → по каждому ИОС5.x: **об+мат = Equipment + Materials**, **полная = Summary/Total**.
- Сверка с Excel-дублями — совпало 1:1 до копейки.
- **НДС ×1.22** (действующая ставка на цены 2026) — приведение к базе B/C (рыночная розница с НДС).
- **Вычленение ТХ** из 02-01-19: раздел «технологическое оборудование» на ~82% — чужая зона
  (лабораторное оборудование, мебель, пищеблок). Наша часть (ТХ1 мультимедиа+ИТ, ТХ2 лифты) отобрана по составу
  нашей СО ТХ1 (утверждённая номенклатура из предыдущего этапа) — как эталон границы «наше/чужое».

## Результат
- **A(об+мат, с НДС) = 32,58 млн ₽**; A(полная сметная, с НДС) = 41,65 млн.
- Против B = 39,63 млн, C = 40,24 млн → смета заказчика ~18% ниже рыночной спецификации по сопоставимой базе.
- Записано в рабочий xlsx: сводный лист (кол. A об+мат + A полная) + отдельный лист детализации (оборуд/матер/монтаж/полная).
- Запись через **win32com** (реальный Excel) — чтобы сохранить кэш формул соседних бюджетов B/C
  (openpyxl при сохранении кэш формул убивает → программное чтение потом даёт None).

## Ревью (обязательный гейт)
- Валидатор формы: PASS.
- Аудитор содержания: **NOT PASSED → PASSED**. Нашёл реальный дефект — одна наша ТХ1-позиция («настенный экран для
  проектора», ~38 тыс) была потеряна. Причина: в смете эта позиция проходит как `<Material>` внутри монтажной расценки,
  а не как `<Equipment>` — сбор оборудования её не видел. Системно проверил все материалы раздела (единственный наш —
  экран), включил. Аудитор подтвердил исправление до копейки.

## Грабли / уроки
1. `.gge` = читаемый XML (не бинарник), итоги в `EstimatePrice`. `.gsfx` — бинарник, не парсить.
2. В сметах ГрандСмета часть оборудования может проходить как **`<Material>`** (расходный материал расценки), а не
   `<Equipment>` — при вычленении «нашей» номенклатуры проверять ОБА тега, иначе позиция теряется. (Поймал аудитор.)
3. **openpyxl рвёт кэш формул** при сохранении → для файла с формулами соседних колонок писать через **win32com**
   (открывает в Excel, пересчитывает, сохраняет с кэшем). Это же чинит уже «пустой» кэш.
4. Смета РИМ: «оборудование» (графа) — только изделия по КАЦ; «материалы» (кабель/крепёж) — внутри монтажных работ
   (Summary/Materials). Для сопоставимости с рыночной спецификацией (оборуд+материалы) брать об+мат, не только оборуд.
5. Библиотека-first грилинг: сначала сам разобрал структуру комплекта смет (ССР/ОСР/ЛСР/ПС), потом вынес пользователю
   ровно ту развилку, которую нельзя вывести из файлов (недостающие данные) — это и привело к перезагрузке комплекта.
