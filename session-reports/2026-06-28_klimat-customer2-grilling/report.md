## Решения грилинга (5 осей, зафиксированы)
1. Видимость задач: заказчик видит **все** задачи своих проектов.
2. Комментарии/ТЗ: видит и пишет (канал общения с заказчиком). Приватной переписки нет —
   уйдёт в будущий мессенджер. (Владелец поправил мой первичный «только карточки» — заказчику
   задают вопросы и он сам меняет ТЗ.)
3. Назначение исполнителя: кнопка «Взять в работу» уже есть; автоподтяг из `projects.executors`.
4. Создание проектов: оба режима (быстрая заявка / подробная) через таблицу `project_requests`.
5. Права на статусы: **приёмка** — заказчик На проверке→Готово / вернуть→В работе;
   производственные статусы — исполнитель.

## Факты БД (проверено)
- `projects.owner_id` — **NOT NULL** (проект-заявка обязан иметь владельца → отдельная таблица заявок).
- `projects.stage` — text БЕЗ CHECK (свободные значения); `client_id` nullable;
  `visibility` CHECK `private|team|selected|marketplace`.
- `is_approved()` = `profiles.approved=true` **без проверки роли** → заказчик проходит его.
- Уже есть (миграция 0007): `is_project_client()`, `am_i_client()`, `get_my_client_projects()`.
- RLS задач: SELECT/INSERT через `can_access_project_comments`; UPDATE USING = author/assignee/admin/editor.

## Архитектура (одобрена)
- **Подход 1**: +`is_project_client` в `can_access_project_comments` (одна правка → задачи+комментарии+ТЗ);
  узкий RPC `client_set_task_status` для приёмки; создание задач — через существующую INSERT-политику.
- **Создание проектов**: таблица `project_requests` (Вариант 1) + RPC `create_project_request` /
  `accept_project_request`; сотрудник принимает → `projects` с owner=сотрудник.
- Новая функция `is_employee()` (роль employee+approved) — для RLS заявок.

## Находка безопасности (НЕ из этой фазы, требует решения владельца)
`is_approved()` не различает client/employee. Ветки `marketplace + is_approved()` в
`projects_select` и `can_access_project_comments` пускают любого заказчика ко ВСЕМ
marketplace-проектам целиком (через прямой `supabase.from('projects')`, в обход портала) —
потенциальная **существующая** утечка. Рекомендация: заменить на `is_employee()` в
marketplace-ветках. Не делать молча — меняет поведение для всех. **Ожидает ответа владельца.**
