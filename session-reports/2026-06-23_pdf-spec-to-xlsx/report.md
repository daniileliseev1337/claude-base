# Извлечение спецификаций оборудования из проектных PDF в Excel (без потерь/галлюцинаций)

Дата: 2026-06-23. Объект: <объект> (раздел рабочей документации, шифр <шифр>-Р-*).

## Задача
Из 7 PDF-разделов РД (СС, СЭС, ЭЧ, ЛВС, СК, СОТ, СКС) вытащить «Спецификацию
оборудования, изделий и материалов» (форма ГОСТ 21.110) в Excel — отдельный .xlsx
на раздел, без потерь, без упущений, без галлюцинаций, с соответствием оригиналу.

## Решение (deterministic-first, инлайн)
- Все 7 PDF — ВЕКТОРНЫЕ (AutoCAD/pdfplot, Adobe, PDFTron, pdf-lib, Ghostscript), с
  текстовым слоем → OCR не нужен. Применён подход скилла **co-verify**: координатное
  извлечение `PyMuPDF find_tables(strategy="lines")` (сетка по векторным линиям).
- co_engine.extract_co НЕ подошёл напрямую: он заточен под сверку и (а) выбрасывает
  строки без количества (групповые заголовки, компоненты «в составе»), (б) берёт 6
  колонок из 9. Написан полный экстрактор `_extract/extract.py` (все 9 колонок формы +
  служебная «Стр. PDF», все строки, маппинг колонок по шапке НА КАЖДОЙ странице).

## Ключевые ловушки и как обойдены
1. **Колонки «плавают»** между страницами (25→23→27 грид-слотов) → колонки маппятся по
   шапке на каждой странице, не по фиксированным индексам. Логические колонки склеиваются
   из грид-слотов по границам соседних заголовков.
2. **Штамп (основная надпись)** попадает в таблицу снизу (графа «Изм. Кол.уч…», подписи,
   «N — Зам. XXXX-26»). Сначала пробовал keyword-фильтр штампа — ОШИБКА: маркер «пров.»
   ложно ловил «С-обр. профиль для **пров.** лотка» (реальная позиция, кол.870!). Урок
   из co-verify подтвердился: лексические правила дают ложные флаги. Решение —
   **структурный фильтр: строка спецификации обязана иметь непустое Наименование**;
   штамп сидит в правых колонках с пустым name. Выброшенное логируется; «выброшено с
   кол-вом» = 0 во всех разделах (контроль непотери).
3. **Числа как текст**: при первом проходе Кол. писалось строкой → Excel не суммирует.
   Исправлено `to_number` (по замечанию excel-validator).

## Верификация (как доказана полнота)
- **Золотой тест полноты**: каждый пропущенный номер позиции искался в ПОЛНОМ тексте
  страниц (get_text). Все пропуски (8.11; 2.4/2.6; 1.17/1.18; 9.19/11.15) = 0 вхождений
  → это позиции, УДАЛЁННЫЕ из оригинала при изменениях, не потеря извлечения.
- **2 рендера-адъюдикации** (co-verify шаг 3): СКС стр.56 и СЭС стр.15 — извлечение
  совпало с печатью 1:1, что доказывает: метод берёт ровно текстовый слой, без потерь/выдумки.
- **Ревьюеры (opus, выбор пользователя)**: auditor — **PASSED** (сверил полноту 100% по
  нумерации + количества буква-в-цифру, отсутствие галлюцинаций, групповые заголовки);
  excel-validator — WARN (нет CRITICAL), все замечания закрыты.

## Аномалии ОРИГИНАЛА (не дефект извлечения — сохранены, доведены пользователю)
- СКС: блок 3.16–3.19 (Розетка/Автомат/SNMP/DIN) повторён дважды (ошибка нумерации проекта,
  но физически на печати).
- СЭС: две позиции с номером 1.7 (разные устройства).
- Пропуски нумерации = удалённые при изм. позиции.
- Колонка «Масса 1 ед.» пуста во ВСЕХ разделах — потому что не заполнена в самих PDF.

## Косметика (по доп. запросу пользователя)
Унификация ед.изм («шт»→«шт.», «м.»→«м»), десятичный разделитель в номерах («9,2»→«9.2»),
оформление (титул с шифром, серая шапка, выделение групп, выравнивание). Содержание не
менялось; оригинальные значения сохранены в JSON (откат возможен).

## Урок для базы (кандидат в co-verify / память)
co_engine.extract_co теряет строки без qty и берёт 6/9 колонок — для задачи «полное
извлечение спецификации в xlsx» нужна полная версия (все строки/колонки + структурный
фильтр штампа по пустому Наименованию + золотой тест полноты по тексту страниц).
Предложить дополнить tools co-verify функцией extract_co_full / extract_spec_to_xlsx.
