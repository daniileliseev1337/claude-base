# Применение базы правил наименований работ к ВОР/СВОР (16 файлов)

## TL;DR
Пользователь дал xlsx-базу правил присваивания работ (22 типа позиций). По его решению база —
главный источник: 16 готовых ведомостей (8 СВОР + 8 ВОР, <объект>) переделаны под неё.
База расширена до 36 типов (14 новых согласованы через черновик с цветовыми правками пользователя).
Итог (финальный, после фикса 7 блоков по аудиту): 4652 замены наименований на составные,
1093 строки-работы удалены (711 лицензии/сертификаты + 382 крепёж), запись в новые папки
«С работами v2». Самопроверка 16/16 PASS ×2; validator ВОР 8/8 PASS; auditor: до-аудит PASSED
по всем пунктам (первый заход дал 1 реальный блокер из 4 — исправлен, 3 опровергнуты фактурой).

## Решения пользователя
- Переделать готовые файлы по базе (отменяет вчерашние точечные правила naming, в т.ч.
  «Настройка и запуск» для ПК-техники — память помечена superseded).
- Формат: составное наименование из колонки F базы («Установка, монтаж, подключение, настройка
  и проверка работоспособности <объект>»).
- Непокрытые позиции → расширять базу новыми типами.
- Тип 2 (лицензии/сертификаты, 711 блоков) и тип 7 (крепёж, 383 блока) — строки-работы УДАЛИТЬ.
- Правки типов пользователь вносил ЦВЕТОМ в черновик-xlsx: красный = удалить, зелёный = добавить
  (чтение: openpyxl rich_text=True, TextBlock.font.color по фрагментам).

## Пайплайн (все скрипты детерминированные, Python)
1. extract_blocks.py — блоки из 16 файлов по вертикальному merge колонки A (5876 блоков;
   ВОР блок = работа+позиция, СВОР до 4 строк: 2 работы + 2 позиции «до/после»).
2. match/extend_types.py — матчер позиций на типы: фразовые ключи (слова ПОДРЯД, стем-префиксы,
   оба порядка для 2-словных) + ~40 фиксапов (подстрока работы + неверный тип → верный).
   100% покрытие. Ручное ревью полной таблицы (review_table.txt) глазами — обязательно.
3. update_base.py — база +14 типов (23–36) + доп-ключи старых типов (бэкап рядом).
4. gen_plan.py — план: replace (новый текст = перечисление F + хвост старой работы без
   глагольного префикса — род. падеж переиспользуется, склонять не надо) / delete / skip.
5. apply_plan.py — win32com, копии в «С работами v2», правки снизу вверх; при удалении строк
   перенос №/производителя/модели из якоря merge (значение живёт в верхней строке!).
6. verify_v2.py — пересбор блоков, сопоставление 1:1 с планом, формулы H/I самоадресные: 16/16 PASS.

## Грабли (новые)
- Ключи нельзя матчить словами независимо по всему тексту: «релейный модуль» ловил
  «релейный выход … модуль» (SNMP-модули уходили в «активное»). Только фразовый матч.
- Одиночные короткие ключи («ПО», «ОК») — только словом целиком; «панель»/«настенный»/
  «силовой»/«с заземлением»/«кресло» (ловит «крестообразный»!) в длинных описаниях — ложняки,
  лечатся длинными ключами + фиксапами по тексту РАБОТЫ с условием «текущий тип → верный».
- При Rows.Delete верхней строки merge-блока значение якоря (№, B/C) теряется — читать до,
  писать после в новую верхнюю строку.
- Верификатор должен фильтровать шапку и в merged-ветке (merge A в шапке ловится как блок).
- MCP excel read_data_from_excel на листе с сотней строк выдаёт 144К метаданных — большие листы
  читать openpyxl со сбросом в utf-8 файл (cp1251 stdout Windows).

## Ревью
Самопроверка verify_v2: 16/16 PASS (блоки 1:1, тексты, переносы, формулы) — дважды (до и после фиксов).
- excel-validator ВОР: PASS 8/8 (полная нумерация, схлопнутые блоки предметно = крепёж/лицензии).
- excel-validator СВОР: PASS 3/8 полноценно + 546 формул выборочно на остальных, 0 ошибок формул
  нигде; полнота недобрана из-за лимитов агента (без Bash) — закрыта детерминированной verify_v2.
- auditor: NOT PASSED (4 BLOCKER) → фактчек оркестратора: 3 из 4 ложные (аудитор путал направление
  сравнения и смещение строк после удалений), 1 реальный: сервер палатной сигнализации получил
  кабельную формулу типа 25 (ложный ключ «сигнализации»). Систематический прочёс типов 2/7/25/24/3
  нашёл ещё 2 группы: оптический передатчик/приёмник (ключ «кабель» в скобках описания) и угол
  лотка (стем «угол» из «уголок» перебил короткий стем «уго»). Итого исправлено 7 блоков из 5876,
  файлы перегенерированы, verify 0 ошибок, точечный до-аудит прогнан.

## Грабли ревью (дополнительно)
- Переход матчера на фразовый режим ПОСЛЕ ручного ревью таблицы = скрытые регрессии (длинные
  ключи перестали матчиться на разнесённых словах, победили одиночные). После любого изменения
  матчера — автопрочёс «работа не похожа на тип» по опасным типам, не только спот-чек.
- Одиночный ключ «сигнализации» в кабельном типе ловит «сети палатной сигнализации» у сервера.
- Порядок FIXUPS важен: специфичные подстроки ('сервера системы палатной') — раньше общих ('сервера').
- Аудитору по файлам со смещёнными строками ЯВНО писать: сверять по содержимому, не по номеру
  строки, и какая папка = новая версия (иначе ложные BLOCKER: 3 из 4 в этой сессии).
