# Сессия 2026-07-08 — сверка 9 КП вендоров со спецификацией СС (сводная книга)

## Задача
Папка с базовой спецификацией СС (<объект>, 7 листов-разделов, уже расценена закуп/РРЦ) + 9 КП
от разных вендоров (каждое в своём формате). Итог: сводная навигационная книга на базе копии
спеки с блоками вендоров (цены как в КП + приведённые ₽ с НДС), пакет вопросов вендорам,
интерактивный дашборд вариантов ответов для сессии с ГИП.

## Пайплайн (за одну сессию, этапы 1–5)
1. **Рекон** (python/openpyxl, дампы 60×16) → вендор-идентификация. Пользователь дал
   первоисточник (папки с именами вендоров) — 2 моих метки из 9 оказались НЕВЕРНЫ
   (метил по производителю товара, а не по поставщику). Урок: имя вендора ≠ бренд номенклатуры.
2. **Глубокий аудит + полные дампы** (все листы, формулы+кэш, merged, скрытое) →
   0 формул без кэша во всех файлах; скрытые строки/колонки у одного вендора; diff
   «копии спеки» вендора против оригинала выявил единственный содержательный лист.
3. **Workflow 10×sonnet** (9 КП + спека → структурированные JSON с контрольными суммами).
   Все 10 успешно, ~1.5M токенов. Пост-чек детерминированным скриптом.
4. **Сопоставление**: 3а — детерминированный матчер (302/447 позиций кодом: «Поз.Замена»,
   req_name+группа, continuation-правило для многострочных предложений); 3б — 5 агентов-снабженцев
   (функциональные замены брендов). Свод покрытия: 222/492 позиции (45%).
5. **Сборка книги** (python, копия xlsx + блоки) + **ревью**: excel-validator PASS,
   auditor NOT PASSED → 1 BLOCKER исправлен, 1 MINOR снят фактической проверкой.

## Ключевые находки метода (переиспользуемо)
- **Формат «Запрос|Предложение»** (вендор заполняет нашу спеку): пустой req = строка-продолжение
  предложения к предыдущей позиции; безымянный блок идентифицируется по последовательности
  имён+количеств против групп спеки (у нас совпало 1-в-1 — доказательное решение, не гипотеза).
- **НДС 22% (с 01.01.2026)**: агенты-sonnet со «знанием 20%» массово флагуют 22% как аномалию —
  проверять эмпирикой файлов (у нас 5 КП независимо подтвердили 22%).
- **Классика ошибок в КП вендоров**: непересчитанный грандтотал после правки количества
  (нашли расхождение ровно в цену одной позиции — сильный аргумент в переговорах);
  MM-оптика при SM-проекте; лицензий меньше, чем устройств; функционал вне ТЗ на ~2 млн.
- **Анти-задвоение** при мультимаппинге (одна позиция вендора закрывает несколько строк спеки):
  сумма в первую строку, в остальные «→ входит в поз. N» — иначе сравнение завышается.
- **Дашборд вопросов с вариантами ответов** (self-contained HTML + localStorage + кнопка
  «скопировать ответы для Claude») — удобный формат для офлайн-сессии с ЛПР; смоук через
  playwright + http.server 127.0.0.1.

## Где сломался / грабли
- Workflow args: 2 фальстарта — (1) случайный JS-код внутри JSON args; (2) args со scriptPath
  не пробросился в скрипт (args=undefined). Лечение: вшить jobs константой в скрипт-файл.
  Подтверждает грабль из памяти «args Workflow только объектом» — надёжнее вообще без args.
- В JS-строках Windows-путей `\0...` = null-символ — пути в скриптах Workflow только с `\\`.
- Ревьюер excel-validator без Bash-канала не видит ширины/заливки/number_format (MCP не отдаёт) —
  честно пометил NOT VERIFIABLE; вижуал проверять открытием файла.
- Валидатор дал ложную тревогу «блоки пустые» (сэмплировал строки без предложений при покрытии
  45%) и аудитор — ложный MINOR «нет маркеров» (смотрел не те строки). Правило: перед правкой
  по замечанию ревьюера — детерминированная проверка факта (у нас 2 из 3 замечаний не подтвердились).
- Настоящий BLOCKER аудитора: строка с ДВУМЯ альтернативными вариантами (решение «оба через /»)
  сломала правило скидки `all(discounted)` → скидка не применилась ни к чему. Смешанные строки —
  считать по долям.
- python http.server в фоне + бесконечный цикл печати = 512KB output за секунды; чинить сразу.

## Продолжение 2026-07-09 — извлечение СО из проектных PDF (этап 5а)

Задача пользователя (вне плана, до ответов ГИП): вынуть спецификации из 7 проектных PDF → Excel → анализ.

- Метод: skill co-verify (координатное извлечение find_tables по векторным линиям) + локальная
  надстройка extract_co_full (+Поставщик/Масса/Примечание — в примечаниях живут пометки «Изм.N»).
  Файлы базы не правились (consumer-ПК), надстройка в скрипте сессии.
- Результат: книга извлечения (10 листов, 503 позиции) + детерминированная сверка со спекой:
  0 расхождений, 467/467 позиций 1:1; «Стр. PDF» спеки совпала 467/467 → спека собрана именно
  с этих редакций и полностью актуальна.
- Находки анализа: раздел СКУД (36 поз., Изм.3) в спеке нет и КП нет (В-11); противоречие в
  проекте СОТ Изм.4 — поз. с примечанием «Аннул.» при необнулённом Кол.=98 (В-12, подтверждено
  рендером); повторная нумерация хвостов шкафов «3.16–3.19» в PDF СКС (грабль §11.3 доказан).

Грабли/уроки:
- Сырой get_text-поток PDF НЕ авторитетен для таблиц: auditor склеил соседние строки и выдал
  ложный BLOCKER «Кол.=0» (значение принадлежало следующей позиции). Таблицы читать только
  find_tables(strategy='lines') или рендером конкретной страницы.
- Ревьюеры на двоих дали 8 замечаний, 6 — ложные тревоги; снова окупилось правило
  «перед правкой — детерминированная проверка факта». Единственное справедливое замечание
  (нет построчной трассировки сверки в самом файле) закрыто листом «Сверка (детально)»
  с пересчитанными кодом вердиктами; повторный вердикт auditor — PASSED.
- Критерий «строка-позиция» в спеке = непустой «Кол.» (на листе СОТ заголовки групп имеют
  «Поз.», но не «Кол.» — критерий «Поз. ИЛИ Кол.» даёт +11 лишних строк).
- Номера позиций в PDF пишутся и через запятую («9,2») — нормализовать до сравнения.

## Продолжение 2026-07-09 (2) — новый свод «спека × КП» + расширение периметра

Старая сводная забракована пользователем как нечитаемая → новый формат (согласован пилотом
на малом разделе, 3 итерации макета): лист = раздел, спека слева (без лишних колонок),
вендорные таблицы ВПРАВО на общей построчной сетке (8 колонок + баннер вендора), freeze
левой части, Arial, «не покрыто КП» серым, лист «Обзор». Затем решение <ГИП>: ВСЕ файлы
папок вендоров актуальны → периметр расширен, diff вендорских копий спеки дал крупные
пласты данных.

Ключевые ходы/уроки:
- Пилот на малом разделе + рендер (win32com PDF → PNG) и проверка ГЛАЗАМИ до показа
  пользователю — 3 итерации вместо возврата от пользователя. Смотреть рендер самому — окупается.
- «У вендора нет цен» ≠ факт: цены нашлись в другом файле его папки (PDF-расчёт оказался
  векторным, find_tables вытащил всё; контрольный пересчёт до копейки).
- Diff вендорской копии спеки против исходной (openpyxl, посимвольно) — дёшево вскрывает
  вклад вендора: построчные цены (144 позиции у одного), замены марок в чужих разделах
  (12 строк у другого), авторитетный маппинг замен (перестановка строк у третьего).
- Ловушка: в копии спеки вендора строки сдвинуты (вставки) → сопоставление ТОЛЬКО по
  наименованию (difflib), не по номеру строки; мой первый дамп по номерам дал ложные цены.
- Упаковочные единицы вендора (Кат.500/Кор.305) → Сумма ≠ Цена×Кол. спеки — не ошибка,
  раскрывать комментарием строки + оговоркой в баннере.
- Скидку вендора (57% на агрегаты КП) НЕ применять построчно к его РРЦ — ставок по
  позициям нет, это была бы выдумка; агрегаты со скидкой — отдельно «вне спеки».
- Ревьюеры: excel-validator честно перечислил, что НЕ смог проверить своим MCP (freeze,
  объём денежных колонок) — эти пробелы закрывать самому openpyxl-скриптом, не считать
  «не проверено» за «ок». Auditor с явным указанием «PDF только координатно» дал PASSED
  без ложных тревог (урок предыдущего раунда сработал).
