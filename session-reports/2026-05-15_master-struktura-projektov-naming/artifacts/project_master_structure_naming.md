Методика структурирования инженерных проектов (фиксация 2026-05-15 на основе
`C:\Users\Deliseev\Desktop\Здадчака\Мастер структура_проектов.xlsx`).

**Why:** на будущее все проекты должны быть структурированы по единой
схеме naming — для автоматического сравнения П↔Р, единых BIM-параметров,
автогенерации спецификаций. Решение пользователя по итогам разбора
эталонной таблицы.

**How to apply:** при старте любого инженерного проекта применять схему
ниже к шкафам и оборудованию. До заполнения мастер-таблицы — запросить
у пользователя список объектов, блоков и этажей.

## Формат человекочитаемого имени шкафа

`[Объект]-[Блок]-[Сегмент_сети]-[Тип_шкафа][Этаж][Порядковый_номер]`

Правило по решению пользователя: **в имя всегда входят все компоненты**,
включая блок (даже для CORE-шкафа). Этаж и порядковый номер сливаются
в двузначный код: `ШД21` = этаж 2, шкаф 1 на этаже.

Примеры:
- `ГК-ГК1-Л2-ШГК01` — ГК, блок 1, Л2, главный кросс, этаж 0, шкаф 1
- `ГК-ГК2-Л2-ШД21` — ГК, блок 2, Л2, доступ, этаж 2, шкаф 1
- `КПП-КПП5-Л2-ШД54` — КПП, блок 5, Л2, доступ, этаж 5, шкаф 4

## System_ID (машинный)

`OBJ_[Объект]_[Блок]_[Сегмент]_[Функция_сокр]_[##]`

Примеры: `OBJ_GK_GK1_L2_CORE_01`, `OBJ_GK_GK2_L2_ACC_21`, `OBJ_KPP_KPP5_L2_ACC_54`.

System_ID — основной ключ для сравнения П↔Р (не имя — оно может
переименоваться).

## Иерархия привязки к месту

```
Проект → Объект → Блок → Этаж (в номере шкафа) → Помещение → Шкаф
```

- **Объект:** ГК / ПАК / КПП / ПП (корпус)
- **Блок:** ГК1, ГК2, КПП5 (блок внутри объекта)
- **Помещение:** код по архитектуре (1.04.021) или текст (Серверная)
- **Сегмент сети:** Л1 / Л2 / Л3 (только для СКС/слаботочки сетевой топологии)
- **Уровень сети:** CORE / DISTRIBUTION / ACCESS

## Классификатор шкафов

| Тип | Описание | Уровень | Назначение |
|---|---|---|---|
| ШГК | Шкаф главного кросса | CORE | Главное ядро объекта |
| ШР | Шкаф распределения | DISTRIBUTION | Агрегация сегментов |
| ШРД | Шкаф распределения и доступа | DISTRIBUTION+ACCESS | Гибрид |
| ШД | Шкаф доступа | ACCESS | Подключение абонентов |

| Код | Функция | Категория | Пример |
|---|---|---|---|

| КР | Коммутатор распределения | ACTIVE | Distribution Switch |

## Алгоритм П → Р (9 этапов)

1. Получение ЧТЗ → определены объекты и системы
2. Создание master structure → шкафы и топология
3. Создание naming → единые названия
4. Создание BIM параметров → единая BIM модель
5. Разработка стадии П → проект внутри структуры
6. Формирование спецификации П → единая Excel
7. Разработка стадии Р → изменения без разрушения naming
8. Сравнение П ↔ Р → автоматический анализ
9. Отчёт изменений

## Статусы изменений П → Р

SAME · NEW · REMOVED · MODEL_CHANGED · QTY_CHANGED · ARCH_CHANGED · MOVED.

## BIM-параметры (для Revit)

Шкаф несёт минимум 8 параметров: Проект, Объект, Система, Сегмент,
Тип_шкафа, Человекочитаемое_имя, System_ID, Функция.

Открытый вопрос: Shared Parameters (с фиксированным GUID между
проектами) vs Project Parameters. Уточнить на первом реальном проекте,
который будет переноситься между файлами.

## Охват систем (приоритет)

1. **СКС/ЛВС** — исходник методики, есть полные данные в эталоне
2. **СПС, СОУЭ, СКУД, СВН** — приоритет №2 (классификаторы похожи на СКС)
3. **ЭО** — щиты ЩР/ЩС/ЩАО (требуется свой классификатор)
4. **ОВ, ВК** — узлы автоматики, насосные/тепловые пункты (свой классификатор)

## Эталон П — где хранится

**Процесс:** пользователь предоставляет вводные через ЧТЗ и уточнения,
Claude фиксирует в мастер-таблице. Перед входом в стадию Р — снимок
(рекомендация Claude: отдельный xlsx `<имя>_P_lock_YYYY-MM-DD.xlsx`
рядом с живой версией). В Р правки идут в живой версии, сравнение через
`excel-helper` по `System_ID`.

**Решение пользователя по механизму:** «Пользователь будет
предоставлять все вводные, твоя задача их запрашивать». То есть Claude
не автоматизирует хранение эталона — запрашивает у пользователя данные
для каждой стадии.
