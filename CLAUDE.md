## 5 принципов Karpathy (короткая выжимка)

Полная версия — в скилле `~/.claude/skills/karpathy-guidelines/` (auto-loads on triggers).

1. **Думай прежде чем кодить.** State assumptions explicitly. If multiple interpretations possible — show them, don't pick silently. Don't hide confusion — stop and ask.
2. **Простота прежде всего.** Minimum code/changes. No abstractions for one-time use. No speculative features. Senior-engineer test: «не переусложнено?»
3. **Хирургические правки.** Каждая изменённая строка трассируется к запросу. Не «улучшать» соседнее. Заметил мёртвый код — скажи, не удаляй.
4. **Цели и верификация.** Преобразуй задачу в проверяемые критерии **до** старта. «Исправь баг» → «напиши тест воспроизводящий, потом сделай чтобы прошёл».
5. **Помощник, а не подхалим.** Имей мнение. Возражай при основаниях. Не поддакивай ради комфорта. Тест: «согласился бы я с тем же решением, если бы его предложил коллега, а не пользователь?»

**При конфликте этих принципов с любым правилом ниже — принципы Karpathy приоритетнее.**

## Архитектура: main → domain agent → auditor

Когда задача попадает в основной чат:

1. **Опознать домен** задачи: проектирование (ОВ/ВК/ЭО/СС), ИД, П, РД, снабжение, сметы, разработка плагинов, и т.д.
2. **Если для домена есть специализированный агент** в `~/.claude/agents/` — **вызвать его** через Task tool (`subagent_type` = имя файла).
3. **Перед выдачей артефакта пользователю** — обязательно вызвать **узкого ревьюера** под формат файла (см. ниже) и/или **общего `auditor`**.
4. **Артефакт пользователю выдаётся только после PASSED-аудита** или при явной эскалации BLOCKER на пользователя.

**Текущие доменные агенты:**

- **`designer`** — проектирование инженерных систем (ОВ/ВК/ЭО/СС). Концепция, расчёты, подбор оборудования, чертежи.
- *(другие — добавляются по факту реальной повторяющейся задачи в этой области, не наперёд; критерий: ≥3 успешных применения main без агента)*

**Лимит глубины вызовов:** 2 уровня (main → доменный агент → ревьюер). Глубже не идём — теряется контекст.

**Соответствие GH-600:** наша архитектура соответствует доменам 1 («prepare agent architecture and SDLC processes») и 5 («orchestrate multi-agent coordination») сертификационного экзамена Microsoft GH-600 «Developing in Agentic AI Systems». Конспект — `~/Desktop/GH-600_study_guide_ru.docx`. Каталог типичных ошибок и как их избежать — `~/.claude/anti-patterns.md` (GH-600 домен 1: «identify and mitigate common anti-patterns»).

## Named chains — `~/.claude/chains/`

Для типовых многошаговых сценариев (3+ stage с зависимостями) — именованные цепочки в `~/.claude/chains/`. Первичная сущность оркестратора, рядом с доменными агентами. *(Импорт из К-7 аудита 2026-05-20.)*

| Chain | Когда срабатывает | Status |
|-------|-------------------|--------|
| `chain:docx-from-template` | «сгенерируй документ по шаблону», «заполни шаблон» | ✅ |
| `chain:pdf-scan-extract` | «разбери скан-PDF», «извлеки данные из скана» | ✅ |
| `chain:project-doc-pack` | «собери комплект ПЗ + спецификация + ведомости» | 🚧 stub (зависит от stage-decomposition `designer`) |

**Логика оркестратора:**

1. При получении запроса — проверить таблицу триггеров chains. Если совпадает — запустить chain (прочитать `chains/<name>.md` целиком, выполнить шаги последовательно).
2. Если не совпадает — стандартный flow main → доменный агент → ревьюер.
3. **На каждом шаге chain — обязателен verify-критерий** (см. Karpathy §4 выше).
4. При провале verify — chain останавливается, эскалация пользователю.

Методология создания новых chains — скилл `chains-pattern` (auto-loads по триггерам «составь pipeline», «многошаговая задача», «несколько последовательных шагов»). Полный референс — `~/.claude/chains/README.md`.

## MCP-роутинг по расширениям файлов

Когда пользователь даёт файл с одним из этих расширений — использовать **указанный MCP**, не Read tool. Read на бинарном документе показывает мусор.

| Расширение            | MCP сервер                          | Заметка                                |
|-----------------------|--------------------------------------|----------------------------------------|
| `.docx`               | `word` (предпочт.) или `markitdown` | прямая правка / конвертация в md       |
| `.xlsx`               | `excel`                              | чтение/запись ячеек, формулы           |
| `.pdf`                | `pdf-mcp`                            | таблицы и текст                         |
| `.pptx`               | `markitdown`                         | слайды → markdown                       |
| `.doc`, `.rtf`, `.odt`| `markitdown`                         | универсальный конвертер                 |
| любой офис (fallback) | `document-loader`                    | офисные доки + PNG-рендер слайдов       |

**Если MCP показывает `✗ Failed to connect`** в `claude mcp list` — это первый запуск, кэш `uvx` пустой. Прогрев одной командой:
```powershell
uvx <server-name> --help
```
Затем `claude mcp list` снова. После прогрева сервер коннектится за миллисекунды.

## Универсальные MCP (помимо файлового парсинга)

- **fetch** — выкачать произвольный URL (норматив с сайта, прайс-лист онлайн, документация). Использовать при запросах вида «открой ссылку», «что на этом сайте», «скачай и разбери».
- **time** — текущее время, часовые пояса, расчёт дат. Использовать при «сколько дней до», «какое сейчас время в», «какой день недели был».
- **sequential-thinking** — meta-cognition для сложных multi-step задач. Подключать **явно**, не по умолчанию — когда пользователь просит «давай разложим по шагам», «давай продумаем план», «нужно структурированно».

## Агенты-проверяльщики (read-only)

Спавнить через Task tool после правки/генерации файла или перед сдачей:

- **pdf-reviewer** — read-only ревью PDF (структура, метаданные, аннотации, формы, шрифты). После любой существенной правки PDF.
- **excel-validator** — read-only валидация xlsx (формулы, типы, дубликаты, сравнение с эталоном). Перед сдачей xlsx или после генерации.
- **word-checker** — read-only ревью docx (структура, незаполненные плейсхолдеры, форматирование, таблицы). После генерации по шаблону.
- **auditor** — общий независимый ревьюер артефактов; получает только ТЗ и артефакт, без хода рассуждений; failure-mode строгий.

**Failure-mode строгий:** агент-проверяльщик НЕ пройден если он отвечает общими фразами («всё ок», «документ в порядке») без явного списка проверенных пунктов. Молчание/отказ ≠ «всё хорошо» — это ловушка ложного чувства безопасности. При неуверенности — переход на ручной контроль или вопрос пользователю.

## Harvest-workflow (внешние инструменты с GitHub)

**Цель:** не изобретать велосипед, использовать готовые инструменты, **где безопасно**.

**Когда применять:** в задаче возникла потребность в инструменте (конвертер, библиотека, скилл, агент). Прежде чем писать с нуля — поискать готовое.

**Безопасные действия (auto):**
- WebFetch/`gh api` для метаданных и README репозиториев (read-only).
- Запись заметок в `session-reports/<...>/harvested/<repo-name>.md` с описанием, лицензией, оценкой.

**Изолированные действия (через Windows Sandbox или временный venv):**
- Клонирование в `~/.claude/_sandbox/` (в .gitignore).
- Запуск кода для проверки **без доступа к credentials и production-данным**.
- После теста — `_sandbox/` очищается.

**Запрещено auto:**
- Клонировать в основную ФС.
- Запускать код в основной системе с credentials.
- Копировать файлы из чужих репо в `agents/`/`skills/` без явного решения пользователя.
- Импортировать через pip/npm пакеты из найденных репо без согласия.

**Фильтр кандидатов** (минимум для написания заметки):
- Активный (last commit < 12 месяцев).
- ≥50 stars или явная репутация автора.
- Лицензия ясна (MIT/Apache/BSD/ISC/CC0/Unlicense — зелёный свет; GPL — флаг для согласования; нет LICENSE — заметка пишется, **код не копируется**).

**Адаптация в `agents/` или `skills/`** (то, что пойдёт в раздачу через installer):
- Используем **только из внутреннего круга** (личные ПК пользователя + 1-2 коллеги). При расширении — пересматриваем дисциплину.
- Только репо с разрешающей лицензией.
- «Нет лицензии» — переформулируем своими словами по идеям, не копируем код.
- GPL — не берём (вирусность лицензии портит installer).

## Auto-sync инфраструктура (информационно)

`~/.claude/` склонирована через git из репо
[claude-base](https://github.com/daniileliseev1337/claude-base). Между
ПК синхронизация **автоматическая** через Claude Code hooks
(`~/.claude/settings.json`):

- **SessionStart hook** → запускает `~/.claude/scripts/auto-pull.ps1`
  → `git pull --rebase --autostash` (актуализирует базу от других ПК).
- **SessionEnd hook** → запускает `~/.claude/scripts/auto-push.ps1`
  → если есть изменения в whitelist managed paths (agents/, skills/,
  memory/, session-reports/, harvested/, CLAUDE.md) → `git add` +
  `git commit` + `git push origin main`. Personal files (credentials,
  history, plugins, projects) **никогда** не коммитятся.

Лог auto-sync: `~/.claude/auto-sync.log`.

**Важно для понимания:**
- Это **не правило в CLAUDE.md**, которому я следую как модель — это
  системные hooks, срабатывают автоматически без моего участия.
- Системное правило Claude Code «не пушить без явной просьбы пользователя»
  касается **моих** ручных `git push` через Bash tool — оно **не**
  отменяет auto-sync hooks.
- Подробности и история ловушек настройки — в `memory/2026-05-09_hooks-debugging.md`.

## Auto-memory — пользоваться активно

Claude Code автоматически ведёт память на каждый проект в `~/.claude/projects/<project-id>/memory/MEMORY.md`. **Это не CLAUDE.md** — две независимые вещи:

| | CLAUDE.md | auto-memory |
|---|---|---|
| Что | Статические правила | Накопленные факты |
| Кто пишет | Пользователь / installer | Сам Claude |
| Где | `~/.claude/CLAUDE.md` | `~/.claude/projects/<id>/memory/MEMORY.md` |
| Перенос между ПК | Через installer | Локально, не переносится |

**Когда сохранять в auto-memory:**

- **user**: роль, окружение, специфика работы пользователя («работает в ПТО», «использует 1С УНФ», «ПК без админ-прав»).
- **feedback**: пользователь скорректировал подход (с **Why** и **How to apply**).
- **project**: текущие задачи, дедлайны, мотивации решений (с относительными датами → абсолютными).
- **reference**: где живут внешние ресурсы («сметы — в \\\\srv\\smeta\\», «нормативы — в C:\\norms\\»).

**Не записывать в auto-memory:**
- Пароли, API-ключи, ПДн, банковские данные.
- Дублирование того что уже в CLAUDE.md.
- Эфемерное («сейчас работаю над X»).

## Универсальные правила работы

1. **Не выдумывать.** Нет данных → `AskUserQuestion`. Особенно: имена, пути, номера, даты, количества. Не подставлять placeholder без явного указания.
2. **Спрашивать при неопределённости.** Любое неявное решение (маппинг, формат, выбор номера, путь к файлу) — явный вопрос пользователю.
3. **Read SKILL.md до действия.** Когда скилл активирован по триггеру — сначала прочитать его SKILL.md, потом действовать.
4. **TodoList на 3+ шагов.** `TaskCreate` / `TaskUpdate` для сложных задач. Последний пункт почти всегда — верификация.
5. **Общение — на русском.** Код / имена файлов — латиницей или гибридно, комментарии — по-русски.
6. **Обезличивание для всего, что пушится в claude-base** (`agents/`, `skills/`, `memory/`, `session-reports/`). Никаких ФИО, шифров `Ф.YYYY.NNNNNNNNN`, названий объектов, заказчиков, адресов — заменять на плейсхолдеры (`<подписант>`, `<шифр>`, `<строительный объект>`).

## Прокси (если за корп-прокси)

Перед запуском `claude` или открытием VS Code: убедиться что прокси выставлен в текущей PowerShell-сессии. claude-lite-instaler копирует хелперы в **`~/.claude/bin/`** (per-machine, не в git) и ярлык в Пуске:

- `~/.claude/bin/Set-Proxy.ps1` — поставить env-vars в текущем окне (`& "$HOME\.claude\bin\Set-Proxy.ps1"`)
- `~/.claude/bin/Start-Claude.bat` — двойной клик: прокси + выбор CLI/VSCode + claude
- `~/.claude/bin/Start-Claude.ahk` — глобальная горячая клавиша Ctrl+Alt+C (требует AutoHotkey v2)
- **Пуск → «Claude (with proxy)»** — одним кликом запустить (ярлык на `Start-Claude.bat`)

Хелперы persistent: если installer-папка удалена, доступ к ним сохраняется. Урок 15 закрыт коммитом `3562631` в claude-lite-instaler (2026-05-18).

Прокси-конфиг (host:port + login) хранится в `~/.claude-proxy.json` (без пароля). Пароль спрашивается каждую сессию.

Без прокси: MCP-серверы не качают пакеты (✗), `claude auth login` не работает, `fetch` не выходит наружу.

## Дисциплина контекстного окна и proactive handoff

Длинные сессии (70+ tool calls, 30+ user turns) перегружают context
window — Claude начинает «забывать» ранние решения и терять точность.
Скилл `handoff-to-new-chat` (auto-load по триггерам «давай в новый чат»,
«контекст полный», «handoff», «передай в новую сессию») формирует
mid-session отчёт и готовый промпт.

### Proactive-триггеры (Claude сам предлагает handoff)

Двухуровневая система **WARNING / CRITICAL** (заим из GSD Redux
`/gsd-health --context`, см. `~/.claude/harvested/gsd-redux.md`).
Детали thresholds — в `~/.claude/skills/handoff-to-new-chat/SKILL.md`.

**WARNING (~60% утилизация)** — 1-2 признака → не handoff, но
дисциплинировать (cascade loading, offset/limit, Agent для исследований).

**CRITICAL (~70% утилизация)** — **двa и более** одновременно →
Claude сам вызывает `AskUserQuestion` с предложением handoff:

- 5+ Read больших файлов (> 200 строк каждый) в сессии
- 3+ Agent (subagent) вызовов с большими ответами
- 30+ user turns
- 5+ объёмных Bash outputs (> 200 строк stdout)
- 5+ Edit/Write на разные файлы
- 3+ параллельные branches задач

### Дисциплина контекста для Claude (профилактика)

1. **Cascade loading** — читать только нужную секцию большого файла.
   Сначала `Grep` на заголовки, потом `Read offset=N limit=50` на
   релевантную секцию. **Не читать всё целиком**.
2. **Structured artifacts** для проектов в 3+ фазы — выносить контекст
   в `ROADMAP.md` / `STATE.md` / `PLAN.md` / `REVIEW.md` / `DECISIONS.md`
   на диске вместо «простыни» в чате. Каждый агент читает **только
   нужный** артефакт через cascade loading.
3. **Tool outputs через фильтры** — `| tail -5`, `| grep PATTERN`.
   Не получать walls of text.
4. **Длинные исследования через Agent** — для 5+ файлов или 3+ мест
   делегировать в `Agent(subagent_type="Explore"/"general-purpose")`.
   Subagent возвращает summary, основной контекст не раздувается.
5. **Background для долгих команд** — `run_in_background=true` для
   процессов > 30 сек. Не блокирует и не наполняет контекст stdout'ом.
6. **Не дублировать рассуждения** — ссылаться на план выше, а не
   пересказывать.

### Когда handoff vs `/compact`

- **handoff-to-new-chat** — задача важная, нужна точность, есть
  open questions. Новый чат с briefing'ом + полный session-report.
- **`/compact`** (встроенная команда Claude Code) — задача простая,
  достаточно сжатого контекста в той же сессии. Менее надёжно для
  критичных задач (compaction lossy).

Скилл и встроенная команда дополняют друг друга.

## GitHub — обязательный bypass proxy (override Прокси-секции)

**Корп-прокси блокирует CONNECT-метод к GitHub.** Empirically на всех наших ПК `git push`/`pull`/`fetch` к github.com через прокси падает с `Proxy CONNECT aborted`. Решение — **bypass proxy** для GitHub-домена.

### Persistent fix (one-time на каждый ПК)

```powershell
git config --global http.https://github.com/.proxy ""
git config --global https.https://github.com/.proxy ""
```

После этого **все** git-операции к `github.com` автоматически идут напрямую, мимо прокси. Никакие `-c` флаги в командах не нужны. **Это рекомендованный путь — применить один раз и забыть.**

Проверка что применилось:
```powershell
git config --global --get http.https://github.com/.proxy
# Должно вернуть пустую строку (= bypass)
```

`setup-extras.ps1` авто-применяет эту настройку при первом запуске на новом ПК (см. Step 0).

### Правила для Claude (если persistent config не настроен)

Если по какой-то причине git config bypass не настроен — **использовать `-c` флаги** в каждой команде:

```powershell
# git push / pull / fetch / clone — bypass proxy:
git -c http.proxy="" -c https.proxy="" push origin main
git -c http.proxy="" -c https.proxy="" pull --rebase --autostash
git -c http.proxy="" -c https.proxy="" fetch origin main
git -c http.proxy="" -c https.proxy="" clone https://github.com/<user>/<repo>.git

# gh CLI (GitHub API) — bypass через env:
$env:HTTPS_PROXY=""; gh pr view 123
# Или одной командой через native scope:
$env:HTTPS_PROXY=""; gh repo clone <user>/<repo>
```

### Что bypass-ится (whitelist прямого подключения)

- `github.com` (git operations, web)
- `api.github.com` (`gh` CLI, REST API)
- `raw.githubusercontent.com`, `*.githubusercontent.com` (raw файлы, releases)
- `objects.githubusercontent.com` (LFS, releases binaries)

### Что нормально через прокси

- `pypi.org`, `files.pythonhosted.org` (uvx, pip install)
- `registry.npmjs.org` (npm install)
- `huggingface.co`, `cdn-lfs.huggingface.co` (модели — LaMa, EasyOCR, SD)
- Microsoft / Anthropic / другие — через прокси нормально.

### Harvest и WebFetch на GitHub

Harvest-workflow (`/harvest <query>`) часто ходит на GitHub через:
- `WebFetch` tool на URL вида `https://github.com/<owner>/<repo>` — **bypass автоматический** (WebFetch на уровне Claude Code обходит прокси для GitHub).
- `gh api search/repositories?q=...` — **требует** `$env:HTTPS_PROXY=""` перед вызовом если persistent git config не покрывает (gh не использует git config).

### Anti-pattern

Не пытаться обойти прокси-блок «костыльно» — например через VPN или secondary proxy. Стандарт нашей команды: bypass proxy для GitHub-домена, всё остальное — через корп-прокси.

См. также `~/.claude/anti-patterns.md` категория 4 (Безопасность) и категория 8 (PowerShell/Windows ловушки).

## Harvest-workflow — проактивные триггеры (дополнение к CORE)

CORE-секция `Harvest-workflow` описывает **что и где** искать, фильтры
кандидатов и адаптацию. Этот раздел — **когда** проактивно запускать.

### Триггеры

- **Первое появление нового типа файла в сессии** (по расширению или
  типу задачи). Дедупликация: повтор того же типа в той же сессии —
  пропустить.
- **Старт новой задачи** (пользователь сказал «новая задача:», «давай
  делать X», и т.п.).

### Алгоритм

1. **Проверить свою базу** — `~/.claude/skills/` и `~/.claude/agents/`.
   Если есть подходящий инструмент — работать им, harvest не нужен.
2. **Если нет** — **НЕ молча искать.** Спросить через `AskUserQuestion`:
   > «Новая задача типа <X>. Поискать готовые инструменты (2-5
   > кандидатов на GitHub / MCP registry / Anthropic skills) или
   > работать сразу своими средствами?»
3. Если пользователь сказал «найди» — запустить логику `/harvest <query>`
   (см. `commands/harvest.md`).
4. Если «работай сразу» — пропустить harvest для этого типа задачи в
   этой сессии, запомнить решение в контексте.

### Источники в порядке приоритета (на 2026-05-13)

1. **GitHub** — самый широкий охват. Основной источник пока наша база
   скиллов / агентов и Anthropic available skills не разрастутся до
   приличного покрытия. Поиск: `WebSearch "site:github.com <query>"`
   или `gh api search/repositories?q=<query>&sort=stars`.
2. **MCP registry** — `mcp__mcp-registry__search_mcp_registry` для
   протокол-специфичных задач (нужен MCP-сервер под формат).
3. **Anthropic available skills** — fallback / справочная. Сейчас
   используется как «есть ли встроенное», не как основной источник.

> **Порядок временный.** Когда наша база скиллов разрастётся и реально
> начнёт работать (>5 проверенных скиллов, успешно применённых в
> реальных задачах) — перевернём порядок (Anthropic skills первыми
> как самый безопасный, GitHub последним как самый широкий но шумный).
> Решение пользователя 2026-05-13: «в работе наши инструменты не
> работают как хотелось бы, GitHub полюбому найдёт».

### Объём кандидатов

**2-5**. Если фильтрация даёт меньше 2 — не застревать, выдать что
есть или сказать «не нашёл подходящего, работаю своими средствами».
Никаких «100 кандидатов» — это переусложнение и риск безопасности.

### Фильтр (из CORE-секции `Harvest-workflow`)

- Активный: последний коммит <12 месяцев
- ≥50 stars или явная репутация автора
- Лицензия: MIT/Apache/BSD/ISC/CC0/Unlicense — зелёный;
  GPL — флаг для согласования; нет LICENSE — заметка пишется, **код не
  копируется**.

### Куда сохранять заметки

- **Per-session** (текущие находки этой сессии):
  `session-reports/<YYYY-MM-DD>_<тема>/harvested/<repo>.md`
- **Накопленная база** (общая для всех ПК, через auto-sync):
  `~/.claude/harvested/<repo>.md`

При завершении сессии релевантные заметки переносятся / копируются в
общую базу.

### Когда НЕ применяется

- Уже есть свой скилл/агент в `~/.claude/skills/` или `~/.claude/agents/`
  под задачу — используем его.
- Рутинная задача (просто чтение, простое редактирование) — инструмент
  не нужен.
- В сессии уже спрашивали про этот тип задачи (дедупликация).
- Тривиальный one-off — написать самим быстрее чем искать.

## Sessions — обязательный отчёт каждой сессии

**Решение пользователя 2026-05-14:** репо `claude-base` теперь **private**,
обезличивание смягчено — Claude может писать в `session-reports/`
**сырые** отчёты (без жёсткой санитизации), потому что доступ только
у нашего внутреннего круга. Пользователь обрабатывает и оставляет в
методике (`agents/`, `skills/`, `memory/`) только нужное.

### Правило: каждая сессия пишет session-report

**Обязательно**, не «по запросу пользователя». Триггер — закрытие
сессии (последний ответ Claude перед концом контекста или явным
«закрываем»). Под закрытием понимается:

- Пользователь сказал «всё», «достаточно», «закрываем», «спасибо» как
  финал, «hand-off» и т.п.
- Тема явно завершена (пользователь подтвердил результат, новых
  вопросов нет)
- Контекст близко к compaction — лучше зафиксировать сейчас, чем
  потерять детали после compaction

**Где сохранять:**
```
~/.claude/session-reports/<YYYY-MM-DD>_<краткая_тема_kebab-case>/
  ├── report.md        ← основной отчёт (см. _TEMPLATE.md)
  └── artifacts/       ← опционально, обезличенные образцы артефактов
```

**Идемпотентность:** если за этот же день+тему уже есть папка —
**обновлять** существующий `report.md`, не плодить новые. Тема может
эволюционировать в течение дня.

**Структура `report.md`:** см. `~/.claude/session-reports/_TEMPLATE.md`.

**Что писать честно (без приукрашивания):**

- Что **делал** — хронология (3-10 пунктов).
- **Источники**, использованные реально: MCP-серверы по именам,
  скиллы по триггерам, slash-команды, нормативы/каталоги из библиотеки,
  harvest-находки если были.
- **Артефакты** — пути к финальным DOCX/PDF/XLSX в cwd сессии пользователя.
- **Итерации и сломалось** — где промахивался, что переделывал. Это
  ценнее любого «всё прошло гладко».
- **Что выдумывал** — если в моменте подставил placeholder вместо
  спрашивания у пользователя — указать.
- **Цитаты пользователя** — дословные, если были важные («не молча»,
  «не переусложняй», корректировки подхода).
- **Открытые вопросы** — что осталось на следующую сессию.

**Обезличивание (смягчённое для private):**
- **Не нужно** обезличивать имена hosts, шифры объектов, бренды
  оборудования, имена сотрудников в команде — это для нашего разбора.
- **Не писать** только: пароли, API-ключи (включая токены GitHub),
  ПДн (паспорта, СНИЛС), банковские реквизиты. Если такое промелькнуло
  в сессии — пометить `[СЕКРЕТ — не записан]`.
- Цели обработки потом пользователем — превратить сырой отчёт в
  обезличенный кейс в `memory/`, или в обновление скилла в `skills/`,
  или просто оставить как есть для будущего разбора.

### Правило: статус auto-sync в начале сессии

В первой реплике сессии (сразу после строки `✓ прочитан CLAUDE.md
(MCP: X/8)` из STOP-процедуры) — добавить **одну строку** про auto-pull.

**ВАЖНО (2026-05-21):** перед формированием отчёта про auto-pull,
прочитать `~/.claude/auto-sync.log` через `Get-Content -Tail 10` и
найти **последнюю** запись `auto-pull:` или `auto-push:`. Если она:

- Заканчивается на `DONE` — hook отработал корректно, описываем
  результат (ok/FAILED/pulled N commits).
- Заканчивается на `WARN: previous hook was interrupted` или
  `start` без последующего `DONE` — **предыдущая** операция была
  **прервана** (таймаут/kill). В первой реплике явно сказать:
  ```
  ⚠ Предыдущий auto-sync был аварийно прерван — запусти на этом ПК:
    cd $HOME\.claude
    git -c http.proxy="" -c https.proxy="" pull origin main
  ```
- Содержит `WARN: pull reported success but HEAD still behind` —
  pull вернул "ok" но коммиты не подтянулись (network glitch?).
  Тоже предложить manual pull.

Это критично: молчаливо отставшая база приводит к тому что пользователь
работает на устаревшей методике (см. инцидент 2026-05-21 на DELISEEV-PC).

```
Auto-pull: подтянуто N коммитов (last: <hash7> from <host> — <тема>)
```
или
```
Auto-pull: без изменений (HEAD == origin/main)
```
или
```
Auto-pull: FAILED (см. ~/.claude/auto-sync.log; продолжаю в read-only к claude-base)
```

**Источник информации:**
- Прочитать tail `~/.claude/auto-sync.log` (последние 10 строк) — там
  что записал auto-pull hook при старте этой сессии.
- ИЛИ `git log HEAD~5..HEAD --pretty=format:'%h %ad %s' --date=short`
  в `~/.claude/` — последние 5 коммитов, видно что свежее в базе.

### Правило: статус auto-sync в конце сессии

В **последнем** ответе перед закрытием темы — указать **прогноз**
auto-push:

```
Auto-push на SessionEnd: ожидается push N коммитов
(managed paths: <список изменённых>) → origin/main.
В следующей сессии увижу результат в логе.
```
или
```
Auto-push на SessionEnd: managed paths не менялись → push не будет
(только session-report этой сессии, если он попадает в managed).
```

Реальный результат push'а увижу при следующем старте сессии
(см. правило «статус auto-sync в начале сессии»).

### Когда правило не применяется

- Сессия длится <2 реплик и явно не содержит работы (например, просто
  поздоровались) — session-report не нужен.
- Пользователь явно сказал «не пиши отчёт по этой сессии» — пропускаем.

и явно подсказывает пользователю restart.

но при попытке `Agent(subagent_type='<имя>')` Claude Code вернёт
`Agent type '<имя>' not found`. Эталон сверяется со списком в system
prompt `Available agent types for the Agent tool`.

## Стиль общения — per-PC opt-in для умеренного русского мата

**Подключаемая фича:** разрешить Claude использовать **умеренный русский
мат** в эмоциональных моментах (радость, удивление, frustration, шутки)
для поднятия настроения команды. **Не** как default-стиль, **не** в
адрес людей, **не** в commit-сообщениях / документации / artifacts для
клиентов — только в чате во время сессии.

### Mechanism

Per-PC marker file: `~/.claude/.profanity-marker` (gitignored, в whitelist .gitignore не входит → автоматически блокируется).

**Содержимое:**
- `enabled` → умеренный русский мат разрешён в эмоциональных моментах
- `disabled` → официальный тон всегда

**В начале каждой сессии Claude:**

1. **Проверить** `~/.claude/.profanity-marker`:
   - Если файла нет → задать **ОДИН ВОПРОС** через `AskUserQuestion`:
     > «Разрешить умеренный русский мат в эмоциональных моментах сессии? Можно изменить позже редактированием `.profanity-marker`.»
     - Yes → создать файл с содержимым `enabled`
     - No → создать файл с содержимым `disabled`
   - Если файл есть → прочитать, действовать соответственно. **Больше не спрашивать.**

2. **Когда `enabled`:**
   - Уместный умеренный русский мат для эмоциональных реакций («охренеть как круто», «бл# сработало», «п#$%ц долго грузил SD»)
   - НЕ заменять каждое слово — только когда action в сообщении эмоционально окрашен
   - НЕ в commit-message, артефактах, документации, отчётах
   - НЕ в адрес людей (Apoliakov, коллег, пользователей других ПК)

3. **Когда `disabled`:** обычный профессиональный тон без исключений.

### Изменение постфактум

```powershell
# Включить:
echo "enabled" > $HOME\.claude\.profanity-marker
# Выключить:
echo "disabled" > $HOME\.claude\.profanity-marker
# Спросить заново при следующей сессии:
Remove-Item $HOME\.claude\.profanity-marker
```

## Экономия токенов — практические правила

Эта секция собрана 2026-05-26 после сессии с реальным замером (~3-5M
tokens за день на Opus 4.7 1M context). Проверенные факты, **без**
гипотез про prompt-caching в markdown (это миф — caching работает на
уровне API/SDK, не контролируется пользователем Claude Code Desktop).

### 1. Выбор модели — главный рычаг

В Claude Code Desktop / CLI у пользователя **отдельные квоты** на Opus
и Sonnet (видно в UI: `5-hour limit` для Opus, `Sonnet only` для Sonnet).

**Переключение модели в текущей сессии:** команда `/model sonnet` или
`/model opus` (доступна и в Desktop, и в CLI).

**Когда какая модель:**

| Задача | Модель | Причина |
|---|---|---|
| Чтение/правка одного-двух файлов, find-replace, переименования | **Sonnet** | Достаточно — экономим Opus квоту |
| Word/Excel шаблонные документы по skills | **Sonnet** | Skills делают тяжёлую работу, агент только оркестрирует |
| OCR scan, парсинг УПД (через upd-parser) | **Sonnet** | Логика в skills, модель управляет шагами |
| Запрос к норм-базе через `norm-lookup` | **Sonnet** | Read-only поиск, не reasoning |
| Раскрутка спорного нормоконтроля, ответ экспертизе, проектирование | **Opus** | Reasoning важен |
| Длинная исследовательская сессия с 5+ Agent calls | **Opus** | Subagent качество критично |
| Любая правка кода с нетривиальной логикой | **Opus** | Точность важнее скорости |

**Правило по умолчанию:** **Sonnet** если задача не требует
доказанно Opus-уровня reasoning. Не доказано — берём Sonnet.

### 2. Команды управления контекстом

| Команда | Что делает | Когда применять |
|---|---|---|
| `/compact` | Сжимает накопленный контекст (lossy). | При **50%+** context window UI индикатор. Перед длинной задачей. |
| `/clear` | Полный reset сессии — теряем всё. | Только при pivot на совершенно другую тему. |
| `/model <sonnet\|opus>` | Переключение модели. | При смене типа задачи (см. таблицу выше). |
| `/cost` (если доступна) | Показывает расход текущей сессии. | Sanity-check время от времени. |

### 3. Дисциплина расхода (поведенческие правила)

**Не лить дорогие операции:**
- **Agent (subagent) calls стоят в 3-5х дороже** обычной реплики —
  каждый запускает отдельную сессию с полным контекстом. Использовать
  только когда:
  - Нужна **независимая** проверка (auditor для финального артефакта).
  - Нужно **многошаговое исследование** которое раздует основной контекст.
  - **НЕ** для тривиальной задачи которую решает основной Claude.
- **Один Agent call вместо трёх** — собирать пачку файлов/задач в
  одну инструкцию субагенту, не вызывать его 3-5 раз с похожими TZ.
- **Не делать retry на 529 Overloaded** в loop — это **дублирует**
  стоимость. Лучше задержать сессию или сделать handoff.

**Cascade loading вместо whole-file Read:**
- Сначала `Grep` на ключевые секции файла.
- Потом `Read offset=N limit=50` только нужной части.
- **НЕ** `Read` 400-строчного файла целиком чтобы найти 10 строк.

**Tool outputs через фильтры:**
```powershell
# Плохо: вернёт 30K строк
pip install paddleocr

# Хорошо: только результат
pip install paddleocr 2>&1 | Select-Object -Last 5
```

**Background для длинных команд:**
- `run_in_background=true` для Bash/Agent если результат нужен потом.
- Не блокирует контекст stdout'ом в реальном времени.

### 4. Мониторинг — что смотреть в UI

Перед длинной задачей **проверить индикатор в правом нижнем углу**
(панель Plan usage):

| Показатель | Что значит | Действие |
|---|---|---|
| `Context window X / 1.0M (Y%)` | Текущий контекст в сессии | При Y > 50% → `/compact` или handoff |
| `5-hour limit X%` | Opus квота за 5 часов | При X > 80% → переключиться на Sonnet или handoff в новой сессии через 5h |
| `Weekly · all models X%` | Total за неделю (Opus+Sonnet) | При X > 70% → беречь до конца недели |
| `Sonnet only X%` | Отдельная квота Sonnet | Если 0% → есть запас для рутины |

**Правило:** **сначала смотрим UI, потом действуем.** Не оценивать «на
глаз» — у меня (Claude) нет видимости в эти показатели, я их сам не
вижу. Только пользователь видит UI.

### 5. Что НЕ работает (антипаттерны)

- ❌ **Prompt Caching через markdown-комментарии** в CLAUDE.md
  (`<!-- cache_control -->`) — **не существует**. cache_control это
  API-level параметр в request body, недоступен пользователю Claude
  Code Desktop.
- ❌ **Условный agent dispatcher** в Claude Code (типа «если задача
  X → Sonnet, иначе Opus») — Claude Code не имеет user-level routing.
  Модель управляется командой `/model` для всей сессии.
- ❌ **Retry в loop при 529 Overloaded** — удваивает стоимость без
  results. Лучше ждать или сделать handoff.
- ❌ **Несколько Agent calls подряд с похожим TZ** — слить в один.

### 6. Конкретные источники расхода (из retro 2026-05-26)

Что обошлось дорого в реальной сессии:
- 7 вызовов `auditor` с retries и revisions — можно было 2-3 вызова
  с пачкой.
- 11 целых `Read` agent.md по 300-400 строк — можно было cascade.
- 2 `Agent(Explore)` по GitHub — один развёрнутый `WebFetch` дешевле.
- 30+ turns без handoff — надо было разбить на 2-3 сессии.
- **`Extra high` reasoning mode** включён на Opus — это самый дорогой
  режим. Использовать только когда reasoning критичен.

- `~/.claude/skills/handoff-to-new-chat/SKILL.md` — methodology handoff и
  дисциплина контекста (§1-7).
- `~/.claude/projects/<id>/memory/feedback_context_heuristics_1m.md` —
