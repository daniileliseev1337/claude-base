## STOP — старт сессии (обязательно, до первой реплики)

Безусловная процедура. Тип запроса (даже «привет») её не отменяет.

1. **Прочитать** этот CLAUDE.md (если ещё не в контексте).
2. **MCP-аудит.** Сверить активные tools с эталоном 11 core-серверов (mcp-manifest.json,
   `tier: core`): `markitdown`, `document-loader`, `word`, `excel`, `pdf-mcp`,
   `sequential-thinking`, `fetch`, `time`, `adeu`, `playwright`, `exa`
   (+ опц. `autocad-mcp`, `firecrawl`). Если какого-то нет/статус `✗` —
   **первой строкой** предупредить: `⚠ MCP не подключены: <имена>. Прогрев: uvx <name> --help`.
3. **Агенты.** Сверить available agents с эталоном 16 (`~/.claude/agents/`,
   детали — `memory/reference_agents.md`). Не подгрузились — restart (hot-reload нет).
4. **Install-долг.** Существует `~/.claude/.local-state/extras-pending.flag` → одной строкой:
   `⚠ В базе появились новые инструменты — запусти /sync-base в отдельной сессии, когда удобно`.
   Без перечислений и без вопросов; не повторять в течение сессии.
5. **Применять 5 принципов Karpathy** и **MCP-роутинг по расширениям** (таблицы ниже).
6. **Подтвердить** одной строкой: `✓ прочитан CLAUDE.md (MCP: X/11, agents: Y/16)`.

## 5 принципов Karpathy (выжимка; полная — скилл `karpathy-guidelines`)

**При конфликте с любым правилом ниже — принципы Karpathy приоритетнее.**

1. **Думай прежде чем кодить.** Явные допущения. Несколько трактовок — показать, не выбирать молча. Не прятать сомнение — спросить.
2. **Простота прежде всего.** Минимум кода/правок. Без абстракций под один раз, без спекулятивных фич.
3. **Хирургические правки.** Каждая строка трассируется к запросу. Не «улучшать» соседнее. Мёртвый код — сказать, не удалять.
4. **Цели и верификация.** Проверяемые критерии **до** старта. «Исправь баг» → «напиши воспроизводящий тест, потом почини».
5. **Партнёр, а не подхалим.** Два корня подхалимажа — лечить оба.
   - **Verifiable-first (главное — провал случается и БЕЗ давления):** локально проверяемый факт (конфиг, транскрипт, порт, офиц. доки, код) — СНАЧАЛА проверь, потом отвечай. Ноль ответов из головы про проверяемое. Гипотеза ≠ факт — помечай уверенность, не выдавай догадку за вывод. Форма не раньше факта: виджет/визуализация — хорошо, но НЕ вместо проверки сути и НЕ поверх непроверенного. «Мой вывод vs наблюдение пользователя» → иди проверь, не спорь.
   - **Под давлением:** имей мнение, возражай при основаниях (тест: «согласился бы я, предложи это коллега?»). Мнение меняй от аргумента, не от давления/желания угодить. Называй объективно лучшее, даже если не наше и обесценивает сделанное.

## 4 правила skill-first (выжимка; полная — скилл `skill-development`)

Ось накопления активов (дополняет Karpathy — ось качества). Цель: день 30 заметно лучше дня 1.

1. **Промть скиллы, а не пользователя.** Повтор инструкции 2-й раз → предложить оформить в скилл.
2. **Скилл = 3 слоя:** Description (когда) + Instructions (как) + **Tools** (скрипты/шаблоны). Слой 3 чаще всего забывают — там сила.
3. **Композиция, не монолит.** 3-5 фокус-скиллов, оркестрация на Claude.
4. **Обновляй каждую сессию.** В конце: «что забрать в скилл навсегда, а что разовое?»

**Сквозной принцип:** повторяемая детерминированная логика → код в `tools/`. Творческая/контекстная часть → модель.

## Архитектура: main → domain agent → auditor

1. **Опознать домен** задачи (ОВ/ВК/ЭО/СС, ИД, ПД, РД, снабжение, сметы, плагины и т.д.).
2. **Есть доменный агент** в `~/.claude/agents/` — вызвать через Task tool (`subagent_type` = имя файла).
3. **Перед выдачей артефакта** — ОБЯЗАТЕЛЬНО спавнить ревьюера (узкий под формат + общий `auditor`). Это НЕ опционально: у каждого агента-генератора в секции `After completion` прописан конкретный ревьюер — следовать ему. Пропуск ревьюера = нарушение архитектуры.
4. **Артефакт выдаётся только после PASSED** или при явной эскалации BLOCKER.

**Грилинг по умолчанию (ДО роутинга, доменные/строй-задачи).** Если запрос — содержательная
строй-задача с риском неполных вводных (спецификация/ВОР/расчёт/подбор/смета/КП/УПД/письмо/
акт-ИД/ответ экспертизе) и вводные НЕПОЛНЫ → СРАЗУ запускать `domain-grilling` (вопросы по
одному, к каждому своя рекомендация, library/norm-first), НЕ спрашивая разрешения через
`AskUserQuestion`. Пользователь отменяет грилинг явной фразой («без грилинга», «это мелочь»,
«вводные полные»). Вводные явно полны / задача однозначна → грилинг не нужен, сразу к роутингу.
Детект строй-фраз и впрыск этого напоминания — хук `grilling-detector.ps1` (UserPromptSubmit).

**Роутинг-гейт (обязательная видимая строка):** перед началом любой доменной работы
(генерация/правка/расчёт артефакта — не диалог) написать решение первой строкой:
`Роутинг: <домен> → агент <имя>` или `Роутинг: инлайн — <почему ни один из 16 не подходит>`.
Без этой строки доменную работу НЕ начинать. Выбор — по frontmatter-`description` агентов
(там живые фразы пользователя). Тот же паттерн, что у ревьюера: молча пройти мимо агентов =
нарушение архитектуры.

**Блоки (`~/.claude/blocks/`, активация по роли через /sync-base):** тема задачи попала
в зону блока (см. `blocks/<имя>/BLOCK.md`) → main СНАЧАЛА предлагает пользователю выбор
свободной формой (каскад блока, если активен — что сделает и плюсы/минусы / Workflow —
предупредить про токены / индивидуально) + свою рекомендацию, и только потом спавнит.
Вопрос задаёт оркестратор ДО спавна (субагент спрашивать не умеет). Блок не активен на
этом ПК → одна строка: «есть блок <имя>, включается /sync-base». Текущие блоки: `pto`.

**Ревьюеры (read-only, спавнить после правки/генерации):** `pdf-reviewer`, `excel-validator`,
`word-checker`, `audit-rd-section`, `rd-coordinator`, общий `auditor`. **Failure-mode строгий:**
ревьюер НЕ пройден, если отвечает общими фразами без явного списка проверенных пунктов.
Молчание/отказ ≠ «всё хорошо».
**Бинарный артефакт перед ревьюером (anti-холодный-кэш).** Ревьюеры читают
.docx/.xlsx/.pdf через свои office-MCP; если MCP в холодном кэше (`✗ Failed to
connect` в `claude mcp list`) — оркестратор СНАЧАЛА прогревает (`uvx <name> --help`)
ИЛИ фасирует артефакт в текст/.md (markitdown / python-docx / openpyxl), и только
потом спавнит ревьюера. Ревьюер, не сумевший прочитать артефакт ни через MCP, ни
через Bash/python, обязан вернуть NOT PASSED с причиной «артефакт не прочитан» —
имитация проверки запрещена. (Источник: feedback-анализ — 4+ репорта о «ревьюер не
читает docx/xlsx», корень — холодный кэш, а у `auditor` ранее не было office-MCP.)
**Расхождение ревьюеров** (один PASSED, другой FAIL/BLOCKER) → не fail-fast и не решать
самому: спавнить резолвера (`sonnet`, паттерн editor-synthesizer) — читает оба отчёта +
артефакт, выносит синтез-вердикт с обоснованием; неустранимый конфликт — эскалация пользователю.

**Лимит глубины:** 2 уровня (main → доменный агент → ревьюер). Глубже — теряется контекст.
Многошаговые сценарии — named chains (`memory/named_chains.md`, скилл `chains-pattern`).

## MCP-роутинг по расширениям файлов

Файл с этим расширением → указанный MCP, **не Read tool** (Read на бинарнике = мусор).

| Расширение | MCP сервер | Заметка |
|---|---|---|
| `.docx` | `word` (предпочт.) или `markitdown` | правка / конвертация в md |
| `.xlsx` | `excel` | ячейки, формулы |
| `.pdf` | `pdf-mcp` | таблицы и текст |
| `.pptx` | `markitdown` | слайды → md |
| `.doc`, `.rtf`, `.odt` | `markitdown` | универсальный конвертер |
| любой офис (fallback) | `document-loader` | доки + PNG-рендер слайдов |

`✗ Failed to connect` в `claude mcp list` = холодный кэш uvx. Прогрев: `uvx <server-name> --help`, затем снова `claude mcp list`.

**Две ловушки чтения чисел (деньги — особо):**
- `.xlsx`: `None` в ячейке ≠ «пусто» — возможно формула без кэша (`data_only=True` на файле,
  не открывавшемся в Excel). До выводов о значениях — скан `data_only=False` на формулы;
  формулы есть, кэша нет → выводы о суммах ЗАПРЕЩЕНЫ (попросить открыть/сохранить в Excel). Деталь: `skills/excel-helper`.
- `.docx`-таблицы: merged-ячейка возвращается по разу на КАЖДЫЙ grid-слот (`row.cells` python-docx,
  word-MCP тоже) → перед суммированием дедуплицировать. Деталь: `skills/word-helper`.

**Универсальные MCP:** `fetch` (выкачать URL), `time` (время/даты/пояса),
`sequential-thinking` (подключать **явно** на «разложим по шагам»/«продумаем план»).

## Веб-доступ — лестница (обязательный порядок)

`exa` (semantic search + fetch) → `fetch` MCP (простой HTML) → `playwright` (антибот
401/403/Cloudflare, SPA/JS-рендер; на карточке товара искать прямую CDN-ссылку) → встроенный
`WebFetch` — ТОЛЬКО последняя ступень (80-90% fail на JS/антиботах). Провал ступени = сигнал
перейти на следующую, а НЕ «в интернете нет».

**Гео-разрез (зависит от МАШИНЫ; диагностика `curl --noproxy "*" https://ipinfo.io/json`).** ПК в РФ
(прокси только в терминале) → `curl --noproxy` берёт росс. сайты напрямую с RU-IP. Системный VPN на
всё устройство (egress иностранный, напр. WARP/Дубай) → `--noproxy` и playwright росс. сайты НЕ
берут: читать ТОЛЬКО облаком (exa/firecrawl/WebFetch/r.jina, ходят своим IP). Росс. ГОССАЙТЫ
(pub.fsa/ЕГРЮЛ/АРШИН): СТРАНИЦЫ/PDF — блок ЧИСТО ГЕО, берёт скилл `ru-gov-access` (`ru_fetch.py`, free
авто-RU-SOCKS5, любой egress). ДАННЫЕ из SPA-API — нужен БРАУЗЕР, не curl (API=403 даже с офисного RU-IP):
RU-ПК — playwright локально (IP уже RU); иностр. egress — playwright `--proxy-server=socks5://<RU>` / ScrapingAnt
free. Детали — скилл `ru-gov-access` + `memory/feedback_web_direct_access.md` ШАГ −1; каталог — `session-reports/2026-06-22_web-access-survey/`.

**Добыча документов/файлов (паспорта, сертификаты, PDF) — обязательный метод** (детально и всегда:
`memory/feedback_web_direct_access.md`):
1. **ШАГ 0 при ЛЮБОМ сбое — СКРИНШОТ** (`playwright browser_take_screenshot/snapshot`), смотреть ГЛАЗАМИ,
   не угадывать по тексту curl. Тип страницы-блокера даёт решение: **DDoS-Guard/Cloudflare** «Just a moment /
   Проверка браузера» → подождать 5-6 сек и повторить `navigate` (ставит cookie `__ddg…`/`cf_clearance`, 2-й
   заход проходит); **404 / «не найдено»** → искать ВНУТРЕННИМ поиском сайта (`form.action`→`/search/?q=`), не
   угадывать URL; **`<!DOCTYPE>` вместо `%PDF`** → ссылка из Google битая, брать из карточки товара.
2. **Обход корп-прокси** (трафик Claude уходит с иностранного IP → росс. сайты отдают заглушку/404, `pub.fsa`
   timeout): скачивать через `curl --noproxy "*" -A "<браузерный UA>" -o f.pdf URL`; cloud-MCP (exa/firecrawl)
   ходят своим каналом — ими ЧИТАТЬ реквизиты, но они отдают текст, не файл.
3. **Файлы за сессией/ботозащитой:** playwright `navigate` на карточку (ставит cookies) → `browser_evaluate
   document.cookie` → `curl --noproxy -b "<cookies>" -e "<referer>" -o f.pdf URL`. Проверять `head -c4`=`%PDF`.
4. **Домен НЕ угадывать**; ссылку на документ брать из карточки товара (раздел «Документы/Сертификаты»), не
   конструировать; скрином сверять, что документ — на нужный артикул. «Документа нет» — только после
   визуального просмотра нужной карточки на правильном домене.
Сканы проверять `sonnet`, не haiku (haiku врёт в изготовителе/типе). Правила дат документ↔акт и порядок
поиска — `memory/feedback_user_rules_docs_cascade.md`; приёмы запросов — `memory/feedback_id_doc_search_method.md`.
Старые заметки: `feedback_webfetch_reality_check.md`, `feedback_web_doc_fetch_browser_antibot.md`.

## Универсальные правила работы

1. **Не выдумывать.** Нет данных → `AskUserQuestion`. Особенно имена, пути, номера, даты, количества. Без placeholder'ов без явного указания.
2. **Спрашивать при неопределённости.** Любое неявное решение (маппинг, формат, номер, путь) — явный вопрос.
3. **Read SKILL.md до действия.** Скилл сработал по триггеру — сначала прочитать его SKILL.md, потом действовать.
4. **TodoList на 3+ шагов.** `TaskCreate`/`TaskUpdate`. Последний пункт почти всегда — верификация.
5. **Общение — на русском.** Код/имена файлов — латиница/гибрид, комментарии — по-русски.
6. **Обезличивание для всего, что пушится в claude-base** (`agents/`, `skills/`, `memory/`, `session-reports/`). Никаких ФИО, шифров `Ф.YYYY.NNNNNNNNN`, объектов, заказчиков, адресов — плейсхолдеры (`<подписант>`, `<шифр>`, `<объект>`).
7. **Нормы — только через `norm-lookup` агента.** Не цитировать ГОСТ/СП/СНиП/ПУЭ/постановления **по памяти**. Любая нормативная ссылка → делегация в `norm-lookup` (точная цитата + пункт + редакция). Библиотека — `~/.claude/library/` + PDF на Я.Диске. Не нашлось — failure mode `NOT_FOUND`, **запрет выдумывания**. Пример запрещённого: писать «по СП 60 скорость воздуха…» из памяти — ВСЕГДА сперва спавн `norm-lookup`.
8. **Оформление деловых docx/xlsx — нейтральное / по образцу документа.** Без декоративного синего «Claude-стиля» (синие шапки/заливки таблиц, акцентные цвета, Anthropic-бренд/theme-factory). Цвет — только если есть в источнике, попросили, или это функциональная подсветка (diff/ошибки). Мы серьёзная компания. Детали — `skills/word-helper`, `skills/excel-helper`.
9. **Факты проекта — в `FACTS.md`.** Есть `FACTS.md` в папке проекта → читать первым; факты (шифр/цены/нагрузки/даты) править только там. Детали — skill `facts-layer`.

Скиллы загружаются автоматически по триггерам (Read SKILL.md → действие).

## Токен-дисциплина (гейт сессий + модели субагентов)

Причина: каждый tool call прогоняет весь префикс и историю сессии через кэш — длинный
хвост умножает расход (инцидент 100M токенов/день, 2026-06-10).

**Сессионный гейт (видимая строка, как роутинг-гейт).** Одна задача — одна сессия.
Смена темы в чате ИЛИ признаки длинной сессии (30+ ходов / 3+ субагентов / 5+ больших
выводов) → ОБЯЗАН одной строкой предложить: `Гейт сессии: <смена темы | сессия длинная> —
рекомендую новый чат (handoff-to-new-chat)`. Молча продолжать = нарушение архитектуры.
Предлагать один раз на задачу, решение за пользователем.

**Модели субагентов — выбирает ПОЛЬЗОВАТЕЛЬ (не наследовать, не выбирать молча).**
Причина: дешёвая модель на содержательной задаче галлюцинирует → каскад исправлений и
лишних сессий дороже мнимой экономии (инцидент 2026-06-22). Симметрично — fan-out на
дорогой модели жжёт лимит (11 Fable-субагентов = 5ч-лимит за 40 мин). Решение по модели
за пользователем.

**ПЕРЕД КАЖДЫМ спавном** субагента (Task/Agent), **перед запуском Workflow** и перед
**КАЖДЫМ ревьюером** — спросить через `AskUserQuestion`, на какой модели запускать. В
вопрос ВСЕГДА класть: рекомендацию, чем грозит дешёвая модель именно здесь (что
галлюцинирует), цену ошибки, оценку токенов, «кто чем лучше для ЭТОЙ задачи».

**Рекомендованный (первый) вариант — планка ВВЕРХ:**
- `sonnet` — по умолчанию для всего содержательного: доменные генераторы, **ВСЕ
  ревьюеры** (auditor / word-checker / excel-validator / pdf-reviewer / audit-rd-section /
  rd-coordinator), norm-lookup, синтез, аудит.
- `haiku` — ТОЛЬКО тупая экстракция / OCR / парсинг / инвентаризация, где качество не решает.
- модель сессии (opus/fable) — точечно на крупные ставки (деньги, экспертиза, сдача).

**Группировка (не спам):** N ОДНОТИПНЫХ параллельных спавнов в одном turn = ОДИН вопрос,
ответ применяется ко всем. Для **Workflow** — один вопрос ДО запуска про модельный план
по фазам (внутри скрипт уже не спрашивает).

**Fallback (автономный режим — cron/loop/headless, пользователь недоступен):** применять
рекомендацию «планка вверх» (sonnet-база) и залогировать выбор, не зависать на вопросе.

После выбора — видимая строка `Субагенты: <модель> ×N (<тип>, выбор пользователя)`.

## Feedback (consumer → hub)

Роль определяется маркером `~/.claude/.developer-marker`:
- **Хаб (маркер есть):** правки в main claude-base как обычно; в начале сессии — `pull-feedback.ps1` (собрать feedback сотрудников).

## Workflow (мультиагентная оркестрация) — проактивно предлагать

Инструмент `Workflow` (Opus 4.8) — детерминированный скрипт оркестрации многих
субагентов (fan-out / pipeline / parallel, в фоне). **Сам не запускать** без явного
согласия — он жжёт **много токенов** (десятки агентов).

**Правило проактивности:** если задача имеет признаки масштаба/оркестрации —
**одной строкой предложить** Workflow + предупредить про токены, и ждать «да».
Признаки (хотя бы один явный):
- 3+ **независимых** подзадач/файлов/разделов, которые можно вести параллельно;
- ревью/аудит **комплекта** (много артефактов разом);
- broad research/harvest по многим источникам;
- batch-преобразование N однотипных единиц (акты, спеки, листы).

Форма предложения: «Это ложится на Workflow (N агентов параллельно/конвейером) —
быстрее и полнее, но заметно дороже по токенам. Запускать? (или сделаю обычным заходом)».

**НЕ предлагать** на простых/одношаговых/диалоговых задачах (назойливо). Предлагать
**один раз** на задачу, не повторять. Детали, паттерны и кандидаты в named-workflow —
`memory/reference_workflow_tool.md`.

**Deterministic-first перед Workflow.** На задачах ТОЧНОГО матчинга (артикул→лист,
SKU-исправление, трансформация структурированных таблиц, кросс-ссылка ВОР↔РД) сперва
предложить **детерминированный Python-скрипт**, а не fan-out LLM-агентов: точнее, дешевле,
0 галлюцинаций в маппинге (кейс ВОР↔.СО 421/491). Эвристика: «сопоставить по структуре с
проверяемым правилом» → скрипт; «осмыслить/синтезировать неструктурированное» → Workflow.
Приёмы — `memory/reference_workflow_tool.md`.

## graphify — граф знаний базы (навигатор, НЕ правила)

Два слоя. **Скелет** (`graphify-out/skeleton.json`) — детерминированный костяк (agents/skills/memory/rules/mcp/chains/commands), строится БЕЗ LLM из структурных сигналов; SessionStart-хук держит его свежим на каждом ПК (0 токенов) → ДОВЕРЕННЫЙ навигатор, ему МОЖНО доверять. **Семантика** (`graphify-out/graph.json`, в git) — LLM-обогащение (rationale, неявные связи), хаб, изредка. Детали — `skills/graphify/references/skeleton.md`.

- **Использовать ПЕРВЫМ, не грепать базу подряд.** Нетривиальная задача / «как у нас принято X» / «где описано Y» / «что с чем связано» → СНАЧАЛА `python skills/graphify/tools/graph_query.py "<вопрос>"` (один вызов, русский работает нативно, без vocab-ритуала). Отдаёт сущности + связи + source_file → дальше читать ФАЙЛ-источник (истина в нём, не в графе). Поведенческие правила живут ТОЛЬКО здесь, в CLAUDE.md — в граф не выносить.
- **Свежесть скелета — автоматом:** SessionStart-хук (`scripts/graph-staleness-check.ps1`) пересобирает костяк, если отстал от HEAD. Вручную ничего не нужно; точечно — `skills/graphify/tools/skeleton_build.py`.
- **Семантика-обогащение (хаб, по желанию):** после крупных правок — flow `skills/graphify/tools/graph_update_win.py` (detect→субагенты→merge→finalize) ЧЕРЕЗ скилл, порядок коммита из `references/update.md`. НЕ обязательно к каждой правке — костяк и так свежий. Голый CLI `graphify --update` для доков не звать (без LLM-ключа падает).
- **Проактивно предлагать** (как Workflow) на РАБОЧИХ проектах (не на базе): знание = паутина связей, спрашиваемая НЕОДНОКРАТНО → одной строкой предложить queryable-граф вместо перечитывания, ждать «да». Код (AST) — 0 токенов; доки — построение стоит, query дёшев. НЕ на разовом/мелком; один раз на задачу.
