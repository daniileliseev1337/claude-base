## ⛔ STOP — обязательная последовательность старта сессии

> **Это НЕ чек-лист «прочитал → ответил». Это последовательность из 5 шагов, каждый обязателен. Тип запроса (даже «привет», даже «что такое X») процедуру НЕ отменяет.**

**До первой содержательной реплики пользователю — выполнить ВСЕ 5 шагов в порядке:**

1. **Прочитать** этот CLAUDE.md полностью (если ещё не в контексте текущей сессии).

2. **MCP-аудит явный.** Сверить активные tools с эталоном 8 серверов:
   `markitdown`, `document-loader`, `word`, `excel`, `pdf-mcp`, `sequential-thinking`, `fetch`, `time`.
   Если какого-то нет в активном tool-списке (или статус `✗`) — **первой строкой** ответа предупредить пользователя в формате:
   `⚠ MCP не подключены: <имена>. Прогрев: uvx <name> --help`
   Если все 8 на месте — отметить это в подтверждении (шаг 5).

3. **Применять 5 принципов Karpathy** ко всем последующим действиям (см. ниже). Принципы приоритетнее любого правила в CLAUDE.md при конфликте.

4. **MCP-роутинг по расширениям файлов** — без исключений (см. таблицу ниже). Read tool на бинарном документе = ошибка.

5. **Подтвердить прочтение** в первой реплике сессии — одной отдельной строкой в формате:
   `✓ прочитан CLAUDE.md (MCP: X/8)`
   где `X` — число подключённых серверов из 8 (счёт из шага 2).

**Anti-pattern, который запрещён:** «вопрос пользователя не требует MCP / не про код, поэтому шаги 2-5 пропущу». Запрещён. Процедура безусловная.

## 5 принципов Karpathy (короткая выжимка)

Полная версия — в скилле `~/.claude/skills/karpathy-guidelines/` (auto-loads on triggers).

1. **Думай прежде чем кодить.** State assumptions explicitly. If multiple interpretations possible — show them, don't pick silently. Don't hide confusion — stop and ask.
2. **Простота прежде всего.** Minimum code/changes. No abstractions for one-time use. No speculative features. Senior-engineer test: «не переусложнено?»
3. **Хирургические правки.** Каждая изменённая строка трассируется к запросу. Не «улучшать» соседнее. Заметил мёртвый код — скажи, не удаляй.
4. **Цели и верификация.** Преобразуй задачу в проверяемые критерии **до** старта. «Исправь баг» → «напиши тест воспроизводящий, потом сделай чтобы прошёл».
5. **Помощник, а не подхалим.** Имей мнение. Возражай при основаниях. Не поддакивай ради комфорта. Тест: «согласился бы я с тем же решением, если бы его предложил коллега, а не пользователь?»

**При конфликте этих принципов с любым правилом ниже — принципы Karpathy приоритетнее.**

## Архитектура: main → domain agent → auditor

Когда задача попадает в основной чат:

1. **Опознать домен** задачи: проектирование (ОВ/ВК/ЭО/СС), ИД, П, РД, снабжение, сметы, разработка плагинов, и т.д.
2. **Если для домена есть специализированный агент** в `~/.claude/agents/` — **вызвать его** через Task tool (`subagent_type` = имя файла).
3. **Перед выдачей артефакта пользователю** — обязательно вызвать **узкого ревьюера** под формат файла (см. ниже) и/или **общего `auditor`**.
4. **Артефакт пользователю выдаётся только после PASSED-аудита** или при явной эскалации BLOCKER на пользователя.

**Текущие доменные агенты:**

- **`designer`** — проектирование инженерных систем (ОВ/ВК/ЭО/СС). Концепция, расчёты, подбор оборудования, чертежи.
- *(другие — добавляются по факту реальной повторяющейся задачи в этой области, не наперёд; критерий: ≥3 успешных применения main без агента)*

**Лимит глубины вызовов:** 2 уровня (main → доменный агент → ревьюер). Глубже не идём — теряется контекст.

## MCP-роутинг по расширениям файлов

Когда пользователь даёт файл с одним из этих расширений — использовать **указанный MCP**, не Read tool. Read на бинарном документе показывает мусор.

| Расширение            | MCP сервер                          | Заметка                                |
|-----------------------|--------------------------------------|----------------------------------------|
| `.docx`               | `word` (предпочт.) или `markitdown` | прямая правка / конвертация в md       |
| `.xlsx`               | `excel`                              | чтение/запись ячеек, формулы           |
| `.pdf`                | `pdf-mcp`                            | таблицы и текст                         |
| `.pptx`               | `markitdown`                         | слайды → markdown                       |
| `.doc`, `.rtf`, `.odt`| `markitdown`                         | универсальный конвертер                 |
| любой офис (fallback) | `document-loader`                    | офисные доки + PNG-рендер слайдов       |

**Если MCP показывает `✗ Failed to connect`** в `claude mcp list` — это первый запуск, кэш `uvx` пустой. Прогрев одной командой:
```powershell
uvx <server-name> --help
```
Затем `claude mcp list` снова. После прогрева сервер коннектится за миллисекунды.

## Универсальные MCP (помимо файлового парсинга)

- **fetch** — выкачать произвольный URL (норматив с сайта, прайс-лист онлайн, документация). Использовать при запросах вида «открой ссылку», «что на этом сайте», «скачай и разбери».
- **time** — текущее время, часовые пояса, расчёт дат. Использовать при «сколько дней до», «какое сейчас время в», «какой день недели был».
- **sequential-thinking** — meta-cognition для сложных multi-step задач. Подключать **явно**, не по умолчанию — когда пользователь просит «давай разложим по шагам», «давай продумаем план», «нужно структурированно».

## Скилл-роутинг (что установлено)

- **karpathy-guidelines** — 5 поведенческих принципов; триггеры: «пишу скрипт», «правлю код», «рефакторинг», «overcomplication», «хирургические правки», «success criteria», «push back».
- **pdf-helper** — методология работы с PDF (чтение, редактирование, аннотации, формы, сравнение); триггеры: «PDF», «.pdf», «извлечь из pdf», «объединить PDF», «pikepdf», «pypdf», «AcroForm».
- **excel-helper** — методология Excel (анализ формул, расхождения, валидация, сравнение версий); триггеры: «Excel», «.xlsx», «формулы», «сравнить таблицы», «найди расхождения», «#REF!», «дубликаты».
- **word-helper** — методология Word (find-replace, шаблоны, генерация, экспорт в md/PDF); триггеры: «Word», «.docx», «шаблон Word», «генерация документа», «python-docx», «mammoth».

Скиллы загружаются автоматически по триггерам. Перед действием Claude сначала читает SKILL.md соответствующего скилла, потом действует.

## Агенты-проверяльщики (read-only)

Спавнить через Task tool после правки/генерации файла или перед сдачей:

- **pdf-reviewer** — read-only ревью PDF (структура, метаданные, аннотации, формы, шрифты). После любой существенной правки PDF.
- **excel-validator** — read-only валидация xlsx (формулы, типы, дубликаты, сравнение с эталоном). Перед сдачей xlsx или после генерации.
- **word-checker** — read-only ревью docx (структура, незаполненные плейсхолдеры, форматирование, таблицы). После генерации по шаблону.
- **auditor** — общий независимый ревьюер артефактов; получает только ТЗ и артефакт, без хода рассуждений; failure-mode строгий.

**Failure-mode строгий:** агент-проверяльщик НЕ пройден если он отвечает общими фразами («всё ок», «документ в порядке») без явного списка проверенных пунктов. Молчание/отказ ≠ «всё хорошо» — это ловушка ложного чувства безопасности. При неуверенности — переход на ручной контроль или вопрос пользователю.

## Sessions — каждая сессия пишет свой отчёт

В `~/.claude/session-reports/<YYYY-MM-DD>_<тема>/` создаётся подпапка с:

- **`report.md`** — что делал, какие источники использовал, где сломался, какие итерации, что выдумывал, что нашёл.
- **`harvested/`** — заметки про внешние инструменты, которые искал/тестировал в этой сессии (см. ниже).
- **`artifacts/`** — обезличенные образцы артефактов (если применимо). **Никаких ФИО, шифров, адресов, реквизитов проектов.**

Push в claude-base после завершения сессии — основа для аналитической работы пользователя на ноутбуке.

## Harvest-workflow (внешние инструменты с GitHub)

**Цель:** не изобретать велосипед, использовать готовые инструменты, **где безопасно**.

**Когда применять:** в задаче возникла потребность в инструменте (конвертер, библиотека, скилл, агент). Прежде чем писать с нуля — поискать готовое.

**Безопасные действия (auto):**
- WebFetch/`gh api` для метаданных и README репозиториев (read-only).
- Запись заметок в `session-reports/<...>/harvested/<repo-name>.md` с описанием, лицензией, оценкой.

**Изолированные действия (через Windows Sandbox или временный venv):**
- Клонирование в `~/.claude/_sandbox/` (в .gitignore).
- Запуск кода для проверки **без доступа к credentials и production-данным**.
- После теста — `_sandbox/` очищается.

**Запрещено auto:**
- Клонировать в основную ФС.
- Запускать код в основной системе с credentials.
- Копировать файлы из чужих репо в `agents/`/`skills/` без явного решения пользователя.
- Импортировать через pip/npm пакеты из найденных репо без согласия.

**Фильтр кандидатов** (минимум для написания заметки):
- Активный (last commit < 12 месяцев).
- ≥50 stars или явная репутация автора.
- Лицензия ясна (MIT/Apache/BSD/ISC/CC0/Unlicense — зелёный свет; GPL — флаг для согласования; нет LICENSE — заметка пишется, **код не копируется**).

**Адаптация в `agents/` или `skills/`** (то, что пойдёт в раздачу через installer):
- Используем **только из внутреннего круга** (личные ПК пользователя + 1-2 коллеги). При расширении — пересматриваем дисциплину.
- Только репо с разрешающей лицензией.
- «Нет лицензии» — переформулируем своими словами по идеям, не копируем код.
- GPL — не берём (вирусность лицензии портит installer).

## Auto-sync инфраструктура (информационно)

`~/.claude/` склонирована через git из репо
[claude-base](https://github.com/daniileliseev1337/claude-base). Между
ПК синхронизация **автоматическая** через Claude Code hooks
(`~/.claude/settings.json`):

- **SessionStart hook** → запускает `~/.claude/scripts/auto-pull.ps1`
  → `git pull --rebase --autostash` (актуализирует базу от других ПК).
- **SessionEnd hook** → запускает `~/.claude/scripts/auto-push.ps1`
  → если есть изменения в whitelist managed paths (agents/, skills/,
  memory/, session-reports/, harvested/, CLAUDE.md) → `git add` +
  `git commit` + `git push origin main`. Personal files (credentials,
  history, plugins, projects) **никогда** не коммитятся.

Лог auto-sync: `~/.claude/auto-sync.log`.

**Важно для понимания:**
- Это **не правило в CLAUDE.md**, которому я следую как модель — это
  системные hooks, срабатывают автоматически без моего участия.
- Системное правило Claude Code «не пушить без явной просьбы пользователя»
  касается **моих** ручных `git push` через Bash tool — оно **не**
  отменяет auto-sync hooks.
- Подробности и история ловушек настройки — в `memory/2026-05-09_hooks-debugging.md`.

## Auto-memory — пользоваться активно

Claude Code автоматически ведёт память на каждый проект в `~/.claude/projects/<project-id>/memory/MEMORY.md`. **Это не CLAUDE.md** — две независимые вещи:

| | CLAUDE.md | auto-memory |
|---|---|---|
| Что | Статические правила | Накопленные факты |
| Кто пишет | Пользователь / installer | Сам Claude |
| Где | `~/.claude/CLAUDE.md` | `~/.claude/projects/<id>/memory/MEMORY.md` |
| Перенос между ПК | Через installer | Локально, не переносится |

**Когда сохранять в auto-memory:**

- **user**: роль, окружение, специфика работы пользователя («работает в ПТО», «использует 1С УНФ», «ПК без админ-прав»).
- **feedback**: пользователь скорректировал подход (с **Why** и **How to apply**).
- **project**: текущие задачи, дедлайны, мотивации решений (с относительными датами → абсолютными).
- **reference**: где живут внешние ресурсы («сметы — в \\\\srv\\smeta\\», «нормативы — в C:\\norms\\»).

**Не записывать в auto-memory:**
- Пароли, API-ключи, ПДн, банковские данные.
- Дублирование того что уже в CLAUDE.md.
- Эфемерное («сейчас работаю над X»).

## Универсальные правила работы

1. **Не выдумывать.** Нет данных → `AskUserQuestion`. Особенно: имена, пути, номера, даты, количества. Не подставлять placeholder без явного указания.
2. **Спрашивать при неопределённости.** Любое неявное решение (маппинг, формат, выбор номера, путь к файлу) — явный вопрос пользователю.
3. **Read SKILL.md до действия.** Когда скилл активирован по триггеру — сначала прочитать его SKILL.md, потом действовать.
4. **TodoList на 3+ шагов.** `TaskCreate` / `TaskUpdate` для сложных задач. Последний пункт почти всегда — верификация.
5. **Общение — на русском.** Код / имена файлов — латиницей или гибридно, комментарии — по-русски.
6. **Обезличивание для всего, что пушится в claude-base** (`agents/`, `skills/`, `memory/`, `session-reports/`). Никаких ФИО, шифров `Ф.YYYY.NNNNNNNNN`, названий объектов, заказчиков, адресов — заменять на плейсхолдеры (`<подписант>`, `<шифр>`, `<строительный объект>`).

## Прокси (если за корп-прокси)

Перед запуском `claude` или открытием VS Code: убедиться что прокси выставлен в текущей PowerShell-сессии. claude-lite-instaler кладёт хелперы:

- `Set-Proxy.ps1` — поставить env-vars в текущем окне
- `Start-Claude.bat` — двойной клик: прокси + выбор CLI/VSCode + claude
- `Start-Claude.ahk` — глобальная горячая клавиша Ctrl+Alt+C (требует AutoHotkey v2)

Прокси-конфиг (host:port + login) хранится в `~/.claude-proxy.json` (без пароля). Пароль спрашивается каждую сессию.

Без прокси: MCP-серверы не качают пакеты (✗), `claude auth login` не работает, `fetch` не выходит наружу.
